# ADR 0004: LLM as Engine, Environment as System

## Status
Accepted

## Date
2026-03-22

## Context

CodeCore не должен становиться оболочкой под одну любимую модель. Если качество держится только на силе конкретной LLM и мастерстве оператора, система не является устойчивой платформой. Она становится лишь тонким клиентом к backend'у.

Для заявленной цели нужен другой принцип: модель - сменный вычислительный двигатель, а инженерное качество обеспечивается самой средой.

## Decision

CodeCore принимает следующий системный тезис:

- LLM отвечает за генерацию, рассуждение и адаптацию;
- среда отвечает за правила, skills, orchestration, контекст, verification, memory и governance;
- оператор усиливает процесс, но не должен быть единственной опорой качества.

Целевой ориентир системы - `quality normalization`.

Это означает:

- разные модели не обязаны быть одинаковыми по ceiling качества;
- но они должны давать сопоставимо успешный результат внутри одной и той же среды, если им заданы одинаковые constraints, context strategy, skills и verification loop;
- при смене модели workflow должен деградировать мягко, а не разваливаться.

Практические следствия:

- все критичные правила выносятся из prompt-фольклора в runtime policies;
- workflow packages живут в skills, а не в памяти оператора;
- verification находится вне модели;
- rankings строятся по outcome, а не по репутации провайдера.

## Consequences

Плюсы:

- provider lock-in становится архитектурно неприемлемым;
- локальные и дешёвые модели могут быть полезны в сильной среде;
- появляется путь к reproducible engineering;
- оператор может делегировать больше без потери контроля.

Минусы:

- выше требования к orchestration, policy engine и telemetry;
- нельзя “спрятать” архитектурную слабость за сильной моделью;
- понадобится больше явных контрактов между подсистемами.

Принятый компромисс:

- часть задач всё равно будут лучше решаться сильными моделями;
- но system design не должен зависеть от этого факта.
