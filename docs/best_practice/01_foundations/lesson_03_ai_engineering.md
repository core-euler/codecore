# Модуль 1, Подмодуль 3: AI Engineering — от исследования к продакшену

## Введение

AI Engineering — это новая дисциплина, которая зародилась на стыке машинного обучения, системного дизайна и инженерной практики. В отличие от традиционного Machine Learning Engineering, сосредоточенного на табулярных данных и обучении моделей с нуля, AI Engineering имеет дело с адаптацией готовых фундаментальных моделей (Foundation Models) к конкретным практическим задачам.

Фундаментальные модели включают большие языковые модели (LLM) и мультимодальные модели (LMM), которые уже обучены на огромных объемах данных и обладают впечатляющими способностями. Задача AI Engineer — не переучивать эти модели с нуля, а эффективно адаптировать их для решения специфических проблем через инженерные техники: prompt engineering, контекстное конструирование, fine-tuning параметров и создание многоагентных систем.

## 1. AI Engineering как дисциплина

### 1.1 Определение и область применения

**AI Engineering** — это системный подход к построению приложений на базе фундаментальных моделей, охватывающий весь цикл: от концептуализации идеи до развертывания в production и непрерывного совершенствования.

Согласно книге Chip Huyen "AI Engineering" (O'Reilly, 2025), AI Engineering отвечает на ключевые вопросы:

- Стоит ли вообще разрабатывать это AI приложение?
- Как корректно оценить результаты? Можно ли использовать AI для оценки AI?
- Что вызывает галлюцинации моделей? Как их обнаружить и снизить?
- Какие best practices существуют для prompt engineering?
- Почему работает RAG? Какие стратегии применять?
- Что такое агент? Как его построить и оценить?
- Когда имеет смысл fine-tuning? Когда его избежать?
- Сколько данных нужно? Как проверить качество аннотаций?
- Как сделать модель быстрее, дешевле и безопаснее?
- Как создать feedback loop для непрерывного улучшения?

### 1.2 Отличия от Traditional ML Engineering

| Аспект | Traditional ML | AI Engineering |
|--------|-----------------|-------------------|
| **Объект фокуса** | Табулярные данные, обучение с нуля | Текст, изображения, готовые модели |
| **Основная задача** | Feature engineering, model training | Prompt engineering, context construction |
| **Валидация** | Metrics на тестовых наборах | Human evaluation, LLM-as-judge |
| **Адаптация** | Full model retraining | Parameter-efficient tuning (LoRA) |
| **Временные рамки** | Недели/месяцы обучения | Часы/дни через prompting |
| **Стоимость** | Дорогостоящие GPU-кластеры | Существующие API (часто дешевле) |
| **Эпизод обучения** | Batch training | Continuous context adaptation |

### 1.3 Базовые принципы AI Engineering

Chip Huyen в своей книге выделяет несколько фундаментальных принципов, которые должны лежать в основе любого AI-инженерного проекта:

1. **Requirements первыми**: начните с четкого определения требований и метрик успеха, а не с выбора модели.

2. **Evalution-driven development**: постоянно измеряйте качество и направляйте разработку результатами тестирования, а не интуицией.

3. **Итеративность**: AI Engineering — это цикл гипотеза → реализация → оценка → рефлексия → улучшение.

4. **Cost-efficiency**: оптимизируйте не только точность, но и скорость, затраты и latency.

5. **Production-mindedness**: с самого начала думайте о том, как система будет работать в production.

6. **Human-in-the-loop**: часто требуется человеческая обратная связь для улучшения системы.

## 2. От research paper к агенту: Paper2Agent

### 2.1 Концепция Paper2Agent

Paper2Agent — это инновационный фреймворк, который автоматически трансформирует научные статьи (вернее, их реализацию в коде на GitHub) в интерактивные AI агенты с минимальным участием человека.

**Основная идея**: научная статья часто содержит:
- Математическую формулировку алгоритма
- Реализацию на Python (tutorials, notebooks)
- Примеры использования

Paper2Agent автоматически:
1. Извлекает все tutorials и notebooks из репозитория
2. Анализирует их содержание
3. Извлекает инструменты и функции
4. Генерирует MCP (Model Context Protocol) сервер
5. Создает агента, готового к использованию в Claude Code

### 2.2 Pipeline Paper2Agent

Paper2Agent следует пятиэтапному процессу трансформации репозитория в агент:

**Этап 1: Обнаружение документации** — сканирование репозитория для поиска всех tutorials и notebooks. Система определяет, какие примеры существуют и как их можно использовать.

**Этап 2: Выполнение и анализ** — запуск найденных примеров в контролируемой среде для понимания входов, выходов и поведения. Это позволяет LLM "испытать" код, а не только прочитать его.

**Этап 3: Извлечение инструментов** — анализ выполненных примеров для выделения key функций, параметров и их влияния на результаты. Система определяет, какие функции будут полезны агенту.

**Этап 4: Генерация MCP сервера** — создание Model Context Protocol сервера с документированными инструментами, готовых к использованию Claude Code.

**Этап 5: Оценка качества** — анализ coverage, убеждение в том, что основной функционал репозитория отражен в MCP сервере.

Ключевая идея: каждый этап обогащает понимание системой того, что может делать репозиторий. В конце имеется структурированное описание, которое LLM может эффективно использовать.

### 2.3 Архитектура генерируемого агента

Результатом Paper2Agent является структурированный проект с четкой организацией инструментов и документации. Главная идея — разделить исходный код на модули, каждый из которых может быть вызван агентом. Таким образом агент не читает весь сырой код, а работает с чистыми функциональными интерфейсами.

Процесс генерации:
1. MCP сервер оборачивает выделенные инструменты
2. Каждый инструмент имеет сигнатуру, документацию и ограничения
3. Вспомогательная информация (reports о покрытии, анализ качества) помогает LLM понять надежность каждого инструмента
4. Claude Code загружает этот MCP сервер и может вызывать инструменты по мере необходимости

Это различается от простого copy-paste кода, потому что:
- Инструменты явно задокументированы и типизированы
- LLM видит только публичный интерфейс, не внутреннюю реализацию
- Каждый инструмент может содержать ошибку handling и валидацию
- Система может отследить, какие инструменты часто используются или вызывают ошибки

### 2.4 Почему Paper2Agent эффективен

Paper2Agent демонстрирует ключевую философию AI Engineering: **не нужен один идеальный prompt, нужна система из специализированных компонентов**.

Вместо того чтобы просить LLM "использовать AlphaGenome", система:
1. Понимает, ЧТО может делать AlphaGenome (через выполнение примеров)
2. Извлекает КАКИЕ инструменты выставить наружу (через анализ)
3. ДОКУМЕНТИРУЕТ инструменты для LLM (через MCP)
4. ВАЛИДИРУЕТ, что все работает (через тесты и coverage)

Многоагентный процесс внутри Paper2Agent позволяет масштабировать анализ даже больших репозиториев, потому что разные агенты могут параллельно анализировать разные части кода.

## 3. Архитектуры reasoning: ROMA и Beyond

### 3.1 Введение в многоагентный reasoning

Когда сложная задача требует нескольких шагов рассуждения, одного LLM часто недостаточно. ROMA (Recursive Open Meta-Agent) — это фреймворк, который структурирует многоагентное решение проблем через иерархическое разложение задач.

### 3.2 ROMA: Reasoning Over Multiple Agents

**ROMA** использует рекурсивный подход к решению сложных задач. Основная идея: вместо того чтобы давать LLM одну большую задачу, система сначала определяет, можно ли ее решить напрямую. Если нет — разбивает на подзадачи, решает каждую параллельно, затем собирает результаты.

**Четыре ключевых компонента ROMA:**

1. **Atomizer** — определяет, может ли задача быть решена напрямую или требует разложения на подзадачи. Это критично, потому что некоторые задачи лучше решать целиком.

2. **Planner** — разбивает сложные задачи на подзадачи с учетом зависимостей. Плохой план может привести к неправильным результатам, поэтому этот компонент использует CoT рассуждение.

3. **Executor** — решает атомарные задачи. Может быть LLM вызов, API запрос, поиск в БД или другой агент. Ключ: у выполнителя есть четкая, хорошо определенная задача.

4. **Aggregator** — собирает результаты подзадач и интегрирует их в финальный ответ. Это не просто конкатенация — нужно учитывать порядок, зависимости и согласованность.

**Дополнительно:**
5. **Verifier** (опционально) — проверяет финальный результат против исходной задачи. Если верификация не пройдена, может переработать или перепланировать.

**Концептуальный поток:**
```
Сложная задача
     ↓
Atomizer: "Это требует разложения"
     ↓
Planner разбивает на 3 подзадачи
     ↓
Executor решает 3 подзадачи параллельно
     ↓
Aggregator комбинирует результаты логически
     ↓
Verifier проверяет, что ответ отвечает на исходный вопрос
     ↓
Финальный результат
```

### 3.3 Когда применяется ROMA

ROMA наиболее полезна для задач, требующих логического разложения:

**Пример 1: Планирование путешествия**
- Исходная задача: "Спланируй выходные в Барселоне"
- Atomizer: "Это требует разложения"
- Planner: подзадачи = (найти рейсы, найти отели, найти рестораны, составить расписание, подготовить список)
- Executor: каждая подзадача решается параллельно
- Aggregator: объединяет в единый план
- Verifier: проверяет, что все элементы согласованы (рейсы и отели в одни дни, рестораны открыты в планируемое время)

**Пример 2: Анализ исследовательской статьи**
- Исходная задача: "Объясни основные contributions этой статьи"
- Atomizer: "Требуется разложение"
- Planner: подзадачи = (прочитай abstract, проанализируй методологию, посмотри результаты, сравни с related work)
- Executor: каждая часть анализируется
- Aggregator: синтезирует в единую нарративную экспозицию

Ключ к успеху ROMA: **хороший Planner**, который выбирает правильные подзадачи и определяет их зависимости. Плохой план приводит к неправильным результатам, даже если каждая подзадача решена верно.

### 3.4 Преимущества ROMA

- **Прозрачность**: каждый шаг четко видим и отлаживаем
- **Масштабируемость**: может обрабатывать очень сложные задачи через рекурсию
- **Гибкость**: каждый компонент может использовать разные модели и стратегии
- **Производительность**: подзадачи решаются параллельно где это возможно
- **Надежность**: верификация на финальном этапе повышает качество

### 3.5 Сравнение reasoning стратегий

| Стратегия | Описание | Сложность | Latency | Cost |
|-----------|---------|-----------|---------|------|
| **Chain-of-Thought (CoT)** | Пошаговое рассуждение в одном пассе | Средняя | Низкая | Низкая |
| **Tree-of-Thought (ToT)** | Исследование дерева возможных путей | Высокая | Средняя | Средняя |
| **Graph-of-Thought (GoT)** | DAG промежуточных состояний | Очень высокая | Высокая | Высокая |
| **ROMA (Hierarchical)** | Рекурсивное разложение + агрегация | Адаптивная | Адаптивная | Адаптивная |

ROMA отличается от классических подходов тем, что **адаптирует структуру рассуждения к самой задаче**, а не используется один фиксированный граф.

## 4. Deep Agents: Глубокие агенты от LangChain

### 4.1 Концепция Deep Agents

**Deep Agents** (от LangChain) — это полнофункциональный фреймворк для создания агентов, готовых к использованию "из коробки". В отличие от ROMA, который фокусируется на архитектуре reasoning, Deep Agents предоставляет полный набор инструментов для практической работы.

**Что включено:**

- **Planning** — `write_todos` для декомпозиции и отслеживания прогресса
- **Filesystem operations** — `read_file`, `write_file`, `edit_file`, `ls`, `glob`, `grep`
- **Shell access** — `execute` для запуска команд (в песочнице)
- **Sub-agents** — `task` для делегирования работы с отдельным контекстом
- **Smart defaults** — готовые промпты, которые учат модель правильному использованию инструментов
- **Context management** — автоматическое сокращение контекста и сохранение больших выходов в файлы

### 4.2 Архитектура Deep Agents

Deep Agents предоставляет готовый агентный framework, включающий несколько слоев:

**Уровень 1: Инструменты** — файловые операции (read, write, edit), shell доступ, поиск в интернете. Все инструменты оборачиваются в унифицированный интерфейс.

**Уровень 2: Контроллер** — логика выбора инструментов. Deep Agents включает smart prompts, которые учат модель правильно использовать инструменты (когда использовать write_file vs edit_file, и т.д.).

**Уровень 3: Менеджмент контекста** — если контекст становится слишком большим, система автоматически сокращает его, сохраняя большие выходы в файлы.

**Уровень 4: Состояние** — Deep Agents может сохранять контрольные точки, позволяя агенту восстановиться после сбоя.

**Интеграция с MCP:** Deep Agents может подключать внешние MCP серверы, расширяя свои возможности за счет специализированных инструментов.

### 4.3 Ключевые компоненты

**Planning tools:** агент может создавать todos для отслеживания сложных задач. Это помогает в больших проектах, где нужно помнить о множестве подзадач.

**File operations:** читать, писать, редактировать файлы. Контроль — агент видит файловую систему и может ориентироваться по проекту.

**Shell execution:** запуск команд в песочнице. Позволяет агенту проверять свои гипотезы (например, запустить тесты для проверки кода).

**Task delegation:** агент может создавать субагентов для отдельных задач с изолированным контекстом. Это помогает при параллельной работе.

### 4.4 Встроенные подходы

Deep Agents основан на **LangGraph** — фреймворке для графов управления потоком. Это означает:

1. **Явная структура:** каждый шаг агента — это узел графа, переходы четко определены
2. **Воспроизводимость:** можно сохранять состояние и возобновлять оттуда
3. **Отладка:** видно, по какому пути идет агент (какие инструменты вызывает)
4. **Расширяемость:** можно добавлять новые узлы и переходы в граф

**Философия:** вместо монолитного агента, Deep Agents структурирует решение как программу с явным контролем потока. Это делает системы более надежными и понятными для отладки.

## 5. ACE Framework: Agentic Context Engineering

### 5.1 Проблема context collapse

При длительной работе агента, контекст часто деградирует через процесс итеративного переписывания. Известные проблемы:

1. **Brevity bias** — система склонна сокращать контекст, теряя важные детали
2. **Context collapse** — информация итеративно "стирается" при переписывании
3. **Token overhead** — очень длинные контексты замедляют работу

### 5.2 ACE: Agentic Context Engineering

**ACE** (из arxiv 2510.04618) — это фреймворк, который трактует контекст как **эволюционирующую систему знаний**, которая накапливает, уточняет и организует стратегии через модульный процесс:

- **Generator** — создает новые стратегии и знания
- **Reflector** — анализирует результаты и их эффективность
- **Curator** — отбирает и организует самую ценную информацию

### 5.3 ACE Pipeline

```
Initial Context
     ↓
┌─────────────────────────┐
│ Generator Module         │ → Генерирует новые стратегии
│ (Generates new facts)    │   на основе текущего контекста
└─────────────────────────┘
     ↓
┌─────────────────────────┐
│ Task Execution           │ → Выполняет задачу с помощью
│ (Apply strategies)       │   сгенерированных стратегий
└─────────────────────────┘
     ↓
┌─────────────────────────┐
│ Reflector Module         │ → Анализирует результаты
│ (Evaluates outcomes)     │   и уроки, извлеченные из выполнения
└─────────────────────────┘
     ↓
┌─────────────────────────┐
│ Curator Module           │ → Выбирает лучшие практики
│ (Organizes knowledge)    │   и организует контекст для следующей итерации
└─────────────────────────┘
     ↓
Evolved Context (для следующей итерации)
```

### 5.4 Как ACE избегает context collapse

Стандартный подход к долгоживущим агентам часто приводит к деградации:

```
Итерация 1: [Полный контекст о стратегиях]
    ↓
Итерация 2: LLM переписывает контекст, теряя детали
    ↓
Итерация 3: [Контекст стал короче и менее точным]
    ↓
Итерация N: Контекст вырождается в "что-то про задачу"
```

ACE решает это **структурированным хранилищем**:

```
Вместо переписывания:
    Сохраняет как JSON структуру:
    {
        "strategies": [
            {"name": "strategy_v1", "effectiveness": 0.87},
            {"name": "strategy_v2", "effectiveness": 0.92}
        ],
        "discovered_patterns": [...]
    }

Затем:
    - Curator выбирает top-K стратегии по effectiveness
    - Новые стратегии добавляются, старые не удаляются
    - Контекст растет структурированно
```

**Ключевое отличие:** вместо свободного текста, который постепенно ухудшается, используется структурированный реестр решений с метаданными (эффективность, дата добавления, применимость).

### 5.5 Результаты ACE

Согласно исследованию:

- **+10.6%** улучшение на agent benchmarks (SWE-bench, GAIA)
- **+8.6%** улучшение на finance-specific tasks
- **Значительное снижение** adaptation latency и стоимости rollout
- На **AppWorld** leaderboard ACE соответствует top-ranked production agents

**Главное преимущество**: ACE может адаптироваться **без labeled supervision**, используя только natural execution feedback.

## 6. Evaluation и benchmarks для агентов

### 6.1 Почему evaluation критична

AI engineering отличается от research тем, что в production нет "правильного ответа" — только бизнес-метрики. Evaluation должна быть:

- **Автоматизирована** (иначе не масштабируется)
- **Надежна** (консистентна между запусками)
- **Многомерна** (не только accuracy)

### 6.2 Метрики для агентов

| Категория | Метрика | Описание | Как мерить |
|-----------|---------|---------|----------|
| **Функциональность** | Task completion rate | % успешно завершенных задач | Manual + LLM-as-judge |
| | Correctness | Корректность результата | Domain-specific metrics |
| **Эффективность** | Latency | Время выполнения | wall-clock time |
| | Cost | Стоимость API вызовов | sum(token_usage * price) |
| **Надежность** | Error rate | Частота ошибок | count / total |
| | Hallucination rate | Выдуманная информация | Human review + LLM judge |
| **Масштабируемость** | Tokens per task | Среднее использование | aggregate per task |
| | Parallel efficiency | Эффективность параллелизма | speedup / ideal_speedup |

### 6.3 Практические benchmarks

**SWE-bench** — Software Engineering benchmark для агентов, решающих реальные GitHub issues. Фокусируется на production-подобных задачах: исправление баков, добавление features. Оценка: тест проходит или нет (четкий критерий успеха).

**GAIA** — General AI Assistant benchmark с факт-ориентированными вопросами. Требует точности и полноты ответа. Сложнее чем простой поиск, потому что нужна синтез из нескольких источников.

**WebArena** — benchmark для агентов, работающих с динамическими веб-интерфейсами. Реальная сложность: страницы меняются, нужна навигация, заполнение форм. Оценка: задача выполнена полностью или нет.

**Отличия между benchmarks:**

| Benchmark | Тип задачи | Сложность | Что оценивает |
|-----------|-----------|-----------|--------------|
| SWE-bench | Инженерные (код) | Высокая | Способность понимать и модифицировать код |
| GAIA | Аналитические (факты) | Средняя | Поиск информации и синтез ответа |
| WebArena | Интерактивные (UI) | Высокая | Навигация и понимание интерфейсов |

### 6.4 LLM-as-judge

Один из ключевых инноваций — использование самого LLM для оценки результатов других LLM-агентов. Вместо жестких правил (что часто неточно для субъективных критериев), используется second LLM как "судья".

**Архитектура LLM-as-judge:**

```
Результат агента (письмо, код, отчет)
       ↓
LLM-judge (обычно дешевая модель)
       ↓
Оценка по критериям (0-10)
+ Feedback с объяснением
```

**Как это работает:**

1. Основной агент генерирует результат
2. Создается prompt для judge LLM с критериями оценки
3. Judge LLM анализирует результат
4. Возвращается JSON с оценками и feedback
5. Основной агент может использовать feedback для улучшения

**Критерии для judge:**

- **Correctness** (0-10) — насколько правильный ответ?
- **Completeness** (0-10) — охватывает ли все требования?
- **Clarity** (0-10) — понятен ли ответ?
- **Hallucination resistance** (0-10) — выдуманы ли факты?
- **Safety** (0-10) — безопасен ли результат?

**Оптимизация стоимости:**

Для judge используйте **дешевую модель** (claude-haiku):
- Judge только оценивает один результат, не решает основную задачу
- Может быть менее умная, чем основной агент
- Экономит деньги при оценке тысяч результатов

### 6.5 Best practices для evaluation

1. **Разнообразьте тесты**: включайте edge cases, long-tail примеры, adversarial inputs
2. **Трекируйте метрики over time**: следите за тенденциями, а не точечными значениями
3. **Используйте несколько судей**: не полагайтесь на один LLM для оценки
4. **А/B тестирование**: сравнивайте версии агента на одном наборе примеров
5. **Human evaluation**: на критичных задачах всегда привлекайте людей
6. **Cost analysis**: оптимизируйте не только точность, но и цену/performance соотношение

## 7. Практическая задача: построить evaluation framework

### 7.1 Архитектура evaluation framework

Evaluation framework должен быть модульным и расширяемым. Основная идея:

```
Evaluation Pipeline:
    1. Загрузить тестовые примеры
    2. Для каждого примера:
       a. Выполнить агент на задаче
       b. Записать время выполнения и стоимость
       c. Запустить набор evaluators (правильность, полнота, безопасность)
       d. Использовать LLM-judge для субъективной оценки
    3. Агрегировать результаты
    4. Сгенерировать отчет
```

**Ключевые компоненты:**

- **Evaluators**: каждый измеряет одну метрику (correctness, latency, cost, hallucination_rate)
- **LLM Judge**: secondary LLM, которая оценивает качество ответа
- **Aggregator**: объединяет все метрики в summary
- **Reporter**: генерирует читаемый отчет

### 7.2 Типы оценки

**Объективные метрики** (можно автоматизировать):
- Execution time (секунды)
- Token usage (стоимость)
- Test pass rate (для кода)
- Exact match (для фактов)

**Субъективные метрики** (нужен LLM или человек):
- Correctness (0-10)
- Completeness (охватывает ли вопрос)
- Clarity (понятен ли ответ)
- Hallucination rate (выдуман ли контент)

### 7.3 Реальная задача: evaluation for RAG

Для RAG систем оценка сложнее, потому что нужно проверить:

1. **Retrieval качество**: найдены ли правильные документы?
2. **Generation качество**: ответ ли LLM хороший, даже если контекст хороший?
3. **End-to-end**: правильный ли финальный ответ пользователю?

**Метрики для RAG:**

| Метрика | Как мерить |
|---------|-----------|
| Recall@10 | Из всех релевантных документов, сколько в топ-10? |
| Precision@3 | Из первых 3 результатов, сколько релевантны? |
| MRR (Mean Reciprocal Rank) | На каком месте первый релевантный результат? |
| Faithfulness | Содержит ли ответ факты из контекста или галлюцинирует? |
| Relevance | Отвечает ли ответ на вопрос? |

### 7.4 Практический подход

**Начните с простых тестов:**
1. Подготовьте 10-20 примеров вопросов с известными правильными ответами
2. Запустите агент на каждом примере
3. Используйте LLM-judge для быстрой оценки (дешевая модель типа haiku)
4. Посчитайте процент успешных ответов
5. Проанализируйте, на каких типах вопросов агент ошибается

**По мере роста:**
1. Увеличивайте размер тестового набора
2. Добавляйте специализированные evaluators
3. Настраивайте LLM-judge для вашего домена
4. Трекируйте метрики over time

## Заключение

AI Engineering — это инженерная дисциплина, которая требует системного подхода к построению, оценке и развертыванию приложений на базе фундаментальных моделей.

**Ключевые takeaways:**

1. **От исследования к продакшену** — Paper2Agent показывает, что можно автоматизировать трансформацию научных работ в практические инструменты

2. **Многоагентный reasoning** — ROMA и Deep Agents демонстрируют, что сложные задачи решаются через рекурсивное разложение и специализированные компоненты

3. **Эволюция контекста** — ACE показывает, что контекст нужно трактовать как обучаемое состояние, а не статичный артефакт

4. **Evaluation is everything** — без надежных метрик и evaluation framework невозможно улучшать систему в production

5. **Практический подход** — настоящий AI Engineer не пишет один perfect prompt, а собирает композитные системы из многих компонентов
