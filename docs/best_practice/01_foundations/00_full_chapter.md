# Модуль 1: Основы LLM-агентов

## Введение

Развитие больших языковых моделей (LLM) привело к революционному сдвигу в способе решения сложных задач. Если раньше мы ограничивались обычным взаимодействием пользователь-чатбот, то теперь LLM могут работать автономно, принимая решения, выполняя действия и улучшая свои результаты в процессе работы. Это качественно новый класс систем — LLM-агенты.

В этой главе мы изучим фундаментальные концепции, которые лежат в основе любого успешного агента. Мы разберем архитектуру агентов, цикл их работы, инструменты, которые им нужны, и практические паттерны для их отладки и верификации результатов.

## 1. Что такое LLM-агент: архитектура и отличие от чат-ботов

### 1.1 Определение LLM-агента

LLM-агент — это система, которая использует большую языковую модель в качестве "мозга" для принятия решений и выполнения действий в определенной среде. Основное отличие от простого чат-бота заключается в способности агента действовать самостоятельно, без постоянного вмешательства пользователя.

**Ключевые характеристики LLM-агента:**

1. **Автономность** — агент может принимать решения и действовать без явного указания пользователя на каждом шаге
2. **Реактивность** — способность реагировать на изменения в окружающей среде и корректировать свои действия
3. **Целенаправленность** — ориентация на достижение конкретной цели или решение задачи
4. **Способность к обучению** — улучшение производительности через обратную связь

### 1.2 Архитектура LLM-агента

Типичная архитектура LLM-агента состоит из следующих компонентов:

```
┌─────────────────────────────────────────┐
│         Пользовательская задача         │
└────────────┬──────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────┐
│    Контекстный менеджер                 │
│  (сбор релевантной информации)          │
└────────────┬──────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────┐
│    LLM (Claude, GPT, и т.д.)            │
│  (принятие решения)                     │
└────────────┬──────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────┐
│    Модуль действия (Tools/API)          │
│  (выполнение решения)                   │
└────────────┬──────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────┐
│    Верификация и обратная связь         │
│  (проверка и улучшение)                 │
└─────────────────────────────────────────┘
```

### 1.3 Сравнение: Чат-бот vs LLM-агент

| Параметр | Чат-бот | LLM-агент |
|----------|---------|-----------|
| **Инициатива** | Реактивная (ответы на вопросы) | Проактивная (самостоятельные действия) |
| **Действия** | Только генерация текста | Выполнение команд в системе |
| **Контекст** | Одно окно разговора | Динамическое управление контекстом |
| **Итерация** | Однократный ответ | Многошаговый процесс с уточнением |
| **Верификация** | Ручная проверка пользователем | Автоматическая проверка и исправление |
| **Примеры** | ChatGPT, Gemini | Claude Code, Anthropic Agent SDK |

## 2. Agent Loop: цикл работы агента

### 2.1 Четырехэтапный цикл агента

Основной паттерн работы LLM-агента известен как **Agent Loop** — цикличный процесс, который повторяется до достижения цели. Claude Code использует следующий четырехэтапный цикл:

**Gather Context → Take Action → Verify Work → Repeat**

```
        ┌──────────────────────┐
        │   Gather Context     │
        │  (сбор информации)   │
        └──────────────┬───────┘
                       │
                       ▼
        ┌──────────────────────┐
        │   Take Action        │
        │  (выполнение)        │
        └──────────────┬───────┘
                       │
                       ▼
        ┌──────────────────────┐
        │   Verify Work        │
        │  (проверка)          │
        └──────────────┬───────┘
                       │
                       ▼
        ┌──────────────────────┐
        │   Ready?             │
        └──────────┬───────┬───┘
                   │       │
              YES  │       │ NO
                   ▼       ▼
                 STOP   REPEAT
```

### 2.2 Фаза 1: Gather Context (сбор контекста)

На этом этапе агент собирает релевантную информацию из доступных источников.

**Основные источники контекста:**

- **Файловая система** — агент может читать и анализировать файлы
- **История разговора** — предыдущие действия и результаты
- **Внешние API** — данные из внешних систем
- **Динамический анализ** — выполнение bash-команд для получения информации

**Пример:**


*Bash скрипт длиннее 2 строк.*


Эта фаза критически важна, потому что качество контекста определяет качество решений, которые примет агент.

### 2.3 Фаза 2: Take Action (выполнение действий)

После сбора контекста LLM анализирует информацию и выбирает подходящие действия.

**Типы действий:**

1. **Tool Calls** — вызов специализированных функций
2. **Bash Commands** — выполнение системных команд
3. **Code Generation** — создание и выполнение кода
4. **API Calls** — обращение к внешним сервисам

**Пример процесса принятия решения:**

```
Агент видит задачу: "Обработай CSV файл и создай отчет"

1. Анализ контекста:
   - Размер файла: 50MB (слишком большой для прямого чтения)
   - Формат: CSV с заголовками
   - Требуемый формат вывода: HTML отчет

2. Выбор действия:
   - Прочитать заголовки файла (grep)
   - Запустить скрипт на Python для обработки
   - Создать HTML отчет
```

### 2.4 Фаза 3: Verify Work (проверка результатов)

Это критическая фаза, где агент сам проверяет качество своей работы.

**Методы верификации:**

1. **Rules-based** — проверка по определенным правилам
2. **Visual feedback** — скриншоты и визуальная проверка
3. **LLM-as-judge** — использование другого LLM для оценки
4. **Linting** — проверка кода на ошибки синтаксиса

**Пример:**


*Python код.*


### 2.5 Фаза 4: Repeat (повтор цикла)

Если проверка не прошла или задача еще не решена, цикл повторяется с новыми данными.

**Условия повтора:**
- Задача еще не выполнена
- Результаты не соответствуют критериям
- Обнаружены ошибки

**Условия завершения:**
- Успешное выполнение задачи
- Достижение максимального количества итераций
- Критическая ошибка, требующая вмешательства пользователя

## 3. Инструменты агента: tools, bash, code generation, MCP

### 3.1 Архитектура инструментов

LLM-агенты нуждаются в различных инструментах для взаимодействия с окружающей средой. Правильный выбор и организация инструментов критична для производительности агента.

**Иерархия инструментов:**

```
┌─────────────────────────────────┐
│    Специализированные Tools     │
│   (fetchInbox, searchEmails)    │
│  (высокий уровень абстракции)   │
└─────────────────┬───────────────┘
                  │
┌─────────────────▼───────────────┐
│      Код (Python, JavaScript)   │
│   (точная, комбинируемая логика)│
└─────────────────┬───────────────┘
                  │
┌─────────────────▼───────────────┐
│   Bash команды (универсально)   │
│  (гибкость, доступ к системе)   │
└─────────────────────────────────┘
```

### 3.2 Custom Tools (специализированные инструменты)

Custom Tools — это специальные функции, разработанные для конкретной задачи. Они представляют собой абстракцию над сырыми API, обеспечивая точно определенный интерфейс, который LLM может безопасно использовать.

**Преимущества Custom Tools:**
- **Высокий уровень абстракции** — скрывают внутреннюю сложность, излагают только релевантные параметры
- **Ясная семантика для LLM** — описание инструмента помогает модели правильно выбрать его
- **Контроль над логикой** — разработчик полностью контролирует, что инструмент может делать
- **Безопасность** — можно валидировать входные данные и ограничивать действия

**Структура определения Tool:**

Каждый инструмент определяется с помощью JSON Schema и содержит:
- **name** — уникальный идентификатор инструмента
- **description** — понятное описание того, что делает инструмент (LLM читает это!)
- **inputSchema** — JSON Schema объект, описывающий параметры

Пример структуры инструмента для email-системы:


*JSON блок длиннее 15 строк.*


**Цикл выполнения инструмента:**

1. Агент анализирует задачу и видит, что ему нужны письма
2. Агент выбирает инструмент `fetchInbox` и заполняет параметры: `filter: "from:boss"`, `limit: 10`
3. Система валидирует параметры по JSON Schema
4. Выполняется фактический код инструмента (это может быть HTTP запрос, вызов БД, etc.)
5. Результат возвращается агенту в виде структурированных данных
6. Агент анализирует результат и может выбрать дальнейшие действия

**Правила проектирования инструментов:**
- Инструмент должен решать **одну конкретную задачу**
- Параметры должны быть **минимальными и необходимыми**
- Описание должно быть понятным для LLM
- Инструмент должен **валидировать входные данные**

### 3.3 Bash как универсальный инструмент

Bash-команды предоставляют максимальную гибкость и прямой доступ к системным ресурсам. В контексте агента bash используется для операций, которые трудно или невозможно сделать через типизированные инструменты.

**Типичные категории bash-операций:**

**Поиск и фильтрация файлов:**
- Поиск файлов по паттерну имени, размеру, времени создания
- Рекурсивный поиск по содержимому (grep)
- Фильтрация результатов по регулярным выражениям

**Обработка текстовых данных:**
- Извлечение определённых строк или столбцов
- Трансформация форматов (CSV ↔ JSON)
- Подсчёт, сортировка, дедупликация

**Системная информация:**
- Состояние процессов, использование памяти, диска
- Список установленных пакетов
- Информация о сети и соединениях

**Интеграция с внешними сервисами:**
- HTTP запросы через `curl`
- Обработка JSON/XML через `jq`, `xmllint`
- Работа с Git, Docker, Kubernetes командами

**Цикл работы с bash в контексте агента:**

```
Агент анализирует задачу
       ↓
Определяет нужна ли bash-команда (vs инструмент, vs код)
       ↓
Генерирует bash-команду
       ↓
Выполняет команду в песочнице
       ↓
Получает stdout и stderr
       ↓
Обрабатывает результат и анализирует результат
       ↓
Решает: достаточно ли информации или нужны дополнительные команды
```

**Почему bash, а не код или инструменты:**
- **Быстрота:** короткие команды выполняются быстрее, чем запуск Python
- **Системный доступ:** некоторые операции доступны только через shell
- **Минимальный overhead:** не нужно загружать интерпретатор языка
- **Интеграция:** естественно работает с системными утилитами

### 3.4 Code Generation (генерация кода)

Code generation — способность агента генерировать и выполнять код для решения задач, требующих сложной логики или трансформации данных. Это один из самых мощных инструментов агента, но требует осторожности в плане безопасности.

**Когда использовать кодогенерацию:**
- **Структурированная обработка данных** — трансформация CSV, JSON, баз данных
- **Математические вычисления** — статистика, линейная алгебра, оптимизация
- **Сложные трансформации** — когда bash недостаточно, а специализированного инструмента нет
- **Создание файлов** — генерация Excel, PDF, HTML отчетов
- **Визуализация данных** — графики, диаграммы, dashboard'ы

**Паттерн работы с кодогенерацией:**

```
Агент анализирует задачу
       ↓
"Мне нужно обработать CSV, агрегировать по группам и создать отчет"
       ↓
Выбирает кодогенерацию (не подходят bash или инструменты)
       ↓
Генерирует Python код с нужными импортами и логикой
       ↓
Выполняет код в безопасной песочнице
       ↓
Получает результаты (числа, файлы, графики)
       ↓
Верифицирует результаты и анализирует
```

**Архитектура кодогенерации:**

Агент не просто генерирует случайный код. Он:
1. **Анализирует задачу** — какие данные нужны, какой формат на выходе
2. **Выбирает библиотеки** — pandas для CSV, matplotlib для графиков, etc.
3. **Пишет структурированный код** — с комментариями, обработкой ошибок
4. **Добавляет валидацию** — проверяет, что входные данные корректны
5. **Обрабатывает ошибки** — gracefully падает с понятным сообщением об ошибке

**Преимущества кодогенерации:**
- **Точность** — логика явно описана в коде, минимизирует амбигуозность
- **Воспроизводимость** — одинаковый код = одинаковые результаты
- **Прозрачность** — разработчик видит ровно что выполнится
- **Комбинируемость** — код можно модифицировать и переиспользовать

### 3.5 Model Context Protocol (MCP)

MCP (Model Context Protocol) — стандартизированный открытый протокол, разработанный Anthropic, для безопасной интеграции LLM-агентов с внешними сервисами, приложениями и данными. MCP позволяет агентам получать доступ к инструментам и информации без необходимости знать внутренние детали каждого сервиса.

**Ключевая идея MCP:**

Вместо того, чтобы агент напрямую интегрировался с каждым сервисом (Slack, GitHub, Asana), существует MCP Server — прослойка, которая:
- **Скрывает детали аутентификации** (OAuth, API ключи находятся на стороне сервера)
- **Предоставляет единый интерфейс** ко всем инструментам
- **Управляет контекстом** и оптимизирует использование токенов
- **Обеспечивает безопасность** — агент не может напрямую обращаться к API

**Архитектура MCP:**

```
┌────────────────────┐
│   Agent (Claude)   │
│  Не знает деталей  │
│  реализации        │
└────────┬───────────┘
         │
         │ Стандартный протокол
         ▼
┌──────────────────────────┐
│  MCP Server              │
│  - Управляет auth        │
│  - Трансформирует API    │
│  - Управляет контекстом  │
└──────┬───────┬───────────┘
       │       │
    ┌──▼──┐ ┌──▼──┐
    │Slack│ │GitHub│  ...
    └─────┘ └─────┘
```

**Типичные MCP Servers:**

- **Slack MCP** — инструменты для чтения сообщений, поиска, отправки нотификаций
- **GitHub MCP** — работа с репозиториями, issues, pull requests, commits
- **Asana MCP** — управление задачами, проектами, командой
- **Linear MCP** — альтернатива Asana
- **Google Drive MCP** — доступ к документам, sheets, files

**Инструменты через MCP vs Custom Tools:**

| Аспект | Custom Tool | MCP Server |
|--------|-----------|-----------|
| **Кто управляет кредами** | Агент | MCP Server (безопаснее) |
| **Где хранятся ключи** | В переменных окружения | На сервере (защищено) |
| **Сложность настройки** | Простая | Средняя |
| **Масштабируемость** | До 10-20 инструментов | Десятки инструментов |
| **Безопасность** | Агент имеет полный доступ | Агент имеет ограниченный доступ |

**Использование MCP в практике:**

Когда агент получает новый запрос, MCP Server предоставляет ему доступные инструменты. Агент может затем:
- "Какие задачи в Asana назначены мне?" → MCP вызывает `get_asana_tasks(assignee="current_user")`
- "Создай PR в GitHub с изменениями" → MCP вызывает `create_pull_request(...)`
- "Отправь сообщение в Slack" → MCP вызывает `post_slack_message(...)`

## 4. Sub-агенты: параллелизм, изоляция контекста, делегирование

### 4.1 Концепция sub-агентов

Sub-агенты — это агенты, которые работают как часть более крупного агента, обычно для решения подзадач. Они позволяют распределить работу и управлять контекстом более эффективно.

**Основные преимущества:**

1. **Параллелизм** — несколько sub-агентов могут работать одновременно
2. **Изоляция контекста** — каждый sub-агент имеет собственное окно контекста
3. **Специализация** — каждый sub-агент оптимизирован для своей задачи
4. **Масштабируемость** — легче управлять большими проектами

### 4.2 Архитектура sub-агентов

```
┌──────────────────────────────────────┐
│      Главный агент (Orchestrator)    │
│         Координирует работу           │
└────────────────┬─────────────────────┘
                 │
        ┌────────┼────────┬─────────┐
        │        │        │         │
        ▼        ▼        ▼         ▼
   ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐
   │ Sub │  │ Sub │  │ Sub │  │ Sub │
   │Agent│  │Agent│  │Agent│  │Agent│
   │  1  │  │  2  │  │  3  │  │  4  │
   └──┬──┘  └──┬──┘  └──┬──┘  └──┬──┘
      │         │        │        │
      └────────────┬─────────────┘
                   │
          (результаты собираются
           главным агентом)
```

### 4.3 Типы встроенных sub-агентов

Claude Code включает несколько встроенных типов sub-агентов:

**1. Explore (исследователь)**

```yaml
name: explore
description: "Быстрый поиск и анализ кодовой базы"
model: haiku  # Быстрая модель
tools:
  - read_file
  - glob
  - grep
disabledTools:
  - write_file
  - edit_file
```

Используется для:
- Поиска файлов по паттерну
- Анализа структуры проекта
- Быстрого сканирования содержимого

**2. Plan (планировщик)**

```yaml
name: plan
description: "Планирование и разработка стратегии"
model: sonnet
tools: all
```

## 5. Hooks: перехватчики жизненного цикла агента

### 5.1 Что такое hooks

Hooks — это функции или скрипты, которые выполняются в определенные моменты жизненного цикла агента. Они позволяют перехватить события и принять решения до того, как действие выполнится, или после.

**Основное назначение:**
- Логирование и мониторинг
- Валидация и проверка безопасности
- Условное выполнение код
- Интеграция с внешними системами
- Аудит

### 5.2 Жизненный цикл и события

```
┌──────────────┐
│ SessionStart │ (начало сессии)
└──────────────┘
       │
       ▼
┌──────────────────────┐
│ UserPromptSubmit     │ (пользователь отправляет запрос)
└──────────────────────┘
       │
       ▼
      ┌─────────────────────────────┐
      │  [Цикл агента]              │
      │                             │
      │  ┌──────────────┐           │
      │  │ PreToolUse   │ (перед инструментом)
      │  └──────────────┘           │
      │        │                    │
      │        ▼                    │
      │  ┌──────────────┐           │
      │  │ Tool Execute │           │
      │  └──────────────┘           │
      │        │                    │
      │        ▼                    │
      │  ┌──────────────┐           │
      │  │PostToolUse   │ (после успеха)
      │  │PostToolUse   │ (после ошибки)
      │  │Failure       │           │
      │  └──────────────┘           │
      │        │                    │
      │        ▼                    │
      │  ┌──────────────┐           │
      │  │ Repeat?      │           │
      │  └──────────────┘           │
      └─────────────────────────────┘
              │
              ▼
       ┌──────────────┐
       │ Stop         │ (конец сессии)
       └──────────────┘
              │
              ▼
       ┌──────────────┐
       │ SessionEnd   │
       └──────────────┘
```

### 5.3 Основные типы hooks

| События | Описание | Когда срабатывает |
|---------|---------|-------------------|
| **SessionStart** | Начало сессии | При запуске агента |
| **UserPromptSubmit** | Отправка запроса | Перед обработкой запроса |
| **PreToolUse** | Перед инструментом | До выполнения tool call |
| **PostToolUse** | После успеха | После успешного выполнения |
| **PostToolUseFailure** | После ошибки | После ошибки в tool call |
| **Notification** | Уведомление | Когда агент отправляет уведомление |
| **SubagentStart** | Старт sub-агента | Когда запускается sub-агент |
| **SubagentStop** | Стоп sub-агента | Когда завершается sub-агент |
| **Stop** | Конец ответа | Когда агент заканчивает ответ |
| **SessionEnd** | Конец сессии | При закрытии сессии |

### 5.4 Конфигурация hooks

Hooks конфигурируются в `settings.json` или в frontmatter sub-агентов.

**Структура конфигурации hook:**

```json
{
  "hooks": [
    {
      "event": "PreToolUse",
      "matcher": "bash",
      "command": "validate_bash_command.sh"
    },
    {
      "event": "PostToolUse",
      "matcher": "write_file",
      "command": "check_file_syntax.sh"
    }
  ]
}
```

**Типы matchers:**
- Название инструмента (точное совпадение)
- Regex паттерн
- Несколько инструментов

### 5.5 Практические примеры use cases

**Use Case 1: Валидация bash команд перед выполнением**

Hook для `PreToolUse` может проверить опасные bash-команды и заблокировать их. Например, "только для чтения" агент блокирует любые команды с `rm`, `DROP`, `DELETE`, etc.

**Процесс:**
1. Агент генерирует bash команду
2. Hook перехватывает её перед выполнением
3. Hook проверяет, содержит ли она опасные паттерны
4. Если да → блокирует (exit code 2)
5. Если нет → разрешает (exit code 0)

**Use Case 2: Логирование и аудит всех действий**

Hook для `PostToolUse` может логировать каждое выполненное действие в audit trail. Это критично для:
- Compliance требований (HIPAA, SOC2, GDPR)
- Отладки и troubleshooting
- Анализа использования агента

**Информация в логе:**
- Временная метка
- Название инструмента
- Параметры
- Результат (успех/ошибка)
- Вызванный пользователь/сессия

**Use Case 3: Валидация сгенерированного кода перед выполнением**

Hook может анализировать Python-код и проверять его на:
- Синтаксические ошибки (парсинг AST)
- Чрезмерную вложенность (признак сложного кода)
- Использование опасных функций (`eval`, `exec`)
- Необычно высокое использование памяти или CPU

### 5.6 Exit codes и их значение

| Exit Code | Значение | Действие |
|-----------|----------|---------|
| 0 | Успех | Инструмент выполняется обычно |
| 1 | Ошибка | Инструмент блокируется, output показывается как error |
| 2 | Блокировка | Инструмент блокируется, агент получает сообщение об отказе |

## 6. Verify Work: правила валидации, визуальная обратная связь, LLM-as-judge

### 6.1 Цикл верификации

Верификация — это процесс проверки, действительно ли агент достиг цели и выполнил задачу правильно.

```
┌─────────────────────┐
│  Результат агента   │
└──────────┬──────────┘
           │
           ▼
┌────────────────────────────┐
│  Rules-based validation    │
│  (проверка правил)         │
└──────────┬─────────────────┘
           │ Passed?
        ┌──┴──┐
        │Yes  │No
        │     ▼
        │  ┌──────────────────────┐
        │  │ Visual validation    │
        │  │ (скриншот/рендер)    │
        │  └──────────┬───────────┘
        │             │
        │             ▼
        │  ┌──────────────────────┐
        │  │ LLM-as-judge         │
        │  │ (другой LLM оценивает)
        │  └──────────┬───────────┘
        │             │
        ▼             │
    ┌────────────────┐│
    │  Успех!        │
    │  Возврат       │
    └────────────────┘
```

### 6.2 Rules-based validation

Это самый надежный и быстрый метод валидации — проверка по определенным правилам. Rules-based validation использует **структурированные проверки**, которые могут быть:

**Типы правил:**

1. **Format validation** — проверка формата (email, URL, phone, JSON, HTML)
2. **Schema validation** — проверка соответствия JSON Schema
3. **Content validation** — проверка наличия обязательных полей
4. **Range validation** — проверка диапазонов (длина, число, размер)
5. **Pattern matching** — проверка по регулярным выражениям

**Примеры областей применения:**

- **Email адреса:** проверка формата, наличие @, правильное имя домена
- **Дата/время:** проверка, что формат соответствует ISO 8601, дата не в прошлом
- **HTML/XML:** синтаксическая корректность, наличие требуемых тегов
- **JSON:** корректная структура, требуемые поля присутствуют
- **Числовые данные:** значения в допустимом диапазоне, правильное количество знаков
def validate_json_schema(data, schema):
    """Проверить соответствие JSON Schema"""
    from jsonschema import validate
    try:
        validate(instance=data, schema=schema)
        return True, None
    except Exception as e:
        return False, f"Schema validation failed: {str(e)}"

# Правило 4: Длина текста
def validate_text_length(text, min_len, max_len):
    """Проверить длину текста"""
    if len(text) < min_len:
        return False, f"Text too short (min {min_len})"
    if len(text) > max_len:
        return False, f"Text too long (max {max_len})"
    return True, None

# Правило 5: Наличие требуемых полей
def validate_required_fields(data, required_fields):
    """Проверить наличие всех требуемых полей"""
    missing = [f for f in required_fields if f not in data]
    if missing:
        return False, f"Missing fields: {', '.join(missing)}"
    return True, None
```

**Интеграция в agent loop:**

> **Удалено:** Python код для верификации черновиков.
>
> **Концепция:** Агент определяет набор правил валидации (email, тема, длина тела) и применяет их последовательно. Если любое правило не пройдёт, возвращается детальное объяснение ошибки. Это базовый паттерн для rule-based верификации в агентных системах.

### 6.3 Visual validation (визуальная обратная связь)

Для визуальных задач (UI, визуализация) необходима визуальная проверка.

**Процесс:**

1. Агент создает визуальный результат (HTML, изображение)
2. Берется скриншот результата
3. Скриншот показывается LLM для визуальной проверки
4. LLM подтверждает или запрашивает изменения

**Пример: верификация HTML письма**

> **Удалено:** Python код для визуальной верификации HTML с Selenium.
>
> **Концепция:** Паттерн визуальной валидации включает три этапа: (1) сохранение результата во временный файл, (2) скриншотирование в браузере, (3) отправка изображения LLM с промптом для анализа. Это позволяет агенту верифицировать внешний вид результатов.

**Использование с Playwright MCP:**

> **Удалено:** Python код для интеграции с Playwright MCP.
>
> **Концепция:** Вместо прямого использования Selenium, можно использовать MCP сервер для браузерной автоматизации. Это позволяет агенту выполнять сложные UI проверки через единую интерфейс MCP инструментов.

### 6.4 LLM-as-Judge

LLM-as-Judge — техника использования другой языковой модели для оценки качества результатов основного агента. Это особенно полезно для субъективных критериев, где простые правила недостаточны.

**Архитектура:**

```
┌──────────────────────┐
│  Main Agent Result   │
│  (письмо, код, etc.) │
└──────────┬───────────┘
           │
           ▼
┌──────────────────────────┐
│  Judge LLM (cheapest)    │
│  (например, claude-haiku)│
│                          │
│  Оценивает:             │
│  - Качество, тон        │
│  - Правильность         │
│  - Соответствие цели    │
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────┐
│  Score + Feedback    │
│  (0-10, комментарии) │
└──────────────────────┘
```

**Когда использовать LLM-as-Judge:**

- **Субъективные критерии** — когда оценка требует понимания контекста и нюансов (тон письма, убедительность, стиль)
- **Сложные требования** — когда простые rules-based проверки недостаточны
- **Детальная обратная связь** — когда нужно объяснить, почему результат плохой
- **Fuzzy критерии** — когда нет четкого определения успеха

**Когда ИЗБЕГАТЬ LLM-as-Judge:**

- **Структурные проверки** — формат, синтаксис, schema (используй rules-based)
- **Критичные задачи** — где нужна 100% надежность
- **Высокий volume** — когда нужно оценить тысячи результатов в день (дорого)
- **Низкий latency** — когда результат нужен за миллисекунды

**Паттерн использования:**

1. Основной агент генерирует результат (письмо, код, отчет)
2. Система проверяет результат простыми rules-based проверками
3. Если rules-based проверки прошли, но есть субъективные критерии → вызывается Judge LLM
4. Judge LLM анализирует результат и возвращает оценку (0-10) и feedback
5. Если оценка < threshold → основной агент переделывает результат
6. Цикл повторяется

**Оптимизация стоимости:**

Для LLM-as-Judge используется **дешевая и быстрая модель** (например, claude-3-5-haiku), не самая мощная. Это не проблема, потому что судья оценивает только один результат, а основной агент может быть мощнее.

## 7. Проектирование агентов: от концепции к спецификации

### 7.1 Что такое спецификация агента

Спецификация агента — это ясное описание того, что должен делать агент, вместо того, чтобы просто писать код и смотреть, что получится. Хорошая спецификация включает:

**Ключевые компоненты спецификации:**

1. **Цель (Purpose)** — зачем этот агент нужен (одно-две строки)
2. **Основные функции (Responsibilities)** — что агент может делать
3. **Требуемые инструменты (Tools)** — какие инструменты нужны для работы
4. **Структуры данных** — какие данные агент обрабатывает (Email, Task, Document)
5. **Поведение (Behavior)** — сценарии использования и expected flow
6. **Критерии качества** — как оценить, хорошо ли агент справляется
7. **Обработка ошибок** — что делать когда что-то идет не так

**Пример спецификации для Email Agenta в сокращении:**

```
Цель: Автоматизировать управление входящей почтой

Функции:
- Поиск писем по критериям (from, subject, keywords)
- Создание черновика ответа
- Архивирование по правилам

Инструменты:
- fetchInbox(filter, limit) → List<Email>
- searchEmails(query) → List<Email>
- createDraft(to, subject, body) → Draft

Сценарии:
1. "Найти письма от john@example.com за неделю"
   → searchEmails("from:john@example.com date:7d")
   → вернуть список писем

2. "Ответить на последнее письмо от Алисы"
   → получить последнее письмо от Alice
   → понять контекст
   → создать draft ответа
   → проверить тон через LLM-judge

Критерии качества:
- Ответ должен быть 20-500 символов
- Ответ должен быть релевантен оригинальному письму
- Тон должен соответствовать истории переписки

Обработка ошибок:
- Письмо не найдено → "Письмо не найдено. Попробуйте другой фильтр."
- Невалидный фильтр → "Синтаксис фильтра не распознан."
```

### 7.2 Почему спецификация важна

**Спецификация first** помогает потому что:
- **Ясность** — разработчик и LLM понимают границы агента
- **Тестируемость** — легче проверить, делает ли агент то, что в спеке
- **Масштабируемость** — легче добавлять новых агентов, расширять функции
- **Переиспользуемость** — спецификация может быть основой для документации и API

### 7.4 Процесс создания агента по спецификации

```
1. WRITE SPEC
   └─ Определи назначение, инструменты, поведение

2. IMPLEMENT TOOLS
   └─ Создай или интегрируй нужные инструменты

3. DEFINE CONSTRAINTS
   └─ Какие действия разрешены/запрещены?

4. CREATE PROMPTS
   └─ Системный prompt на основе спеки

5. SETUP VERIFICATION
   └─ Rules, visual checks, LLM-judge

6. SETUP HOOKS
   └─ Логирование, валидация, аудит

7. TEST & ITERATE
   └─ Проверь против spec'ки
   └─ Итерируй до совпадения

8. DEPLOY
   └─ Запусти в production
```

### 7.5 CLAUDE.md: спецификация на уровне проекта

Каждый агент может иметь файл `CLAUDE.md` с инструкциями:

```markdown
# Email Management Agent

## System Context
This agent manages email for alice@company.com

## Capabilities
The agent can:
- Search emails using Gmail API
- Create and send drafts
- Organize emails with labels
- Set up automatic filters

## Constraints
The agent MUST:
- Never delete emails permanently (only archive)
- Never send emails without user approval
- Respect privacy of CC'd recipients
- Log all actions for audit trail

## Tools Available
```yaml
tools:
  - name: gmail_search
    enabled: true
    scope: read-only

  - name: gmail_create_draft
    enabled: true
    requires_approval: true
```

## Examples

### Example 1: Find urgent emails
User: "Show me emails marked as urgent from last month"
Agent: Searches with label:urgent date:>2024-12-01
Returns: List of 12 urgent emails

### Example 2: Create response
User: "Draft a reply thanking the client for their proposal"
Agent:
1. Gets latest email context
2. Generates professional response
3. Verifies tone (scores 8.5/10)
4. Shows draft to user
5. Waits for approval before sending

## Error Handling
- Network timeout: Retry up to 3 times
- Invalid email: Show validation error
- Missing permissions: Ask user to grant access
```

## 8. Паттерны отладки агентов

### 8.1 Почему отладка агентов сложнее обычного кода

Отладка LLM-агентов отличается от отладки обычного кода тем, что:
- **LLM недетерминирован** — один input может дать разные outputs
- **Контекст зависит от истории** — ошибка в прошлом может проявиться в будущем
- **Инструменты отказывают непредсказуемо** — API может быть недоступен, формат данных изменился
- **Сложно отследить логику** — почему LLM выбрал то или иное действие?

### 8.2 Стратегия 1: Детальное логирование каждого цикла

Каждый цикл агента должен быть залогирован со следующей информацией:

**Фаза "Gather Context":**
- Какие файлы/данные были прочитаны?
- Сколько токенов использовано?
- Какая информация попала в контекст?

**Фаза "Take Action":**
- Какой инструмент выбран?
- Какие параметры переданы?
- Какой был результат?

**Фаза "Verify Work":**
- Какие проверки запустились?
- Какие прошли, какие не прошли?
- Какой feedback получил агент?

**Логирование в JSON** позволяет затем анализировать логи программно и искать паттерны ошибок:

    "result_output_chars": 2540

### 8.3 Стратегия 2: Анализ решений LLM (Extended Thinking)

Когда агент делает странный выбор, используйте Extended Thinking — попросите LLM объяснить свою логику в деталях.

**Паттерн:**
1. Зафиксируйте input и output, которые выглядят неправильно
2. Создайте debug prompt: "Вот что ты делал. Почему ты выбрал именно это действие?"
3. Используйте Extended Thinking для детальной трассировки мысли
4. Анализируйте "thinking" блок, чтобы понять логику

**Это полезно для:**
- Поиска логических ошибок в reasoning
- Понимания, какая информация в контексте привела к неправильному решению
- Итерирования промпта

### 8.4 Стратегия 3: Проверка консистентности состояния

В каждом цикле проверяйте:

- **Контекст не пуст** — есть ли вообще информация для анализа?
- **Бюджет токенов в норме** — не превышен ли лимит?
- **Циклы не превышены** — не зациклился ли агент?
- **Нет бесконечного цикла** — не повторяет ли агент одно и то же действие?
- **Состояние инструментов в норме** — API доступны, креденшалы валидны?

**Проверка бесконечного цикла:**
Если последние 5 действий идентичны → агент зациклился. Остановить и сообщить об ошибке.
