# Модуль 1, Подмодуль 4: Reasoning-фреймворки и архитектуры мышления агентов

## Введение: Почему рассуждение — главная супермощь LLM

Если базовые LLM — это системы с мощной памятью и языковыми навыками, то рассуждающие агенты — это системы, которые **думают вслух**, разбирают сложные задачи на части и корректируют себя в процессе работы. Рассуждение — это не просто способность генерировать текст. Это способность:

- Разложить сложную задачу на управляемые подзадачи
- Показать пошаговую логику решения
- Проверить результаты и исправить ошибки
- Интегрировать информацию из разных источников
- Адаптироваться к новой информации во время выполнения

В контексте LLM-агентов reasoning — это **переход от регургитации к рефлексии**, от ответов к объяснениям, от статичного к динамичному. Данный подмодуль изучает архитектуры и фреймворки, которые превращают LLM в мыслящие системы.

## 1. Reasoning в LLM-агентах: Основные техники

### 1.1 Chain-of-Thought (CoT)

**Chain-of-Thought** — фундаментальная техника, которая инструктирует модель показывать промежуточные шаги мышления перед финальным ответом.

**Принцип:**
```
Задача: Если в классе 15 человек и каждый сделал 3 проекта, сколько всего проектов?

CoT ответ:
1. В классе 15 человек
2. Каждый сделал 3 проекта
3. Всего проектов = 15 × 3 = 45

Ответ: 45 проектов
```

**Преимущества CoT:**
- Улучшает точность на 5-40% (в зависимости от сложности задачи)
- Делает логику прозрачной и проверяемой
- Помогает моделям разбирать сложные многошаговые задачи
- Позволяет идентифицировать и исправлять ошибки в промежуточных шагах

**Когда использовать:**
- Математические задачи с несколькими этапами
- Логические головоломки
- Анализ многосложных сценариев
- Любые задачи, где нужна обоснованность ответа

**Недостатки:**
- Увеличивает длину ответа (больше токенов)
- Может замедлить обработку
- Не гарантирует корректность промежуточных шагов

### 1.2 Tree-of-Thought (ToT)

**Tree-of-Thought** расширяет CoT, позволяя модели исследовать **несколько путей решения параллельно**, как если бы она исследовала дерево возможностей.

**Архитектура ToT:**
```
                   [Исходная задача]
                        |
                    [Шаг 1]
                    /    |    \
                 [1a]  [1b]  [1c]  ← Три возможных пути
                  |      |      |
                [2a]   [2b]   [2c]
                /|\    /|\    /|\
              ...              ...
```

**Компоненты ToT:**
1. **Решатель** (Solver) — анализирует каждый путь
2. **Оценщик** (Evaluator) — оценивает перспективность каждого пути
3. **Стратегия поиска** — решает, продолжать ли, отказаться или переключиться на другой путь

**Пример использования — решение задачи поиска:**

_**Алгоритм solve_with_tree_of_thought**:_
1. Псевдокод ToT
2. Генерируем несколько возможных следующих шагов
3. Оцениваем каждого кандидата
4. Выбираем лучшего кандидата для продолжения


**Результаты:**
- Улучшает решение сложных задач на 20-50%
- Особенно эффективен для комбинаторных проблем
- Требует больше вычислительных ресурсов из-за параллельного исследования

**Трейд-офф:**
```
| Аспект | CoT | ToT |
|--------|-----|-----|
| Скорость | Быстро | Медленнее |
| Точность | Хорошо | Лучше |
| Сложность реализации | Низкая | Высокая |
| Потребление токенов | Среднее | Высокое |
| Лучший для | Линейные задачи | Комбинаторные проблемы |
```

### 1.3 Graph-of-Thought (GoT)

**Graph-of-Thought** — самая сложная архитектура рассуждения, которая представляет решение как направленный граф, где узлы — это идеи, а рёбра — связи между ними.

**Когда ToT использует дерево (иерархия), GoT использует граф (сложные связи):**

```
Понимание проблемы ─→ Анализ ─────┐
       ↓                            ├─→ Синтез ─→ Ответ
   Сбор данных ────→ Оценка ────┘
       ↑
       └── Обратная связь ──────┘
```

**Механизм GoT:**
1. **Инициализация узлов** — каждое предложение/идея — это узел
2. **Определение рёбер** — модель определяет логические связи между идеями
3. **Итеративное уточнение** — граф расширяется и уточняется
4. **Агрегация** — финальный ответ извлекается из граф-структуры

**Преимущества:**
- Обнаруживает и использует скрытые связи между концепциями
- Позволяет агенту "прыгать" между идеями нелинейно
- Очень эффективен для синтеза информации
- Поддерживает циклические рассуждения (обратную связь)

**Применение в research:**
- Синтез научных статей
- Обнаружение противоречий в данных
- Построение причинно-следственных моделей
- Создание новых гипотез из существующих знаний

## 2. Внутренние механизмы современных моделей: Thinking Blocks и Reasoning Tokens

### 2.1 Thinking Blocks в Claude (Anthropic)

Claude 3.5 Sonnet (и более новые версии) имеют встроенную поддержку **explicit thinking** — режим, где модель может "думать" перед отправкой ответа.

**Механизм Thinking Blocks:**

```
Пользователь: "Почему искусственный интеллект может быть опасен?"

[Модель входит в режим Thinking — это внутренний процесс, не видимый пользователю]

<claude_thinking>
Это сложный вопрос. Нужно рассмотреть:
1. Технические риски (alignment, malfunction)
2. Социальные риски (неравенство, безработица)
3. Безопасность и контроль
4. Долгосрочные риски

Не должен быть ни слишком пугающим, ни слишком оптимистичным.
</claude_thinking>

Есть несколько ключевых рисков...
[публичный ответ]
```

**Использование в коде:**

```json
// Claude API: model="claude-3-7-sonnet-20250219", max_tokens=16000, thinking_budget=auto
```
_Параметры thinking: budget_tokens определяет глубину внутреннего рассуждения. Более глубокие рассуждения (10000+) лучше для сложных задач._


**Преимущества Thinking Blocks:**
- Модель может потратить больше "вычислительных ресурсов" на сложные задачи
- Повышает точность на сложных задачах на 30-60%
- Позволяет отловить ошибки до публикации ответа
- Можно контролировать "глубину" мышления через budget_tokens

**Когда использовать:**
- Сложный анализ и критическое мышление
- Дебаты и аргументация
- Исправление собственных ошибок
- Задачи, где нужна творчеств или высокая точность

### 2.2 Reasoning Tokens в OpenAI (o1, o3)

OpenAI идёт дальше и вводит **reasoning tokens** — специальные токены, которые модель использует исключительно для "мышления" без связи с выходом.

**Архитектура o1/o3:**

```
Входные токены
    ↓
[Reasoning Phase]
    ├─ Генерируются reasoning tokens (невидимые для пользователя)
    ├─ Модель исследует множество путей
    ├─ Ошибки исправляются внутри
    └─ Лучший путь решения выбирается
    ↓
[Output Phase]
    └─ Только финальное решение отправляется пользователю
```

**Практическое отличие от Claude:**

| Характеристика | Claude (Thinking Blocks) | OpenAI (o1/o3) |
|---|---|---|
| Видимость мышления | Да, можно увидеть | Нет, скрыто |
| Контроль глубины | Через budget_tokens | Автоматический |
| Скорость | Быстрее | Медленнее |
| Точность | Высокая | Очень высокая |
| Применение | Общие задачи | Критические задачи |

**Примеры использования o1:**

```json
// OpenAI API: model="o1", max_completion_tokens=default
```
_В o1/o3 reasoning происходит внутренне; стандартные параметры sampling (temperature, top_p) игнорируются._

## 3. ROMA: Reasoning Over Multi-Agent архитектура (Sentient AGI)

### 3.1 Проблема, которую решает ROMA

Представьте, что вы хотите решить задачу "Спланировать полномерный европейский тур на 2 недели с бюджетом $5000, включая виза, рейсы, отели и достопримечательности".

**Наивный подход:** Единый агент пытается всё сделать сразу — и часто терпит неудачу, потому что:
- Слишком много деталей для одного "ума"
- Невозможно параллелизировать работу
- Если агент ошибается на одном шаге, весь план рушится
- Нет прозрачности: что именно сломалось?

**Решение ROMA:** Декомпозиция задачи на управляемые подзадачи + иерархический контроль.

### 3.2 Архитектура ROMA

ROMA основана на **recursive plan-execute loop**:

```
def solve(task):
    # Шаг 1: Atomizer — принять решение
    if is_atomic(task):
        return execute(task)

    # Шаг 2: Planner — разбить задачу
    subtasks = plan(task)
    results = []

    # Шаг 3: Executor — выполнить подзадачи (параллельно!)
    for subtask in subtasks:
        results.append(solve(subtask))  # Рекурсия!

    # Шаг 4: Aggregator — собрать результаты
    return aggregate(results)

answer = solve("Спланировать европейский тур")
```

**Пять ключевых компонентов ROMA:**

#### 1. Atomizer — "Это атомарная задача?"

Atomizer определяет, можно ли задачу решить за 1-2 шага (is_atomic=True) или нужна декомпозиция (is_atomic=False, node_type=PLAN). Компонент использует chain-of-thought рассуждение для анализа сложности задачи.

**Логика Atomizer:**
- Если задача простая (1-2 шага) → `is_atomic=True`
- Если сложная (много подзадач) → `is_atomic=False`

#### 2. Planner — "Какие подзадачи нужны?"

Planner разбивает исходную задачу на набор SubTask с полями: goal (цель подзадачи), task_type (тип задачи), dependencies (список зависимостей от других подзадач). Dependency-based система автоматически создает DAG — направленный граф зависимостей, который определяет порядок и возможность параллельного выполнения.

**TaskType в ROMA (MECE: Mutually Exclusive, Collectively Exhaustive):**
- `RETRIEVE` — поиск информации (веб, БД, API)
- `THINK` — анализ, планирование, принятие решений
- `WRITE` — создание контента (текст, документы)
- `CODE_INTERPRET` — выполнение кода, обработка данных
- `IMAGE_GENERATION` — создание изображений

#### 3. Executor — "Выполнить подзадачу"

Executor использует стратегию ReAct (reason + act) с набором инструментов для выполнения каждой подзадачи. Для каждой подзадачи запускается reasoning loop: модель рассуждает о следующем шаге, вызывает инструмент, получает результат, анализирует его и решает, нужны ли дополнительные действия или задача завершена.

#### 4. Aggregator — "Собрать результаты в один ответ"

Aggregator синтезирует результаты из всех подзадач в единый согласованный ответ. Компонент берет исходную цель и результаты всех подзадач, использует chain-of-thought рассуждение для интеграции информации, разрешения противоречий и создания финального ответа, который полностью решает исходную задачу.

#### 5. Verifier — "Проверить результат"

Verifier проверяет, действительно ли синтезированный результат решает исходную задачу. Использует температуру 0.0 для детерминистичных проверок. Если результат не удовлетворяет критериям, возвращает feedback и архитектура может запустить цикл перепланирования (повторный вызов Planner с новыми инструкциями). Это создает feedback loop для итеративного улучшения качества.

### 3.3 Топологии взаимодействия в ROMA

ROMA поддерживает несколько топологий выполнения:

```
1. ЛИНЕЙНАЯ (Linear):
   Task → Subtask1 → Subtask2 → Subtask3 → Result

2. ПАРАЛЛЕЛЬНАЯ (Parallel):
                  ┌→ Subtask1 ┐
   Task →        ├→ Subtask2 ├→ Result
                  └→ Subtask3 ┘

3. ЗАВИСИМАЯ (Dependency-aware):
   Task → Subtask1 ──┐
          ↓         ├→ Subtask3 → Result
          Subtask2 ─┘
```

**Пример: Параллельное выполнение независимых подзадач**

ROMA автоматически определяет, какие подзадачи можно выполнять параллельно на основе графа зависимостей. Для задачи "Организовать конференцию": подзадачи "Забронировать зал" и "Пригласить спикеров" (независимые) выполняются одновременно, а "Организовать кейтеринг" (зависит от количества людей) запускается после завершения предыдущих. Это обеспечивает оптимальное использование вычислительных ресурсов и сокращает общее время выполнения.

### 3.4 Debate, Voting, Consensus в ROMA

ROMA поддерживает **мультиагентные стратегии согласия:**

_**DebateStrategy**: реализует дебатный механизм между агентами_


### 3.5 Бенчмарки и результаты ROMA

ROMA был оценен на трёх ключевых бенчмарках:

| Бенчмарк | Что тестирует | Результат ROMA |
|----------|---|---|
| **SEAL-0** | Поиск при шумных результатах | Превосходит baseline |
| **FRAMES** | Retrieval-Augmented Generation | Верхние позиции |
| **SimpleQA** | Простые факт-вопросы | Высокая точность |

**Ключевые находки:**
- Рекурсивная архитектура = лучше справляется со сложными задачами
- Параллелизм = 30-40% ускорение на независимых подзадачах
- Transparency = легче debug и итеровать

## 4. DeepAgents (LangChain): Глубокие агентные архитектуры

### 4.1 Философия DeepAgents

DeepAgents — это NOT еще один фреймворк. Это **готовый рабочий агент**, который вы берете прямо из коробки и сразу начинаете использовать, кастомизируя по мере необходимости.

**Контраст:**
```
Старый подход (LangChain базовый):
  Вы вкручиваете инструменты + модель + промпт + систему обработки контекста = долго

DeepAgents подход:
  Вы берете готового агента и говорите: "Вот твой промпт, вот инструменты" = быстро
```

### 4.2 Встроенные возможности DeepAgents

DeepAgents предоставляет готового агента с уже интегрированными ключевыми компонентами:
- **Planning** (write_todos) — автоматическое дробление задачи на подзадачи
- **Filesystem** (read_file, write_file, edit_file, ls, glob, grep) — полный доступ к файловой системе
- **Shell** (execute) — запуск команд в изолированной песочнице
- **Sub-agents** (task) — создание вложенных агентов с изолированным контекстом
- **Context management** — автоматическое сжатие больших разговоров через summarization

### 4.3 Мультишаговое планирование в DeepAgents

DeepAgents использует automatic multi-step planning для сложных задач. При получении запроса типа "Исследуй LangGraph и напиши резюме в Markdown", агент автоматически:

1. Создает план подзадач (write_todos)
2. Собирает информацию через файловую систему и shell команды
3. Структурирует контекст для больших объемов данных (summarization)
4. Генерирует результат (write_file)
5. Проверяет качество работы

**Внутренний процесс:**

Для задачи "Напиши курс по Python" агент самостоятельно определяет необходимые этапы: планирование структуры модулей, написание каждого модуля, добавление примеров кода, проверка качества. Выполнение оптимизируется через параллелизм независимых задач и последовательность зависимых, что приводит к структурированному курсу.

### 4.4 Самокорректирующиеся агенты

DeepAgents имеют встроенную способность к самокорректировке. Агент использует цикл write-execute-debug: пишет код, запускает его в изолированной среде через execute(), получает ошибку с traceback, анализирует ее, исправляет код и запускает заново. Этот цикл повторяется до успешного выполнения, что обеспечивает высокую надежность при работе с файлами и вычислениями.

### 4.5 Кастомизация DeepAgents

DeepAgents легко кастомизируется по трем ключевым параметрам:

- **model** — выбор LLM (OpenAI GPT-4o, Claude, другие через LangChain)
- **tools** — добавление своих инструментов (custom functions как search_arxiv)
- **system_prompt** — определение роли и поведения агента ("Ты специалист по ML...")

Все три параметра можно комбинировать: выбрать мощную модель, добавить специализированные инструменты (поиск в научных базах, API для данных) и задать экспертный контекст через промпт.

### 4.6 Sub-agents для делегирования

DeepAgents поддерживает иерархическую архитектуру через sub-agents. Родительский агент может создавать вложенные агенты для параллельного выполнения независимых задач. Каждый sub-agent получает изолированный контекст (отдельная историю, файловую систему логически изолированную) и выполняет свою подзадачу независимо. После завершения всех sub-agents результаты автоматически синтезируются родительским агентом в единый ответ. Это обеспечивает масштабируемость и модульность для сложных многошаговых проектов.

## 5. Paper2Agent: Автоматическое превращение статей в агентов

### 5.1 Что такое Paper2Agent и почему это важно

**Проблема:** Вы прочитали отличную исследовательскую статью с кодом, но:
- Код разбросан по репозиторию
- Примеры использования в Jupyter notebooks
- Нужно потратить дни на интеграцию
- Трудно воспроизвести результаты

**Решение:** Paper2Agent автоматически:
1. Находит все tutorial notebooks в репозитории
2. Исполняет их (разрешает зависимости)
3. Извлекает инструменты и функции
4. Создает MCP server
5. Интегрирует в Claude Code

```
Исследовательская статья
        ↓
  Репозиторий GitHub
        ↓
  [Paper2Agent Pipeline]
        ↓
  MCP Server
        ↓
  Claude Code Ready
```

### 5.2 Pipeline Paper2Agent

**Этап 1: Tutorial Scanner**

Paper2Agent сканирует репозиторий и находит все tutorial notebooks, создавая JSON с метаданными: название туториала, URL к notebook файлу, описание содержания.

Пример структуры:
```json
{
  "tutorials": [
    {"title": "Preprocessing and clustering", "url": "docs/tutorials/preprocessing.ipynb", "description": "Basic preprocessing steps"},
```

## 7. Паттерны рассуждения для production

### 7.1 ReAct: Reason + Act

ReAct — фундаментальный паттерн, используемый в production агентах:

```
Цикл ReAct:

[REASON] Модель размышляет о следующем шаге
  ↓
[ACT] Модель вызывает инструмент
  ↓
[OBSERVE] Получается результат от инструмента
  ↓
[DECIDE] Нужна ли еще информация?
  ├─ ДА → вернуться к REASON
  └─ НЕТ → дать финальный ответ
```

**Пример: Ответить "Когда был основан Stanford?"**

```
Думка: Мне нужна точная дата основания Stanford. Я не уверен, это 1885 или 1891.

Действие: search_web(query="Stanford University founded when")

Наблюдение: Stanford University was founded in 1885 by Leland Stanford.

Вывод: Я имею правильный ответ. Stanford был основан в 1885 году.
```

**Реализация в production:**

В production ReAct реализуется через prediction_strategy='react' с набором tools (инструментов). Агент автоматически чередует reasoning (подумать о следующем шаге и выборе инструмента) и action (вызов инструмента с параметрами). После получения результата цикл повторяется, пока агент не решит, что достаточно информации для финального ответа. Типичные инструменты включают web search, базы данных, вычисления, API вызовы.

### 7.2 Reflection Pattern

Reflection — паттерн, где агент **проверяет и критикует свою собственную работу**:

```
[INITIAL DRAFT]
  ↓
[REFLECTION]
  "Это хорошо? Есть ошибки? Что можно улучшить?"
  ↓
[REVISION]
  "Исправляю на основе критики"
  ↓
[FINAL OUTPUT]
```

**Пример: Написать техническую документацию**

```json
// Claude API: model="claude-3-7-sonnet-20250219", max_tokens=5000, thinking_budget=auto
```
_Параметры thinking: budget_tokens определяет глубину внутреннего рассуждения. Более глубокие рассуждения (10000+) лучше для сложных задач._


### 7.3 Planning + Execution разделение

Разделение фаз планирования и выполнения критично для production:

```
ФАЗА 1: ПЛАНИРОВАНИЕ
  ├─ Что нужно сделать?
  ├─ Какой порядок?
  ├─ Какие зависимости?
  └─ Сколько времени?

         ↓

ФАЗА 2: ВЫПОЛНЕНИЕ
  ├─ Выполнить Plan
  ├─ Обработать ошибки
  ├─ Адаптировать по ходу
  └─ Собрать результаты
```

**Пример: Миграция данных**

Для сложной задачи "Спланировать миграцию 10млн пользователей с PostgreSQL на MongoDB" pattern работает в три фазы:

1. **Планирование:** Planner разбивает на 7 подзадач с зависимостями: анализ схемы (независимая), создание индексов (независимая), валидация моделей (зависит от анализа), миграция метаданных, миграция юзеров батчами, валидация, switchover.

2. **Выполнение:** Executor параллельно запускает независимые подзадачи, последовательно — зависимые. Каждая подзадача использует специализированный инструмент (analyze_postgres_schema, create_mongodb_indexes и т.д.).

3. **Агрегация:** Aggregator собирает результаты всех подзадач в единый отчет о завершении миграции с метриками успеха.

### 7.4 Multi-Agent Debate

Multi-agent debate pattern используется для сложных решений, требующих множественных точек зрения. Создаются несколько специализированных агентов с разными ролями:

- **Optimist Agent** — анализирует положительные стороны и возможности
- **Pessimist Agent** — выявляет риски, проблемы, потенциальные отказы
- **Judge Agent** — мощная модель, которая взвешивает оба аргумента и дает обоснованное решение

Процесс: оба агента отвечают на вопрос независимо, судья получает оба мнения и принимает финальное решение.

**Когда использовать Multi-Agent Debate:**
- Стратегические решения (технологический выбор, инвестиции)
- Анализ риска (нужна разносторонняя оценка)
- Генерация идей (разные углы зрения)
- Проверка решений (дьявольский адвокат для критического анализа)

```
                             ↓

         │  VERIFICATION PHASE (Self-Correction)      │
         │  - Проверить результат                    │

         │  - Если ок → финальный ответ             │
         └───────────────────┬─────────────────────────┘
                             ↓
         ┌───────────────────────────────────────────┐
```

### 8.4 Тестирование агента

Тестирование reasoning-агента включает задачи разной сложности:

- **Basic Math** — простые арифметические задачи (2 + 2 = ?)
- **Complex Logic** — логические выводы (если A > B и B > C, то A > C)
- **Analytical** — открытые вопросы, требующие взвешивания (что важнее для LLM: размер модели или качество данных?)

Для каждой задачи проверяется наличие ключевых слов в ответе (expected_keywords) и подсчитывается количество итераций (циклов reasoning-correction). Успех определяется как совпадение ключевых слов и разумное количество итераций.

## Заключение: Путь к reasoning-ориентированным агентам

Reasoning — это не просто новая фишка, это **фундаментальный сдвиг** в том, как мы строим AI системы:

1. **Explicit thinking** (Claude thinking blocks, OpenAI o1) позволяет моделям показывать свою логику
2. **Hierarchical decomposition** (ROMA) позволяет масштабировать сложность
3. **Deep agent architectures** (DeepAgents) предоставляют готовые к использованию шаблоны
4. **Automated research integration** (Paper2Agent) делает состояние научного прогресса доступным
5. **Multi-agent patterns** (debate, voting, consensus) добавляют надежность и разносторонность

На практике production reasoning-агент сочетает:
- Chain-of-Thought для прозрачности
- Tree-of-Thought для исследования альтернатив
- ReAct для инструментального взаимодействия
- Reflection для самокоррекции
- Verification для надежности

Это не просто улучшает производительность — это создает **объяснимые, проверяемые, адаптивные системы**, которые могут справляться с задачами, которые раньше требовали человеческого участия.
