# Agent Loop — замкнутый цикл агента

## Главный тезис

LLM-приложение становится **агентом** не тогда, когда «умеет tool calling», а когда работает в замкнутом цикле:

1. собрать релевантный контекст,
2. выполнить действие,
3. проверить результат,
4. повторить только при наличии оснований.

Это фундаментальный паттерн, который отличает production-системы от демонстрационных прототипов. Без замкнутого цикла система генерирует предположения; с ним — производит верифицированные результаты.

---

## 1. Что отличает чат-ассистента от агентной системы

### 1.1 Определение LLM-агента

LLM-агент — это система, которая использует большую языковую модель в качестве «мозга» для принятия решений и выполнения действий в определённой среде. Основное отличие от простого чат-бота — способность агента действовать самостоятельно, без постоянного вмешательства пользователя на каждом шаге.

**Ключевые характеристики LLM-агента:**

1. **Автономность** — агент может принимать решения и действовать без явного указания пользователя на каждом шаге
2. **Реактивность** — способность реагировать на изменения в окружающей среде и корректировать свои действия
3. **Целенаправленность** — ориентация на достижение конкретной цели или решение задачи
4. **Способность к обучению** — улучшение производительности через обратную связь и верификацию

### 1.2 Агентная система как контур управления

Агентная система — не «модель с тулзами», а **контур управления выполнением**:

1. постановка цели,
2. план и декомпозиция,
3. выполнение шагов через инструменты,
4. верификация,
5. корректировка или остановка.

Это повторяется до достижения `done criteria`.

### 1.3 Сравнение: чат-бот vs LLM-агент

| Параметр | Чат-бот | LLM-агент |
|----------|---------|-----------|
| **Инициатива** | Реактивная (ответы на вопросы) | Проактивная (самостоятельные действия) |
| **Действия** | Только генерация текста | Выполнение команд в системе |
| **Контекст** | Одно окно разговора | Динамическое управление контекстом |
| **Итерация** | Однократный ответ | Многошаговый процесс с уточнением |
| **Верификация** | Ручная проверка пользователем | Автоматическая проверка и исправление |
| **Примеры** | ChatGPT (базовый режим), Gemini | Claude Code, Anthropic Agent SDK |

### 1.4 Почему это важно

Если нет явной фазы проверки, система превращается в генератор предположений. Результат должен завершаться проверкой (tests/review), а не фразой «должно работать». Чат-бот отвечает на вопрос и ждёт следующий; агент берёт задачу, декомпозирует её, последовательно решает подзадачи и сам проверяет результат.

## 2. Минимальная архитектура agent loop

Типичная архитектура LLM-агента состоит из пяти слоёв, каждый из которых решает конкретную задачу в цикле:

```
┌─────────────────────────────────────────┐
│         Пользовательская задача         │
└────────────┬──────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────┐
│    Orchestrator (маршрутизация)         │
└────────────┬──────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────┐
│    Specialist Layer (subagents/skills)  │
└────────────┬──────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────┐
│    Tool Layer (filesystem, shell, API)  │
└────────────┬──────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────┐
│    Verification Layer (тесты, policy)   │
└────────────┬──────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────┐
│    Memory/Context Layer (compaction)    │
└─────────────────────────────────────────┘
```

### 2.1 Orchestrator

Оркестратор — это «мозг» системы, который принимает цель от пользователя и управляет всем процессом выполнения.

Обязанности:
- Принимает цель, маршрутизирует задачи.
- Выбирает стратегию: `single-pass` (одна попытка) vs `iterative` (цикл с проверками).
- Контролирует остановку: проверяет done criteria после каждой итерации.
- Управляет бюджетом: лимиты по токенам, времени, числу итераций.

**Пример решения оркестратора:**

```
Задача: "Добавить endpoint /users в API"

Оркестратор анализирует:
→ Нужно: изменить код, добавить тесты, проверить lint
→ Стратегия: iterative (код → тесты → lint → fix → repeat)
→ Stop condition: все тесты пройдены + lint чист
→ Budget: max 10 итераций
```

### 2.2 Specialist Layer (subagents/skills)

Specialists (subagents/skills) — это узкоспециализированные компоненты, которые делают конкретные подзадачи.

Используются когда:
- Есть независимые подзадачи, которые можно выполнять параллельно.
- Нужен разный контекст или роль (researcher vs coder).
- Нужно изолировать исследование от модификаций.

**Пример разделения на subagents:**

```
Задача: "Рефакторинг модуля авторизации"

Subagent 1 (Researcher): читает код, строит карту зависимостей
Subagent 2 (Planner): на основе карты предлагает план рефакторинга
Subagent 3 (Coder): вносит изменения согласно плану
Subagent 4 (Tester): запускает тесты, проверяет regression
```

### 2.3 Tool Layer

Tool layer предоставляет агенту возможность взаимодействовать с внешним миром: filesystem, shell, browser, API/MCP.

**Иерархия инструментов:**

```
┌─────────────────────────────────┐
│    Специализированные Tools     │
│   (fetchInbox, searchEmails)    │
│  (высокий уровень абстракции)   │
└─────────────────┬───────────────┘
                  │
┌─────────────────▼───────────────┐
│      Код (Python, JavaScript)   │
│   (точная, комбинируемая логика)│
└─────────────────┬───────────────┘
                  │
┌─────────────────▼───────────────┐
│   Bash команды (универсально)   │
│  (гибкость, доступ к системе)   │
└─────────────────────────────────┘
```

Правила проектирования инструментов:
- Инструменты должны быть **ограничены по правам** (принцип минимальных привилегий).
- **Предсказуемы** по входам/выходам (JSON Schema для параметров).
- **Журналируемы** — каждый вызов должен оставлять след для debugging.
- Один инструмент решает **одну конкретную задачу**.

### 2.4 Verification Layer

Verification layer — это то, что отличает агента от «генератора предположений». Без верификации любой результат — гипотеза.

**Три уровня верификации:**

1. **Статические проверки** — lint, type checking, format validation. Быстро, дёшево, ловит синтаксические ошибки.
2. **Runtime-проверка** — запуск тестов, выполнение сценария, проверка HTTP-ответов. Ловит логические ошибки.
3. **Policy/security-проверка** — проверка перед merge/release: нет secrets в коде, permissions корректны, нет breaking changes.

**Методы верификации:**

- **Rules-based** — проверка по определённым правилам (lint, type check)
- **Visual feedback** — скриншоты и визуальная проверка (для UI-задач)
- **LLM-as-judge** — использование другого LLM для оценки результата
- **Test execution** — запуск unit/integration тестов

### 2.5 Memory/Context Layer

Memory/context layer управляет рабочей памятью агента через compaction и selective retention.

**Что обязательно хранить:**
- Решения и их обоснования (rationale).
- Ограничения и запреты (constraints).
- Незакрытые вопросы и следующие шаги.
- Контекст ошибок — что пробовали и почему не сработало.

**Что можно удалять:**
- Промежуточные результаты tool calls после их обработки.
- Детали exploration, если зафиксировано итоговое решение.
- Устаревшие данные, заменённые свежими.

## 3. Операционная модель: Plan → Execute → Verify → Adapt

Основной паттерн работы LLM-агента известен как **Agent Loop** — цикличный процесс, который повторяется до достижения цели:

```
        ┌──────────────────────┐
        │      Plan            │
        │  (постановка цели)   │
        └──────────────┬───────┘
                       │
                       ▼
        ┌──────────────────────┐
        │      Execute         │
        │  (выполнение)        │
        └──────────────┬───────┘
                       │
                       ▼
        ┌──────────────────────┐
        │      Verify          │
        │  (проверка)          │
        └──────────────┬───────┘
                       │
                       ▼
        ┌──────────────────────┐
        │      Adapt / Done?   │
        └──────────┬───────┬───┘
                   │       │
              DONE │       │ ADAPT
                   ▼       ▼
                 STOP   REPEAT
```

### 3.1 Plan

На этапе планирования агент формирует структурированный план действий. Обязательные поля:

- **`goal`** — что считаем успехом (конкретный, проверяемый критерий).
- **`constraints`** — что нельзя делать (не менять production DB, не удалять файлы).
- **`artifacts`** — что должно появиться (файлы, коммиты, отчёты).
- **`stop conditions`** — когда считаем задачу завершённой.

**Пример плана:**

```
goal: "Endpoint GET /users возвращает список пользователей"
constraints: "Не менять существующие endpoints, не менять схему БД"
artifacts: ["route handler", "unit tests", "API doc update"]
stop_conditions: "Все тесты пройдены, lint чист, endpoint отвечает 200"
```

### 3.2 Execute

Правила выполнения:
- Один шаг = одна проверяемая гипотеза. Не пытаться сделать всё сразу.
- Не смешивать массовый рефакторинг и discovery в одном шаге.
- Каждое действие должно иметь след (лог/дифф/артефакт).

**Пример процесса принятия решения:**

```
Агент видит задачу: "Обработай CSV файл и создай отчет"

1. Анализ контекста:
   - Размер файла: 50MB (слишком большой для прямого чтения)
   - Формат: CSV с заголовками
   - Требуемый формат вывода: HTML отчет

2. Выбор действия:
   - Прочитать заголовки файла (head)
   - Запустить скрипт на Python для обработки
   - Создать HTML отчет

3. Один шаг за раз:
   - Шаг 1: прочитать структуру → проверить
   - Шаг 2: написать скрипт → проверить
   - Шаг 3: сгенерировать отчёт → проверить
```

### 3.3 Verify

Проверка — не только синтаксиса. Три вопроса верификации:

1. **«Запускается ли?»** — базовая проверка работоспособности (компиляция, запуск).
2. **«Делает ли целевое поведение?»** — функциональная проверка (тесты, ручной прогон).
3. **«Не сломало ли соседние сценарии?»** — regression-проверка (полный test suite).

Если агент не может ответить «да» на все три — задача не выполнена.

### 3.4 Adapt

Если проверка не прошла, цикл не просто повторяется — он адаптируется:

- **Фиксируем причину** сбоя (не «что-то не работает», а конкретная диагностика).
- **Корректируем план** на основе новой информации.
- **Повторяем ограниченно** — max turns / budget / time limit.

**Условия завершения:**
- Успешное выполнение всех done criteria.
- Достижение максимального количества итераций (budget exceeded).
- Критическая ошибка, требующая вмешательства пользователя.

## 4. Антипаттерны

### 4.1 Tool Sprawl

Подключили 20 тулов, но без политики доступа. Агент начинает вызывать инструменты случайным образом, запросы к API множатся, стоимость растёт.

**Решение:** ограничить набор инструментов до минимально необходимого для задачи. Использовать принцип «меньше инструментов — лучше выбор».

### 4.2 No Stop Rule

Агент бесконечно «улучшает» результат. Без явного stop condition он может уйти в цикл рефакторинга, оптимизации или переписывания.

**Решение:** всегда задавать `max_turns`, `max_cost`, `done_criteria`. Остановка — не ошибка, а часть дизайна.

### 4.3 No Runtime Verification

Патч есть, но работающего результата нет. Агент написал код, сообщил «готово», но никто не проверил, что код запускается и делает то, что нужно.

**Решение:** фаза Verify обязательна. Никакой результат не считается готовым без runtime-проверки.

### 4.4 Context Pollution

В одной сессии и исследования, и рискованные правки. Агент читает 50 файлов для discovery, контекст переполняется, и при переходе к modification начинаются ошибки.

**Решение:** разделять discovery и implementation. Использовать subagents с отдельным контекстом для каждой фазы.

### 4.5 No Human Gate

Автослияние изменений без проверки критичных зон. Агент мержит PR в main без human review.

**Решение:** для критичных действий (merge в main, deploy в production, удаление данных) всегда требовать human approval.

### 4.6 «Сделал фикс» без проверки

Агент сообщает, что исправил баг, но не запускает тесты и не проверяет, что баг действительно исправлен.

**Решение:** любой фикс должен завершаться запуском теста, который воспроизводил баг, и подтверждением, что тест теперь проходит.

### 4.7 Смешивание исследования и модификации

В одном контексте агент сначала исследует кодовую базу (читает десятки файлов), потом сразу начинает вносить изменения. Контекст перегружен, агент теряет фокус.

**Решение:** разгрузка контекста через compaction между фазами. Или использование отдельных subagents для исследования и модификации.

## 5. Production-правила

1. **Guardrails**: любое действие с риском должно иметь guardrail (hook/policy). Запись в БД — через транзакцию. Деплой — через staging. Merge — через review.
2. **Verification**: любой фикс должен завершаться проверкой (не "should work"). Результат — это evidence, а не заявление.
3. **Decomposition**: большие задачи дробятся через subagents. Одна задача > 30 минут — сигнал к декомпозиции.
4. **Context management**: контекст должен управляться (compaction/editing), а не «надеяться, что хватит окна».
5. **Budget control**: всегда задавать лимиты итераций, токенов, времени. Бесконечный цикл — это баг, не feature.

---

## 6. Минимальный production-чеклист

- [ ] Для задачи задано `done criteria` (конкретное, проверяемое).
- [ ] Инструменты ограничены и прозрачны (принцип минимальных привилегий).
- [ ] Есть верификация до объявления «готово» (тесты, lint, runtime check).
- [ ] Есть лимиты итераций/стоимости (max_turns, max_cost).
- [ ] Есть журнал принятых решений (для debugging и postmortem).
- [ ] Критичные действия требуют human approval.
- [ ] Discovery и modification разделены или изолированы.
- [ ] Контекст управляется (compaction triggers настроены).

## 7. Agent Loop на примере: добавление feature в бота

Рассмотрим полный цикл агента на конкретном примере.

**Задача:** добавить команду `/stats` в Telegram-бота, которая показывает статистику пользователя.

### Iteration 1: Plan

```
goal: "Команда /stats возвращает количество сообщений пользователя за 7 дней"
constraints: "Не менять существующие команды, использовать текущую БД"
artifacts: ["handler для /stats", "SQL query", "unit test"]
stop_conditions: "Тест проходит, бот отвечает на /stats корректно"
```

### Iteration 2: Execute → Verify → Adapt

- Execute: создан handler, написан SQL query.
- Verify: запуск тестов → один тест падает (SQL ошибка в GROUP BY).
- Adapt: зафиксирована причина → исправлен SQL → повторный запуск.

### Iteration 3: Execute → Verify → Done

- Execute: исправленный SQL, обновлённый handler.
- Verify: все тесты проходят, lint чист, бот отвечает корректно.
- Done: все done criteria выполнены → остановка.

## 8. Практическое упражнение

Возьми реальный кейс (например, «добавить feature в бота») и оформи:

1. **Spec** (1 страница) — что, зачем, для кого, acceptance criteria.
2. **Execution plan** (до 10 шагов) — конкретные действия с ожидаемыми артефактами.
3. **Verification plan** (что и как проверяем) — для каждого шага: какие тесты, какие проверки.
4. **Rollback plan** (если что-то пошло не так) — как откатить каждый шаг.
5. **Postmortem** (что улучшить в следующем цикле) — что сработало, что нет, что изменить.

Метрика успеха: новый участник команды может взять твой plan и выполнить задачу без устных пояснений.
