# 4.4. Поисковые API и бэкенды для RAG

## Введение

Архитектура системы RAG (Retrieval Augmented Generation) требует мощного и эффективного уровня поиска для извлечения релевантной информации из обширных коллекций документов. В этой секции мы рассмотрим основные облачные API поиска, специализированные бэкенды с поддержкой векторных баз данных и современные паттерны гибридного поиска, которые обеспечивают высокую точность и масштабируемость RAG-приложений.

На сегодняшний день существует несколько подходов к организации поиска в RAG:
- **Облачные файловые API** (Gemini File Search) с управляемым индексированием
- **Специализированные бэкенды** (Supabase + pgvector) с полной контролируемостью
- **Гибридные маршрутизаторы** (SQL + Vector Search) для интеллектуального выбора метода поиска
- **Автоматизированные crawler-конвейеры** (Site2RAG) для преобразования веб-сайтов в RAG-базы

## 1. Cloud File Search APIs для RAG

### 1.1 Gemini File Search API — Архитектура и возможности

Gemini File Search API предоставляет управляемое решение для индексирования и поиска документов с автоматической генерацией эмбеддингов. Это сервис "всё включено", который избавляет разработчика от необходимости управлять инфраструктурой векторной БД.

#### Основные компоненты архитектуры:

**FileSearchStore** — контейнер для хранения эмбеддингов документов:
- Создается один раз и используется многократно
- Имена хранилищ глобально уникальны
- Данные сохраняются неограниченно долго до явного удаления
- Размер файла: максимум 100 МБ на документ

**Document Lifecycle:**

```
1. Загрузка файла (до 100 МБ)
        ↓
2. Автоматическое chunking
   (система выбирает оптимальный размер)
        ↓
3. Embedding generation
   (модель gemini-embedding-001)
        ↓
4. Индексирование в FileSearchStore
        ↓
5. Готово к retrieval запросам
```

#### Типичный workflow Gemini File Search

**Этап 1: Подготовка** — создать именованное хранилище (FileSearchStore) для документов.

**Этап 2: Загрузка** — загрузить документы (PDF, DOCX, TXT, и т.д.) в хранилище. Система автоматически обработает файлы.

**Этап 3: Автоматическая обработка** — Gemini выбирает оптимальный размер chunking, генерирует embeddings, индексирует в FileSearchStore.

**Этап 4: Использование в LLM** — в prompt добавить инструмент `file_search`. Когда LLM вызывает этот инструмент, API автоматически retrieves релевантные отрывки из документов.

**Этап 5: Генерация ответа** — LLM использует найденный контекст для ответа на запрос пользователя.

**Преимущества Gemini File Search:**
- ✅ No DevOps: полностью управляемое решение
- ✅ Автоматическая оптимизация: chunking и embedding выбирает система
- ✅ Простое API: загрузи файлы, всё работает
- ✅ Встроенная интеграция с LLM: retrieval происходит автоматически
- ✅ Масштабируемость: Google управляет инфраструктурой

**Ограничения:**
- ❌ Максимум 100 МБ на файл (большие документы нужно разбивать)
- ❌ Черный ящик: нет доступа к embeddings или chunking params
- ❌ Меньше контроля: нельзя выбрать embedding модель
- ❌ Нет гибридного поиска: только semantic search
- ❌ Стоимость: плата за хранение + за каждый retrieval запрос

#### Когда использовать Gemini File Search

| Сценарий | Использовать? | Причина |
|----------|---------------|---------|
| Быстрое прототипирование | ✅ | Минимум настроек, быстро запустить |
| Файлы < 100 МБ каждый | ✅ | Естественный лимит сервиса |
| Стартап с ограниченным DevOps | ✅ | No инфраструктуры для управления |
| Нужен гибридный поиск | ❌ | Только semantic search |
| Очень большие файлы (> 100 МБ) | ❌ | Нужно разбивать или другой подход |
| Критична стоимость | ❌ | Может быть дороже self-hosted решений |
| Нужен полный контроль | ❌ | Все параметры выбирает система |

## 2. Специализированные бэкенды для RAG

### 2.1 Supabase как полнофункциональная база данных для RAG

Supabase — это PostgreSQL сервис с встроенной поддержкой pgvector. Это один из лучших выборов для production RAG систем.

#### Архитектура Supabase + pgvector:

```
Supabase PostgreSQL Database
    ├── Documents таблица
    │   ├── id (UUID)
    │   ├── content (TEXT)
    │   ├── embedding (VECTOR[1536])
    │   ├── metadata (JSONB)
    │   └── created_at (TIMESTAMP)
    │
    ├── Vector indices (для быстрого поиска)
    │   └── IVFFlat или HNSW индекс на embedding
    │
    ├── Full-text search (Postgres встроено)
    │   └── BM25 или похожее для keyword поиска
    │
    └── Гибридные queries
        └── SELECT ... WHERE embedding <=> query_vector
            AND metadata->'source' = 'policy.pdf'
```

#### Гибридный поиск в Supabase (концептуально):

```
Запрос: "Когда начинается доставка?"
        ↓
Embedding запроса: [0.23, -0.45, 0.89, ...]
        ↓
SQL query:
SELECT id, content,
       1 - (embedding <=> query_embedding) as similarity
FROM documents
WHERE metadata->>'source' NOT LIKE '%draft%'  -- исключи черновики
ORDER BY similarity DESC
LIMIT 10;
        ↓
Результаты с метаданными и similarity scores
```

#### Преимущества Supabase для RAG

| Преимущество | Описание |
|--------------|---------|
| **Полная мощь SQL** | JOINs, GROUP BY, фильтрация, агрегация |
| **Гибридный поиск** | Комбинирование vector + keyword в одном query |
| **Метаданные фильтрация** | Сложные условия на JSON метаданные |
| **Дешевле облачных** | PostgreSQL дешевле Pinecone на масштабе |
| **Полный контроль** | Своя инфраструктура, свои правила |
| **Надежность** | PostgreSQL — battle-tested технология |

#### Недостатки

| Недостаток | Описание |
|-----------|----------|
| **DevOps требуется** | Нужно управлять базой данных |
| **Масштабирование** | Сложнее чем облачные решения |
| **Производительность** | Медленнее специализированных vector БД при большом scale |
| **Конфигурирование** | Нужно выбрать индекс, параметры, оптимизацию |

### 2.2 Elasticsearch для гибридного поиска

Elasticsearch традиционно известен для full-text поиска, но последние версии поддерживают vector поиск.

#### Гибридный поиск в Elasticsearch

Elasticsearch позволяет комбинировать vector и keyword поиск в одном запросе:

**Архитектура:**

```
Поисковый запрос
    ↓
    ├─→ Vector component (KNN поиск)
    │   Находит K ближайших embeddings
    │
    └─→ Keyword component (BM25 поиск)
        Находит документы с matching терминами
        ↓
Elasticsearch объединяет результаты
    ↓
Возвращает top-K по combined score
```

**Как работает:**
1. Оба компонента выполняются параллельно
2. Результаты объединяются (пересечение или взвешенная сумма)
3. Возвращаются документы с наивысшими combined scores

Это дает лучшие результаты чем один из методов отдельно.

#### Когда использовать Elasticsearch

- Уже используете Elasticsearch для full-text поиска
- Нужен production-grade гибридный поиск
- Готовы управлять кластером

#### Когда использовать что-то другое

- Chroma: простая локальная разработка
- Pinecone: не хотите DevOps
- Supabase: нужна полная SQL мощь

## 3. Маршрутизация запросов: когда использовать какой поиск

### 3.1 Интеллектуальная маршрутизация

Не все запросы требуют одного типа поиска. Система может маршрутизировать запросы на основе их типа.

**Архитектура маршрутизации:**

```
Пользовательский запрос
        ↓
Классификация запроса (LLM или правила)
        ├─→ Структурированный запрос? (числа, даты, ID)
        │   └─→ SQL маршрут
        │       (SELECT * FROM contracts WHERE date = 2024)
        │
        ├─→ Семантический запрос? (смыслы, описания)
        │   └─→ Vector search маршрут
        │       (найти похожие документы)
        │
        └─→ Гибридный запрос? (оба компонента)
            └─→ Hybrid search маршрут
                (vector + SQL фильтр)
                ↓
        Выполнить соответствующий запрос
                ↓
        Объединить результаты если нужно
                ↓
        LLM генерирует ответ
```

### 3.2 Примеры маршрутизации

| Запрос | Классификация | Маршрут |
|--------|---------------|---------|
| "Какой бюджет в 2024?" | Структурированный | SQL |
| "Найти документы похожие на этот" | Семантический | Vector |
| "Контракты в Москве на 2024" | Гибридный | Hybrid |
| "Кто такой Иван Петров?" | ID поиск | SQL (точное совпадение) |
| "Описание политики возврата" | Семантический | Vector |

### 3.3 Реализация Query Router

Router анализирует входящий запрос и решает, какой тип поиска использовать:

**Подход 1: Правила (быстро)**
- Если запрос содержит числа и даты → SQL маршрут
- Если запрос описательный ("похожие", "описание") → Vector маршрут
- Если оба типа компонентов → Hybrid маршрут

**Подход 2: LLM классификация (точнее)**
- Использовать быструю модель для классификации типа запроса
- "Структурированный, семантический или гибридный?"
- Медленнее чем правила, но обрабатывает сложные случаи

**Процесс выполнения:**
```
Пользовательский запрос
    ↓
Классификация (правила или LLM)
    ↓
    ├─→ SQL маршрут (структурированные фильтры)
    ├─→ Vector маршрут (семантический поиск)
    └─→ Hybrid маршрут (оба)
    ↓
Выполнить соответствующий поиск
    ↓
Объединить результаты если нужно
    ↓
LLM использует результаты для ответа
```

**Trade-offs:**
- Простые правила: быстро, может ошибиться на сложных запросах
- LLM классификация: точнее, но добавляет latency

## 4. Web-based RAG и автоматическое обновление

### 4.1 Проблемы web-based RAG

При использовании веб-содержимого в RAG есть специальные требования:

**Вызовы:**
- Динамический контент (JavaScript рендеринг)
- Структурированный noise (меню, реклама, footer)
- Изменяющиеся URL
- Rate limiting и блокировка botов
- SSL/TLS сертификаты

### 4.2 Crawling и индексирование веб-контента

**Pipeline для веб-RAG:**

```
1. Обнаружение (Discovery)
   - Начать с seed URL (например, homepage)
   - Найти все ссылки на сайте
   - Создать граф связей

2. Загрузка (Fetching)
   - Уважить robots.txt
   - Использовать правильный User-Agent
   - Добавить задержки между запросами (rate limit)

3. Парсинг (Parsing)
   - Удалить noise (скрипты, стили, меню)
   - Конвертировать HTML → Markdown
   - Сохранить структуру (заголовки, списки)

4. Нормализация (Normalization)
   - Абсолютные URL (относительные → полные)
   - Дедупликация (одна страница != несколько URL)
   - Очистка от параметров отслеживания (utm, session id)

5. Embedding & Indexing
   - Chunking страниц
   - Embedding generation
   - Индексирование в vector store

6. Мониторинг свежести (Staleness)
   - Периодическая переиндексация (если контент изменился)
   - Отслеживание modified date
```

### 4.3 Site2RAG паттерн (концептуально)

Site2RAG — это идея преобразовать весь веб-сайт в RAG-индекс:

```
Вебсайт (например, документация)
        ↓
Автоматический crawling всех страниц
        ↓
Преобразование в Markdown
        ↓
Удаление дубликатов и noise
        ↓
Chunking и embedding
        ↓
Индексирование в vector store
        ↓
Готовая RAG база для этого сайта
```

**Примеры использования:**
- Документация продукта → RAG для customer support
- Научный сайт → RAG для исследования
- Корпоративный wiki → RAG для сотрудников

**Требования к Site2RAG:**
1. Уважайте robots.txt
2. Не перегружайте сервер (rate limit)
3. Имейте лицензию на использование контента
4. Периодически обновляйте индекс (документация меняется)

## 5. Code RAG: особенности для исходного кода

### 5.1 Почему код требует специального подхода

Исходный код — это специальный случай для RAG из-за:
- Структурной важности (иерархия импортов, зависимости)
- Синтаксической значимости (язык программирования)
- Кросс-ссылок (функция A вызывает функцию B)

### 5.2 Стратегия Code RAG

Исходный код требует специального chunking, потому что структура важнее чем просто размер:

**Правильный подход:**

**Шаг 1: Парсинг** — использовать AST (Abstract Syntax Tree) парсер для структурирования кода.

**Шаг 2: Извлечение entity** — выделить функции, классы, методы, модули как отдельные блоки. Каждая entity получает ID и контекст.

**Шаг 3: Чанкирование** — один чанк = одна function/class/method с:
- Сигнатурой (параметры, возвращаемый тип)
- Docstring (если есть)
- Полной реализацией
- Импортами, которые нужны
- Metadata о сложности, размере, зависимостях

**Шаг 4: Metadata** — сохранить информацию о зависимостях (какие функции вызывает, какие классы использует).

**Преимущества структурного chunking для кода:**
- Не разрывает функцию посередине
- Embedding видит полный контекст функции
- Можно отследить зависимости между функциями
- Retrieve находит полные, работающие блоки кода

### 5.3 Code Graph RAG

Продвинутый подход: использовать граф зависимостей кода.

**Архитектура:**

```
Исходный код
        ↓
Build граф зависимостей
    function_a → function_b → function_c
    class_x → class_y
        ↓
Chunking с контекстом:
    Чанк function_a включает:
    - саму function_a
    - код function_b (которую она вызывает)
    - документацию класса_x (который она использует)
        ↓
Embedding с awareness графа
        ↓
Поиск с contextualized results
```

**Преимущества Code Graph RAG:**
- Найденная функция содержит контекст вызываемых функций
- Лучше понимание зависимостей
- Меньше шансов на incomplete ответ

## 6. SQL + Vector гибридная маршрутизация

### 6.1 Паттерн SQL Router

SQL Router решает задачу: как выбрать между чистым SQL и vector search?

**Сценарий решения:**

```
Запрос: "Найти всех клиентов из Москвы, похожих на 'Tech Startup'"
        ↓
Разбор на компоненты:
    1. Структурированное: city = 'Moscow'
    2. Семантическое: похожий на "Tech Startup"
        ↓
Выполнить оба поиска:
    SQL: SELECT * FROM clients WHERE city = 'Moscow'
    Vector: найти embeddings похожие на "Tech Startup"
        ↓
Пересечение результатов:
    clients ∩ similar_companies = [company1, company3, company5]
        ↓
Возвращение объединенных результатов
```

### 6.2 Выбор между SQL и Vector поиском

Когда LLM обрабатывает запрос, Router решает:

```
Запрос содержит точные критерии?
(конкретная дата, город, ID, категория)
    ├─→ Да → SQL маршрут (быстро и точно)
    └─→ Нет → Vector маршрут (семантический поиск)

Оптимально: если оба компонента → Hybrid маршрут
```

**Примеры:**
- "Все документы с 2024 года" → SQL (точный фильтр на дату)
- "Какие-то про здоровье" → Vector (только семантика)
- "Документы о здоровье из 2024" → Hybrid (семантика + фильтр)

### 6.3 Архитектура SQL Router

```
Пользовательский запрос
    ↓
LLM классифицирует тип
    ↓
    ├─→ Pure SQL:
    │   Генерировать WHERE условия
    │   Выполнить select
    │   Вернуть результаты
    │
    ├─→ Pure Vector:
    │   Embedding запроса
    │   Семантический поиск
    │   Вернуть результаты
    │
    └─→ Hybrid:
        SQL: найти документы по фильтрам
        Vector: найти похожие embeddings
        Merge: пересечение или взвешенная сумма
        ↓
LLM использует merged результаты для ответа
```

**Ключ к успеху:** классификация должна быть быстрой и точной. Используйте простые правила если возможно, LLM классификацию только для сложных случаев.

## 7. Production considerations

### 7.1 Мониторинг и логирование

**Что логировать:**

```
Для каждого retrieval запроса:
- Query text
- Retrieval type (SQL, Vector, Hybrid)
- Execution time
- Number of results
- Top-1 similarity/relevance score
- User satisfaction (если собираете feedback)
```

**Метрики для мониторинга:**

| Метрика | Целевое значение | Что говорит |
|---------|------------------|-----------|
| Avg retrieval latency | < 200ms | Производительность поиска |
| Recall@10 | > 0.7 | Находит релевантные результаты |
| Precision@3 | > 0.8 | Первые результаты релевантны |
| User feedback CTR | > 0.3 | Пользователи довольны результатами |
| Empty result rate | < 0.05 | Поиск находит что-то полезное |

### 7.2 Кэширование для производительности

**Что кэшировать:**

```
1. Часто запрашиваемые embeddings
   - Cache query embeddings (если тот же запрос повторяется)
   - Время жизни: часы-дни

2. Часто используемые chunки
   - Cache popular documents (берите top-100 по frequency)
   - Уменьшает latency retrieval

3. SQL результаты
   - Кэш результаты фильтрации по категориям
   - Например: "все документы из Москвы" может меняться редко
```

### 7.3 Обновление индексов

**Стратегия обновления:**

```
Типы обновлений:

1. Инкрементальное (Real-time)
   - Новый документ загружен → сразу добавить в индекс
   - Требует live indexing capability

2. Батчевое (Periodic)
   - Собрать новые документы за день/неделю
   - Ночью: batch embedding + indexing
   - Надежнее, но медленнее по обновлению

3. Полная переиндексация (Rare)
   - Если изменилась embedding модель
   - Если обнаружена проблема с индексом
   - Занимает время, может быть offline
```

## 8. Практические сценарии и выбор архитектуры

### Сценарий 1: Стартап с документацией

**Требования:**
- 500 PDF документов
- Быстрое протоtiping
- Ограниченный бюджет

**Рекомендуемая архитектура:**

## 9. Практические задачи

### Задача 1: Выбрать backend для RAG

**Дано:**
- 10,000 документов
- Нужна фильтрация по metadata (дата, автор, категория)
- Бюджет есть
- Нужна масштабируемость на 100K документов в будущем

**Решение (анализ):**

1. Объем: 10K сейчас → 100K в будущем
2. Гибридный поиск: да (нужна фильтрация)
3. Бюджет: есть

Выбор: **Supabase + pgvector**
- Поддерживает гибридный поиск
- Масштабируется от 10K до 100K+
- Полный контроль над всем
- Дешевле облачных на масштабе

### Задача 2: Реализовать SQL Router

**Сценарий:** RAG для каталога товаров. Запрос может быть:
- "Найти товары из категории 'электроника'" (SQL)
- "Найти похожие на iPhone товары" (Vector)
- "Электроника в диапазоне цена 10K-50K похожие на смартфон" (Hybrid)

**Архитектурное решение (концептуально):**

```
1. Query Classifier (LLM или правила):
   - "категория" + "электроника" → SQL компонент
   - "похожие" → Vector компонент
   - Оба → Hybrid

2. Execution:
   - SQL: SELECT * FROM products WHERE category = 'electronics'
   - Vector: embedding("смартфон") → search similar
   - Hybrid: SQL результаты ∩ Vector результаты

3. Ranking:
   - Re-rank пересечение по relevance score
   - Return top 10
```

### Задача 3: Оптимизировать retrieval latency

**Дано:**
- Текущее время retrieval: 1200ms (слишком медленно)
- Нужно: < 200ms

**Анализ bottlenecks (концептуально):**

```
1200ms = ?

Профилирование:
- Query embedding: 150ms
- Vector search: 800ms (слишком долго!)
- Re-ranking: 200ms
- Other: 50ms

Проблема: Vector search медленный
Причины:
a) Индекс не оптимальный (linear search?)
b) Слишком много документов в поиске
c) Сеть latency

Решения:
a) Добавить HNSW индекс (если нет)
b) Добавить metadata фильтрация (сужает поиск)
c) Кэшировать embeddings popular документов
d) Батчировать запросы если возможно
```

## 10. Выводы

Выбор поисковой архитектуры критичен для RAG успеха:

1. **Не one-size-fits-all:** выбор зависит от вашего сценария
2. **Гибридный поиск** часто лучше чем один метод
3. **Начните с простого** (Gemini File Search или Chroma)
4. **Итерируйте** по мере роста требований
5. **Мониторьте** latency, recall, precision
6. **Optimize** когда система растет (кэширование, индексирование)
