## Основы векторного поиска

### Что такое векторный поиск?

Векторный поиск — это метод поиска информации, основанный на семантическом подобии содержимого, а не на точном совпадении ключевых слов. В отличие от классического полнотекстового поиска (BM25), который работает с точным или приблизительным совпадением текста, векторный поиск преобразует документы и запросы в высокомерные векторы (embeddings) и находит семантически похожие элементы.

**Основная идея:**

```
Текст документа               Запрос пользователя
        ↓                              ↓
   Embedding модель            Embedding модель
        ↓                              ↓
Вектор: [0.23, -0.45, 0.89, ...] Вектор: [0.21, -0.44, 0.88, ...]
        ↓                              ↓
              ↓────────┬────────↓
                    Сравнение
                  (косинусное
                 подобие, L2
                  расстояние)
                       ↓
                  Результат: 0.97
                  (очень похож!)
```

### Преимущества векторного поиска

| Преимущество | Описание |
|--------------|----------|
| **Семантическое понимание** | Находит смысл, а не ключевые слова |
| **Синонимы и вариации** | "быстро" найдет "на высокой скорости" |
| **Многоязычный поиск** | Вектор одинаковый для разных языков |
| **Скрытый контекст** | Находит по подразумеваемому смыслу |
| **Нечеткие совпадения** | Работает даже если текст не идеален |

### Ограничения векторного поиска

| Ограничение | Описание | Решение |
|-------------|---------|---------|
| **Кодирование bias** | Embedding модель имеет собственные bias | Выберите модель для вашей области |
| **Количественная информация** | Хуже работает с числами и ID | Используйте гибридный поиск |
| **Редкие домены** | На новых темах embedding слабый | Fine-tune embedding модель |
| **Стоимость** | Нужны вычисления для embeddings | Кэшируйте embeddings |

## Embeddings и их свойства

### Что такое embeddings?

Embeddings — это численные представления текста в многомерном пространстве, где семантически похожие тексты находятся близко друг к другу.

```
Dimensionality: 1536 (для OpenAI text-embedding-3-small)

Embedding вектор:
[0.0023, -0.0089, 0.0156, ..., -0.0034] (всего 1536 чисел)
```

### Почему высокая размерность?

- **1536 измерений** может кодировать огромное количество смыслов
- Каждое измерение представляет аспект смысла (понятие, связь, контекст)
- Совместное подпространство позволяет находить отношения между концепциями

**Аналогия:**
- 1D координата: точка на линии (очень мало информации)
- 2D координаты: точка на плоскости (карта)
- 1536D координаты: позиция в 1536-мерном пространстве смысла

### Современные модели embeddings

| Модель | Размерность | Поставщик | Скорость | Мультиязычность | Цена |
|--------|-------------|-----------|----------|-----------------|------|
| **text-embedding-3-small** | 1536 | OpenAI | Быстро | Да | Дешево |
| **text-embedding-3-large** | 3072 | OpenAI | Среднее | Да | Дорого |
| **Voyage AI Embed** | 1024 | Voyage AI | Быстро | Да | Среднее |
| **Cohere embed-english** | 768 | Cohere | Быстро | Нет | Среднее |
| **Multilingual-E5-large** | 1024 | HuggingFace | Медленно | 100+ языков | Бесплатно |

### Выбор модели embedding

**Основные критерии:**

1. **Размерность** (trade-off между качеством и стоимостью)
   - 768-1024: хорошее соотношение цена/качество
   - 1536+: максимальное качество, дорого
   - < 768: экономия, может быть недостаточно

2. **Мультиязычность**
   - Если работаете на одном языке: специализированная модель лучше
   - Если несколько языков: выбирайте мультиязычную
   - Cross-lingual поиск требует мультиязычной модели

3. **Специализация**
   - Для кода: есть специальные модели (code-search-ada)
   - Для научных текстов: есть специализированные embedding'и
   - Для общего текста: text-embedding-3-small хороший выбор

4. **Стоимость vs Скорость**
   - OpenAI: дорого, но лучшее качество, облачное
   - HuggingFace: бесплатно, но медленнее, нужны вычисления
   - Компромисс: использовать оба подхода (облачное для важного, локальное для остального)

**Рекомендация для начала:**
```
Используйте: OpenAI text-embedding-3-small
Причина: хороший баланс качество/цена/скорость
На дороге: экспериментируйте с другими моделями
```

## Типы поиска в RAG

### 1. Semantic Search (Векторный поиск)

Основной тип поиска в RAG. Находит семантически похожие документы.

**Как работает:**
1. Пользовательский запрос конвертируется в embedding
2. Вычисляется расстояние до всех документов в vector store
3. Возвращаются K ближайших соседей

**Метрики расстояния:**

| Метрика | Формула | Смысл | Использование |
|---------|---------|-------|---------------|
| **Cosine similarity** | cos(θ) между векторами | Угол между векторами | Обычно хороший выбор |
| **Euclidean distance** | √(Σ(x-y)²) | Физическое расстояние | Когда важна абсолютная величина |
| **Dot product** | Σ(x·y) | Скалярное произведение | Нормализованные vectors |

**Преимущества semantic search:**
- Находит похожие по смыслу тексты
- Работает с синонимами и перефразировками
- Не требует точного совпадения ключевых слов

**Недостатки:**
- Зависит от качества embedding модели
- Медленнее чем keyword search на больших объемах
- Требует вычислений для embedding запроса

### 2. Keyword Search (BM25)

Полнотекстовый поиск по точному совпадению слов.

**Как работает:**
```
Запрос: "президент России"
         ↓
Найти документы с словом "президент" И/ИЛИ "России"
         ↓
Ранжировать по frequency и importance
         ↓
Вернуть топ-K результатов
```

**BM25 алгоритм:**
- Давит значимость слов в документе (не просто count)
- Учитывает длину документа (longer docs не всегда лучше)
- Стандартный алгоритм для полнотекстового поиска

**Преимущества:**
- Очень быстро (простой индекс)
- Хорошо для точных совпадений (имена, ID, специальные термины)
- Нет необходимости в embedding

**Недостатки:**
- Не работает с синонимами ("быстро" не найдет "быстрый")
- Не понимает контекст
- Может пропустить семантически похожие документы

### 3. Hybrid Search

Комбинирование semantic и keyword поиска.

**Как работает:**

```
Пользовательский запрос
        ↓
        ├─→ Semantic Search (vector подобие)
        │         ↓
        │   Top-K семантических результатов
        │
        ├─→ Keyword Search (BM25)
        │         ↓
        │   Top-K по ключевым словам
        │
        └─→ Re-ranking и объединение
             (voting, weighted score, или LLM re-ranking)
                ↓
          Final Top-K результатов
```

**Стратегии объединения:**

1. **Weighted score** (простое):
   ```
   final_score = 0.6 * semantic_score + 0.4 * keyword_score
   ```

2. **RRF (Reciprocal Rank Fusion)**:
   ```
   score = Σ(1 / (rank_i + 60))
   Объединяет rankings, нейтрально к масштабам
   ```

3. **LLM re-ranking** (продвинутое):
   ```
   Попросить LLM переранжировать результаты
   (медленнее, но лучше качество)
   ```

**Когда hybrid ищет лучше:**

| Запрос | Почему hybrid выиграет |
|--------|----------------------|
| "найти контрагента Иван Иванов" | keyword: точное имя, semantic: контекст |
| "что-то о COVID" | semantic: поймет варианты, keyword: найдет точное |
| "рекомендации по здоровью" | semantic: поймет смысл, keyword: точные термины |

## Выбор и сравнение векторных БД

### Сравнительная таблица

| База | Тип | Масштаб | Гибридный поиск | Управление | Цена | Используйте если |
|------|-----|---------|-----------------|-----------|------|-----------------|
| **Pinecone** | Облачная | Огромный | Нет | Управляемая | Средняя | Простая интеграция, не хотите DevOps |
| **Supabase pgvector** | PostgreSQL | Большой | Да (SQL+vector) | Полный контроль | Дешевая | Нужен SQL, полный контроль, гибридный поиск |
| **Weaviate** | Облачная/self-hosted | Большой | Да | Гибкая | Средняя | Сложный filtering, гибридный поиск |
| **Milvus** | Self-hosted | Огромный | Нет | Полный контроль | Дешевая | Распределенные системы, огромные объемы |
| **Chroma** | Встраиваемая | Мало-Среднее | Нет | Простая | Бесплатная | Локальная разработка, быстрое прототипирование |
| **Qdrant** | Self-hosted/облачная | Большой | Да | Полный контроль | Дешевая | Фильтрация и гибридный поиск |

### Облачные решения

**Pinecone** — наиболее популярное:
- Полностью управляемая (no DevOps)
- Простое API
- Автоматическое масштабирование
- Высокая стоимость

**Когда выбирать Pinecone:**
- Вы не хотите управлять инфраструктурой
- Нужна надежность и масштабируемость
- Бюджет позволяет платить за управляемый сервис

### Self-hosted решения

**Supabase (PostgreSQL + pgvector)** — лучший баланс:
- Полнофункциональная база данных (SQL + vectors)
- Можно делать гибридный поиск
- Дешевле облачных
- Требует управления (но PostgreSQL легко)

**pgvector в PostgreSQL:**

PostgreSQL + pgvector позволяет хранить и искать по embeddings прямо в SQL:

Основная идея:
1. Каждый документ имеет колонку `embedding` с вектором (1536 чисел)
2. Система создает индекс для быстрого поиска (IVFFlat или HNSW)
3. Запрос: найти K ближайших по евклидовому расстоянию

**Как это работает концептуально:**
```
Таблица: documents
- id (номер документа)
- content (текст)
- embedding (вектор 1536D)
- metadata_json (JSON с source, page, date)

Индекс: IVFFlat на embedding колонке
Позволяет быстро найти K ближайших векторов

Запрос: SELECT ... WHERE distance(embedding, query_vector) < threshold
```

Преимущество: можно комбинировать с SQL условиями:
- Найти embeddings похожие на запрос AND metadata->'year' = 2024
- Это гибридный поиск в одном SQL запросе

**Qdrant** — высокооптимизированный:
- Специально для vector search (не универсальная БД)
- Отличная фильтрация
- Встроенный гибридный поиск
- Open-source

## Gemini File Search API

### Архитектура Gemini File Search

Gemini File Search предоставляет управляемое решение для RAG без управления инфраструктурой.

**Поток работы:**

```
1. Создание хранилища (FileSearchStore)
   name = "documents_2024"

2. Загрузка файлов
   upload_file("contract.pdf") → file_id_123
   upload_file("policy.pdf") → file_id_456

3. Автоматическая обработка (Gemini занимается)
   ├─ Chunking (оптимальный размер)
   ├─ Embedding (gemini-embedding-001)
   └─ Индексирование

4. Поиск при вызове LLM
   query = "Когда начинается доставка?"
   tools = [{"type": "google_search", ...}]
   (система автоматически retrieves релевантные chunks)

5. LLM использует контекст для ответа
   response = "Согласно договору, доставка начинается..."
```

### Преимущества Gemini File Search

| Преимущество | Описание |
|--------------|----------|
| **No DevOps** | Google управляет всей инфраструктурой |
| **Автоматическое chunking** | Система оптимизирует размер для контекста |
| **Встроенный embedding** | Не нужно выбирать модель |
| **Простое API** | Загрузи файлы и готово |
| **Интеграция с LLM** | Retrieval происходит автоматически в промпте |

### Ограничения Gemini File Search

| Ограничение | Описание |
|-------------|---------|
| **Размер файла** | Максимум 100 МБ на документ |
| **Черный ящик** | Нет прямого доступа к embeddings |
| **Меньше контроля** | Chunking, embedding выбирает система |
| **Стоимость** | Плата за хранение + за каждый retrieval запрос |
| **Нет гибридного поиска** | Только semantic search |

### Когда использовать Gemini File Search

**Используйте если:**
- Документы меньше 100 МБ
- Не нужен гибридный поиск
- Хотите быстро запустить RAG
- Не критична стоимость

**Используйте что-то другое если:**
- Нужны очень большие файлы
- Требуется гибридный поиск (semantic + keyword)
- Нужен полный контроль над pipeline'ом
- Стоимость критична

## Supabase + pgvector

### Установка и конфигурация (концептуально)

**Компоненты:**

```
PostgreSQL Database
    ↓
pgvector extension (для хранения векторов)
    ↓
БД с таблицами: documents, embeddings, metadata
    ↓
Индексы для быстрого поиска (IVFFlat)
    ↓
Python/JavaScript приложение для RAG
```

### Практический пример работы с pgvector

**Архитектура pgvector RAG:**

PostgreSQL таблица содержит:
- `id` — уникальный ID документа
- `content` — текст чанка
- `embedding` — вектор размерности 1536 (выходы модели embedding)
- `metadata_json` — JSON с информацией о источнике, странице, дате

**Workflow:**

**Этап 1: Загрузка** — каждый документ разбивается на чанки, для каждого генерируется embedding, чанк + embedding + metadata загружаются в таблицу.

**Этап 2: Индексирование** — PostgreSQL создает индекс на embedding колонке для быстрого поиска. Обычно используется IVFFlat или HNSW.

**Этап 3: Запрос** — пользовательский запрос конвертируется в embedding, затем выполняется поиск K ближайших соседей в vector пространстве.

**Преимущество pgvector:**
- Все в одной БД, нет отдельного vector store
- Можно комбинировать vector поиск с SQL условиями: найти similarные embeddings WHERE year = 2024
- Полная мощь SQL: JOINs, GROUP BY, агрегация на результатах
- Дешевле облачных vector БД при масштабировании

### Гибридный поиск в PostgreSQL

**Комбинирование vector + SQL:**

```
Запрос: "Контракты в Москве, подписанные в 2024"
        ↓
    Разделить на части:
    - Semantic: "контракты"
    - Структурированные фильтры: city='Moscow', year=2024
        ↓
    Vector search: найти документы похожие на "контракты"
        ↓
    SQL filter: city='Moscow' AND year=2024
        ↓
    Пересечение: documents matching both условиях
```

**Преимущества PostgreSQL подхода:**
- Полная мощь SQL: JOINs, GROUP BY, агрегация
- Изменяемые фильтры без переиндексирования embeddings
- Стоимость ниже облачных решений
- Полный контроль над данными

## Гибридный поиск

### Архитектура гибридного поиска

```
User Query: "Лучшие продукты для похудения в Москве"
     ↓
Parse запрос (опционально с LLM)
     ↓
     ├─→ Semantic component:
     │   "Лучшие продукты для похудения"
     │   → embedding → vector search
     │   → результаты: [doc1, doc2, doc3]
     │
     └─→ Structured component:
         Location = "Москва"
         → SQL filter
         → результаты: [doc2, doc3, doc5]
         ↓
  Merge результатов
  (intersection: doc2, doc3)

  → Re-rank combined results
  → Return top-K
```

### Стратегии объединения результатов

**1. Пересечение (AND)**
- Возвращает только документы найденные в обоих поисках
- Высокая precision, низкий recall

**2. Объединение (OR)**
- Возвращает все найденные документы
- Высокий recall, низкая precision

**3. Weighted combination (рекомендуется)**
```
score = w_semantic * semantic_score + w_keyword * keyword_score

Пример:
w_semantic = 0.7
w_keyword = 0.3

Обычно семантический поиск важнее в RAG
```

**4. RRF (Reciprocal Rank Fusion)**
```
Объединяет rankings из разных методов
Не зависит от абсолютных значений оценок
```

## Стратегии indexing

### Типы индексов для vector search

**Точный поиск (Exact / Brute Force):**
- Проверяет расстояние до ВСЕ векторов
- Медленно при больших объемах (O(n))
- Идеально точно

**Аппроксимирующие индексы:**

| Индекс | Идея | Скорость | Точность | Использование |
|--------|------|----------|----------|--------------|
| **HNSW** | Иерархическая навигационная структура | Очень быстро | Высокая | Рекомендуется для большинства |
| **IVFFlat** | Инверсные файлы + плоский индекс | Быстро | Средняя-Высокая | PostgreSQL pgvector |
| **LSH** | Locality-Sensitive Hashing | Быстро | Низкая | Когда нужна максимальная скорость |
| **FAISS** | Facebook AI Similarity Search | Очень быстро | Высокая | Научные работы, большие объемы |

### Выбор стратегии indexing

**На малых объемах (<100K документов):**
- Используйте HNSW или даже exact search
- Скорость не критична
- Точность важнее

**На больших объемах (>1M документов):**
- HNSW рекомендуется (лучший баланс)
- IVFFlat если в PostgreSQL
- FAISS если очень большие объемы

**На очень больших объемах (>100M документов):**
- Распределенные индексы (Milvus, Elasticsearch)
- LSH если нужна максимальная скорость
- Шардирование документов по машинам

## Масштабирование и производительность

### Метрики производительности

| Метрика | Что измеряет | Целевое значение |
|---------|-------------|-----------------|
| **Latency** | Время поиска одного запроса | < 100ms |
| **QPS** | Запросов в секунду | Зависит от нагрузки |
| **Memory** | Использование памяти на embedding | < 100MB для 100K документов |
| **Index size** | Размер индекса на диске | 3-5x размер vectors |
| **Build time** | Время создания индекса | < время жизни системы |

### Оптимизация для масштабирования

**1. Кэширование embeddings**
```
Не пересчитывайте embedding одного документа дважды
Кэшируйте: document_id → embedding
```

**2. Батчинг запросов**
```
Вместо: поиск по одному запросу
Лучше: поиск по 100 запросам за раз
Примерно в 10x быстрее
```

**3. Шардирование данных**
```
Если у вас 10M документов:
- Разделите на 10 шардов по 1M
- Каждый шард на отдельной машине
- Parallelize поиск через шарды
```

**4. Асинхронная обработка**
```
Не блокируйте embedding по одному.
Используйте очереди (Redis, Celery, Kafka) для batch processing.
```

### Пример расчета требований

**Для RAG с 1M документов, 1536-размерными embeddings:**

```
Storage requirement:
- Vector data: 1M × 1536 × 4 bytes = 6 GB
- Index overhead (HNSW): ~2-3x = 12-18 GB
- Metadata: ~1-2 GB
- Total: ~20-25 GB

Query latency (с HNSW):
- Без optimization: 200-500ms
- С кэшированием query embedding: 100-200ms
- С батчингом 100 запросов: 10-20ms per query

QPS capacity:
- На single machine: ~100-500 QPS
- На distributed (10 шардов): ~1000-5000 QPS
```

## Практические задачи

### Задача 1: Выбрать правильную vector БД

**Сценарий:** У вас есть 100K научных статей (в сумме 200 ГБ). Нужна система для RAG.

**Анализ факторов:**
1. Объем: 100K статей → средний объем, любая БД подойдет
2. Требования:
   - Гибридный поиск? (может быть статьи с автором "John Smith")
   - Фильтрация по metadata? (год публикации, область знаний)
   - Высокая доступность?
3. Бюджет: есть ли бюджет на облачный сервис?

**Решение (концептуально):**
- Если нужен гибридный поиск → Supabase pgvector
- Если нужна простота → Pinecone
- Если нужна максимальная гибкость → Qdrant
- Если разработка локально → Chroma

### Задача 2: Реализовать гибридный поиск

**Сценарий:** RAG для корпоративной базы знаний. Запрос: "бюджет на маркетинг в 2024 году для Москвы"

**Архитектура решения (концептуально):**

```
Запрос разбивается на:
1. Semantic: "бюджет на маркетинг"
   → vector search → найти документы о бюджетах маркетинга

2. Structured:
   - year = 2024
   - location = "Москва"
   → SQL фильтры в БД

3. Комбинирование:
   - semantic_results = [doc1, doc2, doc3]
   - sql_filtered = [doc2, doc3, doc5]
   - hybrid = intersection = [doc2, doc3]

4. Re-ranking:
   - weighted score = 0.7 * semantic_score + 0.3 * bm25_score

5. Return top 5 документов с высокими scores
```

**Метрики успеха:**
- Документы релевантны по смыслу И по фильтрам
- Никогда не вернуть документ о 2023 году или другому городу
- Время поиска < 200ms

### Задача 3: Оптимизировать производительность

**Сценарий:** RAG система медленная, retrieval занимает 1-2 секунды на запрос. Нужно ускорить.

**Шаги отладки (концептуально):**

1. **Профилирование**: где время тратится?
   - Embedding запроса? (100-200ms)
   - Vector search? (500-1000ms)
   - Другое?

2. **Оптимизации в зависимости от узкого места:**
   - Если embedding: используйте faster модель (text-embedding-3-small вместо large)
   - Если vector search: добавьте индекс (HNSW), уменьшите количество документов (фильтрация)
   - Если другое: профилируйте дальше

3. **Общие оптимизации:**
   - Кэшируйте embeddings часто запрашиваемых документов
   - Батчируйте запросы если возможно
   - Используйте асинхронную обработку

**Целевой результат:** latency < 200ms

## Выводы

Выбор и настройка vector store — критический компонент RAG:

1. **Выбор БД зависит от требований**, нет универсального решения
2. **Гибридный поиск часто лучше** чем только vector или только keyword
3. **Начните с простого** (Chroma или Pinecone), оптимизируйте по мере роста
4. **Embedding модель важна** → выберите подходящую для вашей области
5. **Мониторьте производительность** → latency, recall, precision
