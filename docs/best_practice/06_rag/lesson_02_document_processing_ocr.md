# Модуль 4.2: Обработка документов и OCR для RAG

## Введение

Системы Retrieval-Augmented Generation (RAG) полагаются на качество и структурированность данных, которые они извлекают из документов. В этом подмодуле мы рассмотрим полный цикл обработки документов: от сканов и PDF до структурированных данных, готовых к встраиванию в векторные базы данных. Обработка документов — это критический этап, который определяет эффективность всей RAG-системы.

---

## 1. Зачем RAG нужна обработка документов

### 1.1 Вызовы необработанных документов

RAG-системы работают с документами различных форматов:
- **PDF-документы**: сложные макеты, встроенные изображения, таблицы
- **Word-документы**: форматирование, изображения, диаграммы
- **Сканированные документы**: изображения, требующие OCR
- **Веб-страницы**: динамический контент, JavaScript-рендеринг
- **Электронные таблицы**: сложные структуры данных

Проблемы при прямом использовании необработанных документов:
- Потеря структуры (заголовки, таблицы, списки не разпознаются)
- Некорректный порядок чтения при многоколончных макетах
- Растеризация текста в PDF без возможности извлечения
- Неполное распознавание таблиц и формул
- Смешивание несвязанного контента (колонтитулы, водяные знаки, номера страниц)

Эти проблемы приводят к:
- Низкой точности retrieval (нерелевантные документы в результатах)
- Потере контекста и связей между фрагментами
- Галлюцинациям модели (нет фактической основы для ответа)
- Неэффективному использованию контекстного окна

### 1.2 Цели обработки документов для RAG

1. **Извлечение текста**: качественное распознавание текста с сохранением структуры
2. **Разбор структуры**: определение заголовков, параграфов, таблиц, формул
3. **Нормализация**: приведение к единому формату (обычно Markdown или JSON)
4. **Очистка**: удаление артефактов, колонтитулов, ненужных элементов
5. **Метаданные**: сохранение информации о странице, источнике, контексте

### 1.3 Pipeline обработки

Типичный процесс документ-обработки:

```
Исходный документ (PDF, DOCX, Image)
       ↓
Формат-детекция + парсинг
       ↓
Структурный анализ (заголовки, таблицы, макет)
       ↓
Текстовое извлечение (OCR если необходимо)
       ↓
Нормализация (кодировка, пробелы, структура)
       ↓
Очистка (удаление noise, артефактов)
       ↓
Добавление метаданных
       ↓
Markdown/JSON представление
```

## 2. Парсинг различных форматов

### 2.1 PDF документы

PDF — самый распространённый формат в корпоративной среде. Однако PDF содержит как векторный (векторные примитивы) так и растровый контент (изображения, отсканированные страницы).

**Два типа PDF:**

**Тип 1: Векторный PDF (с встроенным текстом)**
- Текст хранится отдельно от визуализации
- Можно напрямую извлечь текст
- Примеры: PDF созданный из Word, PowerPoint

Процесс обработки:
1. Использовать PDF парсер (PyPDF2, pdfplumber, pypdf)
2. Извлечь текст с координатами
3. Реконструировать структуру (заголовки, параграфы, таблицы)
4. Сохранить макет информацию в метаданные

**Тип 2: Растровый PDF (сканированные документы)**
- Текст только в изображениях
- Требует OCR для извлечения
- Примеры: скан паспорта, отсканированная книга

Процесс обработки:
1. Конвертировать PDF страницы в изображения
2. Применить OCR (PaddleOCR, Tesseract, или Claude Vision)
3. Парсить структуру на основе OCR результатов
4. Сохранить confidence scores для каждого слова

**Сложный случай: Гибридные PDF**
- Страница содержит и встроенный текст и отсканированные области
- Нужно комбинировать оба подхода
- Приоритет: встроенный текст выше (точнее), затем OCR

### 2.2 Word и Office документы

Word документы (.docx) содержат структурированное содержимое внутри ZIP архива:

**Что сохранять:**
- Текст с форматированием (заголовки, полужирный, курсив)
- Таблицы с сохранением структуры ячеек
- Изображения (сохранить или парсить как OCR)
- Комментарии и примечания (если релевантны)

**Подход обработки:**
- Метаданные: заголовки становятся структурой документа
- Таблицы: конвертировать в Markdown таблицы
- Списки: сохранить иерархию маркеров

### 2.3 Веб-контент и HTML

Веб-страницы требуют особой обработки для удаления noise:

**Источники noise в HTML:**
- Навигационное меню (повторяется на каждой странице)
- Сайдбары, реклама, cookie согласие
- Скрипты и стили (не содержат информацию)
- Динамический контент (JavaScript рендеринг)

**Подход парсинга:**
1. Загрузить HTML (requests для простых сайтов, Selenium/Playwright для JS)
2. Удалить теги скриптов, стилей, meta
3. Экстрактить semantic теги (article, main, content)
4. Конвертировать оставшийся HTML в Markdown
5. Дополнить метаданные: URL, publish date, author

### 2.4 Использование MarkItDown от Microsoft

MarkItDown — это утилита для унифицированной конвертации множества форматов в Markdown, оптимизированный для LLM обработки.

**Поддерживаемые форматы:**
- PDF (и векторные и растровые через внутренний OCR)
- Word (.docx, .docm, .doc)
- PowerPoint (.pptx)
- Excel (.xlsx)
- Изображения (через Claude Vision API)
- HTML
- JSON
- XML
- Исходный код

**Как MarkItDown работает:**
1. Определяет формат файла
2. Выбирает подходящий парсер
3. Конвертирует в Markdown с сохранением структуры
4. Возвращает text_content (Markdown) + metadata

**Преимущества использования MarkItDown:**
- Один API для всех форматов
- Оптимизирован для LLM (сохраняет важные структурные элементы)
- Встроенная поддержка OCR для PDF
- Обновляется регулярно

**Пример использования (концептуально):**

Для каждого документа:
1. Загрузить файл
2. Вызвать MarkItDown.convert(file_path)
3. Получить result.text_content (Markdown)
4. Дополнить метаданные (source, filename, page_count)
5. Использовать Markdown как основу для chunking

## 3. OCR системы и их применение

### 3.1 Когда нужен OCR?

OCR требуется когда в документе есть текст, хранящийся как изображение:
- Отсканированные бумажные документы
- PDF из сканов
- Скриншоты, фотографии документов
- Кратовщики от рукописных форм, заполненных вручную
- Исторические документы

**Вызовы OCR:**
- Низкое качество сканов (размытость, шум, искажения)
- Множество языков (латиница, кириллица, иероглифы, иврит и т.д.)
- Сложная структура (таблицы, многоколончный текст, формулы)
- Смешанный контент (текст + диаграммы + математические формулы)
- Правильное разпознавание специальных символов и пунктуации

### 3.2 Обзор OCR решений

**Классификация по подходу:**

| Тип | Примеры | Характеристики |
|-----|---------|-----------------|
| **Традиционные** | Tesseract, ABBYY | Хорошо для печатного текста, слабо для структуры |
| **Deep Learning** | PaddleOCR, EasyOCR | Лучше, мультиязычные, медленнее |
| **Vision LLM** | Claude Vision, GPT-4V | Контекстное понимание, дорого, медленно |

### 3.3 Сравнение основных решений

| Решение | Языки | Структура | Таблицы | Скорость | Стоимость | Использование |
|---------|-------|-----------|---------|----------|-----------|---------------|
| **PaddleOCR** | 109+ | Базовая | Плохо | Быстро | Бесплатно | Стандартные документы |
| **OCRFlux** | Много | Хорошая | Отлично | Медленно | GPU ресурсы | Сложные документы |
| **Dots.OCR** | Все письменные | Отличная | Отличная | Медленно | GPU ресурсы | Специализированные |
| **Claude Vision** | Все | Отличная | Отличная | Медленно | $0.3-3/image | Диаграммы, контекст |
| **Tesseract** | 100+ | Базовая | Плохо | Очень быстро | Бесплатно | Простые случаи |

### 3.4 Рекомендации по выбору OCR

**Используйте PaddleOCR если:**
- Стандартные печатные документы
- Нужна скорость
- Низкий бюджет на вычисления

**Используйте OCRFlux если:**
- Документы с таблицами и структурой
- Много кросс-страничных элементов
- Есть доступ к GPU

**Используйте Claude Vision если:**
- Документы с диаграммами, графиками, аннотациями
- Нужно понять контекст, а не только текст
- Высокая точность критична
- Ограниченное количество документов (стоимость)

**Используйте Tesseract если:**
- Очень простые, четкие сканы
- Минимальная задержка, максимальная скорость
- Любой бюджет на вычисления

### 3.5 Pipeline обработки OCR

Для эффективной OCR обработки:

```
Исходный документ или изображение
       ↓
Предобработка (бинаризация, денойзинг)
       ↓
Ориентация страницы (определение, коррекция)
       ↓
Распознавание текста
       ↓
Структурный анализ (если поддерживается)
       ↓
Post-processing (исправление очевидных ошибок)
       ↓
Markdown форматирование с метаданными
```

**Ключевые шаги:**

1. **Предобработка**: улучшение качества для OCR
   - Бинаризация (преобразование в черно-белое)
   - Денойзинг (удаление шума)
   - Автоматическое выравнивание угла наклона
   - Регулировка контрастности

2. **Основная обработка**: выбранным OCR движком

3. **Post-processing**: улучшение результатов
   - Language-specific spell checking
   - Распознавание названий (капитализация)
   - Исправление типичных ошибок OCR (0 vs O, 1 vs I, l)
   - Валидация через словарь

## 4. Структурные элементы документов

### 4.1 Сохранение макета и структуры

При обработке важно сохранить информацию о структуре документа:

**Структурные элементы:**
- **Заголовки** (уровни: H1, H2, H3...) → сохранить иерархию
- **Параграфы** → сохранить разбиение на абзацы
- **Таблицы** → конвертировать в Markdown или JSON
- **Списки** (маркированные и нумерованные) → сохранить иерархию
- **Цитаты** → выделить специальным форматом
- **Ссылки** → сохранить в Markdown формате [текст](URL)
- **Изображения** → сохранить описание или использовать OCR

### 4.2 Таблицы в RAG

Таблицы — важные структурные элементы, требующие специальной обработки:

**Проблемы таблиц в RAG:**
- Embeddings текста плохо работают для таблиц (структура важнее чем текст)
- При chunking можно разрыть таблицу на невзаимосвязанные части

**Подходы к таблицам:**

1. **Сохранить целиком**: если таблица помещается в один чанк
   - Конвертировать в Markdown таблицу
   - Или в JSON с рядами/колонками
   - Чанк = вся таблица + контекст (предыдущий параграф)

2. **Семантическое разбиение**: если таблица больше
   - Разбить по rows или logical groups
   - Каждая группа = отдельный чанк
   - Добавить header информацию в каждый чанк

3. **Гибридное представление**:
   - Основной чанк: Markdown таблица
   - Вспомогательные чанки: ключевые выводы из таблицы
   - Метаданные: "это таблица", column names

**Пример обработки таблицы:**

Исходная таблица:
```
| Город | 2023 | 2024 |
|-------|------|------|
| Москва| 100  | 120  |
| СПб   | 50   | 65   |
```

Обработанный чанк:
```
Таблица 1: Продажи по городам

| Город | 2023 | 2024 |
|-------|------|------|
| Москва| 100  | 120  |
| СПб   | 50   | 65   |

Вывод: Москва показала рост на 20%, СПб на 30%.

Метаданные: type=table, source=page_5, table_id=sales_by_city
```

### 4.3 Формулы и математика

Математические выражения требуют сохранения точного формата:

**Форматы для сохранения:**
- LaTeX (`$\frac{x}{y}$`)
- MathML (стандарт XML для формул)
- ASCII-art для простых выражений

**Рекомендация для RAG:**
- Сохранить формулу в LaTeX формате
- Добавить текстовое описание: "квадратный корень из x минус y"
- При embedding: комбинировать формулу + описание

## 5. Метаданные и контекст

### 5.1 Какие метаданные сохранять?

Метаданные — информация О документе, которая помогает при retrieval и понимании контекста:

| Метаданные | Пример | Использование |
|-----------|--------|---------------|
| source | "документы/договор_v2.pdf" | Отследить источник |
| page | 5 | Найти в оригинальном документе |
| chapter | "3. Условия доставки" | Контекст в документе |
| author | "Иван Петров" | Кто написал, версионирование |
| date_created | "2024-01-15" | Актуальность, staleness |
| date_modified | "2024-03-10" | Когда было обновлено |
| language | "ru" | Язык документа |
| document_type | "contract" | Специальная обработка |
| size_words | 1250 | Если большой документ → нужна специальная обработка |

### 5.2 Иерархические метаданные

Для сложных документов сохраняйте иерархию:

```
Document: "Ежегодный отчет 2024.pdf"
├── Section 1: "Финансовые показатели"
│   ├── Subsection 1.1: "Доход"
│   │   └── Chunk 1: "Общий доход составил..."
│   │       Metadata: source=report, section=1, subsection=1.1
│   └── Subsection 1.2: "Расходы"
│       └── Chunk 2: "Основные расходы..."
│           Metadata: source=report, section=1, subsection=1.2
└── Section 2: "Стратегия развития"
    └── Chunk 3: "На следующие три года..."
        Metadata: source=report, section=2
```

Такие метаданные помогают:
- Фильтровать по разделам: "только из раздела Финансы"
- Восстанавливать контекст: "это из подраздела Доход, раздел Финансовые показатели"
- Пересчитывать embedding если изменился раздел

## 6. Практический pipeline для RAG

### 6.1 Полный процесс обработки документов

Обработка документа — это последовательность этапов, каждый из которых добавляет структуру и чистоту:

**Этап 1: Загрузка и валидация** — получить документ, проверить формат и размер. Сохранить оригинал для аудита.

**Этап 2: Детекция формата** — определить тип файла (PDF, Word, веб-страница, изображение). Правильная типизация критична для выбора парсера.

**Этап 3: Первичный парсинг** — конвертировать документ в текст. Для PDF может быть прямое извлечение (если текст встроен) или OCR (если это скан).

**Этап 4: Структурный анализ** — выделить элементы структуры (заголовки, параграфы, таблицы, списки). Сохранить иерархию, это важно для контекста.

**Этап 5: Очистка и нормализация** — удалить артефакты, нормализовать символы, исправить кодировку. На этом этапе исчезают колонтитулы, водяные знаки, дублирующийся контент.

**Этап 6: Разбор специальных элементов** — обработать таблицы (конвертировать в структурированный формат), формулы (сохранить в LaTeX), изображения (OCR или описание).

**Этап 7: Добавление метаданных** — аннотировать каждый чанк информацией о источнике, странице, разделе, авторе, дате создания.

**Этап 8: Форматирование** — привести к единому представлению (обычно Markdown). Это оптимизирует для LLM обработки.

**Этап 9-11: Валидация, сохранение, логирование** — убедиться в качестве, сохранить результат, записать метрики для мониторинга.

**Ключевое противодействие:** каждый этап может отказать на каком-то документе, и важно не прерывать весь pipeline. Ошибочные файлы логируются, но обработка продолжается со следующего.

### 6.2 Обработка ошибок и отказоустойчивость

Обработка документов часто встречает проблемы. Ключ к production pipeline — graceful handling без полного отказа:

**Категории ошибок:**
- **Формат ошибки** — файл повреждении, неподдерживаемый формат
- **Парсинг ошибка** — OCR не может распознать, нет структуры
- **Данные ошибка** — кодировка неизвестна, файл слишком большой

**Стратегия обработки ошибок:**
1. **Изолировать** — проблемный файл не должен блокировать обработку других
2. **Логировать с контекстом** — какой файл, какой этап, какая ошибка
3. **Сохранить для анализа** — проблемные файлы собирают для человеческого обзора
4. **Partial результат** — если возможно, вернуть то, что удалось извлечь
5. **Алертить** — если ошибка rate превышает порог (например, 10%), значит что-то сломалось в pipeline

**Мониторинг:**
- % успешно обработанных файлов (целевое значение: > 95%)
- Распределение ошибок по категориям
- Время обработки per file (детектировать outliers)
- Confidence scores из OCR (если доступны)

## 7. Лучшие практики обработки документов

### 7.1 Качество выходных данных

**Проверки качества:**
1. **Полнота**: все ли содержимое было извлечено?
2. **Точность**: правильно ли распознано (особенно OCR)?
3. **Структура**: сохранилась ли иерархия?
4. **Метаданные**: полные ли данные о источнике?
5. **Консистентность**: соответствует ли стандартному формату?

**Метрики для мониторинга:**
- Процент успешно обработанных документов
- Среднее время обработки на документ
- Размер после обработки (улучшение сжатия?)
- Confidence score из OCR (если доступен)

### 7.2 Вычислительная эффективность

**Оптимизация производительности:**
- **Батчинг**: обработать несколько документов параллельно
- **Кэширование**: не переобрабатывать одинаковые файлы
- **Инкрементальное обновление**: обновлять только новые документы
- **Выбор инструментов**: faster парсеры для больших объемов

**Когда запускать обработку:**
- Offline processing: когда документы загружаются, до indexing
- Batch processing: в нерабочие часы для больших объемов
- Streaming: для real-time систем (медленнее, но не блокирует)

### 7.3 Версионирование и воспроизводимость

**Проблема:** Если изменить парсер или параметры OCR, нужно переобработать все документы.

**Решение:**
1. Версионировать pipeline (какой парсер, какие параметры)
2. Хранить оригинальные файлы
3. Логировать версию обработки в метаданные
4. Иметь возможность переобработать специфичные файлы

```
Document metadata:
- pipeline_version: "1.2"
- parser: "markitdown==0.2.1"
- ocr_engine: "paddleocr==2.7.0"
- processing_date: "2024-03-10"
```

## 8. Практические задачи

### Задача 1: Обработка смешанных документов

**Сценарий:** У вас есть папка с документами (PDF, Word, скриншоты). Нужно обработать всё в единый формат.

**Решение (концептуально):**
1. Написать детектор формата для каждого файла
2. Выбрать парсер в зависимости от формата
3. Для растровых PDF/скриншотов использовать OCR
4. Конвертировать всё в Markdown с единообразными метаданными
5. Логировать успехи и ошибки обработки

**Метрики успеха:**
- Все файлы обработаны (или логирована причина ошибки)
- Извлеченный текст читаемый и структурированный
- Метаданные полные и konsistentные

### Задача 2: Оптимизация OCR

**Сценарий:** Много отсканированных PDF. Нужно найти баланс между скоростью и качеством.

**Решение (концептуально):**
1. Выбрать несколько OCR движков (PaddleOCR, Claude Vision)
2. На выборке документов сравнить:
   - Скорость обработки
   - Качество распознавания
   - Стоимость (если облачный OCR)
3. Выбрать оптимальный вариант или гибридный подход
4. Мониторить confidence scores

**Гибридный подход:**
- Для простых сканов: PaddleOCR (быстро, дешево)
- Для сложных случаев: Claude Vision (лучше качество)
- Для очень сложных: ручная валидация и исправление

### Задача 3: Структурное обогащение

**Сценарий:** Исходные документы потеряли структуру при обработке. Нужно восстановить иерархию.

**Решение (концептуально):**
1. Анализировать текст на предмет паттернов заголовков
2. Использовать heuristics: "# ", "## ", ВСЕМ КАПС текст может быть заголовком
3. Для неповторимых случаев: может использовать LLM для классификации
4. Добавить восстановленную структуру в метаданные

## Выводы

Обработка документов — это основание качественной RAG системы:

1. **Формат имеет значение**: выбирайте правильный инструмент для каждого типа документа
2. **OCR не идеален**: проверяйте результаты и используйте validation
3. **Метаданные критичны**: сохраняйте максимум информации о источнике и контексте
4. **Структура важна**: таблицы, заголовки, иерархия существенны для поиска
5. **Итерируйте**: начните с простого pipeline, улучшайте по мере сбора данных о ошибках
