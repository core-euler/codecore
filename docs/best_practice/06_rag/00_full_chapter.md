## Введение в RAG

### Что такое RAG?

Retrieval Augmented Generation (RAG) — это архитектурный подход, который позволяет языковым моделям получать доступ к актуальной информации из внешних источников данных при генерации ответов. Вместо того чтобы полагаться исключительно на знания, содержащиеся в параметрах модели, RAG система динамически извлекает релевантные документы или отрывки текста из хранилища данных и использует их в качестве контекста для генерации более точных и информативных ответов.

### Основные компоненты RAG системы

1. **Ingestion Pipeline** — загрузка и обработка источников данных
2. **Chunking** — разделение документов на логические части
3. **Embedding** — преобразование текста в векторные представления
4. **Vector Store** — хранилище векторов для быстрого поиска
5. **Retriever** — компонент для поиска релевантных документов
6. **LLM** — языковая модель для генерации ответов на основе контекста

```
┌─────────────────┐
│  Документы      │
└────────┬────────┘
         │
         ▼
┌─────────────────────────────┐
│  Ingestion & Chunking       │
└────────┬────────────────────┘
         │
         ▼
┌─────────────────────────────┐
│  Embedding                  │
└────────┬────────────────────┘
         │
         ▼
┌─────────────────────────────┐
│  Vector Store               │
└─────────────────────────────┘
         ▲
         │ Retrieval
         │
┌────────┴─────────────────────┐
│  User Query → Embedding      │
└────────┬─────────────────────┘
         │
         ▼
┌──────────────────────────┐
│  Retrieved Context       │
└────────┬─────────────────┘
         │
         ▼
┌──────────────────────────┐
│  LLM + Prompt            │
└────────┬─────────────────┘
         │
         ▼
┌──────────────────────────┐
│  Generated Response      │
└──────────────────────────┘
```

## Архитектура RAG систем

### Многоуровневая архитектура

Современные RAG системы обычно организованы в несколько уровней:

**Уровень 1: Data Source Layer**
- Разнородные источники (PDF, веб, БД, API)
- Структурированные и неструктурированные данные

**Уровень 2: Ingestion Layer**
- Нормализация формата
- Парсинг (OCR, Markdown extraction)
- Метаданные

**Уровень 3: Processing Layer**
- Chunking
- Embedding generation
- Vector indexing

**Уровень 4: Retrieval Layer**
- Semantic search
- Keyword search (BM25)
- Hybrid search

**Уровень 5: Generation Layer**
- Prompt engineering
- Context management
- Response generation

## RAG vs Fine-tuning

### Сравнительная таблица

| Аспект | RAG | Fine-tuning |
|--------|-----|-------------|
| **Скорость обновления данных** | Мгновенно | Недели |
| **Стоимость** | Умеренная (embedding + storage) | Высокая (переобучение модели) |
| **Актуальность информации** | Высокая | Зависит от данных обучения |
| **Контролируемость** | Полная (управляемые документы) | Частичная |
| **Масштабируемость** | Отличная | Ограниченная объемом параметров |
| **Явный контекст** | Да (видно в источниках) | Нет (черный ящик) |
| **Начало работы** | Быстро (дни) | Долго (недели) |

### Когда использовать RAG?

**Используйте RAG для:**
- Доступа к внешним базам знаний
- Быстрого обновления информации
- Снижения галлюцинаций моделей
- Обеспечения прозрачности источников
- Работы с большими документооборотами

**Используйте Fine-tuning для:**
- Специализированного стиля написания
- Частых задач специфического формата
- Оптимизации под узкую предметную область
- Где важен паттерн, а не конкретные факты

## Pipeline загрузки документов

### Основные этапы

#### 1. Загрузка файлов

Первый этап любого RAG pipeline — это загрузка исходных документов из различных источников: файловой системы, облачного хранилища, веб-адресов или API. Система должна поддерживать разнообразные форматы файлов:

- **Текстовые форматы**: `.txt`, `.md`, `.csv`
- **Документы**: `.pdf`, `.docx`, `.xlsx`
- **Веб-контент**: HTML, JSON с API
- **Специальные**: скан-изображения (JPG, PNG)

Ключевой вызов на этом этапе — сохранить метаинформацию о файле (источник, дату загрузки, оригинальное имя) для последующего отслеживания и валидации.

#### 2. Парсинг документов

После загрузки необходимо конвертировать документ в текстовый формат, пригодный для дальнейшей обработки. Для различных форматов используются разные подходы:

**PDF документы:**
- Векторные PDF (с встроенным текстом) → прямое извлечение
- Растровые PDF (скан) → OCR (Optical Character Recognition)
- Сложные макеты → парсер структуры (таблицы, многоколончные разделы)

**Word/Excel документы:**
- Извлечение текста с сохранением структуры заголовков
- Таблицы конвертируются в структурированный формат
- Изображения и диаграммы требуют OCR

**Веб-контент:**
- HTML → Markdown (удаление скриптов, стилей)
- JavaScript-рендеринг требует браузерной automation
- Обработка навигационного шума (меню, сайдбары, рекламы)

**Инструмент MarkItDown** от Microsoft позволяет унифицировать парсинг всех форматов в единую Markdown репрезентацию, оптимизированную для LLM обработки.

## OCR системы

### Обзор OCR решений и их применение

OCR (Optical Character Recognition) необходим для извлечения текста из отсканированных документов и изображений. Современные системы обрабатывают не только текст, но и структуру документа.

**Основные вызовы OCR:**
- Качество сканов (низкое разрешение, искажения, шум)
- Множество языков (особенно кириллица и иероглифы)
- Сохранение структуры: таблицы, формулы, многоколончные макеты
- Обработка смешанного контента (текст + изображения + диаграммы)

### Классификация OCR решений

| Решение | Специализация | Сильные стороны | Слабые стороны |
|---------|---------------|-----------------|----------------|
| **PaddleOCR** | Универсальное, многоязычное | 109+ языков, быстрое | Базовая структура |
| **OCRFlux** | Сложные документы | Таблицы, кросс-страничные элементы | Требует GPU, медленнее |
| **Dots.OCR** | Все письменные системы | Диаграммы, SVG, формулы | Специализированное |
| **Claude API** (vision) | Смешанный контент | Контекстное понимание, диаграммы | Стоимость per-image |

### Когда что использовать

**PaddleOCR:** Стандартные многоязычные документы с базовой структурой

**OCRFlux:** Сложные документы с таблицами, формулами, кросс-страничными элементами

**Claude Vision:** Документы с диаграммами, графиками, необычными форматами

## Стратегии chunking

### Почему chunking важен?

LLM имеют ограничения контекста (токен window). Если документ больше этого окна, он должен быть разделен на части (чанки). Выбор стратегии chunking критически влияет на качество RAG:

- **Слишком маленькие чанки** → потеря контекста, много нерелевантных результатов
- **Слишком большие чанки** → превышение контекстного окна, высокие затраты на embedding
- **Неправильные границы** → разрыв семантических связей в середине предложения

### Основные стратегии

**1. Fixed-size chunking (фиксированный размер)**

Разделение по количеству символов или токенов:
```
Документ: 100,000 символов
Размер чанка: 1,000 символов
Overlap: 100 символов
```

Преимущества: простота, предсказуемость
Недостатки: может разрыть семантику

**2. Semantic chunking (семантическое)**

Разделение по семантическим границам (конец предложения, параграфа, раздела):
```
Документ разбивается на: параграфы → предложения
Чанк = группа предложений, пока расстояние между embeddings < threshold
```

Преимущества: сохраняет логику, лучшее качество поиска
Недостатки: медленнее (требует embedding промежуточных результатов)

**3. Token-based chunking (на основе токенов)**

Разделение с учетом фактического количества токенов в модели:
```
max_tokens = 512
overlap_tokens = 50
```

Преимущества: точное соответствие контексту модели
Недостатки: требует tokenizer, варьируется между моделями

**4. Document-structure aware (с учетом структуры)**

Разделение с сохранением иерархии документа:
```
Чанк = одна глава или раздел
Метаданные: level, parent_chapter, page_number
```

Преимущества: сохраняет контекст, помогает фильтрации
Недостатки: зависит от качества парсинга структуры

### Таблица сравнения стратегий

| Стратегия | Размер чанка | Скорость | Качество поиска | Использование памяти |
|-----------|--------------|----------|-----------------|----------------------|
| Fixed | 512-1024 симв | Быстро | Среднее | Низкое |
| Semantic | Переменный | Медленно | Отличное | Среднее |
| Token-based | 256-512 токен | Быстро | Хорошее | Среднее |
| Structure-aware | Переменный | Быстро | Отличное | Среднее |

### Практические рекомендации

- **Начните с token-based chunking** размером 512 токенов с overlap 50
- **Для важных документов** используйте semantic chunking
- **Добавляйте метаданные** (источник, раздел, номер страницы) к каждому чанку
- **Экспериментируйте**: тестируйте размер на вашем наборе данных

## Embedding и векторные хранилища

### Что такое embeddings?

Embeddings — это численные представления текста в многомерном пространстве (обычно 1536 измерений для современных моделей). Ключевая идея:

```
Текст → Embedding модель → Вектор чисел
        ↓
Семантически похожие тексты имеют близкие векторы
```

**Пример концепции:**

Текст А: "Кот сидит на доме"
Текст Б: "Кошка лежит на крыше"

Embedding модель вычислит близость эти текстов на основе семантического значения, а не точного совпадения слов.

### Современные модели embeddings

| Модель | Размерность | Провайдер | Стоимость | Мультиязычность |
|--------|-------------|-----------|-----------|-----------------|
| `text-embedding-3-small` | 1536 | OpenAI | $0.02/1M | Да |
| `text-embedding-3-large` | 3072 | OpenAI | $0.13/1M | Да |
| `Voyage AI` | 1024 | Voyage AI | Высокая | Да |
| `Multilingual-E5` | 768 | HuggingFace | Бесплатно | Да (100+ языков) |

### Выбор модели embedding

**Факторы выбора:**
1. **Размерность**: больше = лучше качество, но выше стоимость storage и compute
2. **Стоимость**: облачные API vs локальные модели
3. **Скорость**: требование к real-time vs batch processing
4. **Мультиязычность**: если работаете с несколькими языками
5. **Качество**: на вашем доменном наборе данных

### Векторные базы данных

Embedded чанки нужно хранить и индексировать для быстрого поиска. Для этого используются специализированные базы данных:

| База | Особенности | Использование |
|------|-------------|---------------|
| **Pinecone** | Облачная, управляемая | Простая интеграция, pay-per-use |
| **Supabase (pgvector)** | PostgreSQL + vector extension | Полный контроль, гибридный поиск |
| **Milvus** | Open-source, распределённая | Высокая масштабируемость |
| **Chroma** | Встраиваемая, простая | Локальная разработка |
| **Weaviate** | Облачная и self-hosted | Гибридный поиск, metadata фильтрация |

## Стратегии retrieval

### Типы поиска в RAG

**Semantic Search (вектор-подобие):**
- Запрос конвертируется в embedding
- Поиск K ближайших соседей в векторном пространстве
- Хорошо для понимания смысла
- Чувствителен к quality embeddings

**Keyword/Lexical Search (BM25):**
- Точное или приблизительное совпадение слов
- Быстро, не требует embedding
- Хорошо для специфических терминов
- Не работает для синонимов

**Hybrid Search:**
- Комбинирование векторного и keyword поиска
- Лучшее из обоих миров
- Более сложно настраивать

### Когда что использовать

**Semantic Search:**
- Вопросы с паrafrazable смыслом: "как скоро...?" vs "когда это произойдет?"
- Когда нужно найти контекст, а не конкретный термин

**Keyword Search:**
- Поиск специфических терминов, названий, аббревиатур
- Нормативно-правовые документы
- Когда точность совпадения важна

**Hybrid:**
- Большинство production систем
- Балансирует между recall и precision

## File Search API

### Gemini File Search — управляемое решение

Gemini File Search API предоставляет "всё в одном" решение для индексирования и поиска:

**Что происходит под капотом:**
```
Загрузка файла
     ↓
Автоматический chunking (система выбирает размер)
     ↓
Embedding generation (модель gemini-embedding-001)
     ↓
Индексирование в FileSearchStore
     ↓
API готов к retrieval запросам
```

**Преимущества:**
- Не требует управления инфраструктурой
- Автоматическая оптимизация chunking и embedding
- Встроенная поддержка контекстного окна модели
- Простое API для поиска

**Ограничения:**
- Максимум 100 МБ на файл
- Нет прямого доступа к embeddings
- Меньше контроля над деталями indexing

**Типичное использование:**

1. Создать FileSearchStore: `name = "sales-documents-2025"`
2. Загрузить документы через API
3. Использовать в prompt с tool_choice="file_search"
4. API автоматически retrieves релевантные отрывки

## Web Scraping для RAG

### Вызовы web scraping для RAG

Веб-сайты содержат динамический контент, требуя специальной обработки:

**Основные вызовы:**
- Динамический контент (JavaScript-рендеринг)
- Структурированный noise (меню, реклама, cookie банеры)
- Разнообразная разметка HTML
- Robots.txt и rate limiting
- Изменяющиеся URL структуры

**Подходы к обработке:**

1. **Простой scraping** (статический HTML):
   - Используйте BeautifulSoup или requests
   - Для новостных сайтов, блогов
   - Парсьте структуру, извлекайте контент в Markdown

2. **Динамический scraping**:
   - Требуется браузерная автоматизация (Playwright, Selenium)
   - Для SPA приложений, защищённых сайтов
   - Медленнее, больше ресурсов

3. **API когда доступен**:
   - Предпочитайте официальные API (чище, быстрее)
   - Нет необходимости парсить HTML

### Лучшие практики

- Проверяйте `robots.txt` и terms of service
- Используйте user-agent, respectful headers
- Добавляйте задержки между запросами
- Логируйте URL источники в метаданные
- Обрабатывайте HTTP ошибки и переподключения

## Code RAG

### RAG для исходного кода

Применение RAG к исходному коду имеет специальные требования:

**Вызовы:**
- Важна структурная иерархия (функции, классы, модули)
- Кросс-ссылки между файлами
- Синтаксис нужен для понимания, но не для поиска
- Большие файлы нужно разбивать функционально, не по размеру

**Стратегия chunking для кода:**

```
Подход: Структурный chunking
- Один чанк = одна функция/класс/метод
- Метаданные: тип (function/class), язык, модуль
- Overlap: добавить импорты и сигнатуру класса
```

**Что включать в metadata:**
- Язык программирования
- Тип сущности (function, class, enum)
- Модуль/пакет
- Зависимости (вызовы других функций)
- Версия кодовой базы

**Типичная обработка:**

1. Распарсить AST (Abstract Syntax Tree)
2. Извлечь сущности (функции, классы, методы)
3. Каждая сущность = отдельный чанк
4. Добавить контекст (импорты, сигнатура класса)
5. Создать embeddings

## Hybrid RAG с SQL

### Сочетание SQL и векторного поиска

Во многих случаях релевантные данные уже в SQL базе. RAG может комбинировать структурированный SQL query с векторным поиском:

**Сценарий:**

```
Пользовательский запрос: "Какие продукты продаются в Москве?"
     ↓
Routing: нужна ли структурированная информация?
     ↓
SQL query: SELECT * FROM products WHERE city = 'Moscow'
     ↓
Семантический поиск: найти похожие на query embeddings
     ↓
Комбинировать результаты: пересечение SQL + vector результатов
     ↓
LLM генерирует ответ на основе обоих источников
```

### Гибридный routing

**Query Router** анализирует запрос и решает:
- Нужен ли SQL (структурированные данные, фильтры)?
- Нужен ли vector search (семантический поиск)?
- Нужна ли комбинация?

**Примеры:**

| Запрос | Маршрут | Причина |
|--------|---------|---------|
| "продукты в Москве" | SQL + Vector | Нужна фильтрация + смысл |
| "описание товара X" | SQL | Точный lookup |
| "похожие товары" | Vector | Семантическое подобие |
| "рекомендации" | Vector | Контекстное понимание |

## Data Pipeline

### Организация end-to-end pipeline

RAG система требует непрерывного потока данных от источников к LLM:

```
Data Sources (APIs, Databases, Files)
     ↓
Ingestion (Load, Validate, Transform)
     ↓
Processing (Parse, Structure, Clean)
     ↓
Chunking (Division with Overlap)
     ↓
Embedding (Vector Generation)
     ↓
Indexing (Vector Store Ingestion)
     ↓
Monitoring (Quality, Staleness, Drift)
     ↓
Retrieval & Generation (User Queries)
```

### Ключевые компоненты pipeline

**Data Validation:**
- Проверка форматов
- Дедупликация
- Качество текста (минимальная длина, кодировка)

**Processing:**
- Нормализация (кодировка, пробелы)
- Структура (заголовки, параграфы)
- Очистка (удаление артефактов)

**Quality Monitoring:**
- Средний размер чанка
- Распределение embeddings
- Staleness метрики (когда последний update?)
- Retrieval effectiveness (feedback на поиск)

## Метрики качества RAG

### Оценка retrieval качества

**Recall:** из всех релевантных документов, сколько было найдено?
```
Recall = (найдено релевантных) / (всего релевантных)
```
Важно для: полноты поиска, чтобы не пропустить критическую информацию

**Precision:** из найденных документов, сколько были релевантны?
```
Precision = (найдено релевантных) / (всего найдено)
```
Важно для: качества контекста, чтобы не добавлять шум в prompt

**MRR (Mean Reciprocal Rank):** на каком месте находится первый релевантный результат?
```
MRR = средний рейтинг (1 место = 1.0, 2 место = 0.5, 10 место = 0.1)
```
Важно для: user experience, когда первые результаты критичны

### Оценка generation качества

**Faithfulness:** содержит ли ответ информацию из retriеved контекста (не галлюцинирует)?

**Relevance:** отвечает ли ответ на вопрос пользователя?

**Coherence:** понятен ли ответ, хороша ли структура?

### Практический подход

**Начните с простых метрик:**
1. Recall@10: найдено ли релевантное на первых 10 результатах?
2. Средняя позиция релевантного результата
3. User feedback (пальцы вверх/вниз на результаты)

**Продвинутые метрики (когда система растёт):**
1. NDCG (Normalized Discounted Cumulative Gain)
2. Bertscore на ответы (семантическое качество текста)
3. LLM-based evaluation (попросить модель оценить релевантность)

## Практическая лаборатория

### Задача 1: Построить простую RAG систему

**Цель:** Создать систему, которая отвечает на вопросы на основе загруженных документов.

**Архитектурные решения:**
1. Выбрать источник документов (PDF, Markdown, веб)
2. Выбрать стратегию chunking (начните с фиксированного размера 512 символов)
3. Выбрать модель embedding (OpenAI или локальная)
4. Выбрать vector store (Chroma, Pinecone, или Supabase)
5. Реализовать retrieval + LLM generation

**Метрики успеха:**
- Система загружает документы
- Retrieval находит релевантные отрывки
- LLM генерирует ответ на основе контекста
- Ответы цитируют источники

### Задача 2: Оптимизация chunking

**Цель:** Экспериментировать с размером и стратегией chunking.

**Эксперименты:**
1. Попробуйте размеры: 256, 512, 1024 символов
2. Добавьте overlap: 0%, 10%, 20%
3. Оцените recall на тестовом наборе вопросов
4. Выберите оптимальную конфигурацию

**Замечание:** Не существует универсального оптимума — зависит от типа контента и вопросов.

### Задача 3: Гибридный поиск

**Цель:** Комбинировать semantic и keyword поиск.

**Реализация:**
1. Индексировать документы и в vector store, и в keyword index
2. На запрос: выполнить оба поиска
3. Объединить результаты (re-ranking, weighted score)
4. Сравнить качество vs pure semantic

**Ожидаемый результат:** Гибридный поиск часто лучше, чем один из методов отдельно.

### Задача 4: Оценка качества

**Цель:** Построить простую систему оценки RAG.

**Подход:**
1. Собрать ground truth: вопрос → релевантные документы
2. Запустить retrieval, получить результаты
3. Вычислить Recall@10, средняя позиция релевантного
4. Объяснить результаты: где система ошибается?

## Выводы

RAG — это мощный подход для работы с актуальной информацией и снижения галлюцинаций LLM. Ключевые моменты:

1. **RAG vs Fine-tuning:** Выбирайте RAG для информации, которая меняется; fine-tuning для паттернов
2. **Pipeline важен:** Инvestition в качество loading, parsing, chunking окупается на всех этапах
3. **Это не идеально:** RAG требует мониторинга, итерации и экспериментов
4. **Метрики guide:** Используйте recall, precision и user feedback для улучшений
5. **Гибридный подход:** Комбинирование методов часто лучше, чем один способ
