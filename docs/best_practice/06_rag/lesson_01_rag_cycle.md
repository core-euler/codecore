# RAG — полный цикл: от ingestion до evaluation

## Главный тезис

RAG (Retrieval Augmented Generation) — это не «подключили векторку». Это многоконтурная система с отдельными фазами ingestion, normalization, indexing, retrieval, synthesis и evaluation. Если смешать это в один «чёрный ящик», качество трудно диагностировать и невозможно системно улучшать.

---

## 1. RAG как система

### 1.1 Что такое RAG

Retrieval Augmented Generation — это архитектурный подход, который позволяет языковым моделям получать доступ к актуальной информации из внешних источников данных при генерации ответов. Вместо того чтобы полагаться исключительно на знания, содержащиеся в параметрах модели, RAG система динамически извлекает релевантные документы и использует их в качестве контекста для генерации более точных ответов.

### 1.2 Основные компоненты RAG системы

Рабочий RAG состоит из шести отдельных контуров:

1. **Ingestion** — загрузка и получение данных из источников
2. **Normalization** — парсинг, очистка, приведение к единому формату
3. **Indexing** — chunking, embedding, размещение в хранилище
4. **Retrieval** — поиск релевантных документов по запросу
5. **Synthesis** — генерация ответа с цитированием источников
6. **Evaluation** — измерение качества на всех этапах

### 1.3 Архитектура полного pipeline

```
┌─────────────────┐
│  Документы      │
│  (PDF, Web, DB) │
└────────┬────────┘
         │
         ▼
┌─────────────────────────────┐
│  Ingestion & Normalization  │
│  (парсинг, очистка, OCR)    │
└────────┬────────────────────┘
         │
         ▼
┌─────────────────────────────┐
│  Chunking                   │
│  (разделение на части)      │
└────────┬────────────────────┘
         │
         ▼
┌─────────────────────────────┐
│  Embedding                  │
│  (векторизация)             │
└────────┬────────────────────┘
         │
         ▼
┌─────────────────────────────┐
│  Vector Store               │
│  (индексирование)           │
└─────────────────────────────┘
         ▲
         │ Retrieval
         │
┌────────┴─────────────────────┐
│  User Query → Embedding      │
└────────┬─────────────────────┘
         │
         ▼
┌──────────────────────────┐
│  Retrieved Context       │
│  + Reranking             │
└────────┬─────────────────┘
         │
         ▼
┌──────────────────────────┐
│  LLM + Prompt            │
│  (synthesis с цитатами)  │
└────────┬─────────────────┘
         │
         ▼
┌──────────────────────────┐
│  Generated Response      │
│  + Evaluation            │
└──────────────────────────┘
```

### 1.4 RAG vs Fine-tuning

| Аспект | RAG | Fine-tuning |
|--------|-----|-------------|
| **Скорость обновления данных** | Мгновенно | Недели |
| **Стоимость** | Умеренная (embedding + storage) | Высокая (переобучение модели) |
| **Актуальность информации** | Высокая | Зависит от данных обучения |
| **Контролируемость** | Полная (управляемые документы) | Частичная |
| **Масштабируемость** | Отличная | Ограниченная объемом параметров |
| **Явный контекст** | Да (видно в источниках) | Нет (чёрный ящик) |
| **Начало работы** | Быстро (дни) | Долго (недели) |

**Используйте RAG** для доступа к внешним базам знаний, быстрого обновления информации, снижения галлюцинаций, обеспечения прозрачности источников.

**Используйте Fine-tuning** для специализированного стиля, оптимизации под узкую область, где важен паттерн, а не конкретные факты.

## 2. Ingestion: качество входа определяет потолок качества ответа

### 2.1 Принцип

Если на вход RAG системы подаются плохо обработанные документы, никакая модель не компенсирует это на выходе. Качество ingestion определяет потолок качества всей системы.

### 2.2 Два режима ingestion

По практике, полезно разделять:
- **Discovery scrape** — широкий охват, допускается шум, цель — покрыть максимум источников.
- **Production scrape** — строгий scope + provenance, каждый документ проверен и размечен.

### 2.3 Минимальные метаданные для каждого документа

| Поле | Назначение | Пример |
|------|-----------|--------|
| `source_url` | Откуда получен документ | `https://docs.example.com/api` |
| `timestamp` | Когда загружен/обновлён | `2026-03-15T10:30:00Z` |
| `extraction_method` | Как извлечён текст | `pdf_text`, `ocr_paddleocr`, `html_markdown` |
| `checksum` | Хеш для дедупликации | `sha256:abc123...` |
| `version` | Версия документа | `v2.1` |

### 2.4 Форматы и парсинг

**PDF документы:**
- Векторные PDF (с встроенным текстом) → прямое извлечение
- Растровые PDF (скан) → OCR (PaddleOCR, OCRFlux, Claude Vision)
- Сложные макеты → парсер структуры (таблицы, многоколончные разделы)

**Веб-контент:**
- HTML → Markdown (удаление скриптов, стилей, навигации)
- JavaScript-рендеринг требует браузерной automation (Playwright)
- Обработка навигационного шума (меню, сайдбары, реклама)

**Код:**
- AST-based парсинг для сохранения структуры
- Каждая функция/класс — отдельный документ
- Метаданные: язык, модуль, зависимости

## 3. Стратегии chunking

### 3.1 Почему chunking критичен

LLM имеют ограничения контекста. Если документ больше окна, он должен быть разделён на части (чанки). Выбор стратегии chunking критически влияет на качество retrieval:

- **Слишком маленькие чанки** → потеря контекста, много нерелевантных результатов
- **Слишком большие чанки** → превышение контекстного окна, высокие затраты на embedding
- **Неправильные границы** → разрыв семантических связей в середине предложения

### 3.2 Основные стратегии

**1. Fixed-size chunking (фиксированный размер)**

Разделение по количеству символов или токенов с overlap:
```
Документ: 100,000 символов
Размер чанка: 1,000 символов
Overlap: 100 символов
```
Преимущества: простота, предсказуемость.
Недостатки: может разорвать семантику.

**2. Semantic chunking (семантическое)**

Разделение по семантическим границам (конец предложения, параграфа, раздела):
```
Чанк = группа предложений,
пока расстояние между embeddings < threshold
```
Преимущества: сохраняет логику, лучшее качество поиска.
Недостатки: медленнее (требует embedding промежуточных результатов).

**3. Token-based chunking**

Разделение с учётом фактического количества токенов:
```
max_tokens = 512
overlap_tokens = 50
```
Преимущества: точное соответствие контексту модели.
Недостатки: требует tokenizer, варьируется между моделями.

**4. Document-structure aware (с учётом структуры)**

Разделение с сохранением иерархии:
```
Чанк = одна глава или раздел
Метаданные: level, parent_chapter, page_number
```
Преимущества: сохраняет контекст, помогает фильтрации.
Недостатки: зависит от качества парсинга структуры.

### 3.3 Сравнение стратегий

| Стратегия | Размер чанка | Скорость | Качество поиска | Использование |
|-----------|-------------|----------|-----------------|---------------|
| Fixed | 512-1024 симв | Быстро | Среднее | Прототипирование |
| Semantic | Переменный | Медленно | Отличное | Production |
| Token-based | 256-512 токен | Быстро | Хорошее | Стандартный выбор |
| Structure-aware | Переменный | Быстро | Отличное | Структурированные документы |

### 3.4 Практические рекомендации

- **Начните с token-based chunking** размером 512 токенов с overlap 50
- **Для важных документов** используйте semantic chunking
- **Добавляйте метаданные** (источник, раздел, номер страницы) к каждому чанку
- **Экспериментируйте**: тестируйте разные размеры на вашем наборе данных

## 4. Retrieval: почему «только chunking» часто недостаточно

### 4.1 Проблемы базового подхода

Даже с правильным chunking базовый retrieval (найти top-K ближайших по embedding) имеет проблемы:
- **Разрыв дальних зависимостей** — связанная информация оказывается в разных чанках.
- **Шумовые попадания** — семантически похожие, но нерелевантные результаты.
- **Потеря структурных связей** — заголовок и его содержимое разделены.

### 4.2 Типы поиска

**Semantic Search (вектор-подобие):**
- Запрос конвертируется в embedding
- Поиск K ближайших соседей
- Хорошо для понимания смысла
- Чувствителен к качеству embeddings

**Keyword/Lexical Search (BM25):**
- Точное или приблизительное совпадение слов
- Быстро, не требует embedding
- Хорошо для специфических терминов, аббревиатур
- Не работает для синонимов

**Hybrid Search:**
- Комбинирование векторного и keyword поиска
- Балансирует между recall и precision
- Рекомендуется для большинства production систем

### 4.3 Практики улучшения retrieval

- **Reranking** — переупорядочивание результатов более точной моделью (cross-encoder). Улучшает precision без потери recall.
- **Metadata filtering** — фильтрация по метаданным (дата, источник, тип) до или после vector search.
- **Hybrid retrieval** — комбинация semantic + BM25 с weighted score.
- **Explicit citation constraints** — требование к модели цитировать конкретные чанки.
- **Query expansion** — переформулировка запроса для лучшего покрытия.

### 4.4 Когда что использовать

| Запрос | Лучший метод | Причина |
|--------|-------------|---------|
| "как настроить OAuth?" | Semantic | Понимание смысла |
| "error code 403" | Keyword (BM25) | Точное совпадение |
| "похожие товары" | Semantic | Контекстное подобие |
| "продукты в Москве" | Hybrid (SQL + Vector) | Фильтрация + смысл |

## 5. File Search и managed retrieval

Для managed сценариев полезны hosted-решения (Gemini File Search, OpenAI Assistants API), но важно учитывать:

- **Контроль** — меньше контроля над chunking и embedding.
- **Наблюдаемость** — сложнее инспектировать индекс.
- **Стоимость обновлений** — каждое обновление может быть дорогим.
- **Vendor lock-in** — зависимость от конкретного провайдера.

**Преимущества managed решений:**
- Не требуют управления инфраструктурой
- Автоматическая оптимизация chunking и embedding
- Простое API для интеграции

**Когда использовать managed:** прототипирование, небольшие объёмы данных, когда нет ресурсов на инфраструктуру.

**Когда строить своё:** production с большими объёмами, нужен полный контроль над quality pipeline, специфичные требования к retrieval.

## 6. Quality loop: метрики и evaluation

### 6.1 Оценка retrieval качества

**Recall@K:** из всех релевантных документов, сколько было найдено в top-K?
```
Recall@10 = (найдено релевантных в top-10) / (всего релевантных)
```
Важно для: полноты поиска, чтобы не пропустить критическую информацию.

**Precision@K:** из найденных документов в top-K, сколько были релевантны?
```
Precision@10 = (релевантных в top-10) / 10
```
Важно для: качества контекста, чтобы не добавлять шум в prompt.

**MRR (Mean Reciprocal Rank):** на каком месте находится первый релевантный результат?
```
MRR = среднее(1/позиция первого релевантного)
1 место = 1.0, 2 место = 0.5, 10 место = 0.1
```
Важно для: user experience, когда первые результаты критичны.

### 6.2 Оценка generation качества

| Метрика | Что измеряет | Как проверить |
|---------|-------------|---------------|
| **Faithfulness** | Ответ основан на retrieved контексте | LLM-as-judge: «Есть ли в ответе утверждения без опоры в источниках?» |
| **Relevance** | Ответ соответствует вопросу | Прямая оценка: отвечает ли на вопрос? |
| **Coherence** | Ответ структурирован и понятен | Читабельность, логичность |
| **Groundedness** | Каждое утверждение имеет источник | Проверка цитат |
| **Hallucination rate** | Процент выдуманных фактов | Сравнение с ground truth |

### 6.3 Правило для production

Ответ без проверяемой цитаты для factual-вопросов не считается production-ready. Если модель не может указать источник — это сигнал о галлюцинации.

### 6.4 Практический подход к evaluation

**Начните с простых метрик:**
1. Recall@10: найдено ли релевантное на первых 10 результатах?
2. Средняя позиция релевантного результата
3. User feedback (thumbs up/down)

**Продвинутые метрики (когда система растёт):**
1. NDCG (Normalized Discounted Cumulative Gain)
2. BertScore на ответы (семантическое качество текста)
3. LLM-based evaluation (попросить модель оценить релевантность)

## 7. Common failure modes

### 7.1 «Векторка всё решит»

Подключили vector store, загрузили документы без очистки, без версионирования, без data governance. Результат: шумные retrieval, устаревшие данные, невоспроизводимые результаты.

**Решение:** data governance pipeline — валидация, дедупликация, версионирование, мониторинг свежести.

### 7.2 Нет версии индекса и provenance

Невозможно определить, когда документ был загружен, из какого источника, какой версии. При ошибке невозможно диагностировать причину.

**Решение:** каждый документ хранит полный provenance: source, timestamp, extraction_method, checksum.

### 7.3 Нет оценки retrieval quality

Система работает, ответы генерируются, но никто не измеряет, насколько хорошо работает retrieval. Деградация качества замечается только по жалобам пользователей.

**Решение:** регулярный eval на golden set (набор вопросов с известными правильными ответами).

### 7.4 Нет разграничения свежих и устаревших данных

Документы годовой давности ранкуются наравне со свежими. Пользователь получает устаревшую информацию.

**Решение:** metadata-based filtering по дате, boosting свежих документов, staleness monitoring.

### 7.5 Chunk boundary problems

Ключевая информация разрезана пополам между двумя чанками. Ни один из чанков не содержит полного ответа.

**Решение:** overlap между чанками, semantic chunking, или parent-child indexing (индексировать мелкие чанки, но возвращать при retrieval родительский блок).

## 8. Embedding и векторные хранилища

### 8.1 Современные модели embeddings

| Модель | Размерность | Провайдер | Стоимость | Мультиязычность |
|--------|-------------|-----------|-----------|-----------------|
| `text-embedding-3-small` | 1536 | OpenAI | $0.02/1M | Да |
| `text-embedding-3-large` | 3072 | OpenAI | $0.13/1M | Да |
| `Voyage AI` | 1024 | Voyage AI | Средняя | Да |
| `Multilingual-E5` | 768 | HuggingFace | Бесплатно | Да (100+ языков) |

### 8.2 Векторные базы данных

| База | Особенности | Когда использовать |
|------|-------------|-------------------|
| **Pinecone** | Облачная, управляемая | Простая интеграция, pay-per-use |
| **Supabase (pgvector)** | PostgreSQL + vector extension | Полный контроль, гибридный поиск |
| **Milvus** | Open-source, распределённая | Высокая масштабируемость |
| **Chroma** | Встраиваемая, простая | Локальная разработка, прототипы |
| **Weaviate** | Облачная и self-hosted | Гибридный поиск, metadata фильтрация |

## 9. Lab

Собери минимальный RAG контур:

**Шаг 1: Ingestion**
- Собери 20-50 документов из одной предметной области.
- Для каждого документа сохрани метаданные: source, timestamp, extraction_method.

**Шаг 2: Chunking + Indexing**
- Примени token-based chunking (512 токенов, overlap 50).
- Сгенерируй embeddings, загрузи в vector store.

**Шаг 3: Retrieval + Rerank**
- Реализуй semantic search + reranking.
- Проверь на 15 контрольных вопросах.

**Шаг 4: Synthesis**
- Ответ с citations (каждое утверждение ссылается на источник).

**Шаг 5: Evaluation**
- Вычисли Recall@10 и MRR.
- Найди 5 примеров ошибок и предложи исправления.

**Deliverables:**
1. Ingestion log (список документов + метаданные)
2. Retrieval metrics (Recall@10, MRR, Precision@10)
3. 5 примеров ошибок + анализ причин + план исправлений
