# Agent Skills — что это и зачем

## Главный тезис

Agent Skills — это **модульные, переиспользуемые ресурсы на файловой системе**, которые превращают универсального LLM-агента в специализированного эксперта. Skill — это не промпт. Это папка с инструкциями, скриптами, данными и ассетами, которую агент автоматически обнаруживает и загружает по требованию.

---

## 1. Что такое Agent Skills

### 1.1 Определение

Agent Skill — это набор переиспользуемых, файл-based ресурсов, которые предоставляют агенту узкоспециализированную экспертизу: рабочие процессы, контекст и лучшие практики. Каждый Skill инкапсулирует знания по определённой дисциплине и может использоваться во многих сессиях.

### 1.2 Ключевые характеристики

| Характеристика | Описание |
|----------------|----------|
| **Модульность** | Каждый Skill инкапсулирует знания по одной дисциплине |
| **Переиспользуемость** | Один Skill работает во многих сессиях и контекстах |
| **Загрузка по требованию** | Содержимое загружается только при необходимости |
| **Файловая система** | Skills существуют как директории с структурированным содержимым |
| **Автоматическое обнаружение** | Агент сам определяет, когда нужен конкретный Skill |

### 1.3 Skill — это папка, а не файл

Распространённое заблуждение: Skills — это «просто markdown-файлы». На самом деле самое ценное в Skills — это то, что они представляют собой **папки**, которые могут содержать:

- Markdown-инструкции (SKILL.md)
- Скрипты (Python, Bash, TypeScript)
- Ассеты и шаблоны
- Данные и конфигурации
- Справочную документацию

Агент может обнаруживать эти файлы, читать их и использовать в работе. Это фундаментально отличает Skills от простых текстовых промптов.

## 2. Skills vs промпты: архитектурные преимущества

### 2.1 Сравнение подходов

| Аспект | Промпты | Skills |
|--------|---------|--------|
| **Область применения** | Одна сессия | Многие сессии |
| **Переиспользуемость** | Ручная репликация | Автоматическое обнаружение |
| **Контекстная стоимость** | Всегда в контексте | Загружается при необходимости |
| **Управление версиями** | В сессии, теряется | На файловой системе, версионируется |
| **Распространение** | Скопировать текст | Загрузить Skill один раз |
| **Масштабируемость** | Ограничена контекстом | Практически неограниченна |
| **Состав** | Только текст | Текст + скрипты + данные + шаблоны |

### 2.2 Проблема промптов

Промпты требуют повторного предоставления одной и той же информации в каждой сессии:

```
Сценарий с промптом:
Сессия 1: "Вот как анализировать PDF: ... [200 строк инструкций]"
Сессия 2: "Вот как анализировать PDF: ... [200 строк инструкций]"
Сессия 3: "Вот как анализировать PDF: ... [200 строк инструкций]"

Сценарий со Skill:
Загрузить Skill один раз → Используется во всех сессиях автоматически
```

Каждый промпт потребляет токены из контекстного окна. Когда инструкций много — контекст переполняется, и модель теряет эффективность. Skills решают эту проблему через загрузку по требованию.

### 2.3 Три ключевых преимущества Skills

**Переиспользуемость (Reusability)**. Skill загружается в систему один раз и становится доступным для всех будущих операций. Не нужно каждый раз копировать инструкции.

**Загрузка по требованию (On-Demand Loading)**. Progressive disclosure позволяет агенту загружать только необходимое содержимое. Метаданные (~100 токенов) всегда присутствуют, но полная инструкция загружается только при обращении к Skill.

**Композиция (Composition)**. Skills можно комбинировать и строить на основе друг друга, создавая сложные рабочие процессы без увеличения базовой контекстной нагрузки.

## 3. Архитектура Skills: файловая структура и обнаружение

### 3.1 Минимальная структура

```
my-skill/
├── SKILL.md              # Основной файл (обязателен)
└── [опциональные ресурсы]
```

`SKILL.md` — единственный обязательный файл. Он содержит YAML-метаданные и инструкции для агента.

### 3.2 Полная структура с ресурсами

```
pdf-skill/
├── SKILL.md                 # Основные инструкции
├── FORMS.md                 # Руководство по работе с формами
├── REFERENCE.md             # Детальная документация API
├── examples/
│   ├── simple_extraction.py
│   └── form_filling.py
├── scripts/
│   ├── validate_pdf.py
│   ├── merge_documents.py
│   └── extract_tables.sh
├── templates/
│   ├── form_template.pdf
│   └── report_template.docx
└── assets/
    └── sample_data.json
```

Каждая поддиректория играет свою роль:

- **examples/** — готовые примеры кода для агента
- **scripts/** — исполняемые скрипты, которые агент может запускать
- **templates/** — шаблоны для генерации выходных файлов
- **assets/** — вспомогательные данные
- **references/** — детальная документация, API-спецификации

### 3.3 YAML Frontmatter: метаданные Skill

Каждый SKILL.md начинается с YAML frontmatter:

```yaml
---
name: pdf-processing
description: Extract text and tables from PDF files, fill forms, merge documents. Use when working with PDF files or when the user mentions PDFs, forms, or document extraction.
version: "1.0"
author: "Claude Skills Team"
category: "document-processing"
tags: ["pdf", "forms", "extraction", "merge"]
---
```

**Обязательные поля:**

- `name` — kebab-case идентификатор (уникальный, 15-40 символов)
- `description` — 1-2 предложения: что делает Skill и когда его использовать

**Рекомендуемые поля:**

- `version` — семантическое версионирование
- `author` — создатель или команда
- `category` — область применения
- `tags` — ключевые слова для поиска

### 3.4 Поле name: требования к формату

```yaml
# Неправильно
name: PDF-Processor      # не kebab-case
name: pdfProcessor       # camelCase
name: pdf processing     # пробелы

# Правильно
name: pdf-processing
name: excel-analysis
name: sentiment-analysis-nlp
```

### 3.5 Поле description: критический элемент

Description — это **не резюме для человека**. Это описание для модели: по этому полю агент решает, нужен ли данный Skill для текущего запроса. При старте сессии агент собирает список всех доступных Skills с их описаниями и сканирует его для каждого запроса.

Структура хорошего описания:

```
<Что делает Skill>. <Когда использовать>. [Детали]
```

Примеры:

```yaml
# Хорошо: что делает + когда использовать
description: Extract text and tables from PDF files, fill forms, merge documents.
  Use when working with PDF files or when the user mentions PDFs, forms,
  or document extraction.

# Хорошо: для code review
description: Review code for security vulnerabilities, performance issues,
  and best practices. Use when the user asks for code review, security audit,
  or code quality assessment.

# Плохо: слишком коротко
description: Analyzes PDFs

# Плохо: нет условий срабатывания
description: Works with various document formats

# Плохо: слишком технически
description: Implements pdfplumber wrapper with table extraction using pandas
```

## 4. Progressive Disclosure: трёхуровневая загрузка контента

### 4.1 Концепция

Progressive Disclosure — ключевая архитектурная инновация Skills. Она минимизирует использование контекста при сохранении полной функциональности через трёхуровневую загрузку:

```
┌─────────────────────────────────────────────┐
│ Уровень 1: Метаданные (всегда загружены)    │
│ name + description (~100 токенов)           │
│ Ответ: "Какие Skills доступны?"             │
└─────────────────────────────────────────────┘
                      ↓
            [Агент обнаруживает релевантность]
                      ↓
┌─────────────────────────────────────────────┐
│ Уровень 2: Инструкции (при триггере)        │
│ SKILL.md тело (~3-5K токенов)               │
│ Ответ: "Как использовать этот Skill?"       │
└─────────────────────────────────────────────┘
                      ↓
            [Агент использует инструкции]
                      ↓
┌─────────────────────────────────────────────┐
│ Уровень 3: Ресурсы (по ссылке)              │
│ Scripts, templates, reference files         │
│ (0 токенов если не используются)            │
│ Ответ: "Как запустить validate.py?"         │
└─────────────────────────────────────────────┘
```

### 4.2 Таблица уровней

| Уровень | Когда загружается | Стоимость | Содержимое |
|---------|------------------|-----------|------------|
| **1. Metadata** | Всегда (при запуске) | ~100 токенов | `name`, `description` из YAML |
| **2. Instructions** | При триггере | ~3-5K токенов | SKILL.md body с инструкциями |
| **3. Resources** | По требованию | Через bash, не в контексте | Скрипты, шаблоны, данные |

### 4.3 Почему это важно для context engineering

Для Skill с 10 вспомогательными файлами:

- **Skill не используется**: ~100 токенов (только метаданные)
- **Skill активирован**: ~3-5K токенов (метаданные + инструкции)
- **Полное раскрытие**: инструкции + ресурсы через bash (ресурсы не занимают основной контекст)

Это означает, что можно иметь десятки и сотни Skills, и они не будут загромождать контекстное окно. Агент загрузит только те, которые реально нужны.

### 4.4 Алгоритм обнаружения Skills

1. При старте сессии система сканирует директорию Skills
2. Индексирует метаданные (`name`, `description`) каждого SKILL.md
3. При запросе пользователя — semantic search по описаниям
4. Если сходство превышает порог → загрузка инструкций в контекст
5. Агент получает полный текст инструкций и может использовать ресурсы

## 5. Встроенные Skills: примеры

### 5.1 Document Processing Skills

В Claude Code встроены Skills для работы с документами:

| Skill | Возможности |
|-------|-------------|
| **PDF Processing** | Извлечение текста, работа с формами, объединение документов, OCR |
| **Excel** | Анализ и манипуляция табличными данными, pivot tables |
| **PowerPoint** | Создание и редактирование презентаций |
| **Word** | Работа с текстовыми документами |

### 5.2 Пример: PDF Processing Skill

Типичный SKILL.md для PDF-обработки включает:

1. **Quick Start** — минимальный код для немедленного использования
2. **Core Capabilities** — список возможностей (text extraction, table detection, form filling, merging, metadata)
3. **Step-by-Step Workflows** — пошаговые инструкции для типовых задач
4. **Advanced Techniques** — OCR для отсканированных PDF, batch processing
5. **Troubleshooting** — решение распространённых проблем

Такая структура позволяет агенту быстро найти нужную информацию на любом уровне сложности задачи.

## 6. Типы Skills: таксономия Anthropic

Проанализировав сотни Skills внутри Anthropic, команда выделила 9 категорий. Лучшие Skills чётко попадают в одну категорию; те, что размываются между несколькими, обычно вызывают путаницу.

### 6.1 Library & API Reference

Skills-справочники по библиотекам, CLI или SDK. Содержат примеры кода и список типичных ошибок (gotchas).

**Примеры**: `billing-lib` (внутренняя библиотека с edge cases), `internal-platform-cli` (все подкоманды CLI с примерами), `frontend-design` (дизайн-система).

### 6.2 Product Verification

Skills для тестирования и верификации кода. Часто работают в связке с Playwright, tmux и другими инструментами. Могут записывать видео для визуальной проверки и включать программные assertions на каждом шаге.

**Примеры**: `signup-flow-driver`, `checkout-verifier`, `tmux-cli-driver`.

> Верификационные Skills крайне ценны. В Anthropic считают оправданным выделить инженера на неделю, чтобы довести их до совершенства.

### 6.3 Data Fetching & Analysis

Skills для подключения к стекам данных и мониторинга. Содержат библиотеки для получения данных, идентификаторы дашбордов, инструкции по workflow.

**Примеры**: `funnel-query`, `cohort-compare`, `grafana`.

### 6.4 Business Process & Team Automation

Skills для автоматизации повторяющихся рабочих процессов. Часто зависят от других Skills или MCP. Полезно сохранять результаты прошлых запусков в лог-файлах.

**Примеры**: `standup-post`, `create-ticket`, `weekly-recap`.

### 6.5 Code Scaffolding & Templates

Skills для генерации boilerplate-кода. Особенно полезны, когда scaffolding включает требования на естественном языке.

**Примеры**: `new-workflow`, `new-migration`, `create-app`.

### 6.6 Code Quality & Review

Skills для обеспечения качества кода и code review. Могут включать детерминированные скрипты и запускаться автоматически через hooks или GitHub Actions.

**Примеры**: `adversarial-review` (подагент для критики), `code-style`, `testing-practices`.

### 6.7 CI/CD & Deployment

Skills для CI/CD-процессов: сборка, тестирование, деплой.

**Примеры**: `babysit-pr` (мониторит PR, повторяет flaky CI, решает merge-конфликты), `deploy-service`, `cherry-pick-prod`.

### 6.8 Runbooks

Skills, которые принимают симптом (alert, ошибка, тред в Slack) и проводят расследование с формированием структурированного отчёта.

**Примеры**: `service-debugging`, `oncall-runner`, `log-correlator`.

### 6.9 Infrastructure Operations

Skills для рутинного обслуживания инфраструктуры с предохранителями для деструктивных действий.

**Примеры**: `resource-orphans`, `dependency-management`, `cost-investigation`.

## 7. Когда использовать Skills vs другие подходы

### 7.1 Skills vs System Prompt

Используйте **system prompt**, когда инструкции:
- Нужны в каждой сессии
- Короткие (< 500 токенов)
- Не меняются между сессиями

Используйте **Skill**, когда инструкции:
- Нужны только для определённого типа задач
- Объёмные (> 500 токенов)
- Содержат скрипты или другие ресурсы
- Должны переиспользоваться в разных проектах

### 7.2 Skills vs MCP Tools

**MCP Tools** — для внешних интеграций (API, базы данных, сервисы). Они выполняют действия.

**Skills** — для знаний и рабочих процессов. Они предоставляют экспертизу и инструкции.

На практике Skills и MCP Tools часто работают вместе: Skill содержит знания о том, *как* решать задачу, а MCP Tool предоставляет возможность *выполнить* конкретное действие.

### 7.3 Skills vs Subagents

**Subagents** — для параллельного выполнения независимых задач с изолированным контекстом.

**Skills** — для предоставления знаний текущему агенту. Skill может использоваться как основным агентом, так и subagent-ом.

### 7.4 Матрица выбора

| Ситуация | Рекомендация |
|----------|-------------|
| Нужно вызвать внешний API | MCP Tool |
| Нужно дать агенту экспертизу по предметной области | Skill |
| Нужно параллельно решить несколько задач | Subagent |
| Нужно переиспользовать инструкции между проектами | Skill |
| Нужно выполнить действие + знать как | Skill + MCP Tool |
| Инструкции короткие и нужны всегда | System Prompt |

## 8. Память и хранение данных в Skills

### 8.1 Skills с памятью

Некоторые Skills могут иметь **форму памяти**, сохраняя данные внутри себя. Варианты:

- Append-only текстовый лог
- JSON-файлы
- SQLite-база данных

Пример: Skill `standup-post` ведёт `standups.log` со всеми опубликованными стендапами. При следующем запуске агент читает историю и определяет, что изменилось со вчерашнего дня.

### 8.2 Стабильное хранение

Данные в директории Skill могут быть удалены при обновлении. Для долговременного хранения используйте `${CLAUDE_PLUGIN_DATA}` — стабильную директорию, предоставляемую системой для каждого плагина.

## 9. Конфигурация и хуки

### 9.1 Опции конфигурации

В Claude Code Skills имеют широкий набор конфигурационных опций, включая регистрацию динамических hooks. Самые эффективные Skills используют эти опции нестандартно.

### 9.2 On-Demand Hooks

Skills могут включать hooks, которые активируются только при вызове Skill и действуют до конца сессии:

- `/careful` — блокирует `rm -rf`, `DROP TABLE`, `force-push`, `kubectl delete` через `PreToolUse` matcher на Bash. Нужен только при работе с production
- `/freeze` — блокирует Edit/Write вне указанной директории. Полезен при отладке, когда нужно добавить логи, но не трогать остальной код

Хуки по требованию — мощный паттерн для безопасности и контроля, без постоянной нагрузки на агента.

## Итоги

1. **Skills** — это папки с инструкциями, скриптами и данными, а не просто markdown-файлы
2. **Progressive Disclosure** оптимизирует контекст через 3 уровня загрузки
3. **Поле description** — для модели, не для человека; определяет когда Skill срабатывает
4. **9 категорий** покрывают большинство use cases: от справочников API до runbooks
5. Skills дополняют, а не заменяют MCP Tools, subagents и system prompts
6. Skills могут хранить данные и регистрировать on-demand hooks
