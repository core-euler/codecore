# Модуль 7, Раздел 2: Локальный и облачный runtime моделей

## Введение

Выбор runtime для запуска языковых моделей — один из критических решений при разработке AI-приложений и агентов. Современный ландшафт предлагает два основных подхода: локальное выполнение (local inference) и облачные сервисы (cloud services). Каждый подход имеет уникальные характеристики, преимущества и ограничения.

---

## 1. Локальный Inference: архитектура и инструменты

### 1.1 llama.cpp: ядро локального выполнения

**llama.cpp** — это проект с открытым исходным кодом, который предоставляет эффективную реализацию вывода моделей на C/C++ без зависимостей. Основная цель — максимизировать производительность на различном оборудовании.

**Ключевые характеристики:**
- **Кроссплатформенность**: поддержка ARM (Apple Silicon), AVX/AVX2 (Intel/AMD), RISC-V
- **GPU ускорение**: CUDA для NVIDIA, HIP для AMD, Metal для Apple Silicon
- **Архитектурная универсальность**: работает на CPU, GPU, или гибридный режим
- **Маленький размер кода**: простая реализация на C/C++ упрощает интеграцию

**GGUF формат:**
GGUF (GGML Universal Format) — это стандартизированный формат для хранения моделей, который позволяет:
- Хранить метаинформацию о модели (архитектура, параметры)
- Эффективно загружать части модели по требованию
- Применять квантизацию на уровне формата

**Квантизация в llama.cpp:**

| Формат | Бит | Качество | Использование |
|--------|-----|----------|---------------|
| Q4_K_M | 4.3 | Хороший баланс | Рекомендуется по умолчанию |
| Q5_K_M | 5.3 | Выше качество | Когда качество критично |
| Q8_0 | 8 | Почти без потерь | Критичные приложения |
| Q3_K_M | 3 | Очень маленький | Мобильные устройства |

**OpenAI-совместимый API:**
llama.cpp предоставляет HTTP сервер с OpenAI-совместимым API:
- Запускается с флагом `--server`
- Доступен на `http://localhost:8080/v1/`
- Поддерживает те же endpoints, что и OpenAI API

### 1.2 Ollama: пакетный менеджер моделей

**Ollama** — это высокоуровневый инструмент для управления открытыми LLM, напоминающий Docker для моделей.

**Основные возможности:**
- **Простой интерфейс**: скачать модель командой `ollama pull mistral`
- **Versioning**: управление разными версиями одной модели
- **Modelfile**: конфигурация модели (как Dockerfile)
- **REST API**: встроенный API для интеграции

**Modelfile концепция:**
Подобно Dockerfile, `Modelfile` определяет конфигурацию модели:
- Какую base модель использовать
- Системный промпт (default behavior)
- Параметры (температура, top_p)
- Размер контекста

**Преимущества Ollama:**
- Не нужно понимать квантизацию — Ollama выбирает оптимальную версию
- Можно легко переключаться между моделями
- Интегрируется с IDE плагинами (VS Code, Zed)

### 1.3 LM Studio: графический интерфейс

**LM Studio** — это десктопное приложение с GUI для запуска локальных моделей.

**Основной workflow:**
1. Открыть LM Studio
2. Найти и скачать модель (например, Mistral 7B)
3. Загрузить модель в память
4. Выбрать режим работы:
   - Chat — интерактивный чат
   - Server — OpenAI-совместимый API

**Преимущества для разработчиков:**
- Минимальная конфигурация
- Визуальный мониторинг performance (GPU usage, tokens/sec)
- Подходит для прототипирования
- Кроссплатформенна (macOS, Linux, Windows)

## 2. Открытые модели для агентов

### 2.1 Kimi K2: MoE архитектура для агентов

**Kimi K2** от MoonshotAI — это state-of-the-art Mixture-of-Experts модель для агентических приложений.

**Архитектура:**
- **1 триллион параметров** всего, но активны только **32 миллиарда** на токен
- **384 expert'а**, из которых выбираются лучшие 8 для каждого токена
- **128K context** для анализа больших файлов
- **Специализирована для агентов**: лучше в tool-calling, рассуждении

**Когда использовать:** Kimi K2 оптимальна для:
- Агентических приложений, требующих рассуждения
- Tool-calling (функций/API вызовы)
- Сложных многошаговых задач

**Развертывание:**
Рекомендуемые runtime'ы для inference:
- **vLLM** — для максимальной пропускной способности
- **SGLang** — для оптимизации задержки
- **KTransformers** — для эффективного использования памяти

**Лицензирование:** Modified MIT License позволяет использование как в открытых, так и в коммерческих проектах.

### 2.2 Qwen-Code: специализированная модель для разработки

**Qwen-Code** от Alibaba — это открытый AI агент в терминале, оптимизированный для программирования и анализа больших кодовых баз.

**Характеристики:**
- **Специализация**: кодирование, понимание codebases, рефакторинг
- **Размер**: доступны версии от 7B до 235B параметров
- **Контекст**: поддержка огромного контекста для анализа целых проектов
- **Встроенные инструменты**: Skills, SubAgents для сложных workflows

**Когда использовать:** Qwen-Code лучша для:
- Анализа и понимания больших кодовых баз
- Code completion и generation
- Автоматического рефакторинга
- Поиска ошибок и security issues в коде

### 2.3 Сравнение моделей: выбор для разных сценариев

| Модель | Параметры | Context | Архитектура | Лучше всего для |
|--------|-----------|---------|-------------|-----------------|
| Kimi K2 | 1T (32B активных) | 128K | MoE | Агенты, reasoning |
| Qwen-Code | 235B | Variable | Dense | Code analysis |
| Mistral 7B | 7B | 32K | Dense | General purpose |
| LLaMA 3.1 | 70B | 128K | Dense | Large context |
| Claude Opus | Proprietary | 200K | Proprietary | Best quality |

## 3. Облачные Coding Assistants

### 3.1 Gemini Code Assist: IDE интеграция

**Gemini Code Assist** — это AI-powered коллаборатор, встроенный в IDE для помощи в разработке.

**Поддерживаемые IDE:**
- VS Code
- IntelliJ IDEA
- PyCharm
- WebStorm
- Android Studio
- Другие JetBrains IDE

**Основные функции:**

**Code Generation:**
- Команда `/generate` для создания кода по описанию
- Может генерировать функции, классы, целые модули

**Code Completions:**
- Встроенные подсказки при печати
- Context-aware suggestions основаны на файле и project

**Code Transformation:**
- `/fix` для исправления ошибок
- `/doc` для добавления документации
- `/simplify` для рефакторинга

**Next Edit Predictions:**
- Предсказывает следующее редактирование разработчика
- Экономит несколько нажатий клавиш

**Smart Actions:**
- Контекстные команды для выбранного кода
- Generate unit tests, explain, refactor, optimize

**Remote Repository Context:**
- Может анализировать код из конкретных GitHub репозиториев
- Использовать `@REPOSITORY_NAME` в чате для reference

### 3.2 Gemini CLI: терминальный интерфейс

**Gemini CLI** — это мощный open-source агент для терминала (Apache 2.0 лицензия).

**Основные возможности:**
- Интерактивный режим: `gemini` открывает чат
- Анализ кодовой базы: просто запустите в папке проекта
- Встроенные инструменты: Google Search, file operations, shell commands
- MCP (Model Context Protocol) интеграция: расширяемость через сервера

**Аутентификация:**
- Google аккаунт (free tier: 60 req/min, 1000 req/day)
- API ключ (больше лимитов)
- Vertex AI (для enterprise)

**Возможности:**
- Встроенные инструменты: Google Search, file operations, shell execution
- MCP интеграция для расширения (GitHub, Slack, databases)

### 3.3 Kiro: spec-driven development платформа

**Kiro** (от AWS) — это платформа для spec-driven разработки, которая трансформирует вербальные требования в точные спецификации и автоматизирует реализацию.

**Workflow Kiro:**

1. **Natural Language** — опишите требование человеческим языком
2. **EARS Specification** — Kiro преобразует в структурированную спецификацию
3. **Architecture Design** — предлагает архитектурные решения
4. **Implementation Plan** — разбивает на дискретные задачи
5. **Task Dependencies** — определяет зависимости между задачами
6. **AI Agents** — каждую задачу выполняет агент (Autopilot)

**EARS формат для спецификаций:**
```
Given <context>
When <action>
Then <outcome>
```

Пример:
```
Given: User logged in
When: User clicks "Delete Account"
Then: Account deleted AND user logged out AND confirmation email sent
```

**Особенности:**
- Spec-driven: от требований к коду через спецификации
- Steering files: глобальная конфигурация для проектов
- Agent hooks: автоматические агенты на события (file save, test failure)
- Smart context: управление контекстом для больших проектов

**Поддерживаемые модели:**
- Claude Sonnet 4.5: надежные рассуждения
- Auto: смесь моделей для оптимизации цены/качества

### 3.4 Cursor CLI: агенты из терминала

**Cursor CLI** — это интерфейс командной строки для доступа к Cursor агентам прямо из терминала.

**Основные возможности:**
- Frontier моделей: доступ к лучшим моделям (Claude, GPT-4, Gemini)
- Интеграция с IDE: работает с любым редактором
- Shell Mode: выполнение команд и git операций

## 4. Выбор runtime: Decision Framework

### 4.1 Локальное выполнение

**Когда выбирать:**
- ✓ Критична приватность данных
- ✓ Нет интернета
- ✓ Очень высокий volume (cost-effective)
- ✓ Real-time требования (low latency)

**Когда NOT выбирать:**
- ✗ Нужна best-in-class модель (GPT-4, Claude Opus)
- ✗ Нет мощного GPU оборудования
- ✗ Быстрый time-to-market (setup требует времени)
- ✗ Скалирование на миллионы пользователей (сложнее, чем облако)

### 4.2 Облачное выполнение

**Когда выбирать:**
- ✓ Быстрый старт (no setup)
- ✓ Доступ к лучшим моделям
- ✓ Автоматическое масштабирование
- ✓ Встроенный мониторинг и логирование

**Когда NOT выбирать:**
- ✗ Огромный volume (очень дорого)
- ✗ Приватные/конфиденциальные данные
- ✗ Критична latency
- ✗ Нет интернета

### 4.3 Гибридные стратегии

**Routing по сложности:**

Определить complexity, выбрать подходящий runtime:
- Simple: Local (быстро, дешево)
- Medium: Local или Cloud Sonnet
- Complex: Cloud Opus

**Fallback стратегия:**

Начать с local, если timeout/error → fallback to cloud:
1. Пробуем local (быстро, дешево)
2. Если нет ответа за 10 секунд → переключаемся на cloud
3. Возвращаем ответ пользователю
4. Логируем какой путь был использован (для анализа)

**Cost optimization:**

- **Кэширование**: сохраняем результаты для одинаковых запросов
- **Batching**: обрабатываем несколько запросов за раз
- **Compression**: используем более дешевую модель для easy tasks

## 5. Квантизация и оптимизация производительности

### 5.1 Квантизация: что это и почему нужна

**Квантизация** уменьшает размер модели и требования к памяти путем представления весов в меньшем числе бит.

| Формат | Размер | Memory | Speed | Quality |
|--------|--------|--------|-------|---------|
| FP32 | 100% | 100% | 100% | 100% |
| FP16 | 50% | 50% | 200% | 99% |
| INT8 | 25% | 25% | 300% | 95% |
| Q4 | 25% | 25% | 300% | 88% |
| Q3 | 19% | 19% | 400% | 82% |

**Баланс quality vs speed vs память — главное решение.**

### 5.2 Оптимизация производительности

**Flash Attention:**
Оптимизация алгоритма внимания, снижает сложность с O(N²) памяти до O(N).

**KV-Cache оптимизация:**
Кэширование промежуточных вычислений ускоряет generation токенов.

**Batch обработка:**
Параллельная обработка множества запросов улучшает пропускную способность (throughput).

**Speculative Decoding:**
Используется меньшая модель для предсказания токенов, что ускоряет generation.

## 6. Практические рекомендации

### 6.1 Для разработчиков (Development)

**Лучший выбор:**
- **Модель**: Claude Opus (лучший качество для идей и анализа)
- **Runtime**: Cloud (Claude API)
- **IDE**: Cursor (оптимален для работы с агентами)
- **Стоимость**: ~$10-50/месяц за development

**Практика:** Используйте облако для development. Это позволяет:
- Не тратить время на setup
- Использовать best-in-class модели
- Итеративно разрабатывать
- Понять требования перед optimization

### 6.2 Для production (Medium Traffic)

**Лучший выбор:**
- **Модель**: Claude Sonnet (best balance)
- **Runtime**: Hybrid (local for simple, cloud for complex)
- **Мониторинг**: полное отслеживание cost и quality
- **Стоимость**: ~$500-2000/месяц

**Практика:**
- Начните с облака (Sonnet)
- Мониторьте cost
- Если очень высокий cost → добавьте local fallback

### 6.3 Для High-Volume приложений

**Лучший выбор:**
- **Модель**: Mix (Haiku for simple, Sonnet for complex)
- **Runtime**: Local + Cloud (smart routing)
- **Infrastructure**: Kubernetes для масштабирования
- **Стоимость**: $5000+/месяц

**Практика:**
- Инвестируйте в GPU hardware
- Установите local inference (Ollama/llama.cpp)
- Используйте cloud только для complex tasks
- Регулярно переоцензите cost/quality tradeoff

## Резюме

При выборе локального vs облачного runtime:

- **Cloud** лучше для: development, best models, простоте
- **Local** лучше для: privacy, high volume, latency
- **Hybrid** часто оптимален: использует оба в зависимости от задачи

**Начните с облака** (простота, quality), затем оптимизируйте к local если нужно (cost, privacy, latency).

Неправильный выбор runtime может стоить тысячи долларов в неоптимальных расходах или месяцы в разработке инфраструктуры.
