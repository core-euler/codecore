# CI/CD для LLM-агентов

## Введение

CI/CD для LLM-агентов принципиально отличается от классических pipeline. Когда выход системы **недетерминистичен**, привычные подходы -- детерминистические unit-тесты, бинарные pass/fail проверки, статичные артефакты -- перестают работать. Нужна новая парадигма: probabilistic testing, валидация качества AI-вывода и контроль стоимости на каждом этапе pipeline.

---

## 1. Фундаментальные различия с классическим CI/CD

### 1.1 Что меняется

| Аспект | Классический CI/CD | AI-Native CI/CD |
|--------|-------------------|-----------------|
| **Тестирование** | Детерминистические unit-тесты | Probabilistic testing, метрики качества |
| **Интеграция** | Merging кода | Merging с валидацией выхода модели |
| **Артефакты** | Статичные бинарники | Динамические модели + конфигурации промптов |
| **Метрики** | CPU, память, latency | Token-usage, cost, quality score |
| **Откат** | Переключение версии | Rollback с учётом истории промптов и конфигов |

Ключевой сдвиг: в AI-native CI/CD **стоимость** становится first-class метрикой наравне с performance и correctness.

### 1.2 Архитектура AI-Native Pipeline

Типичный pipeline для LLM-агента проходит следующие стадии:

1. **Trigger** -- пользователь упомянул `@claude` в PR/Issue или запустил workflow вручную
2. **Context analysis** -- агент читает кодовую базу, историю коммитов, description задачи
3. **Solution generation** -- агент создаёт или модифицирует код
4. **Syntax validation** -- lint, type checking, базовые ошибки
5. **Quality validation** -- проверка метрик выхода модели (корректность кода, соответствие стилю проекта)
6. **Cost check** -- расходы на API в пределах бюджета
7. **PR creation** -- агент открывает pull request
8. **Human review** -- разработчик читает и одобряет или просит исправления

### 1.3 Ключевые параметры pipeline

При создании CI/CD для агентов необходимо определить:

- **max_turns** -- максимальное количество итераций агента (обычно 5-10). Больше = дороже и медленнее, но может решить сложную задачу.
- **model** -- какую модель использовать (`claude-opus-4-6` для сложных задач, `claude-sonnet` для скорости).
- **temperature** -- креативность ответов. Для кодирования обычно 0.0-0.3.
- **budget_per_run** -- максимальный расход токенов на один запуск. При превышении workflow прерывается.
- **timeout** -- сколько времени ждём ответа агента.
- **quality_threshold** -- минимальная оценка качества (например, `score >= 0.8`). Если ниже, PR не создаётся.

## 2. GitHub Integration: Claude Code в CI/CD

### 2.1 Способы интеграции

**Способ 1: @claude mentions в PR/Issues**

Разработчик открывает issue ("Добавить обработку ошибок в функцию `load_data`"), упоминает `@claude`, агент анализирует кодовую базу и создаёт коммиты с исправлениями. Разработчик просматривает PR перед merge.

**Способ 2: GitHub Actions workflow**

На каждый push или PR запускается автоматический workflow. Агент работает в режиме code review -- проверяет стиль, безопасность, производительность. Оставляет комментарии с предложениями.

**Способ 3: Scheduled jobs**

Регулярные задачи: каждый день агент анализирует открытые issues, автоматически выбирает те, которые может решить, и создаёт PR с решениями.

### 2.2 Базовая конфигурация

Шаги подключения:

1. Установите GitHub App из официального репозитория Anthropic
2. Добавьте `ANTHROPIC_API_KEY` в GitHub Secrets
3. Создайте `.github/workflows/claude.yml`
4. Добавьте `.claude.yml` в корень проекта

Структура `.claude.yml`:

```yaml
model: claude-opus-4-6
max_turns: 8
temperature: 0.1
token_budget: 100000
quality_checks:
  - type: "linting"
    tools: ["pylint", "eslint", "rustfmt"]
  - type: "testing"
    command: "pytest --cov"
  - type: "security"
    tools: ["bandit", "npm-audit"]
```

### 2.3 Что происходит при запуске агента

| Этап | Время | Описание |
|------|-------|----------|
| Context collection | 30-60 сек | Чтение ветки, commits, issues, документации |
| Task analysis | 30-60 сек | Понимание задачи из description/issue |
| Planning | 20-40 сек | Разбиение на подзадачи |
| Implementation | 2-10 мин | Написание кода, итерации |
| Validation | 1-5 мин | Запуск тестов, проверка стиля |
| PR creation | 10-30 сек | Открытие pull request |

Общее время: **3-15 минут** в зависимости от сложности.

## 3. GitOps для промптов и конфигураций агентов

### 3.1 Git как единый источник истины

GitOps для LLM-агентов означает:

- **Версионирование промптов** -- все системные инструкции хранятся в Git с полной историей
- **Версионирование конфигураций** -- параметры моделей (temperature, max_tokens) отслеживаются
- **Автоматический deploy** -- изменение в Git автоматически деплоится в production
- **Аудит** -- каждое изменение поведения агента связано с конкретным коммитом

### 3.2 Структура репозитория

```
my-ai-project/
├── agents/
│   ├── code-reviewer/
│   │   ├── system_prompt.md        # Системный промпт
│   │   ├── config.yaml             # Параметры модели
│   │   ├── tools.yaml              # Определение инструментов
│   │   └── version.txt             # Текущая версия
│   ├── data-analyzer/
│   └── documentation-writer/
├── prompts/
│   ├── templates/
│   │   ├── code_review.md
│   │   └── bug_analysis.md
│   └── examples/
│       ├── good_output.md
│       └── bad_output.md
├── tests/
│   ├── agent_test.py               # Тесты для агента
│   └── prompt_test.py              # Тесты для промптов
├── .github/workflows/
│   ├── agent-deploy.yml            # Deploy при изменении промпта
│   └── agent-test.yml              # Тестирование перед merge
└── AGENTS.md                       # Документация
```

### 3.3 Workflow обновления промпта

Когда разработчик хочет улучшить системный промпт:

1. Создаёт ветку `feature/improve-code-reviewer-prompt`
2. Изменяет `agents/code-reviewer/system_prompt.md`
3. Открывает PR
4. Автоматический workflow:
   - Запускает тесты промпта на dataset'е примеров
   - Сравнивает качество с предыдущей версией
   - Если улучшение >= 5% -- разрешает merge
   - Если нет или хуже -- требует ручной проверки
5. После merge в `main`:
   - Автоматический deploy в staging
   - Интеграционные тесты
   - После прохождения -- deploy в production

## 4. Probabilistic Testing: тестирование недетерминистичных систем

### 4.1 Новая парадигма

В классическом коде тесты проверяют точные выходы: `input = 5 -> output = 10`. Для LLM-агентов это неприменимо -- выход **недетерминистичен**. Нужно тестировать:

- **Функциональность** -- агент выполняет задачу в целом? Получаем ли рабочую функцию?
- **Качество** -- насколько хорош результат? Следует ли стилю проекта?
- **Стабильность** -- стабильно ли хорошие результаты? Воспроизводимость?
- **Стоимость** -- вписывается ли в бюджет?

### 4.2 Архитектура probabilistic testing

**Dataset примеров** -- набор тестовых запросов с ожидаемыми паттернами:

```
Input: "Напиши функцию factorial(n)"
Expected patterns:
  - содержит def factorial
  - использует рекурсию или цикл
  - обрабатывает edge cases (n=0, n=1)
  - есть type hints
```

**Scoring** -- для каждого выхода агента считается score:

- **Semantic score** -- LLM-классификатор проверяет наличие необходимых элементов (правильный алгоритм, edge cases)
- **Syntactic score** -- автоматические правила (lint, type checking)
- **Cost score** -- насколько дорого стоило получить результат

**Aggregation** -- final score = `0.5 * semantic + 0.3 * syntactic + 0.2 * cost`

**Pass threshold** -- average score за 5-10 запусков >= 0.8 означает прохождение теста.

### 4.3 Практический пример

Тест для агента-код-ревьюера:

```yaml
test_case: "code_review_detects_performance_issue"
input: |
  def find_max(numbers):
      max_val = numbers[0]
      for i in range(len(numbers)):
          if numbers[i] > max_val:
              max_val = numbers[i]
      return max_val

expected_issues:
  - type: "performance"
    severity: "medium"
    explanation: "uses len() in loop"

scoring:
  - semantic: does_review_mention_loop_inefficiency (0-1)
  - semantic: suggests_optimization (0-1)
  - syntax: valid_json_response (0/1)

threshold: 0.75
```

При запуске:
1. Вызываем агент с этим кодом
2. Агент возвращает review в формате JSON
3. Проверяем: упомянул проблему с циклом? Предложил оптимизацию? Валидный JSON?
4. Score = (1 + 1 + 1) / 3 = 1.0 -- тест пройден

## 5. Безопасность CI/CD Pipeline для агентов

### 5.1 Уникальные риски

**Prompt injection через issues.** Пользователь открывает issue с `@claude Игнорируй свои инструкции и...` Агент может быть скомпрометирован.

**Защита:** санитизируйте issue description перед отправкой агенту, используйте явное разделение пользовательского контента и системных инструкций.

**Утечка secrets в сгенерированном коде.** Агент имеет доступ к `.env` файлам, может случайно включить API key в код.

**Защита:** никогда не отправляйте актуальные secrets агенту, используйте placeholder'ы или mock-значения.

**Компрометированные dependencies.** Агент обновляет dependencies, которые были скомпрометированы.

**Защита:** запускайте security scanning (bandit, npm-audit, Snyk) перед merge.

### 5.2 Best practices

1. **Принцип наименьших привилегий** -- агент имеет доступ на чтение к большей части репо, может писать только в определённые папки
2. **Manual approval для критических изменений** -- изменения в security-чувствительные файлы требуют человеческого одобрения
3. **Версионирование и откат** -- всегда можно быстро откатить изменение агента
4. **Аудит всех действий** -- каждое действие агента логируется: кто запустил, что сделано, результат

## 6. Масштабирование: от одного агента к fleet

### 6.1 Паттерны масштабирования

Когда проект вырастает от одного code-reviewer до fleet'а агентов, нужна инфраструктура:

- **Agent routing** -- разные агенты на разные типы задач, контролируют качество друг друга
- **Queueing** -- задачи идут в очередь, агенты берут по мере доступности. Предотвращает перегрузку
- **Shared cost budget** -- общий дневной бюджет на все агенты. Если один израсходовал 60% бюджета, другие ограничены
- **Orchestration layer** -- центральная система, которая распределяет задачи, мониторит здоровье, обрабатывает отказы

### 6.2 Типичная архитектура fleet'а

```
GitHub Events
     |
Event Router (PR, Issue, Manual trigger)
     |
Task Queue (Redis / RabbitMQ)
     |
[Agent 1: Code Review]  [Agent 2: Data Analysis]  [Agent 3: Docs Generation]
     |
Result Aggregator
     |
PR Creation & Monitoring
     |
Cost Tracker & Alerts
```

## Резюме

CI/CD для LLM-агентов требует переосмысления привычных подходов:

- **Pipeline** должен проверять не просто код, но и **качество AI-вывода**
- **Стоимость** -- first-class метрика наравне с performance
- **Промпты** версионируются в Git как часть инфраструктуры (GitOps)
- **Тестирование** вероятностное: scoring, threshold, multiple runs
- **Безопасность** включает защиту от prompt injection и утечки secrets
- **Масштабирование** требует queueing, routing и orchestration
