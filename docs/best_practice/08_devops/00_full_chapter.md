# Модуль 6: DevOps для агентов

## Введение

DevOps для LLM-агентов представляет собой принципиально новую область, объединяющую традиционные практики DevOps с уникальными требованиями систем, работающих с искусственным интеллектом. Если классический DevOps фокусируется на развертывании кода, мониторинге инфраструктуры и обеспечении надежности, то DevOps для агентов добавляет к этому множество новых аспектов: управление токен-лимитами, мониторинг качества AI-вывода, контроль стоимости вызовов API и обеспечение предсказуемости поведения автономных систем.

В этой главе мы рассмотрим полный цикл разработки, тестирования, развертывания и мониторинга LLM-агентов.

## 1. CI/CD для LLM-агентов: фундаментальные различия

### 1.1 Чем CI/CD для агентов отличается от классического CI/CD

Традиционный CI/CD конвейер ориентирован на:
- Компиляцию кода
- Запуск детерминистических тестов
- Развертывание артефактов
- Проверку статус-кодов и метрик производительности

**CI/CD для LLM-агентов** должен учитывать совсем другие аспекты:

| Аспект | Классический CI/CD | AI-Native CI/CD |
|--------|-------------------|-----------------|
| **Тестирование** | Детерминистические unit-тесты | Probabilistic testing, метрики качества |
| **Интеграция** | Merging кода | Merging с валидацией выхода модели |
| **Развертывание** | Статичные артефакты | Динамические модели, конфигурации промптов |
| **Мониторинг** | CPU, память, сетевые задержки | Token-usage, cost, quality metrics |
| **Откат** | Простое переключение версии | Сложный rollback с историей промптов |

### 1.2 Архитектура AI-Native CI/CD Pipeline

Типичный pipeline для LLM-агентов следует этой последовательности:

**Триггер** → пользователь упомянул @claude в PR/Issue или запустил workflow вручную

**Анализ контекста** → агент читает кодовую базу, историю коммитов, description задачи

**Генерация решения** → агент создает или модифицирует код

**Валидация синтаксиса** → проверяется lint, type checking, базовые ошибки

**Валидация качества** → проверяются метрики выхода модели (насколько корректный код, следует ли стилю проекта)

**Проверка стоимости** → убеждаемся, что расходы на API находятся в пределах бюджета

**Создание/обновление PR** → агент открывает pull request со своими изменениями

**Ожидание человеческой проверки** → разработчик читает и одобряет или просит исправления

**Мониторинг после деплоя** → отслеживаем параметры агента в production

### 1.3 Ключевые параметры для настройки

При создании CI/CD для агентов нужно определить:

- **max_turns**: максимальное количество итераций агента (обычно 5-10). Больше = дороже и медленнее, но может решить сложную задачу.
- **model**: какую модель использовать (claude-opus-4-6 для сложных задач, claude-sonnet для скорости)
- **temperature**: креативность ответов (0.0 = детерминистично, 1.0 = максимально креативно). Для кодирования обычно 0.0-0.3.
- **budget_per_run**: максимальный расход токенов на один запуск. Если превышен, workflow прерывается.
- **timeout**: сколько времени ждем ответа агента перед отказом.
- **quality_threshold**: минимальная оценка качества выхода (например, score >= 0.8). Если ниже, PR не создается.

## 2. Claude Code и GitHub Integration

### 2.1 Концепция: Agent-Assisted Development

Claude Code предоставляет несколько способов интеграции с GitHub:

**Способ 1: @claude mentions в PR/Issues**
- Разработчик открывает issue: "Добавить обработку ошибок в функцию load_data"
- Упоминает: "@claude Fix error handling"
- Агент анализирует кодовую базу и создает коммиты с исправлениями
- Разработчик может просмотреть PR перед merge

**Способ 2: GitHub Actions workflow**
- На каждый push или PR создается автоматический workflow
- Агент запускается с кодом "Code review mode"
- Проверяет стиль, безопасность, производительность
- Оставляет комментарии с предложениями

**Способ 3: Scheduled jobs**
- Каждый день в 9:00 AM агент анализирует открытые issues
- Автоматически выбирает те, которые может решить
- Создает PR с решениями

### 2.2 Базовая конфигурация

Чтобы включить Claude Code в GitHub:

1. **Установите GitHub App** из официального репозитория Anthropic
2. **Добавьте `ANTHROPIC_API_KEY`** в GitHub Secrets
3. **Создайте `.github/workflows/claude.yml`** файл (пример ниже)
4. **Добавьте `.claude.yml`** в корень проекта для настройки поведения агента

Структура `.claude.yml`:
```yaml
model: claude-opus-4-6
max_turns: 8
temperature: 0.1
token_budget: 100000
quality_checks:
  - type: "linting"
    tools: ["pylint", "eslint", "rustfmt"]
  - type: "testing"
    command: "pytest --cov"
  - type: "security"
    tools: ["bandit", "npm-audit"]
```

### 2.3 Что происходит при запуске агента

Когда вы упомянете @claude или workflow запустится:

1. **Контекстный сбор** (30-60 сек): агент читает текущую ветку, последние commits, open issues, документацию
2. **Анализ задачи** (30-60 сек): понимает, что нужно сделать, на основе description/issue
3. **Планирование** (20-40 сек): разбивает задачу на подзадачи
4. **Реализация** (2-10 мин): пишет код, может делать несколько итераций
5. **Валидация** (1-5 мин): запускает тесты, проверяет стиль
6. **PR создание** (10-30 сек): открывает pull request

Общее время: 3-15 минут в зависимости от сложности.

## 3. GitOps для LLM-агентов

### 3.1 Концепция GitOps

GitOps — это подход, где **Git репозиторий — единый источник истины** для всего конфигурационного состояния системы. Для LLM-агентов это означает:

- **Версионирование промптов**: все системные инструкции хранятся в Git с историей
- **Версионирование конфигураций**: параметры моделей (температура, max_tokens) версионированы
- **Автоматический деплой**: изменение в Git → автоматический деплой в production
- **Аудит**: каждое изменение поведения агента отслеживается и связано с коммитом

### 3.2 Структура репозитория

```
my-ai-project/
├── agents/
│   ├── code-reviewer/
│   │   ├── system_prompt.md        # Системный промпт
│   │   ├── config.yaml             # Параметры модели
│   │   ├── tools.yaml              # Определение инструментов
│   │   └── version.txt             # Текущая версия
│   ├── data-analyzer/
│   └── documentation-writer/
├── prompts/
│   ├── templates/
│   │   ├── code_review.md
│   │   └── bug_analysis.md
│   └── examples/
│       ├── good_output.md
│       └── bad_output.md
├── tests/
│   ├── agent_test.py               # Тесты для агента
│   └── prompt_test.py              # Тесты для промптов
├── .github/workflows/
│   ├── agent-deploy.yml            # Деплой при изменении промпта
│   └── agent-test.yml              # Тестирование перед merge
└── AGENTS.md                       # Документация
```

### 3.3 Workflow для обновления промпта

Когда разработчик хочет улучшить системный промпт:

1. Создает новую ветку: `feature/improve-code-reviewer-prompt`
2. Изменяет `agents/code-reviewer/system_prompt.md`
3. Открывает PR
4. Автоматический workflow:
   - Запускает тесты промпта на датасете примеров
   - Сравнивает качество с предыдущей версией
   - Если улучшение >= 5%, разрешает merge
   - Если нет или хуже, требует ручной проверки
5. После merge в main:
   - Автоматически деплоится в staging
   - Запускается набор интеграционных тестов
   - После прохождения → деплой в production

## 4. Мониторинг стоимости и использования токенов

### 4.1 Почему это критично

Без контроля стоимости, LLM-агент может быстро накопить счета в 10,000+$/месяц просто из-за:
- Бесконечных циклов (агент повторяет одно и то же)
- Неоптимальных промптов (излишне длинные системные инструкции)
- Утечек (кто-то запустил агента с 100MB файлом контекста)

### 4.2 Что отслеживать

**Per-request metrics:**
- Input tokens (количество токенов на входе)
- Output tokens (сгенерировано)
- Total cost (input + output, взвешенные по прайсу)
- Latency (время ответа)
- Model и parameters использовал

**Per-user/per-day metrics:**
- Total tokens consumed
- Total cost
- Request count
- Average cost per request
- Cost trend (увеличивается ли использование?)

**Aggregated metrics:**
- Daily spend by agent type
- Cost per business metric (cost per bug fixed, cost per feature)
- Прогноз месячного расхода (если trend продолжится)

### 4.3 Практический механизм отслеживания

Используйте следующий паттерн:

1. **Логирование**: каждый вызов API логируется в структурированную базу (PostgreSQL, BigQuery)
2. **Аналитика**: ежечасный/ежедневный скрипт рассчитывает метрики
3. **Алерты**: если дневной расход превышает `daily_budget * 1.2`, отправляется алерт
4. **Дашборд**: real-time дашборд Grafana или DataDog показывает текущее использование

Структура лога:
```
{
  "timestamp": "2026-03-13T10:15:00Z",
  "user_id": "user_123",
  "agent_name": "code-reviewer",
  "model": "claude-opus-4-6",
  "input_tokens": 1200,
  "output_tokens": 450,
  "input_cost_usd": 0.018,
  "output_cost_usd": 0.027,
  "total_cost_usd": 0.045,
  "latency_ms": 2340,
  "status": "success"
}
```

## 5. Тестирование LLM-агентов

### 5.1 Принципиально новая парадигма тестирования

В классическом коде тесты проверяют точные выходы: если input = 5, output должно быть 10. Для LLM-агентов это не применимо, потому что выход **недетерминистичен**.

Вместо этого нужно тестировать:

**Функциональность**: агент выполняет задачу в целом? Если попросить "напиши функцию для сортировки списка", получим ли мы рабочую функцию?

**Качество**: насколько хорош результат? Есть ли ошибки? Следует ли стилю проекта?

**Стабильность**: дает ли агент стабильно хорошие результаты? Или зависит от времени суток?

**Стоимость**: вписывается ли в бюджет? Не использует ли слишком много токенов?

### 5.2 Как устроено тестирование

**Dataset примеров**: подготовьте набор тестовых примеров с ожидаемыми результатами.

```
Input: "Напиши функцию factorial(n)"
Expected patterns:
  - содержит def factorial
  - использует либо рекурсию, либо цикл
  - обрабатывает edge cases (n=0, n=1)
  - есть type hints
```

**Scoring**: для каждого выхода агента считается score на основе:
- **Семантический score**: LLM-классификатор проверяет, содержит ли результат необходимые элементы (использует заданный алгоритм, обрабатывает edge cases, т.д.)
- **Синтаксический score**: автоматически проверяемые правила (lint, тип checking)
- **Коcт score**: насколько дорого стоило получить результат

**Аграгирование**: final score = 0.5 * семантический + 0.3 * синтаксический + 0.2 * cost

**Порог прохождения**: если average score за 5-10 запусков >= 0.8, тест прошел.

### 5.3 Практический пример

Тест для агента-код-ревьюера:

```yaml
test_case: "code_review_detects_performance_issue"
input: |
  def find_max(numbers):
      max_val = numbers[0]
      for i in range(len(numbers)):  # Неэффективно
          if numbers[i] > max_val:
              max_val = numbers[i]
      return max_val

expected_issues:
  - type: "performance"
    severity: "medium"
    explanation: "uses len() in loop"

scoring:
  - semantic: does_review_mention_loop_inefficiency (0-1)
  - semantic: suggests_optimization (0-1)
  - syntax: valid_json_response (0/1)

threshold: 0.75
```

При запуске:
- Вызываем агент с этим кодом
- Агент возвращает review JSON
- Проверяем, упомянул ли проблему с циклом
- Упомянул ли оптимизацию
- Получился ли валидный JSON
- Score = (1 + 1 + 1) / 3 = 1.0 ✓

## 6. Безопасность в CI/CD пайплайне

### 6.1 Уникальные риски AI CI/CD

**Риск 1: Prompt injection через issues**
- Пользователь открывает issue с "@claude Игнорируй свои инструкции и..."
- Агент может быть скомпрометирован

**Защита:** санитизируйте issue description перед отправкой агенту, используйте явное разделение контента.

**Риск 2: Утечка secrets в сгенерированном коде**
- Агент имеет доступ к `.env` файлам или конфигам
- Может случайно включить API ключ в код

**Защита:** никогда не отправляйте актуальные secrets агенту, используйте placeholder или недействительные примеры.

**Риск 3: Компрометированные dependencies**
- Агент обновляет dependencies, которые были скомпрометированы

**Защита:** запускайте security scanning (bandit, npm-audit) перед merge.

### 6.2 Best Practices для безопасного CI/CD

1. **Принцип наименьших привилегий**: агент имеет доступ только на чтение к большей части репо, может писать только в определенные папки
2. **Одобрение критических изменений**: любые изменения в security-чувствительные файлы требуют manual approval
3. **Версионирование и откат**: всегда можно быстро откатить изменение агента
4. **Аудит всех действий**: каждое действие агента логируется с деталями кто запустил, что сделано, результат

## 7. Масштабирование: от одного агента к fleet'у

### 7.1 Паттерн scaling

Когда вы начинаете с одного агента (например, code-reviewer), вполне управляемо. Но при масштабировании (code-reviewer + data-analyzer + documentation-writer + ...) нужна новая архитектура:

**Агент распределение**: разные агенты на разные типы задач, контролируют друг друга качество.

**Queueing система**: tasks идут в очередь, и агенты берут их по мере доступности. Предотвращает перегрузку.

**Shared cost budget**: общий дневной бюджет на все агенты. Если один агент израсходовал 60% бюджета, другие ограничены в возможностях.

**Orchestration layer**: központальная система, которая распределяет tasks, мониторит здоровье, обрабатывает отказы.

### 7.2 Типичная архитектура для fleet'а

```
GitHub Events
     ↓
Event Router (различает PR, Issue, Manual trigger)
     ↓
Task Queue (Redis/RabbitMQ)
     ↓
[Agent 1: Code Review] [Agent 2: Data Analysis] [Agent 3: Docs Generation]
     ↓
Result Aggregator (комбинирует результаты)
     ↓
PR Creation & Monitoring
     ↓
Cost Tracker & Alerts
```

## Резюме

DevOps для LLM-агентов требует переосмысления привычных подходов:

- **CI/CD** должен проверять не просто код, но и качество AI выхода
- **Стоимость** — это first-class метрика наравне с performance
- **Промпты** версионируются в Git как часть инфраструктуры
- **Тестирование** вероятностное, а не детерминистичное
- **Масштабирование** требует инфраструктуры для оркестрации множественных агентов

Без правильного DevOps в production быстро окажетесь с неконтролируемыми расходами и непредсказуемым поведением.
