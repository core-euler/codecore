# Подмодуль 4: MCP на практике и параллельные агенты

## Введение

В предыдущих уроках мы разобрали MCP как протокол и его экосистему. Этот урок — про практику: как MCP работает в реальных сценариях как мост между LLM-приложениями и внешними источниками, и как масштабировать работу с AI-агентами через параллельное выполнение задач.

Мы рассмотрим два взаимосвязанных направления:

1. **MCP на практике** — как сущности протокола (resources, tools, prompts) реализуются в реальных интеграциях, типичные потоки взаимодействия, транспорты и их trade-offs.
2. **Параллельные агенты** — зачем запускать несколько AI-агентов одновременно, какие инструменты для этого существуют, и как организовать изоляцию через git worktrees.

## 1. MCP как мост: практическая архитектура

### 1.1 Трёхуровневая модель на практике

MCP определяет три уровня взаимодействия, которые важно различать при проектировании интеграций:

| Уровень | Роль | Практический пример |
|---------|------|---------------------|
| **Host** | LLM-приложение, в котором живёт пользовательский опыт | Claude Desktop, Cursor, VS Code с Copilot |
| **Client** | Коннектор внутри host, поддерживающий 1:1 соединение с MCP server | Один экземпляр клиента на каждый подключённый сервер |
| **Server** | Процесс/сервис, предоставляющий tools, resources и prompts | MCP-сервер для GitHub, PostgreSQL, файловой системы |

Ключевой принцип: **один client — одно соединение с одним server**. Host может поднять множество client-ов для работы с разными серверами, сохраняя изоляцию и независимый lifecycle каждого соединения.

### 1.2 Типичный поток взаимодействия

На практике цикл работы через MCP выглядит так:

1. **Выбор источника** — host определяет, что нужно обратиться к внешним данным/действиям, и выбирает подходящий MCP-сервер (на основе конфигурации, эвристик или подсказки от LLM).

2. **Установка сессии** — обязательный handshake `initialize` → `notifications/initialized`. Сервер сообщает свои capabilities: какие resources, tools и prompts доступны.

3. **Discovery** — host запрашивает каталоги:
   - `resources/list` — что можно прочитать (файлы, схемы БД, документы)
   - `tools/list` — что можно выполнить (API-вызовы, вычисления, записи)
   - `prompts/list` — какие готовые шаблоны диалогов доступны

4. **Планирование** — LLM (и/или пользователь) решает, какие capabilities использовать, а host применяет политику безопасности.

5. **Получение данных / выполнение действий** — `resources/read`, `tools/call`, `prompts/get` возвращают результаты.

6. **Сборка контекста** — host упаковывает результаты в контекст для LLM.

7. **Итерация** — если LLM нужно больше данных, цикл повторяется.

### 1.3 Resources, Tools, Prompts — практические различия

На уровне протокола эти три сущности чётко разделены по назначению:

**Resources** — контекстные данные для чтения. Каждый resource идентифицируется URI (не обязательно веб-адрес — может быть `file://`, `git://`, `db://` и т.д.). Поддерживают подписки на изменения: сервер может уведомить клиента через `notifications/resources/updated`, когда содержимое изменилось.

```
Примеры URI:
file:///project/src/main.rs     — файл проекта
db://production/users/schema    — схема таблицы
git://repo/main/README.md       — файл из репозитория
```

**Tools** — действия с побочными эффектами. В отличие от resources, у tool есть `inputSchema` (JSON Schema) — формальный интерфейс функции. Tools описаны как **model-controlled**: LLM может инициировать вызов, но спецификация рекомендует human-in-the-loop для безопасности.

**Prompts** — готовые шаблоны диалогов. Описаны как **user-controlled**: пользователь явно выбирает prompt (например, как slash-команду). Сервер возвращает массив `messages`, который host может вставить в диалог с LLM.

### 1.4 Транспорты: когда что использовать

| Транспорт | Когда использовать | Особенности безопасности |
|-----------|-------------------|--------------------------|
| **stdio** | Локальные интеграции, desktop-приложения | Минимальная поверхность атаки; сервер — subprocess хоста |
| **Streamable HTTP** | Удалённые серверы, cloud-сервисы | Обязательна валидация Origin (защита от DNS rebinding); listen только на localhost для локальных серверов; аутентификация для удалённого доступа |

Практическое правило: если MCP-сервер работает на той же машине — stdio проще и безопаснее. Если сервер удалённый — Streamable HTTP с авторизацией.

### 1.5 Подписки и realtime

MCP предлагает два механизма реального времени:

1. **List changed** — уведомление о том, что каталог resources/tools/prompts изменился. Клиент в ответ перечитывает каталог через `*/list`. Работает для всех трёх сущностей.

2. **Resource subscriptions** — подписка на конкретный resource по URI. Сервер шлёт `notifications/resources/updated {uri}`, клиент перечитывает содержимое через `resources/read`. Работает только для resources.

На практике это важно для сценариев, когда внешние данные меняются в процессе работы: обновилась схема БД, появился новый файл в проекте, изменился статус задачи в трекере.

## 2. Реальные сценарии использования MCP

### 2.1 Интеграция с базами данных

MCP-сервер для PostgreSQL может предоставлять:
- **Resources**: схемы таблиц (`db://prod/users/schema`), результаты частых запросов
- **Tools**: выполнение SQL-запросов, миграции, создание индексов
- **Prompts**: шаблоны для анализа производительности запросов

LLM получает схему через `resources/read`, строит SQL-запрос и выполняет его через `tools/call` — всё в рамках одной MCP-сессии.

### 2.2 Работа с файловой системой и репозиториями

Типичный flow для code review через MCP:

1. MCP-сервер для GitHub предоставляет `tools/list` с инструментами: `get_pr_diff`, `list_pr_comments`, `create_review`
2. LLM вызывает `tools/call("get_pr_diff", {pr_number: 123})` — получает diff
3. Анализирует код и вызывает `tools/call("create_review", {comments: [...]})` — оставляет комментарии

### 2.3 Комбинирование серверов

Сила MCP — в возможности подключить несколько серверов одновременно. Один host может работать с:
- MCP-сервером файловой системы (чтение/запись кода)
- MCP-сервером для GitHub (PR, issues)
- MCP-сервером для Slack (уведомления команде)
- MCP-сервером для PostgreSQL (миграции)

Каждый сервер — изолированное соединение через отдельный client. Host агрегирует capabilities и предоставляет LLM единый «каталог возможностей».

## 3. Параллельные агенты: зачем и как

### 3.1 Проблема последовательности

Один AI-агент (например, Claude Code) работает последовательно: пока пишет тесты — frontend стоит; пока рефакторит API — документация ждёт. При работе над несколькими задачами одновременно это становится bottleneck.

**Что нужно от параллельности:**

1. **Изоляция** — агент A рефакторит auth-модуль, агент B пишет новый API endpoint. Они не должны конфликтовать на уровне файлов.
2. **Контроль** — возможность видеть, что каждый агент делает, переключаться между ними, смотреть diff.
3. **Merge** — когда оба закончили, собрать изменения вместе без конфликтов.

### 3.2 Git Worktrees — механизм изоляции

Все современные инструменты параллельных агентов используют один и тот же механизм — **git worktrees**:

```
# Обычный git — одна рабочая директория
~/my-project/             ← тут живёт код

# Git worktrees — несколько рабочих директорий, один репозиторий
~/my-project/             ← основная ветка (main)
~/my-project-wt-auth/     ← worktree для задачи auth
~/my-project-wt-api/      ← worktree для задачи api
~/my-project-wt-tests/    ← worktree для тестов
```

Каждый worktree — полноценная копия проекта со своей веткой, но **общим `.git`**. Один агент работает в одном worktree и не видит изменений в соседних. По завершении — создаётся PR и делается merge.

**Важный нюанс:** `node_modules`, `.env`, виртуальные окружения не копируются автоматически. Каждый worktree нужно «поднимать» заново — и именно здесь инструменты различаются.

### 3.3 Обзор инструментов для параллельных агентов

#### Claude Squad — стандарт для терминала

- **Интерфейс:** TUI (Terminal UI), требует tmux
- **Установка:** `brew install claude-squad`
- **Платформы:** macOS, Linux (Windows только через WSL2)
- **Лицензия:** AGPL-3.0

Управление через горячие клавиши:

| Клавиша | Действие |
|---------|----------|
| `n` | Новая сессия |
| `Enter` | Войти в сессию (видеть Claude Code вживую) |
| `Esc` | Выйти в менеджер |
| `d` | Показать diff |
| `p` | Создать PR |
| `D` | Удалить сессию |

**Уникальная фича** — мульти-агентность. Можно запускать не только Claude Code, но и другие агенты:

```bash
cs -p "aider --model ollama_chat/gemma3:1b"  # Aider с локальной моделью
cs -p "codex"                                  # Codex CLI
cs -p "gemini"                                 # Gemini CLI
```

Можно одновременно: Claude Code на рефакторинг, Aider на тесты, Gemini на документацию — всё в одном TUI.

**YOLO-режим** (`cs -y`) автоматически принимает все запросы на подтверждение. Экспериментальный — не рекомендуется для production-репозиториев.

#### CCManager — минимализм и кросс-платформенность

- **Интерфейс:** TUI (tmux НЕ нужен)
- **Установка:** `npx ccmanager`
- **Платформы:** все (включая Windows нативно)
- **Лицензия:** MIT

**Ключевые отличия от Claude Squad:**

1. **Safe Auto-Approval** — вместо YOLO, каждый промпт проверяется через Claude Haiku на безопасность. Безопасные команды принимаются автоматически, подозрительные — требуют подтверждения.

2. **Worktree Hooks** — автоматизация настройки окружения:
```json
{
  "worktreeHooks": {
    "postCreate": ["npm install", "cp ../.env .env"]
  }
}
```

3. **Копирование контекста** — при создании нового worktree можно скопировать session data из существующего.

4. **7 поддерживаемых агентов** — Claude Code, Gemini CLI, Codex CLI, Cursor Agent, Copilot CLI, Cline CLI, OpenCode.

#### Crystal — лучший open-source GUI

- **Интерфейс:** GUI (Electron)
- **Установка:** `brew install --cask stravu-crystal`
- **Платформы:** все
- **Лицензия:** MIT

Позиционируется как «Integrated Vibe Environment». Предоставляет:
- Diff viewer с syntax highlighting
- Desktop notifications (когда агент закончил или ждёт ввода)
- Auto-commit каждой итерации
- SQLite для хранения conversation history между сессиями

#### Auto-Claude — максимальная автономность

- **Интерфейс:** GUI (Electron) + CLI headless
- **Stars:** ~11 200 (самый популярный)
- **Лицензия:** AGPL-3.0

Единственный инструмент, реализующий полный pipeline «fire and forget»:

```
Фаза 1: Discovery      → Анализ кодовой базы
Фаза 2: Requirements    → Формулировка требований
Фаза 3: Context         → Сбор контекста из проекта
Фаза 4: Spec            → Создание спецификации
Фаза 5: Implementation  → Написание кода
Фаза 6: QA Review       → Проверка acceptance criteria
Фаза 7: QA Fix          → Автоматическое исправление
Фаза 8: Validate        → Финальная валидация
```

**QA Reviewer Agent** проверяет результат, **QA Fixer Agent** исправляет ошибки. Self-correction loop предотвращает бесконечные повторы.

Встроенный **Graphiti + LadybugDB** — knowledge graph для persistent memory между сессиями. Нативная интеграция с Linear и GitHub Issues.

#### Conductor — venture-backed решение

- **Интерфейс:** GUI (macOS native)
- **Лицензия:** проприетарный, бесплатный
- **Финансирование:** $2.8M от Y Combinator

Единственный инструмент с нативной интеграцией Linear. Автоматически создаёт worktree, запускает агента, показывает diff в реальном времени и создаёт PR. Только macOS, только GitHub.

#### Cursor 2.0 — параллельные агенты в IDE

- **Интерфейс:** GUI (VS Code fork)
- **Цена:** Free / $20 Pro / $60 Pro+ / $200 Ultra

До 8 агентов одновременно, каждый в изолированном git worktree. Aggregated diff view. Self-healing: агент запускает код, проверяет output, при ошибке — исправляет и повторяет.

| Критерий | Claude Squad | CCManager | Crystal | Auto-Claude | Conductor | Cursor 2.0 |
|----------|-------------|-----------|---------|-------------|-----------|------------|
| **Установка** | `brew install` | `npx ccmanager` | `brew --cask` | 15-20 мин | DMG | Installer |

## 4. Практические паттерны параллельной работы

### 4.1 Декомпозиция задач

Эффективная параллельность начинается с правильной декомпозиции. Задачи для параллельных агентов должны быть:

- **Изолированы по файлам** — агенты не должны редактировать одни и те же файлы
- **Независимы по логике** — результат одного агента не должен блокировать другого
- **Объединяемы** — результаты должны merge-иться без сложных конфликтов

Хорошие кандидаты для параллелизации:
- Backend API + Frontend UI (разные директории)
- Код + Тесты для другого модуля
- Рефакторинг модуля A + Новая функциональность в модуле B
- Код + Документация

Плохие кандидаты:
- Два агента редактируют один и тот же файл
- Задача B зависит от результата задачи A
- Глобальный рефакторинг (затрагивает весь проект)

### 4.2 Workflow с worktrees

Типичный workflow параллельной работы:

```bash
# 1. Запуск менеджера
cs                          # Claude Squad
# или
npx ccmanager               # CCManager

# 2. Создание сессий с задачами
# Сессия 1: "Рефакторинг auth-модуля: перевести на JWT"
# Сессия 2: "Написать API endpoint для /api/v2/users"
# Сессия 3: "Покрыть тестами модуль payments"

# 3. Мониторинг прогресса
# Переключение между сессиями, просмотр diff

# 4. Review и merge
# Каждая сессия → PR → code review → merge в main
```

### 4.3 Решение проблемы зависимостей worktree

Главная практическая проблема worktrees — зависимости проекта не копируются автоматически. Решения:

**CCManager (рекомендуемый путь):**
```json
{
  "worktreeHooks": {
    "postCreate": [
      "npm install",
      "cp ../.env .env",
      "cp -r ../venv ./venv"
    ]
  }
}
```

**Claude Squad (ручной или скриптовый подход):**
```bash
# Скрипт setup-worktree.sh
#!/bin/bash
npm install
cp ../main-project/.env .env
```

**Symlinks для тяжёлых зависимостей:**
```bash
ln -s /path/to/main/node_modules ./node_modules
```

### 4.4 Review bottleneck

Запустить 10 агентов — легко. Ревьюить 10 PR — нет. Как выразился Simon Willison: *«The natural bottleneck is how fast I can review the results.»*

Стратегии смягчения:
- **Маленькие задачи** — проще ревьюить 10 маленьких PR, чем 3 больших
- **QA-агенты** — Auto-Claude использует отдельного QA Reviewer Agent для автоматической проверки
- **Стандартизация** — чёткие acceptance criteria в промпте уменьшают количество итераций

## 6. Тренды и куда всё движется

1. **Git worktrees — стандарт де-факто.** Все инструменты параллельных агентов используют один механизм изоляции.

2. **От мультиплексеров к платформам.** Ранние инструменты (Claude Squad, CCManager) просто мультиплексировали сессии. Новые (Auto-Claude, Emdash) строят полные pipeline с планированием, QA и memory.

3. **Рост экосистемы.** Год назад инструментов для параллельных агентов не существовало. Сейчас их десятки, включая venture-backed проекты (Conductor — $2.8M от YC, Emdash — YC W26).

4. **Claude Code Tasks** (v2.1.16) — Anthropic встроил Tasks для длительных автономных проектов прямо в Claude Code. Для простых сценариев внешний оркестратор может стать не нужен. Но для параллельности — пока нет встроенной альтернативы.

5. **MCP как единый интерфейс.** Параллельные агенты всё активнее используют MCP-серверы для интеграций: GitHub (PR, issues), Linear (задачи), файловая система — всё через стандартизированный протокол.

## Итоги

**MCP на практике** — это не абстрактный протокол, а рабочий мост между LLM и внешними системами. Три типа сущностей (resources для чтения, tools для действий, prompts для шаблонов) покрывают основные сценарии интеграции. Выбор транспорта (stdio vs Streamable HTTP) определяется тем, локальный сервер или удалённый.

**Параллельные агенты** решают проблему масштабирования: один агент — один разработчик, несколько агентов в изолированных worktrees — команда. Ключевые инструменты:
- **Claude Squad** — для терминала и мульти-агентных сценариев
- **CCManager** — для минимализма и кросс-платформенности
- **Crystal** — для GUI с open-source
- **Auto-Claude** — для максимальной автономности
- **Conductor** — для macOS + Linear
- **Cursor 2.0** — для тех, кто не хочет покидать IDE

Главный bottleneck параллельных агентов — не запуск, а review результатов. Декомпозиция на маленькие изолированные задачи и автоматизация QA — ключ к эффективности.
