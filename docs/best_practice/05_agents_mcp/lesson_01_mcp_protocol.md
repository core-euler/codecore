# Модуль 3, Подмодуль 1: MCP — Протокол и архитектура

## Введение

Model Context Protocol (MCP) — это революционный протокол, который переопределяет способ взаимодействия больших языковых моделей с внешним миром. Если вы когда-либо задумывались о том, как искусственный интеллект может безопасно и эффективно подключаться к различным инструментам, API, базам данных и сервисам, то ответ находится именно здесь.

MCP существует как универсальный переводчик для AI-приложений. Подобно тому, как разъемы USB позволяют подключать любое устройство к компьютеру стандартным способом, MCP позволяет языковым моделям подключаться к любому инструменту или сервису унифицированным способом. Это фундаментальное изменение в архитектуре AI-систем, которое устраняет необходимость в написании уникального кода интеграции для каждой пары модель-сервис.

## 1. Что такое MCP — Model Context Protocol

### 1.1 Определение и назначение

Model Context Protocol (MCP) — это стандартизированный протокол, определяющий способ коммуникации между AI-клиентом (чаще всего языковой моделью) и различными серверами инструментов. Протокол был разработан для решения критической проблемы в экосистеме AI: отсутствия унифицированного способа для моделей получать доступ к внешним ресурсам.

До появления MCP каждый раз, когда разработчик хотел дать языковой модели доступ к новому сервису, инструменту или API, требовалось:

1. Написать специальный код для интеграции этого конкретного сервиса
2. Адаптировать этот код для каждой конкретной модели (Claude, GPT, etc.)
3. Обрабатывать специфичные для этого сервиса ошибки и сложности
4. Поддерживать и обновлять код при изменении API сервиса
5. Реализовывать безопасность и аутентификацию отдельно для каждого случая

Это приводило к экспоненциальному росту сложности и затрат.

### 1.2 Основная проблема, которую решает MCP

**Проблема: Фрагментация инструментов**

Представьте себе ситуацию: вы разработали мощного AI-агента, который может работать с GitHub, Notion, Slack, базой данных PostgreSQL и несколькими другими сервисами. Без MCP вы столкнулись бы с множеством проблем:

- Каждый сервис требует собственного кода интеграции
- Если вы захотите использовать другую модель, придется переписывать все интеграции
- Безопасность реализуется по-разному для каждого сервиса
- Обновления в API одного сервиса могут сломать всю систему
- Тестирование и отладка становятся чрезвычайно сложными

**Решение: Стандартизированный протокол**

MCP решает эту проблему, введя единый стандартизированный способ описания того, что может делать инструмент:

- **Унификация**: Один и тот же MCP-сервер работает с Claude, ChatGPT, и любой другой моделью, поддерживающей MCP
- **Масштабируемость**: Добавление нового инструмента — это вопрос создания нового MCP-сервера, а не переписывания существующего кода
- **Безопасность по умолчанию**: MCP включает встроенные механизмы безопасности и контроля доступа
- **Стандартизация**: Все инструменты описываются единообразно, что упрощает тестирование, отладку и документацию

### 1.3 Ключевые преимущества MCP

1. **Модульность и масштабируемость**: Каждый инструмент или сервис может быть реализован как отдельный MCP-сервер, не влияя на другие
2. **Независимость от модели**: Один MCP-сервер работает со всеми моделями, поддерживающими протокол
3. **Безопасность**: MCP включает встроенные механизмы для управления доступом и аутентификации
4. **Стандартизация**: Единая спецификация для всех инструментов упрощает разработку и интеграцию
5. **Экономичность**: Вместо разработки N × M интеграций (где N — количество моделей, M — количество сервисов), требуется только M MCP-серверов
6. **Надежность**: Стандартный протокол позволяет создавать надежные и предсказуемые системы

## 2. Архитектура MCP: Host, Client, Server

Архитектура MCP основана на классической парадигме «клиент-сервер», но с определенными специфическими расширениями для работы с AI-моделями. Понимание этой архитектуры критично для работы с MCP.

### 2.1 Три компонента архитектуры

#### 2.1.1 MCP Server (Сервер)

**Что это**: MCP Server — это приложение, которое предоставляет определенные возможности (tools, resources, prompts) другим приложениям через протокол MCP.

**Ключевые характеристики**:
- Реагирует на запросы от MCP Client
- Предоставляет описание своих возможностей (capabilities)
- Выполняет требуемые действия и возвращает результаты
- Может работать локально или удаленно
- Может использовать различные транспорты для коммуникации (stdio, HTTP, SSE, etc.)

**Примеры MCP-серверов в экосистеме**:
- Filesystem Server — предоставляет инструменты для работы с файловой системой
- GitHub Server — управление репозиториями, issues, pull requests
- Notion Server — доступ к базам данных Notion
- Database Server — выполнение SQL запросов к PostgreSQL, MySQL, SQLite
- Web Search Server — поиск информации в интернете
- Stripe Server — работа с платежной системой Stripe

**Как работает**: MCP Server слушает входящие соединения, обрабатывает запросы от Client, выполняет требуемые операции и отправляет результаты обратно.

#### 2.1.2 MCP Client

**Что это**: MCP Client — это компонент, который инициирует соединение с MCP Server и использует предоставляемые им возможности. Обычно MCP Client встроен в приложение или языковую модель.

**Ключевые характеристики**:
- Инициирует соединение с Server
- Запрашивает список доступных возможностей (tools, resources, prompts)
- Вызывает инструменты и получает результаты
- Может управлять множественными Server соединениями
- Обрабатывает ошибки и переподключения

**Примеры MCP Client**:
- Claude (от Anthropic) — встроенная поддержка MCP
- ChatGPT (от OpenAI) — поддержка через MCP интеграцию
- VS Code — встроенная поддержка MCP для расширений
- Custom AI приложения, использующие mcp-use SDK

**Как работает**: MCP Client устанавливает соединение с одним или несколькими Server, получает информацию об их возможностях, и затем использует эти возможности по мере необходимости.

#### 2.1.3 MCP Host (Хост)

**Что это**: MCP Host — это приложение или платформа, которая содержит MCP Client и управляет его жизненным циклом, а также жизненным циклом соединений с MCP Servers.

**Ключевые характеристики**:
- Запускает и управляет MCP Server процессами
- Запускает MCP Client и управляет его операциями
- Обрабатывает конфигурацию соединений
- Управляет ресурсами и подключениями
- Может запускать множественные Server процессы

**Примеры MCP Host**:
- VS Code (когда работает с MCP расширениями)
- Claude (когда подключается к MCP серверам)
- Custom приложение на Python или TypeScript, использующее MCP SDK
- Облачная платформа, развертывающая MCP сервер

**Как работает**: MCP Host читает конфигурацию, запускает необходимые Server процессы, создает Client соединения, и координирует всю коммуникацию.

### 2.3 Коммуникационный цикл

1. **Инициализация**: MCP Host создает Client, который устанавливает соединение с Server
2. **Discover**: Client запрашивает список доступных tools, resources, и prompts у Server
3. **Execute**: При необходимости Client вызывает конкретный tool на Server
4. **Response**: Server обрабатывает вызов и возвращает результат
5. **Processing**: Client обрабатывает результат и возвращает его в Host
6. **Shutdown**: При завершении Host закрывает соединения и останавливает Server процессы

## 3. Транспорты: способы коммуникации

Один из ключевых аспектов MCP — это возможность использования разных транспортных протоколов для коммуникации между Client и Server. Это обеспечивает гибкость в различных сценариях использования.

### 3.1 Stdio (Standard Input/Output)

**Что это**: Stdio транспорт использует стандартные потоки ввода-вывода процесса для коммуникации между MCP Client и Server.

**Как это работает**:
- MCP Host запускает MCP Server как дочерний процесс
- Server читает JSON-RPC сообщения из stdin (стандартный ввод)
- Server пишет JSON-RPC ответы в stdout (стандартный выход)
- Client и Server обмениваются структурированными JSON сообщениями

**Преимущества**:
- Локальный транспорт — идеален для локальных инструментов
- Нет необходимости в сетевой конфигурации
- Простота реализации для локальных серверов
- Автоматическое управление жизненным циклом процесса

**Недостатки**:
- Работает только для локальных процессов
- Не подходит для удаленных сервером
- Один Client может подключиться к одному Server процессу

**Пример конфигурации в VS Code mcp.json**:

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "node",
      "args": ["/path/to/filesystem-server.js"]
    }
  }
}
```

### 3.2 SSE (Server-Sent Events)

**Что это**: SSE транспорт использует HTTP Server-Sent Events для установки однонаправленного соединения от Server к Client, с обычными HTTP POST запросами от Client к Server.

**Как это работает**:
1. Client отправляет HTTP POST запрос к Server
2. Server обрабатывает запрос и отправляет ответ в HTTP ответе
3. Server отправляет асинхронные события клиенту через SSE соединение
4. Client получает события через SSE потока

**Преимущества**:
- Работает с удаленными серверами через HTTP
- Хорошо работает за прокси и файрволами (использует HTTP)
- Поддерживает множество одновременных клиентов
- Слабее требования к сетевой конфигурации, чем Websockets

**Недостатки**:
- Требует настройки HTTP сервера
- Однонаправленная коммуникация от сервера (требует POST для запросов)
- Может быть менее подходящим для высокочастотной коммуникации

**Реализация MCP-сервера:**

MCP-сервер — это отдельный процесс, который инкапсулирует функциональность и предоставляет её через стандартный протокол MCP. Сервер определяет набор инструментов, ресурсов и промптов, регистрирует их и делает доступными для клиента через интерфейс MCP.

При создании MCP-сервера на практике используются специальные SDK (Software Development Kit) для выбранного языка программирования. Эти SDK предоставляют удобный API для определения инструментов, обработки запросов и отправки результатов. Разработчик фокусируется на бизнес-логике (что делает инструмент), а SDK заботится о протоколе и коммуникации.

Жизненный цикл сервера простой: инициализация → регистрация инструментов → запуск слушателя → обработка запросов в цикле → выключение. Этот процесс одинаков независимо от языка программирования или транспортного механизма.


### 3.3 Streamable HTTP (HTTP с потоковой передачей)

**Что это**: Streamable HTTP — это более современный транспорт, который использует HTTP для коммуникации с поддержкой потоковой передачи данных в обоих направлениях.

**Как это работает**:
1. Client и Server используют HTTP для основной коммуникации
2. Ответы могут быть потоковыми для поддержки передачи больших объемов данных
3. Поддерживает как запросы от Client к Server, так и асинхронные события от Server к Client
4. Более современный и гибкий подход, чем SSE

**Преимущества**:
- Двусторонняя коммуникация через HTTP
- Поддержка потоковой передачи данных
- Лучше подходит для больших объемов данных
- Работает с удаленными серверами
- Хорошо работает за прокси и файрволами

**Недостатки**:
- Требует правильной конфигурации HTTP сервера
- Может быть более сложным в отладке, чем stdio
- Требует аутентификации для удаленных серверов

### 3.4 Сравнение транспортов

| Критерий | Stdio | SSE | Streamable HTTP |
|----------|-------|-----|-----------------|
| Локальный | Да | Нет | Нет |
| Удаленный | Нет | Да | Да |
| Двусторонняя коммуникация | Да | Ограниченно | Да |
| Сложность | Низкая | Средняя | Средняя |
| Масштабируемость | Низкая | Средняя | Высокая |
| Потоковая передача | Нет | Нет | Да |
| Требует аутентификации | Нет | Опционально | Рекомендуется |

## 4. Capabilities — возможности MCP-сервера

MCP-сервер может предоставлять различные типы возможностей (capabilities) для использования Client'ами. Каждый тип возможности представляет собой разные способы взаимодействия с серверов.

### 4.1 Tools (Инструменты)

**Что это**: Tools — это функции или операции, которые может выполнить MCP Server по запросу Client'а.

**Характеристики**:
- Tools имеют четко определенную схему входных параметров (JSON Schema)
- Tools выполняют определенное действие и возвращают результат
- Tools могут иметь побочные эффекты (модифицировать файлы, отправлять запросы и т.д.)
- Tools описываются во время инициализации

**Примеры Tools**:
- `read_file` — прочитать содержимое файла
- `write_file` — записать в файл
- `execute_sql_query` — выполнить SQL запрос
- `search_github_issues` — поиск issues на GitHub
- `send_slack_message` — отправить сообщение в Slack
- `get_weather` — получить прогноз погоды

**Вызов Tool'а из Client'а**:

Когда модель решает вызвать инструмент, она выполняет следующий процесс. Сначала модель анализирует описание и схему инструмента, подготавливает аргументы согласно схеме, затем клиент отправляет запрос серверу с именем инструмента и аргументами. Сервер находит соответствующий обработчик, выполняет функцию с полученными аргументами и возвращает результат.

На уровне протокола MCP это выглядит как JSON-RPC запрос: клиент отправляет объект с методом "tools/call", именем инструмента и аргументами, сервер обрабатывает и возвращает результат либо ошибку. Этот обмен зашифрован и подписан для безопасности.


### 4.2 Resources (Ресурсы)

**Что это**: Resources — это статичные или динамичные данные, которые Client может читать от Server'а. В отличие от Tools, Resources предназначены для чтения данных, а не для выполнения действий.

**Характеристики**:
- Resources представляют статичные или медленно меняющиеся данные
- Resources имеют URI (универсальный идентификатор ресурса)
- Resources могут быть текстовыми, бинарными, или иметь другой формат
- Клиент может запросить содержимое Resource'а
- Resources могут быть перечислены (list operation)

**Примеры Resources**:
- `file://` — содержимое файла
- `notion://database123` — таблица в Notion
- `database://users` — метаданные таблицы базы данных
- `docs://api-reference` — документация API
- `git://repo/readme` — содержимое README файла

## 5. Жизненный цикл соединения MCP

Соединение между MCP Client и Server проходит через несколько четко определенных этапов. Понимание этих этапов важно для правильной имплементации и отладки MCP систем.

### 5.1 Initialize (Инициализация)

**Что происходит**:
1. MCP Client инициирует соединение с MCP Server
2. Client отправляет `initialize` запрос с информацией о себе
3. Server получает запрос и инициализирует свое состояние
4. Server отправляет `initialize` ответ с информацией о своих возможностях

**Пример JSON-RPC сообщения (инициация от Client'а)**:

При инициализации соединения клиент отправляет JSON-RPC сообщение с указанием своего имени, версии и поддерживаемых протокольных версий. Сообщение имеет стандартную JSON-RPC структуру с полями jsonrpc (версия спецификации), id (идентификатор запроса для корреляции), method (название операции) и params (параметры).

**Пример ответа Server'а**:

Сервер отвечает аналогичным JSON-RPC сообщением, в котором указывает свое имя, версию и свои возможности (capabilities) — список типов инструментов, ресурсов и других функций, которые он предоставляет. Это позволяет клиенту узнать, что доступно и как это использовать.


### 5.2 Capabilities Exchange (Обмен возможностями)

**Что происходит**:
1. После инициализации обе стороны знают о возможностях друг друга
2. Client может запросить полный список доступных tools, resources, и prompts
3. Server предоставляет детальное описание каждой возможности

**Получение списка Tools**:

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "tools/list",
  "params": {}
}
```

**Ответ сервера со списком tools**:

Сервер отвечает JSON-RPC сообщением, в котором содержится массив инструментов. Каждый инструмент описывается объектом с полями: name (идентификатор инструмента), description (человеко-читаемое описание назначения), и inputSchema (JSON Schema, определяющая структуру входных параметров — типы, обязательные поля, диапазоны значений и т.д.). Эта информация позволяет модели понять, когда и как вызвать каждый инструмент.


### 5.3 Normal Operation (Обычная работа)

**Что происходит**:
1. Client вызывает различные tools, читает resources, получает prompts
2. Server выполняет запрошенные операции
3. Server отправляет результаты обратно Client'у
4. Процесс повторяется столько раз, сколько необходимо

**Пример вызова tool'а**:

```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "method": "tools/call",
  "params": {
    "name": "read_file",
    "arguments": {
      "path": "/home/user/document.txt"
    }
  }
}
```

**Ответ Server'а**:

```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "This is the content of the file..."
      }
    ]
  }
}
```

**Обработка ошибок**:

```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "error": {
    "code": -32001,
    "message": "File not found",
    "data": {
      "path": "/nonexistent/file.txt"
    }
  }
}
```

### 5.4 Shutdown (Завершение)

**Что происходит**:
1. Client отправляет запрос на завершение соединения
2. Server прерывает текущие операции
3. Server выполняет cleanup (закрывает файлы, соединения и т.д.)
4. Соединение закрывается

**Пример shutdown сообщения**:

```json
{
  "jsonrpc": "2.0",
  "method": "shutdown"
}
```

**Диаграмма жизненного цикла соединения**:

В production среде скрипты управления обычно становятся частью систем оркестрации (systemd, supervisor, Kubernetes), которые гарантируют, что сервер всегда запущен и перезапускается при сбое.
