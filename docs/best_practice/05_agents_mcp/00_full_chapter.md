# Модуль 3: Агенты и MCP (Model Context Protocol)

## Введение

В предыдущих модулях мы изучили основы работы с языковыми моделями и процесс их взаимодействия с инструментами. Теперь мы поднимемся на следующий уровень абстракции: создание интеллектуальных агентов, которые могут автономно решать сложные задачи, используя стандартизированный протокол взаимодействия с инструментами.

Этот модуль охватывает два критически важных компонента современной экосистемы AI:

1. **Model Context Protocol (MCP)** — универсальный стандарт для подключения моделей к инструментам, сервисам и данным
2. **Agent Skills** — модульные, переиспользуемые пакеты знаний, которые трансформируют общие агенты в специалистов

На практике мы создадим собственный MCP-сервер, разработаем Skills для своих агентов и интегрируем их с популярными фреймворками.

## 1. Model Context Protocol (MCP): Основы

### 1.1 Что такое MCP?

Model Context Protocol — это открытый стандарт, разработанный Anthropic, который унифицирует взаимодействие между AI-моделями и различными инструментами, сервисами и источниками данных.

Представьте USB как стандарт: вместо того чтобы разные устройства имели свои уникальные разъемы, USB обеспечивает единый способ подключения любых периферийных устройств к компьютеру. MCP работает аналогично для AI: вместо написания уникального кода для подключения каждой модели к каждому инструменту, MCP определяет универсальный способ установления этого соединения.

**Основные преимущества MCP:**

- **Стандартизация**: Одна реализация MCP-сервера работает с любой совместимой моделью или клиентом
- **Модульность**: Инструменты, ресурсы и функции кодируются отдельно и комбинируются по необходимости
- **Масштабируемость**: От простых локальных инструментов до распределенных облачных сервисов
- **Безопасность**: Явный контроль над тем, какие инструменты доступны и при каких условиях

### 1.2 Архитектура MCP: Host, Client, Server

MCP использует трехуровневую архитектуру, которая обеспечивает четкое разделение ответственности:

```
┌─────────────────────────────────────────────────────────┐
│                    MCP Host                              │
│              (Claude Code, IDE, приложение)             │
└─────────────────────────────────────────────────────────┘
                    ↓ ↑ (MCP протокол)
┌─────────────────────────────────────────────────────────┐
│                   MCP Client                             │
│           (встроено в host для коммуникации)            │
└─────────────────────────────────────────────────────────┘
           ↓ ↑ (stdio, SSE, HTTP)
┌─────────────────────────────────────────────────────────┐
│                   MCP Server                             │
│         (инструменты, ресурсы, промптовые наборы)      │
└─────────────────────────────────────────────────────────┘
```

**Host (Хост)**: Приложение, использующее MCP-клиент для доступа к инструментам
- Claude Code, IDE (VS Code, Cursor)
- Веб-приложения, desktop-приложения
- Кастомные агенты

**Client (Клиент)**: Инициирует запросы и управляет соединением с сервером
- Обнаруживает доступные инструменты через `list_tools`
- Отправляет запросы на выполнение через `call_tool`
- Управляет ресурсами и контекстом

**Server (Сервер)**: Экспортирует функциональность через инструменты и ресурсы
- Определяет и реализует инструменты (tools)
- Предоставляет ресурсы и контекстную информацию (resources)
- Обрабатывает запросы от клиента

### 1.3 Транспорты MCP

MCP поддерживает несколько транспортных механизмов для гибкого развертывания:

#### Stdio (Standard Input/Output)

Используется для локального взаимодействия через стандартные потоки ввода-вывода.

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "node",
      "args": ["mcp-server-filesystem.js"],
      "env": {
        "LOG_LEVEL": "debug"
      }
    }
  }
}
```

**Преимущества:**
- Простота развертывания на локальных машинах
- Естественная изоляция процессов
- Поддержка переменных окружения

**Недостатки:**
- Работает только на одной машине
- Требует управления жизненным циклом процесса

#### SSE (Server-Sent Events)

Используется для асинхронного потока данных от сервера к клиенту через HTTP.

```json
{
  "mcpServers": {
    "remote-tools": {
      "type": "sse",
      "url": "https://api.example.com/mcp/sse",
      "headers": {
        "Authorization": "Bearer your-token"
      }
    }
  }
}
```

**Преимущества:**
- Работает через сетевые границы
- Поддерживает аутентификацию и авторизацию
- Низкая задержка для удаленных сервисов

**Недостатки:**
- Требует HTTPS для безопасного взаимодействия
- Одностороннее соединение (сервер → клиент инициирует)

#### Streamable HTTP

Расширенная версия HTTP для потоковой передачи данных и обработки длительных операций.

```json
{
  "mcpServers": {
    "advanced-tools": {
      "type": "streamable",
      "url": "https://api.example.com/mcp",
      "headers": {
        "Authorization": "Bearer token"
      }
    }
  }
}
```

**Преимущества:**
- Поддержка потоковой передачи и долгих операций
- Двусторонняя коммуникация через HTTP
- Совместимость с облачными сервисами

### 1.4 Основные концепции MCP

#### Tools (Инструменты)

Функции, которые модель может вызывать для выполнения действий.



**Реализация MCP-сервера:**

MCP-сервер — это отдельный процесс, который инкапсулирует функциональность и предоставляет её через стандартный протокол MCP. Сервер определяет набор инструментов, ресурсов и промптов, регистрирует их и делает доступными для клиента через интерфейс MCP.

При создании MCP-сервера на практике используются специальные SDK (Software Development Kit) для выбранного языка программирования. Эти SDK предоставляют удобный API для определения инструментов, обработки запросов и отправки результатов. Разработчик фокусируется на бизнес-логике (что делает инструмент), а SDK заботится о протоколе и коммуникации.

Жизненный цикл сервера простой: инициализация → регистрация инструментов → запуск слушателя → обработка запросов в цикле → выключение. Этот процесс одинаков независимо от языка программирования или транспортного механизма.


**Каждый инструмент определяет:**
- `name` — уникальный идентификатор
- `description` — когда и как использовать
- `schema` — входные параметры (JSON Schema)
- `handler` — реализация функции

#### Resources (Ресурсы)

Структурированные данные, которые модель может читать для получения контекста.

```typescript
server.resource({
  uri: "documents://api-spec.md",
  name: "API Documentation",
  mimeType: "text/markdown"
}, async (uri) => {
  return "# API Specification\n\n...";
});
```

#### Prompts (Промптовые наборы)

Переиспользуемые шаблоны промптов с параметрами для специальных случаев.


Промптовые наборы в MCP — это переиспользуемые шаблоны с параметрами для специальных случаев. Сервер может предоставить готовые промпты для различных задач (code review, debugging, planning и т.д.), которые модель может использовать для лучшего выполнения работы. Параметры в промптах могут быть подставлены динамически на основе контекста.

## 2. Экосистема MCP-серверов

### 2.1 Категории серверов

Существует богатая экосистема готовых MCP-серверов, организованных по категориям:

#### Работа с файлами и документами

| Сервер | Описание |
|--------|---------|
| **filesystem** | Навигация по файловой системе, чтение/запись файлов |
| **pdf** | Извлечение текста из PDF, работа с формами |
| **office** | Excel, Word, PowerPoint документы |
| **markdown** | Преобразование форматов в Markdown |

#### Разработка и контроль версий

| Сервер | Описание |
|--------|---------|
| **git** | Git репозитории, коммиты, ветки |
| **github** | GitHub API, issues, PR, workflows |
| **gitlab** | GitLab интеграция |
| **npm** | Поиск пакетов, анализ зависимостей |

#### Базы данных и хранилище

| Сервер | Описание |
|--------|---------|
| **postgres** | PostgreSQL запросы и управление |
| **mongodb** | MongoDB операции и аналитика |
| **supabase** | Supabase платформа |
| **sqlite** | SQLite базы данных |
| **elasticsearch** | Поиск и аналитика текстовых данных |

#### Web и интеграции

| Сервер | Описание |
|--------|---------|
| **web-search** | Поиск в интернете через Tavily |
| **fetch** | Загрузка и парсинг веб-страниц |
| **stripe** | Stripe платежи и управление |
| **slack** | Slack интеграция |
| **notion** | Notion базы данных |

#### Визуализация и анализ

| Сервер | Описание |
|--------|---------|
| **antv-chart** | Генерация диаграмм и графиков |
| **mermaid** | Диаграммы Mermaid |
| **plotly** | Интерактивные графики |

### 2.2 Установка и настройка серверов

#### Через npm (TypeScript/JavaScript)



**Скрипты развертывания и управления:**

Bash скрипты автоматизируют процесс установки, конфигурации и управления MCP-сервером. Типичный скрипт может: загрузить зависимости, создать необходимые директории, установить переменные окружения, запустить сервер и его зависимости.

Хорошие скрипты обеспечивают идемпотентность — результат должен быть одинаковым независимо от того, сколько раз выполнить скрипт. Это означает проверку, установлен ли пакет, прежде чем пытаться установить его, удаление старых данных перед созданием новых, и т.д.

В production среде скрипты управления обычно становятся частью систем оркестрации (systemd, supervisor, Kubernetes), которые гарантируют, что сервер всегда запущен и перезапускается при сбое.

## Заключение

Мы изучили комплексную экосистему для создания интеллектуальных агентов:
