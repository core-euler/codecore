# Мультиагентные системы и фреймворки

## Введение: Эволюция от одного агента к системам

Одного агента часто недостаточно для решения сложных задач. Если один LLM-агент может отправить письмо, десять координированных агентов могут провести полный маркетинговый кампейн — анализируя конкурентов, генерируя контент, планируя расписание и отслеживая результаты. Мультиагентные системы (Multi-Agent Systems, MAS) — это архитектурный подход, где несколько специализированных агентов работают вместе для достижения общей цели.

---

## 1. Почему нужны мультиагентные системы

### 1.1 Ограничения одного агента

Один LLM-агент сталкивается с несколькими фундаментальными проблемами:

1. **Перегруженность контекста**: агент должен знать всё о каждой задаче, что приводит к потере качества при росте сложности. При 100k+ токенов контекста модель начинает терять фокус (middle token problem).
2. **Отсутствие специализации**: один универсальный агент хуже специализированного. Агент, который одновременно пишет код, тестирует и деплоит — делает каждую задачу хуже, чем три отдельных агента.
3. **Масштабируемость**: невозможно распределить нагрузку. Один агент обрабатывает задачи последовательно.
4. **Отказоустойчивость**: если один агент неудачен, вся система падает. Нет возможности изолировать сбои.
5. **Отсутствие взаимодействия**: нет возможности диалога между компонентами системы. Один агент не может «обсудить» решение с другим.

### 1.2 Преимущества мультиагентных систем

- **Специализация**: каждый агент отвечает за свою область (анализ, планирование, выполнение). Специализированный промпт и набор инструментов повышают качество.
- **Параллелизм**: агенты работают одновременно, ускоряя выполнение.
- **Отказоустойчивость**: отказ одного агента не разрушает систему; задачу можно переназначить.
- **Сложность решаемых задач**: можно решать задачи уровня «полный проект», а не «один шаг».
- **Интерпретируемость**: легче понять, почему система приняла решение — каждый агент отвечает за свою часть.
- **Переиспользование**: компоненты легко переиспользуются в других системах.

### 1.3 Когда НЕ нужны мультиагентные системы

Мультиагентность — не всегда правильный выбор. Один агент лучше, когда:

- **Задача линейна и проста**: написать функцию, ответить на вопрос, обработать файл.
- **Контекст помещается**: вся информация для задачи умещается в одно контекстное окно.
- **Нет параллелизма**: все шаги строго последовательны и зависимы.
- **Стоимость координации выше стоимости задачи**: overhead на передачу данных между агентами превышает выигрыш.
- **Нужна скорость**: latency мультиагентной системы всегда выше (orchestration overhead).

**Правило:** если задачу можно решить одним агентом за < 10 шагов с < 50k токенов контекста — используй одного агента.

## 2. Паттерны мультиагентных систем

### 2.1 Sequential (Последовательный) паттерн

Агенты выполняются поочередно, один за другим. Выход одного агента становится входом для следующего.

```
Агент 1 → Агент 2 → Агент 3 → Результат
```

**Когда использовать:** когда результат каждого агента необходим для следующего. Каждый шаг зависит от предыдущего.

**Примеры:**
- Анализ → Написание → Редактирование → Публикация
- Сбор требований → Дизайн → Код → Тесты
- OCR → Извлечение данных → Валидация → Форматирование

**Реализация:** в LangGraph определяются узлы (nodes) для каждого агента и связи (edges) между ними. Граф выполняет узлы в порядке, определённом связями: выход первого узла становится входом для второго. Состояние (state) передаётся через общий объект.

**Преимущества:** простота, предсказуемость, лёгкость отладки.
**Недостатки:** медленно (нет параллелизма), bottleneck на каждом шаге.

### 2.2 Parallel (Параллельный) паттерн

Несколько агентов работают одновременно независимо друг от друга. Результаты собираются агентом-координатором.

```
        ├─ Агент 1 (анализ цены)    ─┐
Вход ─→ ├─ Агент 2 (анализ отзывов)  ─┼→ Координатор → Результат
        └─ Агент 3 (анализ конкурент) ─┘
```

**Когда использовать:** когда агенты решают независимые части задачи. Результаты можно объединить без зависимостей.

**Примеры:**
- Анализ цены + Анализ отзывов + Анализ конкурентов → Итоговый отчёт
- Перевод на 5 языков одновременно → Сборка мультиязычного документа
- Сканирование 10 репозиториев → Сводный отчёт по безопасности

**Реализация:** используется механизм асинхронного выполнения (async/await). Система запускает несколько агентов одновременно, каждый работает с частью входных данных. Координатор собирает результаты и выполняет синтез.

**Преимущества:** скорость (время = время самого медленного агента), масштабируемость.
**Недостатки:** сложнее обрабатывать ошибки, нужен координатор для синтеза.

### 2.3 Hierarchical (Иерархический) паттерн

Один агент (менеджер) координирует работу нескольких подчинённых агентов, каждый из которых может иметь своих подчинённых.

```
         ┌─ Менеджер проекта
         │  ├─ Менеджер фронтенда
         │  │  ├─ React разработчик
         │  │  └─ CSS специалист
         │  └─ Менеджер бэкенда
         │     ├─ Backend разработчик
         │     └─ DBA
```

**Когда использовать:** при наличии иерархии задач и естественного разделения ответственности. Когда задача декомпозируется на уровни абстракции.

**Примеры:**
- Менеджер проекта → Менеджер разработки, Менеджер QA, Менеджер документации
- CTO-агент → Backend team, Frontend team, DevOps team
- Редактор → Авторы по разделам → Корректоры

**Реализация:** менеджер-агент получает задачу, декомпозирует её и распределяет между подчинёнными. Подчинённые могут выполнять задачи параллельно или последовательно. Менеджер собирает результаты, проверяет качество и при необходимости возвращает на доработку.

**Преимущества:** естественная декомпозиция, контроль качества на каждом уровне.
**Недостатки:** overhead менеджерских агентов, потенциальные bottleneck на уровне менеджера.

### 2.4 Peer-to-Peer (Равноправный) паттерн

Все агенты имеют равные права и могут инициировать диалог друг с другом.

```
┌─ Агент 1 ─────────────────┐
│    ↔    ↔    ↔            │
├─ Агент 2 ─────────────────┤
│         ↔    ↔            │
└─ Агент 3 ─────────────────┘
```

**Когда использовать:** когда агенты должны сотрудничать на равных условиях без чёткой иерархии. Когда решение рождается из дискуссии.

**Примеры:**
- Коллегиум юристов (каждый высказывает мнение, система синтезирует)
- Дебатные клубы (аргументы за и против)
- Исследовательские группы (каждый эксперт дополняет картину)

**Реализация с A2A протоколом:** агент-инициатор отправляет сообщение соответствующему агенту через Agent-to-Agent протокол. Целевой агент обрабатывает сообщение и отправляет результат обратно. Нет центрального координатора.

**Преимущества:** гибкость, нет single point of failure.
**Недостатки:** сложность координации, риск зацикливания, сложнее отлаживать.

### 2.5 Сравнение паттернов

| Паттерн | Latency | Сложность | Отказоустойчивость | Лучший сценарий |
|---------|---------|-----------|-------------------|----------------|
| Sequential | Высокая | Низкая | Низкая | Pipeline обработки |
| Parallel | Низкая | Средняя | Средняя | Независимые подзадачи |
| Hierarchical | Средняя | Высокая | Высокая | Крупные проекты |
| Peer-to-Peer | Непредсказуемая | Очень высокая | Высокая | Экспертные дискуссии |

## 3. Фреймворки для мультиагентных систем

### 3.1 LangGraph

LangGraph — фреймворк для построения агентных систем на основе графов состояний. Часть экосистемы LangChain.

**Ключевые концепции:**
- **State** — общий объект состояния, доступный всем узлам
- **Nodes** — функции или агенты, которые обрабатывают состояние
- **Edges** — связи между узлами (условные и безусловные)
- **Checkpointing** — автоматическое сохранение состояния для recovery

**Когда использовать:**
- Сложные workflows с ветвлением и условной логикой
- Когда нужен fine-grained контроль над потоком данных
- Когда важна возможность recovery после сбоев (checkpointing)
- Интеграция с экосистемой LangChain

**Сильные стороны:** гибкость, визуализация графа, streaming, human-in-the-loop.
**Слабые стороны:** сложный API, крутая кривая обучения, verbose код.

### 3.2 CrewAI

CrewAI — высокоуровневый фреймворк, построенный вокруг метафоры «команды» (crew) из агентов с ролями.

**Ключевые концепции:**
- **Agent** — агент с ролью, целью и backstory
- **Task** — задача с описанием, ожидаемым результатом и привязкой к агенту
- **Crew** — команда агентов с определённым процессом выполнения
- **Process** — порядок выполнения (sequential, hierarchical)

**Когда использовать:**
- Быстрое прототипирование мультиагентных систем
- Задачи с естественным разделением на роли
- Когда не нужен low-level контроль над графом

**Сильные стороны:** простота, быстрый старт, интуитивная модель ролей.
**Слабые стороны:** ограниченная гибкость, сложнее кастомизировать поведение, менее зрелая экосистема.

### 3.3 AutoGen (Microsoft)

AutoGen — фреймворк для построения мультиагентных систем через conversation patterns.

**Ключевые концепции:**
- **ConversableAgent** — базовый агент, способный вести разговор
- **AssistantAgent** — агент с LLM-бэкендом
- **UserProxyAgent** — агент, представляющий пользователя (может выполнять код)
- **GroupChat** — контейнер для группового диалога агентов

**Когда использовать:**
- Задачи, которые естественно решаются через диалог
- Когда нужен human-in-the-loop через UserProxy
- Исследовательские задачи, brainstorming

**Сильные стороны:** естественная модель диалога, good для research, активное развитие Microsoft.
**Слабые стороны:** менее предсказуемое поведение (диалог может уйти в сторону), сложнее контролировать стоимость.

### 3.4 Сравнение фреймворков

| Аспект | LangGraph | CrewAI | AutoGen |
|--------|-----------|--------|---------|
| **Модель** | Граф состояний | Команда с ролями | Диалог агентов |
| **Сложность** | Высокая | Низкая | Средняя |
| **Гибкость** | Максимальная | Ограниченная | Средняя |
| **Production-ready** | Да | Растёт | Растёт |
| **Контроль стоимости** | Хороший | Средний | Сложный |
| **Debugging** | Хороший (визуализация) | Средний | Сложный |
| **Лучший для** | Complex workflows | Quick prototyping | Conversational tasks |

## 4. Composio: Платформа инструментов для AI-агентов

### 4.1 Обзор и архитектура

Composio — это платформа для подключения AI-агентов к инструментам и сервисам. Она решает задачу «инструментизации» агентов с поддержкой более 250 интеграций.

**Ключевые компоненты:**
- **Core SDK** (Python/TypeScript) — основной API для интеграции
- **Provider Layer** — поддержка OpenAI, Claude, LangChain, CrewAI и др.
- **Tools Catalog** — 250+ предварительно сконфигурированных интеграций
- **Authentication Manager** — управление ключами и OAuth

### 4.2 Зачем нужна Composio

Без Composio каждый агент должен сам реализовывать интеграции:
- Написать обёртку для Slack API
- Реализовать OAuth flow для Google
- Обработать rate limiting для GitHub
- Управлять ключами для каждого сервиса

Composio предоставляет всё это из коробки:
- Агент получает ready-to-use инструменты
- Аутентификация управляется централизованно
- Rate limiting и retry обрабатываются автоматически

### 4.3 Интеграция с фреймворками

Composio интегрируется с популярными LLM фреймворками:
- LangChain / LangGraph
- OpenAI SDK (function calling)
- Anthropic SDK (tool use)
- CrewAI

После настройки агент может использовать 250+ инструментов без дополнительной разработки.

## 5. Оркестрация и Handoffs

### 5.1 Передача контекста между агентами

Одна из ключевых проблем мультиагентных систем — передача контекста. Когда агент A передаёт задачу агенту B, нужно передать:

- **Цель подзадачи** — что конкретно нужно сделать
- **Релевантный контекст** — не весь контекст, а только нужную часть
- **Constraints** — ограничения, которые нужно соблюдать
- **Артефакты** — файлы, данные, промежуточные результаты

**Антипаттерн:** передавать весь контекст оркестратора каждому subagent. Это приводит к token dilution и росту стоимости.

**Правильный подход:** формировать минимальный handoff summary для каждого subagent, включающий только то, что нужно для его конкретной подзадачи.

### 5.2 Error handling в мультиагентных системах

Каждый агент может упасть. Стратегии обработки:

1. **Retry** — повторить задачу (с ограничением на количество попыток)
2. **Fallback** — передать задачу другому агенту (backup)
3. **Escalation** — эскалировать к человеку или к агенту более высокого уровня
4. **Graceful degradation** — вернуть частичный результат вместо полного сбоя

### 5.3 Мониторинг мультиагентных систем

Что мониторить:
- **Latency каждого агента** — кто bottleneck?
- **Стоимость каждого агента** — кто потребляет больше всего токенов?
- **Error rate** — какие агенты падают чаще?
- **Quality** — какие агенты дают наименьшее качество?

## 6. Примеры реальных мультиагентных систем

### 6.1 Контент-маркетинг pipeline

```
Исследователь → Писатель → Редактор → SEO-оптимизатор → Публикатор

Исследователь: собирает данные, анализирует конкурентов
Писатель: создаёт контент на основе research
Редактор: проверяет качество, стиль, факты
SEO-оптимизатор: оптимизирует keywords, meta
Публикатор: публикует в CMS, расшаривает в соцсетях
```

**Паттерн:** Sequential (каждый шаг зависит от предыдущего)
**Фреймворк:** CrewAI (естественные роли)

### 6.2 Code Review система

```
         ┌─ Менеджер Review
         │  ├─ Security Reviewer (SAST, secrets detection)
         │  ├─ Performance Reviewer (complexity, bottlenecks)
         │  ├─ Style Reviewer (lint, conventions)
         │  └─ Logic Reviewer (bugs, edge cases)
         │
         └─ Synthesizer (итоговый отчёт)
```

**Паттерн:** Hierarchical (менеджер + параллельные reviewers)
**Фреймворк:** LangGraph (нужен fine-grained контроль)

### 6.3 Customer Support routing

```
        ├─ Sales Agent       ─┐
Router ─├─ Tech Support Agent ─┼→ Escalation Agent (если нужно)
        └─ Billing Agent     ─┘
```

**Паттерн:** Parallel routing (router определяет, к кому направить)
**Фреймворк:** LangGraph или AutoGen

### 6.4 Анализ документов

```
Classifier → Extractor → Analyzer → Summarizer

Classifier: определяет тип документа (invoice, contract, report)
Extractor: извлекает ключевые данные в зависимости от типа
Analyzer: проводит анализ (compliance, risk, anomalies)
Summarizer: создаёт структурированное резюме
```

**Паттерн:** Sequential с условным ветвлением (тип документа определяет extraction strategy)
**Фреймворк:** LangGraph (условные edges)

## 7. Production checklist для мультиагентных систем

- [ ] Определён паттерн оркестрации (sequential/parallel/hierarchical/peer)
- [ ] Каждый агент имеет чёткую роль, scope и constraints
- [ ] Handoff между агентами минимален (только нужный контекст)
- [ ] Error handling настроен для каждого агента (retry/fallback/escalation)
- [ ] Budget control на уровне системы (max tokens, max cost, max time)
- [ ] Мониторинг: latency, cost, error rate, quality для каждого агента
- [ ] Human-in-the-loop для критичных решений
- [ ] Тесты: end-to-end сценарии с golden set
- [ ] Logging: полный trace от запроса до ответа через все агенты

## Заключение

Мультиагентные системы — мощный подход к решению сложных задач через координацию специализированных компонентов. Ключевые выводы:

1. **Не всегда нужны**: начните с одного агента, переходите к мультиагентности только при явной необходимости.
2. **Паттерн определяет архитектуру**: выбор между sequential, parallel, hierarchical и peer-to-peer зависит от природы задачи.
3. **Фреймворк определяет скорость разработки**: CrewAI для быстрого старта, LangGraph для production, AutoGen для conversational задач.
4. **Оркестрация — главная сложность**: передача контекста, error handling и мониторинг — основные инженерные вызовы.
5. **Стоимость растёт**: каждый дополнительный агент = дополнительные токены. Мониторинг cost обязателен.
