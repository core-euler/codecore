# Модуль 2, Подмодуль 2: Системы памяти и граф знаний для агентов

## Введение

Одна из фундаментальных проблем современных LLM-агентов — это отсутствие памяти. В то время как люди могут помнить события, факты и опыт из прошлого, большие языковые модели работают в состоянии "забывчивости", обрабатывая только текущий контекст. Эта глава посвящена архитектурным решениям, которые позволяют агентам сохранять, восстанавливать и использовать прошлый опыт для улучшения своей производительности.

---

## 1. Проблема памяти у LLM-агентов

### 1.1. Stateless природа больших языковых моделей

LLM по своей архитектуре — это stateless преобразователи текста. Каждый вызов модели начинается с «чистого листа» — модель не имеет встроенного механизма для сохранения информации между сессиями. Это означает:

- **Отсутствие долгосрочной памяти**: Модель не помнит предыдущие разговоры и опыт
- **Ограниченность контекста**: Память ограничена размером окна контекста (например, 8K, 128K или 200K токенов)
- **Повторяющиеся вычисления**: Одна и та же информация должна передаваться в каждом запросе
- **Возрастающие затраты**: Длинные истории разговоров требуют больше токенов и вычислительных ресурсов

Это контрастирует с человеческой памятью, которая:
- Автоматически сохраняет важное
- Забывает рутинное
- Может восстанавливать информацию по ассоциациям
- Эффективно используется (мы не помним каждое слово, но суть остается)

### 1.2. Виды памяти в AI-системах

Хотя LLM сами не имеют встроенной памяти, мы можем построить многоуровневую архитектуру памяти:

#### Краткосрочная память (Short-term Memory)
- **Контекст окна**: Текущий набор токенов в контексте модели
- **Объем**: Ограничен размером окна контекста (обычно 4K-200K токенов)
- **Время жизни**: Существует только во время одного обмена сообщениями
- **Использование**: Текущий разговор, анализ текущей задачи
- **Управление**: Автоматическое, встроено в архитектуру LLM

#### Эпизодическая память (Episodic Memory)
- **Что хранит**: Конкретные события, разговоры, решения из прошлого
- **Объем**: Может быть большим (всё, что пользователь хочет помнить)
- **Время жизни**: Дни, месяцы, годы
- **Использование**: "Вспомни, что мы обсуждали на прошлой неделе"
- **Пример**: База данных всех предыдущих разговоров

#### Семантическая память (Semantic Memory)
- **Что хранит**: Факты, правила, структурированные знания
- **Объем**: Может быть очень большим (энциклопедия, кодовая база)
- **Время жизни**: Постоянно
- **Использование**: "Дай определение", "Объясни концепцию"
- **Пример**: Граф знаний, база фактов

#### Процедурная память (Procedural Memory)
- **Что хранит**: Как что-то делать, паттерны решения
- **Объем**: Различный (от простых алгоритмов до сложных стратегий)
- **Время жизни**: Постоянно
- **Использование**: "Как решить эту проблему", "Какие шаги предпринять"
- **Пример**: Сохраненные стратегии решения задач, best practices

### 1.3. Отличия от человеческой памяти

Человеческая память имеет свойства, которые мы стремимся воспроизвести в AI:

| Свойство | Человеческая память | AI системы (сейчас) | AI системы (с архитектурой) |
|----------|-------------------|-------------------|---------------------------|
| Ассоциативность | Высокая | Низкая | Высокая (через векторный поиск) |
| Консолидация | Автоматическая | Ручная | Частично автоматизирована |
| Забывание | Экспоненциальное (старое забывается) | Нет встроенного механизма | Можно реализовать через TTL |
| Воспроизведение | Фрагментарное | Полное или ничего | Фрагментарное (через поиск) |
| Компрессия | Неточная, но эффективная | Точная, но дорогая | Комбинированная |

## 2. Архитектура персистентной памяти

### 2.1. Многоуровневая модель памяти

Эффективная память требует нескольких слоев с разными характеристиками:

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  УРОВЕНЬ 0: Контекстное окно (transient)                   │
│  ────────────────────────────────────────────              │
│  - Размер: 4K-200K токенов                                │
│  - Скорость: Мгновенная (уже в памяти модели)            │
│  - Содержимое: Текущий разговор (10-50 сообщений)        │
│  - Жизненный цикл: Одна сессия или день                  │
│                                                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  УРОВЕНЬ 1: Кеш сессии (temporary)                        │
│  ─────────────────────────────────                         │
│  - Размер: 100K-1M токенов                               │
│  - Скорость: Быстрая (память сервера)                    │
│  - Содержимое: Последние 10-20 сообщений из других сеансов│
│  - Жизненный цикл: Несколько часов                       │
│                                                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  УРОВЕНЬ 2: Семантическая память (vector DB)             │
│  ──────────────────────────────────────────               │
│  - Размер: 100M-10B токенов (практически неограниченно)  │
│  - Скорость: Медленная (требует поиска)                  │
│  - Содержимое: Сжатые факты, идеи, решения              │
│  - Жизненный цикл: Месяцы, годы                         │
│                                                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  УРОВЕНЬ 3: Граф знаний (episodic + semantic)            │
│  ─────────────────────────────────────────                │
│  - Размер: Неограниченно                                 │
│  - Скорость: Медленная (граф-запросы)                    │
│  - Содержимое: Связи между сущностями, события           │
│  - Жизненный цикл: Всегда                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2. Поток информации между уровнями

```
1. Входящее сообщение (от пользователя или результат tool_use)
        ↓
2. Добавляется в УРОВЕНЬ 0 (контекстное окно)
        ↓
3. Модель обрабатывает и выдает ответ
        ↓
4. Кратко обобщение ответа (через LLM) сохраняется в УРОВЕНЬ 2
        ↓
5. Если контекст переполнен, старые сообщения архивируются в УРОВЕНЬ 1
        ↓
6. Когда требуется информация, поиск идет:
   сначала УРОВЕНЬ 0 → УРОВЕНЬ 1 → УРОВЕНЬ 2 → УРОВЕНЬ 3
```

## 3. Системы памяти: Mem0

### 3.1. Обзор Mem0

Mem0 — это платформа для добавления персистентной памяти к AI агентам. Она автоматически:

1. **Извлекает ключевую информацию** из разговоров
2. **Структурирует информацию** в виде фактов и отношений
3. **Сохраняет в базу знаний** для последующего использования
4. **Восстанавливает релевантную информацию** перед каждым ответом

### 3.2. Типы информации, которую сохраняет Mem0

#### Факты (Facts)
- Простые утверждения, которые остаются верными
- Пример: "Пользователь живет в Москве", "Он работает инженером"
- Экономия: 1-3 токена вместо полного предложения

#### Предпочтения (Preferences)
- Личные выборы и стиль пользователя
- Пример: "Предпочитает краткие ответы", "Говорит на русском"
- Использование: Настройка системного промпта для каждого пользователя

#### Решения (Decisions)
- Выбор, которые было принято
- Пример: "Выбрал архитектуру A вместо B, потому что..."
- Использование: Не повторять старые обсуждения

#### История (History)
- Временно-ориентированные события
- Пример: "15 апреля: начал проект X", "20 апреля: завершил фазу 1"
- Использование: Ответы типа "что мы обсуждали в прошлый раз?"

### 3.3. Как Mem0 работает

**Фаза 1: Добавление информации**
```
Пользователь: "Я работаю на Python и предпочитаю FastAPI для backend."
    ↓
LLM анализирует: Извлекает факты
    ↓
Сохраняет:
  - Факт: "Использует Python для разработки"
  - Факт: "Предпочитает FastAPI для backend"
  - Связь: Python ← (используется для) → backend разработка
    ↓
При следующем запросе этой информация используется в системном промпте
```

**Фаза 2: Восстановление информации**
```
Пользователь: "Какой фреймворк выбрать для нового проекта?"
    ↓
Mem0 ищет в памяти: "Python", "backend", "фреймворк"
    ↓
Находит ранее сохраненные предпочтения о FastAPI
    ↓
Добавляет в контекст: "Пользователь предпочитает FastAPI"
    ↓
Модель учитывает это при ответе
```

### 3.4. Параметры конфигурации Mem0

| Параметр | Назначение |
|----------|-----------|
| `auto_extract` | Автоматически извлекать факты из каждого сообщения |
| `extract_interval` | Как часто запускать извлечение (каждое сообщение vs. батчи) |
| `max_memory_size` | Максимальный размер базы знаний |
| `retention_policy` | Как долго хранить информацию (TTL) |
| `embedding_model` | Какая модель использовать для эмбеддингов |

### 3.5. Когда использовать Mem0

**Хорошо подходит для:**
- Personal assistants (персональные помощники)
- Chat-bots с долгосрочными отношениями
- Customer service с историей клиентов
- Аналитические агенты, которые собирают информацию о пользователе

**Не подходит для:**
- Анонимных чат-ботов без персонализации
- Систем с очень большими объемами информации (лучше граф)
- Real-time систем с очень низкой задержкой

## 4. Графы знаний для агентов

### 4.1. Обзор графов знаний

Граф знаний (Knowledge Graph) — это структура данных, которая представляет информацию как:
- **Узлы (Nodes)**: сущности (Person, Company, Concept)
- **Рёбра (Edges)**: отношения между сущностями (works_for, is_author_of, relates_to)
- **Атрибуты**: Дополнительные свойства узлов и рёбер

**Пример:**
```
Узел: "Alice" (Person)
  ├─ атрибут: name = "Alice"
  ├─ атрибут: title = "Data Scientist"
  └─ ребро: works_for → [Компания: "TechCorp"]

Узел: "TechCorp" (Company)
  ├─ атрибут: founded = 2015
  ├─ атрибут: location = "San Francisco"
  └─ ребро: builds → [Продукт: "Analytics Platform"]
```

### 4.2. Отличие графов от Vector DB

| Характеристика | Vector DB | Graph KB |
|----------------|-----------|----------|
| Структура | Плоский (вектор) | Структурированный (граф) |
| Отношения | Семантические (похожесть) | Явные (связи) |
| Запрос | "Найди похожее" | "Найди все, связанные с X" |
| Экономия памяти | Хорошая (эмбеддинги) | Отличная (структура) |
| Сложность запросов | Простые | Сложные |
| Идеально для | Поиск и рекомендации | Сложный анализ и рассуждение |

### 4.3. Graphiti: специализированная система графов

Graphiti — система для построения и управления графами знаний в AI агентах.

**Основные возможности:**
1. **Автоматическое создание узлов и рёбер** из текста
2. **Обновление существующих узлов** при появлении новой информации
3. **Граф-запросы** для нахождения связанной информации
4. **Интеграция с LLM** для интерпретации результатов

### 4.4. Как Graphiti работает

**Шаг 1: Экстракция сущностей**
```
Текст: "Алиса работает в TechCorp. TechCorp делает платформу для аналитики."
    ↓
Graphiti извлекает:
  - Сущности: Alice (Person), TechCorp (Company), "Analytics Platform" (Product)
  - Отношения: Alice works_for TechCorp, TechCorp builds "Analytics Platform"
```

**Шаг 2: Обновление графа**
```
Если узел "TechCorp" уже существует:
  - Добавляет новые рёбра (builds)
  - Обновляет атрибуты

Если это новая информация о Alice:
  - Может обновить title или добавить новые отношения
```

**Шаг 3: Запрос к графу**
```
Вопрос пользователя: "Что Алиса делает?"
    ↓
Graphiti выполняет граф-запрос:
  1. Найди узел "Alice"
  2. Найди все рёбра от этого узла
  3. Верни связанные сущности и отношения
    ↓
Результат: "works_for → TechCorp", потом можно получить "TechCorp builds Analytics Platform"
```

### 4.5. Типы отношений в графах

| Отношение | Пример | Использование |
|-----------|--------|---------------|
| `works_for` | Person works_for Company | Поиск сотрудников |
| `knows` | Person knows Person | Сетевые анализы |
| `owns` | Company owns Product | Портфолио продуктов |
| `discusses` | Conversation discusses Topic | История тем |
| `solved_by` | Problem solved_by Solution | База решений |
| `depends_on` | Code depends_on Library | Анализ зависимостей |
| `similar_to` | Concept similar_to Concept | Рекомендации |

## 5. Интеграция памяти с контекстом

### 5.1. Полный поток с памятью

```
ИНИЦИАЛИЗАЦИЯ:
│
├─ Загружаем системный промпт
├─ Инициализируем память (Mem0 или Graphiti)
└─ Инициализируем контекстное окно

НОВОЕ СООБЩЕНИЕ:
│
├─ Добавляем сообщение в контекст (Уровень 0)
│
├─ Поиск в памяти:
│  └─ Извлекаем релевантные факты, решения, историю
│
├─ Добавляем найденную информацию в контекст:
│  └─ "Воспоминания: [факты из памяти]"
│
├─ ОТПРАВЛЯЕМ МОДЕЛИ:
│  └─ [Системный промпт] + [Воспоминания] + [Текущий контекст]
│
├─ ОБРАБОТКА ИНСТРУМЕНТОВ (если нужны):
│  └─ Выполняем tool_uses, добавляем результаты в контекст
│
├─ ПОЛУЧЕНИЕ ОТВЕТА от модели
│
├─ СОХРАНЕНИЕ В ПАМЯТИ:
│  ├─ Извлекаем новые факты из ответа
│  ├─ Обновляем граф знаний
│  └─ Сохраняем эпизод разговора
│
└─ УПРАВЛЕНИЕ КОНТЕКСТОМ:
   ├─ Проверяем размер
   ├─ Если > порога: применяем очистку
   └─ Архивируем старые части в кеш сессии
```

### 5.2. Пример интеграции: Agent с памятью

**Сценарий:** Долгоживущий аналитический агент

```
День 1:
  Юзер: "Запусти анализ для компании X"
    → Agent запускает анализ
    → Сохраняет: "Анализ X запущен 1 декабря"

День 2:
  Юзер: "Как дела с анализом?"
    → Agent ищет в памяти о "компании X"
    → Находит: "Анализ X запущен 1 декабря"
    → Может получить текущий статус и обновить память

День 10:
  Юзер: "Сравни результаты X и Y"
    → Agent имеет полную историю обоих анализов в памяти
    → Может сравнивать даже если контекст был очищен много раз
```

### 5.3. Когда использовать какую систему памяти

| Сценарий | Mem0 | Graphiti | Vector DB | Простой JSON |
|----------|------|----------|-----------|--------------|
| Личный помощник | ★★★★★ | ★★★ | ★★★ | ★ |
| Многопользовательский чат | ★★★★ | ★★ | ★★★★ | ★★ |
| Аналитический агент | ★★★★ | ★★★★★ | ★★★ | ★★ |
| Рекомендер | ★★★ | ★★★★ | ★★★★★ | ★ |
| Простой инструмент | ★ | ★ | ★★ | ★★★★★ |
| Прототип | ★★ | ★ | ★★ | ★★★★★ |

## 6. Практические паттерны

### 6.1. Паттерн 1: Персональный ассистент с Mem0

**Цель:** Создать помощника, который помнит предпочтения и историю пользователя.

**Архитектура:**
```
Пользователь → Input
              ↓
         Mem0: Поиск фактов о пользователе
              ↓
         Контекст: [Системный промпт] + [Факты из Mem0] + [Текущее сообщение]
              ↓
         LLM обрабатывает и отвечает
              ↓
         Mem0: Сохранить новые факты и предпочтения
              ↓
         Пользователь ← Output
```

**Ключевые параметры:**
- Mem0 extraction: каждое сообщение
- TTL: никогда (предпочтения не устаревают)
- Типы памяти: факты, предпочтения, решения

### 6.2. Паттерн 2: Аналитический агент с Graphiti

**Цель:** Создать агента, который анализирует сложные системы с множеством сущностей.

**Архитектура:**
```
Новый запрос
      ↓
Graphiti: Найти все узлы, связанные с запросом
      ↓
Контекст: [Архитектурный контекст] + [Граф-результаты] + [Текущий запрос]
      ↓
LLM анализирует граф и отвечает
      ↓
Graphiti: Обновить граф новыми сущностями и отношениями
      ↓
Ответ агента
```

**Ключевые параметры:**
- Граф-запросы: поиск в глубину до 3 уровней
- Обновление: после каждого анализа
- Типы узлов: зависит от домена (код, люди, продукты и т.д.)

### 6.3. Паттерн 3: Гибридная система (Mem0 + Vector DB)

**Цель:** Быстрая структурированная информация (Mem0) + поиск по смыслу (Vector DB)

**Архитектура:**
```
Вопрос пользователя
      ↓
Mem0: Быстрый поиск фактов и структурированной информации
Vector DB: Семантический поиск похожих концепций
      ↓
Контекст: [Структурированные факты] + [Семантические результаты] + [Текущий вопрос]
      ↓
LLM получает наилучшую информацию из обоих источников
```

**Когда использовать:**
- Нужна скорость (Mem0) + качество поиска (Vector DB)
- Большая база знаний
- Много разных типов запросов

## 7. Управление памятью и контекстом

### 7.1. Проблемы при росте памяти

По мере того, как агент накапливает память:

**Проблема 1: Галлюцинация из памяти**
- Старые факты могут противоречить новым
- Модель может использовать устаревшую информацию
- Решение: версионирование, TTL, явные метаданные свежести

**Проблема 2: Перегруженный контекст**
- Много релевантной информации → не вся влезает в контекст
- Решение: ranked retrieval (выбирать top-K, а не всё)

**Проблема 3: Стоимость памяти**
- Каждое сохранение + каждый поиск = вычисления
- Решение: батчинг, кеширование результатов поиска

### 7.2. Best Practices

**Для Mem0:**
- Не сохраняйте дублирующуюся информацию
- Используйте консистентный формат для фактов
- Периодически очищайте старые факты (TTL)
- Проверяйте качество извлечения (модель может ошибаться)

**Для Graphiti:**
- Не создавайте слишком много типов отношений (сложно управлять)
- Используйте явные метаданные для рёбер (уверенность, дата)
- Периодически переиндексируйте граф
- Проверяйте консистентность (нет циклических противоречий)

**Для всех систем памяти:**
- Мониторьте размер памяти (бесконечный рост = проблема)
- Реализуйте механизм забывания (как люди забывают старое)
- Разделяйте важное от рутинного
- Не полагайтесь только на память (основной контекст важнее)

## 8. Метрики и мониторинг

### 8.1. Что измерять

**Метрика 1: Memory Hit Rate**
```
Определение: Сколько раз в памяти нашлась релевантная информация?
Целевое значение: >70%
Как считать: (удачные поиски) / (всего поисков)
```

**Метрика 2: Memory Accuracy**
```
Определение: Насколько используемые факты релевантны вопросу?
Целевое значение: >85%
Как считать: (релевантные результаты) / (всего возвращено)
```

**Метрика 3: Memory Size Growth**
```
Определение: Как быстро растет база знаний?
Целевое значение: Замедляющийся рост (экспоненциальное замедление)
Как считать: (новые факты в неделю)
```

**Метрика 4: Context Reduction**
```
Определение: На сколько процентов памятина помогает сокращать контекст?
Целевое значение: 30-50% сокращение
Как считать: (размер контекста без памяти - размер с памятью) / размер без памяти
```

### 8.2. Индикаторы проблем

**Индикатор 1: Память не используется**
- Hit Rate < 30%
- Причина: неправильный поиск, плохие эмбеддинги, мало информации в памяти
- Решение: отладить поиск, улучшить эмбеддинги

**Индикатор 2: Галлюцинации из памяти**
- Модель использует неправильные старые факты
- Причина: противоречивые факты, устаревшая информация
- Решение: добавить TTL, версионирование, явные метаданные

**Индикатор 3: Медленные ответы**
- Время ответа растет с размером памяти
- Причина: граф-запросы становятся дороже, поиск медленнее
- Решение: индексирование, кеширование, архивирование

## 9. Практическое задание

### 9.1. Задача 1: Построить Mem0-подобную систему

**Что нужно сделать:**
1. Создать простое хранилище фактов (JSON или SQLite)
2. После каждого ответа извлекать ключевые факты
3. Перед новым вопросом вставлять релевантные факты в контекст
4. Отслеживать, какие факты используются

**Ожидаемый результат:**
- Система помнит предпочтения пользователя
- При новом вопросе включает релевантные факты
- Отчет показывает, какие факты помогли

### 9.2. Задача 2: Простой граф знаний

**Что нужно сделать:**
1. Создать простую структуру узлов и рёбер

4. Использовать граф для улучшения ответов

**Ожидаемый результат:**
- Граф содержит хотя бы 10-20 узлов
- Можно выполнить граф-запросы на расстояние 1-2 уровня
- Использование графа улучшает ответы

## 10. Выводы и рекомендации

### Ключевые идеи

1. **Архитектурное решение для памяти необходимо** — контекстное окно не достаточно для долгосрочных агентов

2. **Многоуровневая память эффективнее**, чем монолитное хранилище — краткосрочное (окно) + среднесрочное (кеш) + долгосрочное (граф)

3. **Типы информации требуют разных подходов:**
   - Факты → Mem0 или Graphiti
   - История → эпизодическая память
   - Сложные отношения → граф
   - Свободный текст → Vector DB

4. **Интеграция с контекстом критична** — память должна дополнять, а не заменять контекст

5. **Мониторинг и управление памятью важны** — бесконечный рост = деградация производительности

### Рекомендуемые комбинации

**Для персонального ассистента:**
- Mem0 для фактов и предпочтений
- Простой Vector DB для истории разговоров
- TTL для всех данных (старое забывается)

**Для аналитического агента:**
- Graphiti для сущностей и отношений
- Кеш результатов анализа
- Explicit архивирование для долгих проектов

**Для масштабируемой системы:**
- Mem0 для структурированных фактов
- Vector DB для полнотекстового поиска
- Graphiti для сложного анализа (опционально)
