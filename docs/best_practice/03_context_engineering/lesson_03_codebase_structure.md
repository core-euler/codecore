# Модуль 2, Подмодуль 3: Структура кодовой базы для агентов

## Введение

Когда AI-агент впервые подключается к проекту, он оказывается в ситуации нового разработчика, который не знает кодовую базу. Без ориентиров агент тратит значительную часть context window на чтение файлов, которые могут оказаться нерелевантными, или, наоборот, не находит нужные фрагменты кода. Эта глава посвящена практическим подходам к организации кодовой базы так, чтобы агент мог эффективно в ней ориентироваться, тратить меньше токенов и быстрее находить нужную информацию.

---

## 1. Проблема холодного старта агента

### 1.1. Что происходит при первом запуске

При первом подключении к проекту агент не знает ни структуру, ни назначение файлов. Это приводит к двум типичным проблемам:

1. **Перерасход контекста**: агент читает слишком много файлов подряд, быстро заполняя context window нерелевантной информацией
2. **Потеря нужных файлов**: агент не находит ключевые модули и принимает решения на основе неполной картины проекта

Обе проблемы ведут к одному результату — низкое качество генерации кода, увеличение числа итераций и рост расходов на API-вызовы.

### 1.2. Наивное решение: справочный файл

Первое, что приходит в голову — создать файл вроде `./docs/reference.md`, в котором описаны ключевые файлы проекта и их назначение. Это работает, но имеет существенные недостатки:

- **Сложность поддержки вручную**: справочник быстро устаревает, особенно при активной разработке
- **Недостаток деталей при автогенерации**: если попросить агента сгенерировать описание проекта, результат часто получается слишком общим и бесполезным для конкретных задач

Более устойчивый подход — сделать так, чтобы **сама кодовая база была собственной справкой**.

## 2. Правила оформления кода для агентов

Ключевая идея: вместо внешней документации закладываем правила оформления файлов, которые позволяют автоматически генерировать структуру проекта. Эти правила фиксируются в `AGENTS.md` (или аналогичном файле инструкций для агента).

### 2.1. Одна ответственность на файл

Каждый файл должен решать одну задачу. Имя файла должно прямо описывать его назначение:

```
# Плохо
utils.ts          — что именно внутри?
helpers.ts        — какие хелперы?
misc.ts           — содержимое непредсказуемо

# Хорошо
input-arguments.ts    — парсинг аргументов CLI
language-registry.ts  — реестр языковых движков
outline-renderer.ts   — рендеринг структуры файла
```

**Почему это важно для агента**: при поиске нужного кода агент часто сканирует список файлов. Чем точнее имя файла описывает содержимое, тем меньше лишних файлов агенту нужно открывать.

### 2.2. Header comment — комментарий в начале файла

В первых строках каждого файла размещается комментарий, описывающий:

- Назначение файла (что он делает)
- Что в файл **не** входит (разграничение ответственности)

```typescript
// CLI argument parsing and input path normalization helpers.
// Does NOT handle execution logic — see modes.ts for that.

import { resolve } from 'path';
...
```

При каждом изменении файла комментарий должен обновляться. Это правило можно явно прописать в инструкциях для агента.

**Расширения header comment**: в зависимости от потребностей проекта, в комментарий можно добавлять:
- Зависимости и связи с другими модулями
- Побочные эффекты (I/O, мутации состояния)
- Теги для навигации

## 3. Двухуровневая навигация по проекту

После внедрения правил оформления можно автоматически генерировать навигационные индексы на двух уровнях.

### 3.1. Уровень 1: Структура папки (folder outline)

Это индекс по файлам: путь к файлу и его header comment. Скрипт проходит по `src/**/*` и собирает пары **путь → описание**.

```
src/cli/input-arguments.ts: CLI argument parsing and input path normalization helpers.
src/cli/modes.ts: CLI execution modes for file, glob, and debug workflows.
src/cli/types.ts: Shared CLI runtime types for dependency injection and content processing.
src/core/declaration-lines.ts: Shared helpers for converting parsed declarations into outline lines.
src/core/formatter.ts: Declaration formatting utilities for generating outline output.
src/core/language-engine.ts: Shared interface for language-specific outline generators.
src/core/language-registry.ts: Registry and resolver for language-specific outline engines.
src/core/outline-renderer.ts: Shared helpers for rendering structured outline lines.
src/core/types.ts: Core type definitions for outline generation.
```

**Что это даёт агенту**: компактный обзор всего проекта за минимальное количество токенов. Агент может быстро определить, какие файлы ему нужно прочитать для решения конкретной задачи.

### 3.2. Уровень 2: Структура файла (file outline)

Это список элементов верхнего уровня внутри файла: функции, классы, константы, экспорты. Скрипт извлекает только объявления (declarations), без тел реализаций.

```
function stripBom(content: string): string
function findFirstContentLineIndex(lines: readonly string[]): number
function getLineStartPosition(lines: readonly string[], lineIndex: number): number
function extractSingleLineComments(lines: readonly string[], startIndex: number): string
export function extractTopComment(content: string): string | null
export function extractTopCommentLineNumber(content: string): number | null
export function cleanCommentText(comment: string): string
```

**Что это даёт агенту**: понимание содержимого файла без необходимости читать его целиком. Агент видит сигнатуры функций и может решить, нужно ли ему детальное чтение.

## 4. Интеграция в рабочий процесс

### 4.1. Агент вызывает скрипт перед чтением файлов

Правило для агента прописывается в промпте или в `AGENTS.md`:

> Перед тем как читать файлы проекта, вызови скрипт генерации структуры. Сначала получи folder outline, затем file outline для нужных модулей. Только после этого открывай исходники.

Это превращает навигацию по проекту из случайного блуждания в целенаправленный поиск.

### 4.2. Генерация на каждый коммит

Структуру `src/` можно генерировать автоматически на каждый коммит (через pre-commit hook или CI) и хранить в `docs/`:

```
docs/
  folder-outline.txt   — индекс файлов проекта
  file-outlines/       — outline каждого модуля
```

**Дополнительный бонус**: изменения в outline-файлах видны в git diff. Это позволяет отслеживать архитектурные изменения проекта — добавление новых модулей, переименования, разрастание файлов — прямо в истории коммитов.

### 4.3. Пример правила в AGENTS.md

```markdown
## Навигация по проекту

1. Перед началом работы с кодом запусти `npm run outline:folder` для получения
   структуры папок.
2. Для детального изучения файла запусти `npm run outline:file <path>`.
3. Только после анализа outline читай исходный код.

## Правила оформления

1. Каждый файл — одна ответственность. Имя файла = назначение.
2. Первая строка каждого файла — header comment с описанием назначения.
3. При изменении файла обнови header comment.
```

## 5. Практическая реализация

### 5.1. Создание скрипта генерации

Есть два пути:

**Вариант 1: Попросить агента написать скрипт**

Попросите вашего AI-агента создать скрипт, который:
1. Проходит по файлам в `src/`
2. Извлекает header comment из каждого файла
3. Выводит объявления верхнего уровня (функции, классы, константы)

Для разбора кода хорошо подходит **tree-sitter** — библиотека инкрементального парсинга, поддерживающая множество языков.

**Вариант 2: Готовый инструмент**

Можно использовать готовые решения, например [outln](https://github.com/ptol/outln) — инструмент, поддерживающий TypeScript, JavaScript, Go, Rust и Markdown.

### 5.2. Пример минимального скрипта на Node.js

```javascript
// generate-folder-outline.js
// Собирает header comments из всех файлов в src/

const fs = require('fs');
const path = require('path');
const glob = require('glob');

function extractHeaderComment(filePath) {
  const content = fs.readFileSync(filePath, 'utf-8');
  const lines = content.split('\n');

  // Ищем первый однострочный комментарий
  for (const line of lines) {
    const trimmed = line.trim();
    if (trimmed.startsWith('//')) {
      return trimmed.slice(2).trim();
    }
    if (trimmed && !trimmed.startsWith('//')) break;
  }
  return '(no header comment)';
}

const files = glob.sync('src/**/*.{ts,js}');
for (const file of files) {
  const comment = extractHeaderComment(file);
  console.log(`${file}: ${comment}`);
}
```

## 6. Измеримый результат

Практические тесты показывают, что подход с двухуровневой навигацией снижает расход токенов **до 20%** в зависимости от задачи. Экономия складывается из:

- Меньше лишних файлов прочитано агентом
- Меньше итераций «прочитал — не то — прочитал другой файл»
- Более точные правки благодаря лучшему пониманию контекста проекта

Для крупных проектов с сотнями файлов эффект может быть ещё значительнее, поскольку без навигации агент теряется в объёме кода экспоненциально сильнее.

## Итоги

Структурирование кодовой базы для AI-агентов — это не отдельная задача, а расширение обычных практик чистого кода. Ключевые принципы:

1. **Кодовая база — собственная справка**: header comments и чёткие имена файлов позволяют автоматически генерировать навигацию
2. **Двухуровневая навигация**: folder outline (что где лежит) + file outline (что внутри файла) дают агенту компактную карту проекта
3. **Интеграция в workflow**: генерация outline на каждый коммит и правило «сначала outline» для агента превращают навигацию из проблемы в преимущество
4. **Измеримая экономия**: снижение расхода токенов до 20% и повышение точности генерации кода
