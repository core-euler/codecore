# Контекстное окно, editing и compaction

## Главный тезис

Управление контекстом — это отдельная инженерная дисциплина. Без неё длинные задачи деградируют: растёт стоимость, падает точность, появляется дрейф решений. Инженерия контекста важнее prompt engineering: качество данных в контексте влияет на результат больше, чем формулировка промпта.

---

## 1. Контекст — рабочее состояние системы

### 1.1 Что такое контекстное окно

Контекстное окно (context window) — это максимальное количество токенов, которое может обработать языковая модель за один проход. Это не просто техническое ограничение; оно определяет возможности агента, стоимость операций и качество результата.

Контекст включает не только prompt, но и:
- рабочую память (текущее состояние задачи),
- ограничения и constraints,
- историю решений и их обоснования,
- результаты инструментов (tool_uses),
- незавершённые ветки задач.

### 1.2 Понимание токенизации

Токен — не символ. Один токен примерно равен 4 символам в английском тексте. В русском языке токены часто меньше из-за различий в морфологии:

- "Hello, how are you?" — примерно 6 токенов
- "Привет, как ты?" — примерно 8 токенов

**Ключевой вывод:** при оценке размера контекста нельзя просто делить количество символов на 4. Необходимо учитывать язык, структуру данных и форматирование.

### 1.3 Экономика токенов

Понимание стоимости токенов — ключ к эффективной разработке агентов. Ориентировочные цены (2026 год):

| Модель | Входные токены | Выходные токены | Cache Write | Cache Read |
|--------|----------------|-----------------|-------------|-----------|
| Claude Opus 4.6 | $3 за 1M | $15 за 1M | $3.75 за 1M | $0.30 за 1M |
| Claude Sonnet 4.6 | $3 за 1M | $15 за 1M | $3.75 за 1M | $0.30 за 1M |
| Claude Haiku 4.5 | $0.80 за 1M | $4 за 1M | $1 за 1M | $0.08 за 1M |

**Практический пример:** агент обрабатывает 100,000 токенов контекста 100 раз в день:
- Без кеша: 100,000 x 100 x $3 / 1,000,000 = **$30 в день**
- С prompt caching: одна запись ($3.75) + 99 чтений ($0.30 каждый) = **~$33.50 за 100 запросов** (но при 1000+ запросов выигрыш кратный)

Стоимость контекста растёт линейно. Удвоение контекста удваивает затраты. Это делает context engineering не только вопросом качества, но и вопросом экономики.

### 1.4 Проблемы большого контекста

Большое окно — не панацея. Исследования показывают серьёзные проблемы:

1. **Attention heads degradation** — модель хуже использует информацию в середине контекста (phenomenon «middle token problem»).
2. **Token dilution** — релевантная информация «размывается» нерелевантными данными. 100 релевантных предложений в 900 нерелевантных — потеря точности.
3. **Cost explosion** — линейный рост стоимости с размером контекста.
4. **Latency increase** — больший контекст требует больше вычислений, что критично в real-time сценариях.

### 1.5 Что происходит без управления контекстом

Если контекстом не управлять, появляются:
- **Drift** — агент начинает отклоняться от первоначальной цели.
- **Повторение шагов** — агент заново делает то, что уже было сделано.
- **Конфликтующие решения** — новые решения противоречат ранее принятым.
- **Рост стоимости без роста качества** — больше токенов, но не лучше результат.

## 2. Инженерия контекста как дисциплина

### 2.1 Почему это важнее prompt engineering

Традиционный prompt engineering сосредоточен на формулировке инструкции. Инженерия контекста фокусируется на том, **какие данные и в каком порядке** подаются модели:

- Какую историю разговора включать
- Какие примеры релевантны
- Как организовать информацию
- Когда очищать контекст
- Какие метаданные включать

Модель, получившая хороший контекст с плохим промптом, часто даёт лучший результат, чем модель с идеальным промптом, но плохим контекстом.

### 2.2 Три столпа контекстной инженерии

**1. Качество (Quality)**
- Только релевантная информация в контексте
- Отсечение шума и дублей
- Структурированное представление данных
- Минимизация противоречивой информации

**2. Эффективность (Efficiency)**
- Минимизация использования токенов
- Оптимизация стоимости вычислений
- Ускорение инференса через кеширование
- Компрессия без потери смысла

**3. Надёжность (Reliability)**
- Поведение модели остаётся предсказуемым
- Не происходит неожиданных «забываний»
- Сохранение контекста критических решений
- Воспроизводимость результатов

## 3. Два базовых механизма

### 3.1 Context Editing

Целенаправленно обновляет «рабочую картину мира» агента. Это хирургическая операция: конкретно что удалить, что оставить, что обновить.

Действия:
- **Удалить устаревшее** — результаты старых tool calls, которые уже обработаны.
- **Зафиксировать инварианты** — решения, которые не должны пересматриваться.
- **Обновить факты** — заменить устаревшую информацию свежей.

### 3.2 Server-side Context Editing

Server-side Context Editing позволяет давать указания API Anthropic на какие части контекста смотреть перед отправкой запроса к модели. Это происходит на стороне сервера, до того как контекст достигнет нейросетевой модели.

**Ключевое преимущество:** клиентское приложение сохраняет полную историю беседы, но сервер видит отредактированную версию. Полная история на клиенте для восстановления + экономия на токенах.

**Пример: `clear_tool_uses`**

Когда агент использует web_search 50 раз в рамках одной задачи, все 50 результатов попадают в контекст. Но для финального ответа нужны только последние несколько. Остальные занимают место и отвлекают модель.

Параметры конфигурации:

| Параметр | Описание | Рекомендация |
|----------|----------|--------------|
| `trigger` | Когда запускается очистка (по количеству токенов) | 80-90% от размера окна |
| `keep` | Сколько последних результатов сохранять | Минимум 3-5 |
| `clear_at_least` | Минимум токенов для удаления | 15,000-20,000 |
| `exclude_tools` | Какие инструменты никогда не удалять | critical_search, user_input |

**Стратегии:**

- **Aggressive Cleanup** (для длинных сеансов): trigger 80k, keep 3, clear_at_least 20k
- **Conservative Cleanup** (для критичных разговоров): trigger 150k, keep 10, clear_at_least 5k
- **Smart Hybrid**: разные параметры для разных инструментов

### 3.3 Client-side Context Editing

Client-side editing выполняется в приложении до отправки запроса к API. Приложение самостоятельно формирует контекст:

- Фильтрует историю сообщений по релевантности
- Удаляет или сжимает старые tool_use результаты
- Добавляет summary вместо полной истории
- Контролирует общий размер перед отправкой

**Когда что использовать:**

| Аспект | Server-side | Client-side |
|--------|-------------|-------------|
| Контроль | Ограниченный (параметры API) | Полный |
| Сложность | Низкая (настройка параметров) | Высокая (своя логика) |
| Надёжность | Высокая (гарантии API) | Зависит от реализации |
| Гибкость | Ограниченная | Максимальная |

### 3.4 Compaction

Compaction сжимает историю без потери причинно-следственных связей. В отличие от editing (который удаляет), compaction — суммаризирует.

**Различие:** Server-side editing удаляет ненужные части полностью. Compaction сжимает информацию, сохраняя суть. Вместо "Юзер спросил о погоде. API вернула температуру 25°C, влажность 60%, давление..." → "Погода: 25°C, облачно".

**Что нельзя потерять при compaction:**
1. Цель и критерии done.
2. Ключевые решения и почему они приняты.
3. Ограничения/риски/запреты.
4. Незакрытые вопросы и следующие шаги.

**Стратегии компрессии:**

| Стратегия | Описание | Экономия | Потери |
|-----------|----------|----------|--------|
| Summarization | Текстовое резюме всей истории | 60-80% | Детали реализации |
| Structural | Только структура: список решений | 80-90% | Контекст обсуждений |
| Semantic Clustering | Группировка по темам + резюме каждой | 70-80% | Минимальные |

**Рекомендации:**
- Для compaction используйте более дешёвую модель (Claude Haiku) — суммаризация не требует мощной модели.
- Компактируйте батчами, не по одному сообщению.
- Избегайте многократного компактирования (суммари от суммари теряет информацию).

## 4. ACE — эволюция контекста

### 4.1 Что такое ACE

ACE (Agentic Context Engineering) — исследовательский подход, описывающий как контекст агента должен эволюционировать со временем. Ключевая идея: контекст должен **расти как playbook**, а не «усыхать в один абзац».

### 4.2 Проблемы классических подходов

- **Brevity bias** — при compaction модель стремится сделать суммари как можно короче, теряя важные детали.
- **Context collapse** — после нескольких compaction весь контекст схлопывается в один параграф, из которого невозможно восстановить ход работы.

### 4.3 Модульный цикл ACE

ACE предлагает трёхфазный цикл:

1. **Generator** — создаёт новый контент на основе текущей работы (решения, факты, ошибки).
2. **Reflector** — анализирует качество контекста: что устарело, что противоречит, что избыточно.
3. **Curator** — обновляет контекст: удаляет устаревшее, добавляет новое, реструктурирует.

### 4.4 Incremental обновление

Обновление контекста — **incremental (`delta`)**, а не полная перезапись. Каждый цикл добавляет/удаляет конкретные элементы, сохраняя основу.

**Практический смысл:** долгие задачи (часы/дни) можно вести стабильно, если контекст обновляется структурно, а не пересоздаётся с нуля.

## 5. Паттерн long-running задачи

### 5.1 Четыре фазы

1. **Bootstrapping context** — цель, scope, ограничения, критерии stop. Формируется один раз в начале.
2. **Work chunks** — маленькие итерации, в конце каждой: status + evidence. Размер chunk: 5-15 минут работы.
3. **Compaction trigger** — по длине (80% окна), по шуму (повторы), по числу противоречий.
4. **Handoff summary** — что сделано, что осталось, где риск. Формируется при передаче другому агенту/человеку.

### 5.2 Practical pattern

```
1. План: scope, допущения, stop criteria
   ↓
2. Рабочие итерации с короткими checkpoints
   ↓
3. Trigger compaction каждые N шагов / при росте шума
   ↓
4. Перед handoff — финальная выжимка с action items
```

### 5.3 Сигналы, что пора делать compaction

- Агент повторяет уже принятые решения.
- Увеличилось число «уточняющих» шагов без прогресса.
- В ответах появляются противоречия к ранее зафиксированным правилам.
- Стоимость запросов выросла, а качество — нет.
- Агент начинает «забывать» constraints, заданные в начале.

### 5.4 Жизненный цикл контекста

```
Инициализация (системный промпт, начальные данные)
    ↓
Накопление (разговоры, tool calls, метаданные)
    ↓
Мониторинг размера (проверка порогов)
    ↓
Триггер очистки (контекст перегружен)
    ↓
Стратегия сохранения (что оставить, что удалить)
    ↓
Editing или compaction
    ↓
Восстановление из памяти (если нужна удалённая информация)
    ↓
Продолжение работы с очищенным контекстом
```

### 5.5 Когда инженерия контекста критична

| Сценарий | Критичность | Причина |
|----------|-------------|---------|
| Короткие задачи (< 10 мин) | Средняя | Контекст ещё не переполнен |
| Долгоживущие агенты (часы/дни) | Критическая | Контекст будет расти, нужна стратегия |
| Многопользовательские системы | Очень высокая | Контекст может конфликтовать |
| Real-time системы | Высокая | Задержки при росте контекста |
| Анализ больших документов | Критическая | Большой контекст изначально |

## 6. Prompt Caching

Prompt caching позволяет кешировать статичные части контекста (инструкции, примеры, документацию), чтобы не отправлять их с каждым запросом.

**Механизм:**
1. Отправляете промпт с флагом кеширования
2. API хеширует и кеширует промпт на сервере
3. Последующие запросы ссылаются на кеш
4. Cached токены стоят ~10% от обычной цены

**Когда эффективно:**
- Одна системная инструкция используется 100+ раз в день
- Большие статичные документы (API docs, knowledge base)
- Few-shot примеры, которые не меняются

**Когда неэффективно:**
- Промпты постоянно меняются
- Инструкции используются один раз
- Динамичные данные, которые часто обновляются

## 7. Шаблон compaction summary

```md
## Objective
[Текущая цель задачи]

## Done criteria
[Конкретные, проверяемые критерии завершения]

## Decisions (with rationale)
- Решение 1: [что] → [почему]
- Решение 2: [что] → [почему]

## Constraints / policies
- [Ограничение 1]
- [Ограничение 2]

## Open risks
- [Риск 1: описание + вероятность]

## Next actions (ordered)
1. [Действие 1]
2. [Действие 2]
```

## 8. Антипаттерны

1. **«Сохраняем всё»** → контекст шумит, модель теряет фокус, стоимость растёт.
2. **Сжатие без причин решений** → потеряно обоснование, агент может принять противоречащее решение.
3. **Обнуление ошибок** → теряется обучение, агент повторяет те же ошибки.
4. **Нет разделения discovery и implementation** → контекст перегружен промежуточными данными.
5. **Нет инвентаря фактов** → только «разговор», из которого невозможно извлечь структуру.
6. **Хранить «всё подряд» в основном контексте** → token dilution, потеря точности.
7. **Смешивать discovery и implementation без разгрузки** → context pollution.
8. **Слишком низкий trigger для compaction** → очистка срабатывает слишком часто, теряется свежий контекст.
9. **Многократное compaction (суммари от суммари)** → каскадная потеря информации.

---

## 9. Мини-чеклист качества

- [ ] После compaction можно восстановить ход ключевых решений.
- [ ] Есть явные next actions (не «продолжить работу», а конкретные шаги).
- [ ] Нет конфликтов с текущими constraints/policies.
- [ ] Контекст стал короче **и** полезнее, а не просто короче.
- [ ] Критичные tool results сохранены (exclude_tools настроен).
- [ ] Стоимость запросов после compaction снизилась.
- [ ] Агент не повторяет ранее принятые решения.

---

## 10. Практикум

Сделай на реальной задаче:
- минимум 3 итерации работы агента,
- минимум 1 compaction (с сохранением compaction summary),
- 1 handoff summary (передача задачи другому исполнителю).

**Метрика успеха:** после compaction новый исполнитель может продолжить работу без устных пояснений. Проверь: покажи handoff summary коллеге и спроси, может ли он продолжить задачу.

**Дополнительное задание:** сравни стоимость (в токенах) до и после compaction. Насколько уменьшился контекст? Что потерялось?
