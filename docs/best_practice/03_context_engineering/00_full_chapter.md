## Контекстное окно: фундамент

### Что такое контекстное окно?

Контекстное окно (context window) — это максимальное количество токенов, которое может обработать языковая модель за один проход. Это не просто техническое ограничение; это ограничивает возможности агента, стоимость операций и качество выходного результата.

Например, модель Claude Opus 4.6 имеет контекстное окно **200k токенов**. Это означает:
- Максимум 200,000 токенов входных данных + выходных токенов
- За пределами этого окна модель попросту не может видеть информацию

### Понимание токенизации

Токен — это не символ. Один токен примерно равен 4 символам в английском тексте. Например, фраза "Hello, how are you?" разбивается на ~6 токенов.

В русском языке ситуация иная — токены часто меньше из-за различий в морфологии. Фраза "Привет, как ты?" будет занимать примерно 8 токенов.

**Ключевой вывод**: при оценке размера контекста нельзя просто делить количество символов на 4. Необходимо учитывать язык, структуру данных и форматирование.

### Экономика токенов

Понимание стоимости токенов — ключ к эффективной разработке агентов. Рассмотрим модель цены (2026 год):

| Модель | Входные токены | Выходные токены | Cache Write | Cache Read |
|--------|-----------------|-----------------|-------------|-----------|
| Claude Opus 4.6 | $3 за 1M | $15 за 1M | $3.75 за 1M | $0.30 за 1M |
| Claude Sonnet 4.6 | $3 за 1M | $15 за 1M | $3.75 за 1M | $0.30 за 1M |
| Claude Haiku 4.5 | $0.80 за 1M | $4 за 1M | $1 за 1M | $0.08 за 1M |

Практический пример: если вы обрабатываете 100,000 токенов 100 раз в день:
- Без кеша: 100,000 × 100 × $3 / 1,000,000 = $30 в день
- С prompt caching: одна запись ($3.75) + 99 чтений ($0.30 каждый) = ~$33 за 100 запросов

Хотя это кажется дороже в краткосрочной перспективе, в долгосрочной перспективе кеш значительно снижает затраты, особенно при многократном использовании одного и того же набора данных.

### Ограничения и прогрессия

Современные модели расширяют контекстные окна:

| Квартал | Модель | Размер окна | Статус |
|---------|--------|-------------|--------|
| Q4 2023 | Claude 2.1 | 200k | Архив |
| Q2 2024 | Claude Opus 4 | 200k | Архив |
| Q2 2024 | Claude Sonnet 4 | 200k | Архив |
| Q4 2024 | Claude Opus 4.6 | 200k | Текущая |

Но большое окно — не панацея. Исследования показывают серьезные проблемы:

1. **Attention heads degradation** — модель хуже использует информацию в середине контекста. Феномен известен как "middle token problem": когда контекст растет, модель начинает хуже сосредотачиваться на информации в центре длинного контекста.

2. **Token dilution** — релевантная информация "размывается" нерелевантными данными. Если в контексте есть 100 релевантных предложений и 900 нерелевантных, модель теряет в точности.

3. **Cost explosion** — линейный рост стоимости с размером контекста. Удвоение контекста удваивает затраты.

4. **Latency increase** — больший контекст требует больше вычислений. Это может быть проблемой в real-time сценариях.

## Инженерия контекста как дисциплина

### Почему инженерия контекста важнее prompt engineering?

Традиционный prompt engineering сосредоточен на поиске идеальной формулировки инструкции. Вы изменяете слова, структуру инструкции, примеры в инструкции.

Инженерия контекста фокусируется на том, какие данные и в каком порядке подаются модели. Это включает:
- Какую историю разговора включать
- Какие примеры релевантны
- Как организовать информацию
- Когда очищать контекст
- Какие метаданные включать

**Почему это важнее?** Качество данных в контексте часто влияет на результат больше, чем формулировка промпта. Модель, получившая хороший контекст с плохим промптом, часто дает лучший результат, чем модель с идеальным промптом, но плохим контекстом.

### Три столпа контекстной инженерии

**1. Качество (Quality)**
- Только релевантная информация в контексте
- Отсечение шума и дублей
- Структурированное представление данных
- Минимизация противоречивой информации

**2. Эффективность (Efficiency)**
- Минимизация использования токенов
- Оптимизация стоимости вычислений
- Ускорение инференса
- Использование кеша и компрессии

**3. Надежность (Reliability)**
- Поведение модели остается предсказуемым
- Не происходит неожиданных "забываний"
- Сохранение контекста критических решений
- Воспроизводимость результатов

### Жизненный цикл контекста в агенте

Долгоживущий агент проходит через следующие этапы работы с контекстом:

```
Инициализация (системный промпт, начальные данные)
    ↓
Накопление информации (разговоры, результаты tool calls, метаданные)
    ↓
Мониторинг размера контекста (проверка порогов)
    ↓
Триггер очистки контекста (обнаружение того, что контекст перегружен)
    ↓
Стратегия сохранения (выбор: что оставить, что удалить?)
    ↓
Применение контекстного редактирования или компрессии
    ↓
Восстановление из персистентной памяти (если нужна информация об удаленных элементах)
    ↓
Продолжение работы агента с очищенным контекстом
```

Каждый этап требует решений:

| Этап | Вопросы | Решения |
|------|---------|----------|
| Накопление | Сохранять ли результаты tool calls? Все ли сообщения? | Selective persistence: сохранять только критичное |
| Мониторинг | Как часто проверять размер? | После каждого ответа или после N новых сообщений |
| Триггер | При какой точке срабатывает? | 80% окна или когда остается <20k токенов |
| Стратегия | Что удалять: старое или нерелевантное? | Комбинация LRU (удалять старое) + семантический анализ |
| Редактирование | Где реализовать логику? | Server-side API для надежности, client-side для контроля |
| Восстановление | Использовать базу знаний? | Да, если есть критичная информация вне контекста |

### Когда инженерия контекста критична

**Краткосрочные задачи (< 10 минут):** Контекстная инженерия важна, но менее критична. Контекст еще не переполнен.

**Долгоживущие агенты (часы/дни):** Критически важна. Контекст будет расти, нужна стратегия управления.

**Многопользовательские системы:** Очень важна. Контекст может конфликтовать между пользователями.

**Real-time системы (чаты, помощники):** Важна. Задержки при рост контекста недопустимы.

**Анализ больших документов:** Критична. Большой контекст изначально, нужна умная фильтрация.

## Server-side Context Editing

### Обзор и назначение

Server-side Context Editing позволяет вам давать указания API Anthropic *на какие части контекста смотреть* перед отправкой запроса к модели. Это происходит на стороне сервера, до того как контекст достигнет нейросетевой модели.

**Ключевое преимущество**: ваше клиентское приложение сохраняет полную историю беседы, но сервер видит отредактированную версию. Это дает вам лучшее из обоих миров: полная история на клиенте для восстановления информации, но экономия на использовании токенов.

### clear_tool_uses_20250919: очистка результатов инструментов

Самая частая стратегия — удаление результатов `tool_uses`. Когда агент использует инструмент (веб-поиск, чтение файла, запрос базы данных), результат добавляется в контекст. После обработки результата старые результаты часто становятся ненужными.

Например, агент может использовать web_search 50 раз в рамках одной задачи. Все 50 результатов попадают в контекст. Но для финального ответа нужны только последние несколько результатов. Остальные занимают место и отвлекают модель.

#### Как это работает

Вы передаете API инструкцию, в какой момент и как удалять старые результаты инструментов. API:

1. Отслеживает размер контекста по мере добавления новых сообщений
2. Когда размер достигает порога (например, 100k токенов), запускается очистка
3. Удаляются старые результаты tool_uses, но сохраняются последние N результатов
4. Это происходит на сервере, перед отправкой контекста модели

**Что сохраняется на клиенте:** Полная история всех tool_uses. Если позже потребуется информация из удаленного результата, можно восстановить из памяти агента.

**Что видит модель:** Очищенный контекст без старых результатов.

#### Параметры конфигурации

| Параметр | Тип | Дефолт | Описание |
|----------|-----|--------|----------|
| `trigger` | Объект | 100k токенов | Когда запускается очистка (по количеству входных токенов) |
| `keep` | Объект | 3 tool uses | Сколько последних результатов сохранять |
| `clear_at_least` | Объект | Не установлено | Минимум токенов для удаления (иначе пропускаем очистку) |
| `exclude_tools` | Список | Пусто | Какие инструменты никогда не удалять (например, critical_search) |
| `clear_tool_inputs` | Булево | False | Удалять ли параметры вызовов инструментов (обычно нет) |

#### Пример API запроса

При использовании context_management API вы указываете параметры в JSON:

```json
{
  "context_management": {
    "edits": [
      {
        "type": "clear_tool_uses_20250919",
        "trigger": {
          "type": "input_tokens",
          "value": 100000
        },
        "keep": {
          "type": "tool_uses",
          "value": 5
        },
        "clear_at_least": {
          "type": "input_tokens",
          "value": 15000
        },
        "exclude_tools": ["critical_search"],
        "clear_tool_inputs": false
      }
    ]
  }
}
```

**Что это означает:** Когда контекст достигнет 100k токенов, удалить результаты старых tool_uses, но сохранить 5 последних результатов и гарантировать, что удалятся минимум 15k токенов. Инструмент critical_search никогда не удалять.

#### Стратегии использования

**Стратегия 1: Aggressive Cleanup** — для длинных сеансов с множеством tool_uses
- `trigger`: 80,000 токенов
- `keep`: 3 последних результата
- `clear_at_least`: 20,000 токенов

Подходит для: Исследовательские агенты, web scrapers, data aggregators.

**Стратегия 2: Conservative Cleanup** — для критичных разговоров
- `trigger`: 150,000 токенов
- `keep`: 10 последних результатов
- `clear_at_least`: 5,000 токенов
- `exclude_tools`: ["critical_database", "user_input"]

Подходит для: Финансовые советы, медицинские рекомендации, legal analysis.

**Стратегия 3: Smart Hybrid** — комбинация подходов
- Разные параметры для разных инструментов
- Критичные инструменты исключены
- Менее важные инструменты очищаются агрессивнее

### Рекомендации и anti-patterns

**✓ Делайте:**
- Устанавливайте `trigger` на 80-90% от размера вашего окна
- Сохраняйте достаточно результатов для контекста (минимум 3-5)
- Используйте `exclude_tools` для критичных инструментов
- Мониторьте, какие инструменты генерируют больше всего токенов

**✗ Не делайте:**
- Не удаляйте критичные инструменты (user_input, decision_tools)
- Не используйте слишком низкий порог `keep` (модель потеряет контекст)
- Не игнорируйте `clear_tool_inputs` для функций с чувствительными параметрами
- Не устанавливайте `trigger` слишком низко (очистка будет срабатывать слишком часто)

## Compaction: автоматическое сжатие

### Различие между editing и compaction

**Server-side Context Editing** удаляет ненужные части контекста полностью.

**Compaction** сжимает информацию, сохраняя суть. Вместо удаления результата "Юзер спросил о погоде. API вернула температуру 25°C, влажность 60%, давление...", компактор создаст "Погода: 25°C, облачно".

Компактор использует LLM для суммаризации, что добавляет затраты, но сохраняет информацию в более экономном виде.

### Когда использовать compaction

Используйте компрессию когда:
1. Информация в контексте полезна, но многословна
2. Модель может самостоятельно восстановить детали, если нужно
3. Есть бюджет на дополнительные токены для суммаризации
4. Разговор долгий, и нужно сохранить всю историю в сжатом виде

Не используйте когда:
1. Нужна полная точность каждого деталя (финансы, медицина)
2. Компрессия стоит дороже, чем просто расширить окно
3. Информация уже в компактной форме

### Архитектура компактора

Типичный процесс выглядит так:

```
Контекст агента (растет со временем)
    ↓
Детектор: размер > 150k?
    ↓
Да → Выбрать старые сообщения для компактования
    ↓
Генератор суммари: LLM создает краткое резюме
    ↓
Замена: старые сообщения → резюме
    ↓
Результат: контекст сжат, но информация сохранена
```

### Стратегии компрессии

**Стратегия 1: Summarization** — создать текстовое резюме
- Входная информация: "Юзер обсудил A, потом B, потом C. Модель ответила X, потом Y."
- Выходное резюме: "Обсуждены вопросы A, B, C. Ответы основаны на X и Y."
- Экономия: часто 60-80% токенов

**Стратегия 2: Structural Compression** — оставить только структуру
- Входная информация: "Полный разговор с 20 сообщениями"
- Выходная структура: "Ключевые решения: [список из 5 пунктов]"
- Экономия: 80-90% токенов, но теряется некоторая информация

**Стратегия 3: Semantic Clustering** — сгруппировать по смыслу
- Входная информация: "30 обменов на разные темы"
- Выходная структура: "Тема 1 (10 обменов) → суммари. Тема 2 (15 обменов) → суммари."
- Экономия: 70-80% токенов, лучше сохраняется структура

### Обработка компактированного контекста

Когда контекст скомпактирован, модель видит суммари вместо оригинальных сообщений. Если модели нужны детали, она может:

1. **Запросить детали**: "Дай мне расширенную информацию о теме 1"
2. **Использовать контекст из суммари**: Суммари содержит достаточно информации для большинства задач
3. **Восстановить из памяти**: Если агент сохраняет оригинальные данные, их можно вернуть

### Рекомендации

**Для компактирования используйте:**
- Более слабую модель (Claude Haiku) для экономии (суммари не требует большой способности)
- Батching: компактируйте разом множество сообщений, не по одному
- Явные инструкции модели о том, что сохранять

**Избегайте:**
- Многократного компактирования (суммари от суммари теряет информацию)
- Компактирования критичных данных без сохранения оригинала
- Использования компактирования для реального времени (это медленно)

## Prompt Caching и взаимодействие

### Что такое prompt caching?

Prompt caching позволяет кешировать часть контекста (обычно статичные элементы вроде инструкций, примеров, документации), чтобы не отправлять их с каждым запросом.

**Механизм:**
1. Вы отправляете промпт с флагом кеширования
2. API Anthropic хеширует и кеширует этот промпт на сервере
3. Последующие запросы ссылаются на кеш вместо отправки всего промпта заново
4. Вы платите сниженный тариф за cached токены

**Экономия:** Cached токены стоят ~10% от обычной цены (например, $0.30 за 1M вместо $3).

### Когда кеширование эффективно

**Высокая эффективность:**
- Одна система инструкций, используется 100+ раз в день
- Большие статичные документы (API docs, knowledge base), которые используются многими пользователями
- Few-shot примеры, которые не меняются

**Низкая эффективность:**
- Промпты, которые постоянно меняются
- Личные инструкции, которые используются только один раз
- Динамичные данные, которые часто обновляются

### Архитектура кеширования

```
Запрос 1: "Вот инструкции + примеры + вопрос юзера"
    ↓
API: Хеширует инструкции + примеры, кеширует их
    ↓
Ответ 1 (оплачивается полная цена за инструкции)

Запрос 2: "Вот кешированные инструкции (ссылка) + новый вопрос"
    ↓
API: Использует кеш, не пересчитывает инструкции
    ↓
Ответ 2 (оплачивается только новый вопрос + 10% для кеша)
```

### Стратегии кеширования

**Стратегия 1: System Prompt Caching** — кешировать systemPrompt
- Что: Основные инструкции агента
- Как часто: Меняется редко (раз в неделю)
- Экономия: 20-30% затрат для типичного агента

**Стратегия 2: Example Caching** — кешировать примеры (few-shot)
- Что: 10-20 примеров вход-выход
- Как часто: Не меняется
- Экономия: 40-50% затрат, если много примеров

**Стратегия 3: Knowledge Base Caching** — кешировать статичную базу знаний
- Что: Документация, FAQ, reference материалы
- Как часто: Обновляется раз в месяц
- Экономия: 60-70% затрат для knowledge-heavy систем

### Взаимодействие с Context Editing

Context editing и prompt caching работают вместе:

```
Кеш (статичные инструкции) → API
                             ↓
                        Кеш применен
                             ↓
Dynamic контекст (история разговора) → API
                                       ↓
                                Context editing (удалить старое)
                                       ↓
Финальный контекст → Модель
```

**Порядок имеет значение:** Кеширование происходит перед Context Editing. Это означает:
1. Кеш не может быть отредактирован (это статичная инструкция)
2. Context Editing может удалять динамичные части после кешированного промпта

### Best Practices

**✓ Делайте:**
- Кешируйте системный промпт + инструкции (они не меняются)
- Кешируйте примеры и few-shot данные
- Разбивайте контекст на статичные (кешируемые) и динамичные (редактируемые) части
- Мониторьте hit rate кеша (сколько запросов используют кеш)

**✗ Не делайте:**
- Не кешируйте информацию, которая часто меняется (сегодняшнюю дату, цены)
- Не пытайтесь кешировать user input (это поражает весь смысл кеша)
- Не полагайтесь на кеш для чувствительных данных (используйте шифрование)

## Системы персистентной памяти

### Проблема с контекстным окном

Контекстное окно — это краткосрочная память. Оно удобно для текущего разговора, но:
- **Ограничено по размеру**: 200k токенов — это ~400 страниц текста
- **Энергозатратно**: Каждый токен требует вычислений
- **Дорого**: Большой контекст — большие счета
- **Хрупко**: При очистке контекста теряется информация

Для долгоживущих агентов нужна *персистентная память* — долгосрочное хранилище, из которого можно извлекать релевантную информацию.

### Архитектура многоуровневой памяти

```
Уровень 1: Контекстная память (краткосрочная)
  - Текущий разговор (4-10 последних сообщений)
  - Размер: 5-20k токенов
  - Скорость доступа: Максимальная (уже в контексте)
  - Жизненный цикл: Сессия или день

         ↓ (выгрузка перед удалением)

Уровень 2: Семантическая память (среднесрочная)
  - Сжатая информация о разговорах, решения, факты
  - Размер: 100k+ токенов
  - Скорость доступа: Медленнее (нужен семантический поиск)
  - Жизненный цикл: Месяцы, годы

         ↓ (обобщение)

Уровень 3: Эпизодическая память (долгосрочная)
  - Дате-ориентированные события, важные моменты
  - Размер: Неограниченно
  - Скорость доступа: Медленная (поиск по индексам)
  - Жизненный цикл: Всегда
```

### Системы памяти: обзор

**Mem0** — умная персистентная память для AI агентов
- Автоматически извлекает ключевые информацию из разговоров
- Сохраняет факты, предпочтения, контекст
- Пример: "Юзер предпочитает краткие ответы" → сохраняется как факт, затем включается в системный промпт

**Graphiti** — граф знаний для AI агентов
- Представляет информацию как граф (узлы = сущности, рёбра = отношения)
- Позволяет моделировать сложные отношения между объектами
- Пример: "Юзер работает в компании X, компания X делает продукт Y, продукт Y конкурирует с Z"

**Встроенная векторная база (Vector DB)** — простой подход
- Каждое сообщение кодируется в вектор
- При новом вопросе ищется релевантные векторы
- Быстро, дешево, но не такой умный, как граф

### Когда использовать какую систему

| Сценарий | Mem0 | Graphiti | Vector DB |
|----------|------|----------|-----------|
| Личный ассистент (1 юзер) | ✓ Хорошо | ✓ Отлично | Хорошо |
| Чат-бот (много юзеров) | ✓ Отлично | Хорошо | ✓ Хорошо |
| Аналитический агент | Хорошо | ✓ Отлично | Хорошо |
| Реко-движок | Хорошо | ✓ Очень хорошо | Хорошо |
| Быстрая, дешевая система | Хорошо | Хорошо | ✓ Отлично |

### Интеграция памяти с контекстом

Типичный поток:

```
1. Новый вопрос от юзера
   ↓
2. Поиск в персистентной памяти по релевантности
   ↓
3. Извлечение топ-K фактов/эпизодов
   ↓
4. Добавление в контекст (перед историей разговора)
   ↓
5. Отправка модели (контекст = воспоминания + текущий разговор)
   ↓
6. Модель отвечает с учетом воспоминаний
   ↓
7. После ответа: сохранить новую информацию в память
```

### Best Practices

**Для всех систем памяти:**
- Не сохраняйте шумные данные (окончание фраз, повторения)
- Нормализуйте информацию перед сохранением (удалите дубли)
- Периодически переиндексируйте (для Vector DB)
- Используйте TTL (time-to-live) для старой информации

**Специфично для Mem0:**
- Используйте для извлечения структурированных фактов
- Хорошо для личных предпочтений и персонализации

**Специфично для Graphiti:**
- Используйте, если отношения между объектами важны
- Требует больше вычислений, но дает лучшие результаты

**Специфично для Vector DB:**
- Используйте для быстрого прототипирования
- Будьте осторожны с качеством эмбеддингов

## ACE: Agentic Context Engineering

### Что такое ACE?

ACE (Agentic Context Engineering) — это методология управления контекстом специально для агентов. На отличие от chatbot, где контекст меняется медленно, в агентах контекст может расти быстро (많이 tool_uses, многих итераций).

ACE определяет:
1. Как аккумулировать контекст (что сохранять)
2. Как выбирать контекст для каждого запроса (relevance ranking)
3. Когда очищать контекст (trigger strategy)
4. Как восстанавливать контекст из памяти (retrieval strategy)

### Компоненты ACE

**1. Context Accumulation Policy** — что добавлять в контекст?

| Тип информации | Сохранять? | Причина |
|----------------|-----------|---------|
| Tool calls | Да, но сжимать | Нужно понять, что делал агент |
| Tool results | Зависит | Большие результаты убирать после обработки |
| Prompts (система) | Да, кешировать | Не меняется, можно кешировать |
| Промежуточные выводы | Выборочно | Сохранять только важные |
| Ошибки агента | Да | Нужно для обучения и отладки |

**2. Relevance Ranking** — для каждого нового запроса выбирать релевантный контекст

При большом контексте не всё нужно передавать модели. Вместо этого:
- Ранжируйте контекст по релевантности
- Передавайте только топ-K элементов (например, 10 из 100)
- Остаток оставляйте в памяти на случай, если нужен

**Как ранжировать?**
- Семантический поиск (вектора)
- Временной приоритет (более свежие = важнее)
- Явные метки (пользователь помечает, что важно)

**3. Cleanup Triggers** — когда очищать контекст?

Вместо простого порога размера, используйте комбинацию:
- Размер контекста > 150k токенов
- ИЛИ количество tool_uses > 30
- ИЛИ время сессии > 2 часов
- ИЛИ обнаружена смена темы разговора

**4. Recovery Strategy** — как восстановить информацию после очистки?

Когда контекст очищен, но модель нуждается в информации:
1. **Explicit recovery**: модель просит "Дай мне информацию о X"
2. **Implicit retrieval**: система автоматически ищет в памяти
3. **Hybrid**: комбинация — система предлагает релевантное

### Примеры ACE в действии

**Пример 1: Исследовательский агент**

```
Агент должен исследовать 50 компаний и написать отчет.

Accumulation: Сохранять название компании, основные метрики, но не сырые данные
Relevance: При обобщении — выбирать компании, похожие на текущую
Cleanup trigger: Когда исследовано 10 компаний, архивировать старые результаты
Recovery: Если нужна информация о компании #1, поиск в памяти
```

**Пример 2: Коде-генератор**

```
Агент должен написать код для сложного проекта (100+ файлов).

Accumulation: Сохранять архитектуру, зависимости, интерфейсы, но не полный код
Relevance: Для каждого файла — найти связанные файлы в памяти
Cleanup trigger: Когда генерировано 20 файлов, архивировать детали старых файлов
Recovery: Если нужны детали — поиск в памяти + загрузка в контекст
```

### Фреймворк принятия решений

При проектировании ACE для вашего агента, ответьте на вопросы:

| Вопрос | Ответ влияет на |
|--------|-----------------|
| Как часто меняется контекст? | Trigger strategy |
| Насколько большой может быть контекст? | Cleanup strategy |
| Какая информация критична? | Exclusion list |
| Сколько можно потратить на память? | Storage choice (Vector DB vs Graph) |
| Нужна ли точность или скорость? | Relevance ranking method |

## Code Context: управление контекстом кодовых баз

### Проблема: большие кодовые базы в контексте

Когда вы просите моделью помочь с кодом, возникает проблема:
- Полная кодовая база может быть 10MB+ (слишком большая для контекста)
- Релевантные части могут быть разбросаны по разным файлам
- Модель нуждается в контексте (архитектура, зависимости), а не в сыром коде

**Решение:** Code Context — специализированная система управления контекстом для кода.

### Структурирование кода для контекста

**Уровень 1: Architecture Overview**
```
Проект структурирован как:
- Backend: FastAPI приложение в /src/backend
- Frontend: React в /src/frontend
- Database: PostgreSQL, миграции в /migrations
- Tests: pytest в /tests

Основные зависимости: FastAPI, SQLAlchemy, React, PostgreSQL
```

Это тонкий слой контекста (~1k токенов), который помогает модели понять структуру.

**Уровень 2: Relevant Files Index**
```
Когда юзер спрашивает про "аутентификацию", система автоматически идентифицирует:
- /src/backend/auth/models.py
- /src/backend/auth/handlers.py
- /src/backend/auth/middleware.py
- /src/backend/auth/utils.py

И выбирает только эти файлы для контекста, пропуская /src/backend/search, /src/backend/analytics и т.д.
```

**Уровень 3: Semantic Code Selection**
```
Не весь код файла нужен. Из файла /src/backend/auth/handlers.py выбираются:
- Функции, относящиеся к текущему вопросу
- Вызываемые ими функции
- Импорты и тип-определения

Пропускаются: логирование, мониторинг, старый код в комментариях
```

### Инструменты и подходы

**Подход 1: AST-based Selection** — анализ синтаксического дерева
- Инструменты: tree-sitter, Roslyn (для C#), AST парсеры
- Преимущество: точное понимание структуры кода
- Недостаток: нужны парсеры для каждого языка

**Подход 2: Semantic Search** — поиск по смыслу
- Инструменты: CodeBERT, code2vec embeddings
- Преимущество: работает для любого языка
- Недостаток: может выбрать ненужные файлы

**Подход 3: Dependency Graph** — анализ зависимостей
- Инструменты: custom static analysis
- Преимущество: находит все связанный код
- Недостаток: может выбрать слишком много

### Стратегия для разных сценариев

**Сценарий: "Помоги с bagfix"**
1. Используй semantic search (найди похожий на ошибку код)
2. Выбери файлы из найденных
3. Используй dependency graph (найди код, который вызывает найденные файлы)
4. Добавь юнит-тесты для этого функционала

**Сценарий: "Рефакторь эту функцию"**
1. Найди функцию (AST-based)
2. Найди все вызывающие эту функцию (dependency graph)
3. Покажи интерфейс функции и реализацию

**Сценарий: "Поясни архитектуру"**
1. Покажи структуру папок (simple tree)
2. Покажи основные классы/модули (AST overview)
3. Не вводи в контекст фулл код файлов

### Context Management для кода

Когда агент работает с кодом:

```
1. Юзер спрашивает про фичу/баг
   ↓
2. Поиск релевантных файлов (по названию, semantic search)
   ↓
3. Выбор необходимых частей кода (AST, dependencies)
   ↓
4. Добавление архитектурного контекста (README, docs)
   ↓
5. Отправка модели (архитектура + релевантный код)
   ↓
6. Модель генерирует решение
   ↓
7. Если модель просит больше контекста, повторить шаг 2-3
```

### Best Practices

- **Не отправляйте весь репозиторий** — выбирайте только релевантное
- **Показывайте архитектуру перед кодом** — контекст важен
- **Включайте тесты** — модель видит expected behavior
- **Используйте типизацию** — type hints помогают модели
- **Избегайте минифицированного кода** — читаемость важна

## Практические паттерны и пороги

### Пороги для различных сценариев

Использование контекстного окна зависит от типа задачи:

| Сценарий | Рекомендуемое окно | % используемых токенов | Причина |
|----------|-------------------|----------------------|---------|
| Чат (1 юзер) | 32k-64k | 40-60% | Много истории не нужно |
| Помощник (памятка) | 64k-128k | 60-75% | Нужна память + контекст |
| Аналитика (аген) | 128k-200k | 70-85% | Много данных, tool_uses |
| Кодовая база | 100k-200k | 75-90% | Архитектура + код |
| Обучение модели | Зависит | 90%+ | Максимум информации |

### Сигналы, что контекст переполнен

**Сигнал 1: Качество ответов падает**
- Модель начинает забывать детали
- Начинаются противоречия в ответах
- Появляются "галлюцинации"

**Сигнал 2: Скорость растет, но вниз**
- Инференс становится медленнее (больше обработки большего контекста)
- Но качество ответов не улучшается

**Сигнал 3: Стоимость растет экспоненциально**
- Каждое сообщение дороже, чем раньше
- Нет корреспондирующего улучшения качества

### Метрики для мониторинга

Отслеживайте эти метрики в production:

```
1. Context Size Per Request (средний размер контекста)
   Целевое значение: 80% от окна модели

2. Token Count Growth Rate (как быстро растет контекст)
   Целевое значение: <10% в час для чат-бота

3. Cost Per Request
   Целевое значение: <50% от бюджета на нижнее квартиле

4. Context Cleanup Frequency
   Целевое значение: не чаще, чем раз в N сообщений (зависит от сценария)

5. Information Retrieval Accuracy
   Целевое значение: >90% нужной информации в контексте
```

### Шаблоны для разных задач

**Шаблон 1: Долгосрочный chat-bot**
- Context window: 100k
- Server-side editing: yes (clear_tool_uses)
- Persistent memory: yes (Mem0)
- Trigger: 80% контекста или 500 сообщений
- Keep: 10 последних tool_uses
- Recovery: Vector DB поиск

**Шаблон 2: Аналитический агент**
- Context window: 200k
- Server-side editing: yes (aggressive)
- Persistent memory: yes (Graphiti)
- Trigger: 150k токенов или 100 tool_uses
- Keep: 5 последних важных результатов
- Recovery: Graph retrieval + semantic search

**Шаблон 3: Code assistant**
- Context window: 128k
- Server-side editing: selective (не трогать код)
- Persistent memory: maybe (индекс файлов)
- Trigger: отдельная логика для кода
- Keep: последние успешные компиляции
- Recovery: AST-based retrieval

2. Проверить, что контекст управляется правильно (логи показывают очистку)
3. Проверить, что память сохраняет и использует факты
4. Вычислить общую стоимость в токенах vs. без управления контекстом

**Бонус:**
- Добавить сохранение истории разговора
- Добавить восстановление из истории (новая сессия может использовать старую память)
- Добавить метрики качества (использование памяти, точность поиска)

## Выводы

Инженерия контекста — это дисциплина, которая становится всё более важной по мере того, как агенты становятся сложнее и работают дольше.

**Основные идеи:**

1. **Context is a resource** — как память в старых компьютерах, контекст нужно управлять
2. **Quality > Quantity** — 50k высокорелевантных токенов > 200k токенов с шумом
3. **Layered approach** — краткосрочный контекст + среднесрочная память + долгосрочная база
4. **Tooling matters** — правильные инструменты (server-side editing, caching, compaction) решают 80% проблем
5. **Monitoring is key** — вы не можете управлять тем, что не мер меритесь

На этом модуль завершен. Далее следует модуль 3, посвященный специальным техникам для различных типов агентов.
