# AI-Driven Development — LLM как команда ролей

## Введение

В предыдущем уроке мы разобрали Spec-Driven Development — подход, где спецификация становится артефактом намерения. Теперь пойдём дальше: как организовать саму работу с AI так, чтобы LLM была не «одним большим мозгом», а полноценной **командой специалистов** с понятными ролями, зонами ответственности и quality-гейтами на каждом шаге.

Этот подход называется **AI-Driven Development (AIDD)** — и он превращает хаотичный вайб-кодинг в инженерный процесс, встроенный в привычный SDLC.

---

## 1. От «одного мозга» к команде ролей

### 1.1. Проблема «героя-одиночки»

Типичный режим работы с LLM: написал промпт, нажал Enter, получил код, запустил. Быстро, удобно и наполнено магией. Но для полноценной продуктовой разработки это не работает: архитектура, тесты, ревью и релизы не ложатся на слабо предсказуемый код.

**Что идёт не так при подходе «сделай всё»:**

- Модель одновременно пытается быть аналитиком, архитектором, разработчиком и тестировщиком
- Нет разделения ответственности — каждый запрос начинается с нуля
- Нет контроля качества между этапами — ошибка на этапе анализа проникает в код
- Результат непредсказуем и невоспроизводим

### 1.2. Идея AIDD

AI-Driven Development — это не волшебная методология, а приземлённый процесс:

> LLM — не оракул «сделай всё», а **команда специалистов**, у каждого из которых своя зона ответственности и выходные артефакты. Эта команда помогает на каждом шаге привычного SDLC.

Суть подхода: вместо одного промпта «написать эндпоинт» мы проходим цепочку:

```
соглашения → спецификация → архитектура → задачи →
→ реализация → ревью и тесты → финальное QA → документация
```

На каждом шаге работает отдельная «роль» — это может быть отдельный subagent, отдельный промпт или отдельная slash-команда.

## 2. Восемь ролей AI-команды

### 2.1. Полный набор ролей

| Роль | Зона ответственности | Вход | Выход |
|------|---------------------|------|-------|
| **Analyst** | Сбор требований, Q&A, формирование PRD | Идея, контекст | `docs/prd/<ticket>.prd.md` |
| **Researcher** | Исследование кодовой базы и ограничений | PRD, кодовая база | `docs/research/<ticket>.md` |
| **Planner** | Архитектура, ADR, план реализации | PRD, research | `docs/plan/<ticket>.md` |
| **Task Planner** | Декомпозиция плана на задачи | PRD, план | `docs/tasklist/<ticket>.md` |
| **Implementer** | Написание кода маленькими шагами | План, tasklist, conventions | Код + тесты, обновлённый tasklist |
| **Reviewer** | Code review, архитектурные и стилевые проблемы | Diff, PRD, план, conventions | Замечания, follow-up задачи |
| **QA** | Тест-план, поиск уязвимостей | PRD, план, tasklist | `reports/qa/<ticket>.md` |
| **Tech Writer** | Документация | PRD, план, QA-отчёты | CHANGELOG, summary |
| **Validator** | Контроль quality-гейтов | Все артефакты | Статусы гейтов |

### 2.2. Analyst — аналитик

**Задача:** Превратить сырую идею и артефакты из репозитория в понятный, структурированный PRD.

**Что делает:**
- Создаёт `docs/prd/<ticket>.prd.md` по шаблону
- Заполняет разделы: контекст, цели, user stories, сценарии, метрики, ограничения, риски
- Задаёт уточняющие вопросы
- Явно выписывает нерешённые вопросы в «Открытых вопросах»

**Правило:** Не придумывает бизнес-требования, если они не следуют из контекста. Если информации не хватает — фиксирует пробелы.

**Пример вызова:** `/idea T-104 "GET /invoices/{id} в сервисе billing-api"`

### 2.3. Researcher — исследователь

**Задача:** Понять, как текущий код и инфраструктура влияют на реализацию тикета.

**Что делает:**
- Сканирует ключевые директории проекта (src, docs, конфиги)
- Находит связанные модули, сущности и зависимости
- Описывает существующие эндпоинты и контракты
- Фиксирует ограничения и риски

**Ключевое отличие от Analyst:** Researcher работает с кодом и технической реальностью, Analyst — с бизнес-требованиями. Researcher не меняет код, только собирает информацию.

### 2.4. Planner — архитектор

**Задача:** На основе PRD и исследования спроектировать архитектуру решения.

**Что делает:**
- Читает PRD и research
- Проектирует компоненты, интерфейсы, потоки данных
- Описывает NFR (Non-Functional Requirements)
- Фиксирует риски и альтернативы
- При необходимости создаёт ADR (Architecture Decision Record)

**Правило:** Соблюдает слои и запреты из `conventions.md`. Явно описывает принятые trade-offs.

### 2.5. Implementer — разработчик

**Задача:** Реализовать задачи из tasklist небольшими, согласованными шагами.

**Как работает:**

1. Находит первую незакрытую задачу в `docs/tasklist/<ticket>.md`
2. Формулирует план изменений (файлы, сущности, тесты)
3. Ждёт явного подтверждения (опционально)
4. Вносит минимально необходимые изменения
5. Добавляет / обновляет тесты
6. Отмечает задачу как выполненную
7. Показывает diff
8. **Останавливается** — не переходит к следующей задаче без команды

**Правила:**
- Без явного подтверждения пользователя не вносит правок
- Соблюдает conventions.md и существующую архитектуру
- Всегда добавляет тесты для реализуемой задачи

### 2.6. Reviewer — ревьюер

**Задача:** Проверить изменения на соответствие PRD, плану и conventions.

**Что проверяет:**
- Архитектурные инварианты
- Безопасность
- Читаемость
- Соответствие плану и спецификации

**Формат замечаний:**
- **Blocking** — что нужно исправить до мержа
- **Important** — желательно поправить
- **Nit** — косметика

**Правило:** Не придирается к стилю, если он не противоречит conventions.md. Фокус на архитектуре и безопасности.

### 2.7. QA — тестировщик

**Задача:** Подумать, как сломать то, что сделали, и зафиксировать результат.

**Что делает:**
- Собирает список сценариев (позитивных и негативных)
- Делит их на автотесты и ручные проверки
- Определяет edge-cases
- Оформляет отчёт в `reports/qa/<ticket>.md`
- Даёт рекомендацию: выпускать / с оговорками / не выпускать

### 2.8. Tech Writer — техписатель

**Задача:** Привести документацию в порядок после всех реализаций.

**Что создаёт:**
- `<ticket>-summary.md` — краткое описание сделанного и принятых решений
- Обновлённый `CHANGELOG.md`

**Правило:** Пишет так, чтобы новому разработчику и дежурному по инцидентам было понятно без чтения кода.

### 2.9. Validator — валидатор

**Задача:** Следить за прохождением quality-гейтов.

**Что делает:**
- Ищет связанные артефакты (PRD, план, tasklist, QA)
- Оценивает каждый гейт: `PRD_READY`, `PLAN_APPROVED`, `TASKLIST_READY`, `IMPLEMENT_STEP_OK`, `REVIEW_OK`, `RELEASE_READY`, `DOCS_UPDATED`
- Возвращает сводку: что OK, что missing, что сыро

**Правило:** Не изменяет артефакты, только читает. Консервативен: если не уверен, что гейт пройден, помечает как требующий внимания.

## 3. Quality-гейты: стоп-сигналы между этапами

### 3.1. Зачем нужны гейты

Без quality-гейтов AIDD превращается в тот же вайб-кодинг, только с красивыми файлами. Гейты — это **стоп-сигналы**, которые не дают перескочить через важные этапы.

### 3.2. Набор гейтов

| Этап | Гейт | Что проверяем |
|------|-------|---------------|
| Agreements (проект) | `AGREEMENTS_ON` | Есть conventions, CLAUDE.md, workflow, базовые агенты |
| PRD | `PRD_READY` | PRD заполнен, нет блокирующих вопросов, есть метрики |
| Архитектура + ресерч | `PLAN_APPROVED` | Архитектура описана, учтены риски и NFR, ADR согласован |
| Задачи | `TASKLIST_READY` | Есть tasklist, задачи мелкие, с acceptance-критериями |
| Реализация (по шагу) | `IMPLEMENT_STEP_OK` | Код + тесты + обновлённый tasklist по текущей задаче |
| Review | `REVIEW_OK` | Изменения прошли CI и ревью, нет blocking-замечаний |
| QA / релиз | `RELEASE_READY` | Релиз прошёл QA, критических багов нет |
| Документация | `DOCS_UPDATED` | Архитектура / runbook / changelog обновлены |

### 3.3. Три уровня реализации гейтов

**Уровень 1 — Ручной:**
Статус в Markdown + чек-лист. Пример: `Status: PRD_READY` в `docs/prd/T-104.prd.md`.

**Уровень 2 — AI-валидатор:**
Команда `/validate`, которая смотрит PRD, план, tasklist, QA и пишет, что уже ок, а что не дотянуто.

**Уровень 3 — Жёсткий (hooks):**
Hooks в `.claude/settings.json`, которые физически блокируют `Edit/Write`, если не выполнены гейты по активному тикету.

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": ".claude/hooks/gate-workflow.sh"
          }
        ]
      }
    ]
  }
}
```

## 4. Практическая реализация в Claude Code

### 4.1. Agreements — конституция процесса

Agreements — это договоры между вами и LLM: как писать код, в каком порядке работать, когда можно переходить на следующий шаг.

**Файловая структура:**

```
conventions.md          # Общие конвенции для людей
CLAUDE.md               # Инструкция для Claude Code
workflow.md             # Описание процесса

.claude/
  agents/               # Роли (subagents)
    analyst.md
    researcher.md
    planner.md
    implementer.md
    reviewer.md
    qa.md
    validator.md
    tech-writer.md
    task-planner.md
  commands/             # Slash-команды
    idea.md
    researcher.md
    plan.md
    review-prd.md
    tasks.md
    implement.md
    review.md
    qa.md
    validate.md
    feature-development.md
    docs-update.md
  settings.json         # Hooks и настройки
  hooks/
    gate-workflow.sh

docs/
  .active_ticket
  prd/
  plan/
  tasklist/
  research/

reports/
  qa/
```

### 4.2. Разделение ответственности по файлам

| Файл | Для кого | Содержание |
|------|----------|------------|
| `conventions.md` | Для людей | Кодстайл, архитектурные принципы, требования к тестам, правила работы с секретами |
| `CLAUDE.md` | Для Claude Code | Как использовать conventions, как работать с тикетами, что делать до/после правок |
| `workflow.md` | Для всех | Шаги pipeline, роли, quality-гейты, критерии «готово» |

**Пример фрагмента CLAUDE.md:**

```markdown
- Всегда опирайся на conventions.md.
- Перед любыми правками кода:
  1. Найди активный тикет в docs/.active_ticket.
  2. Посмотри PRD, план и tasklist по нему.
  3. Сформулируй план изменений и дождись моего «ОК».
- После правок:
  - Покажи diff
  - Предложи текст коммита с тикетом
  - Обнови чекбоксы в docs/tasklist/<ticket>.md
  - Остановись и жди ревью
```

### 4.3. Роли как subagents

Роли оформляются в `.claude/agents/*.md`. У каждого файла:

- YAML-фронтматтер (`name`, `description`, `tools`, `model`, `permissionMode`)
- «Должностная инструкция»: входные артефакты, ожидаемый выход, ограничения

**Пример `.claude/agents/analyst.md`:**

```yaml
---
name: analyst
description: "Собирает исходную идею, уточняет требования
и оформляет PRD по тикету."
tools: Read, Glob, Grep, Write
model: inherit
---
```

```markdown
## Роль
Ты — продуктовый аналитик. Твоя задача — превратить сырую идею
и артефакты из репозитория в понятный, структурированный PRD.

## Входные артефакты
- docs/.active_ticket — текущий тикет.
- docs/prd/<ticket>.prd.md — черновик PRD (если есть).
- docs/research/<ticket>.md — отчёт ресерча (если есть).

## Выход
- Обновлённый docs/prd/<ticket>.prd.md:
  - цель и контекст,
  - user stories и сценарии,
  - метрики и критерии успеха,
  - ограничения и риски,
  - открытые вопросы.

## Правила
- Не придумывай бизнес-требования, если они не следуют из контекста.
- Если информации не хватает — явно выпиши вопросы
  в «Открытых вопросах».
```

### 4.4. Команды vs. Агенты

Важное различие:

- **Агент (subagent)** — **кто** делает (роль / персона). Пример: `analyst`, `planner`, `qa`
- **Команда (/slash-команда)** — **что и когда** делаем. Пример: `/idea`, `/plan`, `/qa`

Связка команд и агентов:

| Команда | Агент | Действие |
|---------|-------|----------|
| `/idea` | analyst | Создать PRD по идее |
| `/researcher` | researcher | Исследовать кодовую базу |
| `/plan` | planner | Спроектировать архитектуру |
| `/tasks` | task-planner | Декомпозировать на задачи |
| `/implement` | implementer | Реализовать следующую задачу |
| `/review` | reviewer | Code review |
| `/qa` | qa | Финальное QA |
| `/validate` | validator | Проверить гейты |
| `/docs-update` | tech-writer | Обновить документацию |
| `/feature-development` | (оркестратор) | Весь pipeline |

### 4.5. Артефакты тикета

Для каждого тикета заводим мини-набор файлов — это **структурированная память проекта**, а не «последние 20 сообщений в чате»:

```
docs/prd/T-104.prd.md       # Product Requirements Document
docs/plan/T-104.md          # Архитектура и план
docs/tasklist/T-104.md      # Чеклист задач
docs/research/T-104.md      # Отчёт по кодовой базе
reports/qa/T-104.md          # Отчёт QA-агента
```

## 5. Пошаговый пример: тикет T-104

Рассмотрим весь workflow на конкретной задаче:

> **T-104:** `GET /invoices/{id}` в сервисе `billing-api`

### 5.1. Шаг 1: Идея и PRD

```
/idea T-104 "GET /invoices/{id} в сервисе billing-api"
```

Analyst создаёт `docs/prd/T-104.prd.md` по шаблону, задаёт вопросы, помогает докрутить требования.

**Гейт:** `PRD_READY` — PRD заполнен, нет блокирующих вопросов, есть метрики.

### 5.2. Шаг 2: Ресерч и архитектура

```
/researcher T-104
/plan T-104
```

Researcher сканирует кодовую базу, описывает связанные модули. Planner проектирует решение: компоненты, интерфейсы, потоки данных, NFR, риски.

**Гейт:** `PLAN_APPROVED` — архитектура описана, учтены риски, статус `PLAN_APPROVED`.

### 5.3. Шаг 3: Tasklist

```
/tasks T-104
```

Task-planner создаёт `docs/tasklist/T-104.md`:

```markdown
# T-104: GET /invoices/{id}
Status: TASKLIST_READY

- [ ] OpenAPI: docs/api/T-104.yaml
  - Описание эндпоинта, коды ошибок, примеры.
- [ ] Controller: InvoiceController.getById
  - Обработка ошибок, маппинг DTO.
- [ ] Service: InvoiceService.findByIdWithPermissions
  - Проверка прав + доступ к репозиторию.
- [ ] IT-тесты
  - Позитивные и негативные сценарии.
```

**Гейт:** `TASKLIST_READY` — задачи мелкие, с acceptance-критериями.

### 5.4. Шаг 4: Итеративная реализация

```
/implement T-104
```

Implementer находит первую незакрытую задачу, формулирует план, ждёт OK, реализует, показывает diff, останавливается.

```
/implement T-104  (→ следующая задача)
/implement T-104  (→ следующая задача)
...
```

**Гейт:** `IMPLEMENT_STEP_OK` — код написан, тесты обновлены, tasklist обновлён.

### 5.5. Шаг 5: Ревью

```
/review T-104
```

Reviewer смотрит diff, проверяет соответствие конвенциям, подсвечивает проблемы.

**Гейт:** `REVIEW_OK` — CI зелёный, нет blocking-замечаний.

### 5.6. Шаг 6: QA

```
/qa T-104
```

QA формирует сценарии, определяет что покрыто автотестами, даёт рекомендацию.

**Гейт:** `RELEASE_READY` — критических багов нет.

### 5.7. Шаг 7: Документация

```
/docs-update T-104
```

Tech Writer создаёт summary и обновляет CHANGELOG.

**Гейт:** `DOCS_UPDATED`.

## 6. Оркестратор: «одна кнопка»

### 6.1. Как работает оркестратор

После того как отдельные команды и агенты обкатаны, можно создать одну команду, которая знает про весь pipeline:

```
/feature-development T-104 "GET /invoices/{id} в billing-api"
```

Оркестратор:

1. Вызывает `/validate T-104`
2. Показывает, какие гейты пройдены
3. Предлагает следующие шаги:
   - Нет PRD → предлагает `/idea`
   - Нет плана → предлагает `/plan`
   - Нет tasklist → предлагает `/tasks`
   - Реализация в процессе → предлагает `/implement`
   - Нет QA → предлагает `/qa`
   - Нет документации → предлагает `/docs-update`

Оркестратор **не изменяет код** — только описывает состояние и next steps.

### 6.2. Headless-режим и CI

Claude Code умеет работать в headless-режиме (`claude -p`, `--permission-mode plan`). Это позволяет:

- По cron прогонять `/validate` по всем тикетам релиза
- Автоматически собирать сводки по гейтам
- Готовить PRD/планы перед ревью
- Интегрировать AIDD в CI-пайплайн

## 7. Hooks: жёсткий контроль процесса

### 7.1. Зачем нужны hooks

Hooks — опциональный уровень, когда вы хотите, чтобы LLM **физически не могла** обойти этапы.

### 7.2. Пример gate-workflow.sh

```bash
#!/usr/bin/env bash
set -euo pipefail

TICKET=$(cat docs/.active_ticket 2>/dev/null || true)
if [ -z "$TICKET" ]; then
  echo "Workflow gate failed: нет активного тикета." >&2
  exit 2
fi

PRD_STATUS=$(grep -m1 '^Status:' "docs/prd/$TICKET.prd.md" 2>/dev/null || true)
PLAN_STATUS=$(grep -m1 '^Status:' "docs/plan/$TICKET.md" 2>/dev/null || true)
TASK_STATUS=$(grep -m1 '^Status:' "docs/tasklist/$TICKET.md" 2>/dev/null || true)

if [[ "$PRD_STATUS" != *"PRD_READY"* ]] || \
   [[ "$PLAN_STATUS" != *"PLAN_APPROVED"* ]] || \
   [[ "$TASK_STATUS" != *"TASKLIST_READY"* ]] ; then
  echo "Workflow gate failed: гейты не выполнены для $TICKET." >&2
  echo "Сначала пройди /idea, /plan и /tasks." >&2
  exit 2
fi

exit 0
```

Код возврата `2` воспринимается Claude как блокирующий: правки не будут применены, пока гейты не пройдены.

## 8. Чек-лист внедрения: три уровня

### 8.1. Уровень 1: Minimal AIDD

Цель — перестать вайб-кодить и ввести базовый AIDD без сложной автоматизации.

- [ ] Создать `CLAUDE.md` — единственный «договор» с LLM
- [ ] Добавить шаблоны PRD / плана / tasklist (простые markdown-шаблоны)
- [ ] Завести `docs/prd/<ticket>.prd.md` и `docs/tasklist/<ticket>.md` для одной фичи
- [ ] Подружить LLM с файлами: просить не «написать эндпоинт», а «собери PRD по этим заметкам»
- [ ] Прогнать одну фичу по цепочке: PRD → план → tasklist → реализация → ревью
- [ ] Ретроспектива: что помогло, где мешала, подправить CLAUDE.md

### 8.2. Уровень 2: Full AIDD без жёстких гейтов

Цель — полноценный SDLC с AI.

- [ ] Завести `.claude/agents` с ролями: analyst, researcher, planner, task-planner, implementer, reviewer, qa, validator, tech-writer
- [ ] Описать команды workflow: `/idea`, `/researcher`, `/plan`, `/tasks`, `/implement`, `/review`, `/qa`, `/validate`, `/docs-update`
- [ ] Стандартизировать артефакты тикетов: шаблонный набор файлов в `docs/` и `reports/`
- [ ] Использовать `/validate` и оркестратор для проверки гейтов

### 8.3. Уровень 3: Строгий AIDD

Цель — процесс, который не зависит от настроения LLM.

- [ ] Подключить hooks в `.claude/settings.json` — блокировать изменения кода при невыполненных гейтах
- [ ] Довести оркестратор до «одной команды»
- [ ] Добавить headless-скрипты в CI: проверки гейтов перед релизом, ночные отчёты, автоматические подсказки

## 9. Итоги

### Ключевые тезисы

1. **LLM — не «один мозг», а команда ролей.** Analyst, Researcher, Planner, Implementer, Reviewer, QA, Tech Writer, Validator — каждая роль со своими входами, выходами и ограничениями.

2. **Quality-гейты** не дают перескочить через этапы. Без них AIDD превращается в вайб-кодинг с красивыми файлами.

3. **Артефакты тикета** — структурированная память проекта, а не «последние 20 сообщений в чате». PRD, plan, tasklist, research, qa report — всё версионируется в репозитории.

4. **Три уровня внедрения:** от минимального (CLAUDE.md + шаблоны) до строгого (hooks + CI + headless). Начинайте с малого.

5. **В Claude Code** роли оформляются как subagents в `.claude/agents/*.md`, а workflow — как slash-команды в `.claude/commands/*.md`.

### Что делать дальше

- Начните с Minimal AIDD: создайте `CLAUDE.md` и пройдите одну фичу по цепочке
- Попробуйте описать хотя бы 3 роли (analyst, implementer, reviewer) как subagents
- Переходите к уроку 3 для изучения практик работы с двумя AI-агентами (Триада) и DRY-flow
