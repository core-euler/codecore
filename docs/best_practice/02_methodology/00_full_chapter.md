# Методология разработки с AI — полная глава

Модуль покрывает четыре темы: Spec-Driven Development (SDD), AI-Driven Development с ролями (AIDD), практики ежедневной работы с агентами и 8 уровней зрелости AI-разработчика. Вместе они образуют полный стек методологий — от «как писать спецификации» до «как оркестрировать автономные команды агентов».

---

## 1. Spec-Driven Development (SDD)

### 1.1 Проблема вайб-кодинга

Вайб-кодинг — режим, когда разработчик описывает желаемое на естественном языке, AI генерирует код, и он сразу запускается без спецификации, декомпозиции и тестов. Для прототипов это ускоритель, для продакшена — источник хаоса:

- Проект превращается в разрозненные куски кода без архитектурных соглашений
- Модель додумывает детали — появляются небезопасные решения
- AI ломает старые фичи при создании новых, расходует контекст впустую

Когда код дешёв в производстве, узким местом становится **контроль качества**.

### 1.2 Концепция SDD

SDD добавляет к скорости AI-генерации недостающий контроль. Конвейер:

```
идея -> спецификация -> план реализации -> декомпозиция задач -> код -> проверки
```

Спецификация — **артефакт намерения**: хранится в репозитории, версионируется, ревьюится через PR, превращается в acceptance criteria и служит единым источником правды для человека и агента.

### 1.3 Негативный промпт

Критически важный элемент спецификации — чёткое описание того, чего НЕ делаем. Ограничивает «творчество» модели и не даёт генерировать лишний код.

### 1.4 Конвейер SDD

**Спецификация** (specification.md): контекст, цели, не-цели, пользовательские сценарии, бизнес-правила, критерии приёмки.

**План реализации** (plan.md): стек технологий, модель данных, архитектурные решения, стратегия тестирования.

**Декомпозиция** (tasks.md): мелкие задачи, каждая — отдельный PR с отдельным циклом ревью.

**Распределение ответственности**: человек утверждает спецификацию и план, агент реализует задачи, CI проверяет качество.

### 1.5 Фреймворки SDD

| Фреймворк | Тип | Сложность | Для кого |
|-----------|-----|-----------|----------|
| **Spec-Kit** (GitHub) | Шаблоны + CLI | Средняя | Команды, большие проекты |
| **OpenSpec** | Лёгкий каркас | Низкая | Старт с SDD, инди-проекты |
| **Kiro** (Amazon) | AI IDE | Средняя | Те, кто не привязан к IDE |
| **BMAD** | Полный SDLC | Высокая | Agile-команды |

Различия между инструментами — не в «магии генерации», а в том, **где живёт контроль**: в репозитории, CLI, IDE или корпоративных пайплайнах.

### 1.6 Сопутствующие инструменты

- **Cursor Memory Bank** — сохранение памяти о проекте между сессиями
- **TaskMaster** — разбиение PRD на задачи с отслеживанием выполнения
- **Claude-Flow** — агенты + память + задачник + MCP в одном пакете

### 1.7 Правило синхронизации

> Изменилось поведение — обновилась спецификация.

Без этого правила артефакт намерения рассинхронизируется с реальностью.

## 2. AI-Driven Development — LLM как команда ролей

### 2.1 От «одного мозга» к команде

AIDD — подход, где LLM рассматривается не как «один большой мозг», а как **команда специалистов** с понятными ролями, зонами ответственности и quality-гейтами.

Вместо одного промпта «написать эндпоинт» проходим цепочку:

```
соглашения -> спецификация -> архитектура -> задачи ->
-> реализация -> ревью и тесты -> финальное QA -> документация
```

### 2.2 Восемь ролей AI-команды

| Роль | Вход | Выход |
|------|------|-------|
| **Analyst** | Идея, контекст | PRD (`docs/prd/<ticket>.prd.md`) |
| **Researcher** | PRD, кодовая база | Research-отчёт |
| **Planner** | PRD, research | Архитектура, ADR |
| **Task Planner** | PRD, план | Tasklist с acceptance criteria |
| **Implementer** | План, tasklist | Код + тесты, обновлённый tasklist |
| **Reviewer** | Diff, PRD, план | Замечания (blocking / important / nit) |
| **QA** | PRD, план, tasklist | QA-отчёт, рекомендация |
| **Tech Writer** | Все артефакты | CHANGELOG, summary |
| **Validator** | Все артефакты | Статусы quality-гейтов |

Ключевое: Implementer **останавливается** после каждой задачи и не переходит к следующей без команды.

### 2.3 Quality-гейты

Гейты — стоп-сигналы, которые не дают перескочить через этапы:

| Гейт | Что проверяем |
|------|---------------|
| `PRD_READY` | PRD заполнен, нет блокирующих вопросов |
| `PLAN_APPROVED` | Архитектура описана, риски учтены |
| `TASKLIST_READY` | Задачи мелкие, с acceptance criteria |
| `IMPLEMENT_STEP_OK` | Код + тесты + обновлённый tasklist |
| `REVIEW_OK` | CI зелёный, нет blocking-замечаний |
| `RELEASE_READY` | QA пройден, критических багов нет |
| `DOCS_UPDATED` | Документация обновлена |

Три уровня реализации: ручной (статус в markdown), AI-валидатор (команда `/validate`), жёсткий (hooks в `.claude/settings.json`, блокирующие Edit/Write).

### 2.4 Реализация в Claude Code

**Файловая структура:**

- `conventions.md` — конвенции для людей
- `CLAUDE.md` — инструкция для Claude Code
- `workflow.md` — описание процесса
- `.claude/agents/*.md` — роли как subagents (YAML frontmatter + «должностная инструкция»)
- `.claude/commands/*.md` — slash-команды (`/idea`, `/plan`, `/implement`, `/review`, `/qa`)

**Артефакты тикета** — структурированная память проекта: PRD, plan, tasklist, research, qa report.

### 2.5 Оркестратор

Команда `/feature-development` знает весь pipeline: вызывает `/validate`, показывает пройденные гейты, предлагает следующий шаг. Не изменяет код — только описывает состояние.

### 2.6 Три уровня внедрения AIDD

1. **Minimal** — CLAUDE.md + шаблоны PRD/plan/tasklist + одна фича по цепочке
2. **Full** — `.claude/agents` с ролями, slash-команды, стандартизированные артефакты, `/validate`
3. **Строгий** — hooks в settings.json, оркестратор, headless-скрипты в CI

## 3. Практики и процессы разработки с AI

### 3.1 Триада: человек + два агента

| Участник | Роль | Что делает |
|----------|------|------------|
| **Человек** | Владелец продукта | Решает что делаем, принимает результат, проводит ревью |
| **Claude Opus** | Стратег | Думает, планирует, пишет инструкции для исполнителя |
| **Claude Code** | Исполнитель | Пишет код, запускает команды, фиксирует изменения |

Принцип: Opus думает, Code делает, человек направляет и проверяет. Два агента — не недостаток, а защита: норма управляемости с агентами ниже, чем с людьми.

**Три файла триады:**

1. **CLAUDE.md** (200-400 строк) — описание проекта, стек, структура, правила, текущий статус
2. **settings.json** — разрешения и запреты (`allow`/`deny` для bash-команд)
3. **docs/instructions/*.md** — детальные инструкции к задачам с контекстом, шагами, критериями и «чего НЕ делать»

**Рабочий цикл:** утро — планирование с Opus (план в md-файле), день — выполнение с Code (короткие команды), вечер — ревью.

### 3.2 DRY-flow: итеративный workflow со Spec-Kit

Цель — сэмулировать проект со зрелыми процессами, где агент работает строго по спецификациям.

Пошаговый workflow:

1. **Constitution.md** — железобетонные правила проекта (создаётся один раз, итеративно улучшается)
2. **User Stories в md-файлах** — вместо CLI-ввода, с декомпозицией и правильным порядком зависимостей
3. **Генерация бизнес-требований** (`/speckit.specify`)
4. **Git-ветка** — изоляция от вреда агента
5. **План внедрения** (`/speckit.plan`) — data-model.md и openapi.yaml как самые ценные артефакты
6. **Нарезка задач** (`/speckit.tasks`)
7. **Реализация по фазам** (`/speckit.implement Phase 1-2`) — чистить контекст, коммитить чекпоинты
8. **Финальное код-ревью** — диффы, архитектура, зависимости, тесты, обновление constitution.md

### 3.3 Шесть практик AI-кодинга

| # | Практика | Суть |
|---|----------|------|
| 1 | Ошибки агента — ваша ответственность | Проблема в контексте, инструкциях или критериях, а не в модели |
| 2 | Контекст — это топливо | Принцип «сверху-вниз»: от архитектуры к подсистемам, потом узкий фокус |
| 3 | Планирование обязательно | Нет плана — нет реализации. План фиксируется в md-файле |
| 4 | Приёмка через тесты | E2E сценарий важнее code review глазами. Критерий оформлен тестом |
| 5 | Верификация — сверка с реальностью | Агент берёт план и по пунктам сверяет с кодом. «Готово» — не верьте |
| 6 | Git-гигиена | Коммит после плана, после реализации пункта, после прохождения тестов |

### 3.4 Антипаттерны

- **«Сделай всё сразу»** — декомпозируйте на 50-200 строк с отдельными коммитами
- **«Починить починку»** — после 3 итеративных фиксов `git checkout -- .` и заново с правильным ТЗ
- **«Абстрактные инструкции»** — конкретные указания с примерами, а не «улучши производительность»
- **«Контекстная перегрузка»** — CLAUDE.md 200-400 строк, детали в отдельных файлах
- **«Автопилот без страховки»** — всегда отдельная ветка, коммиты после фаз

### 3.5 Сводная таблица подходов

| Аспект | Триада | DRY-flow | 6 практик |
|--------|--------|----------|-----------|
| Фокус | Распределение ролей | Итеративный workflow | Повседневные привычки |
| Агенты | 2 (Opus + Code) | 1 (Code + Spec-Kit) | Любой |
| Подходит для | Средние/крупные проекты | Проекты со зрелыми процессами | Любой AI-кодинг |

Подходы не взаимоисключающие. Триада — **кто** работает, DRY-flow — **как** организовать артефакты, 6 практик — **повседневные привычки**.

## 4. Восемь уровней зрелости AI-разработчика

### 4.1 Мультиплеерный эффект

Ваша продуктивность ограничена уровнем коллег. Если вы на уровне 7, а коллега на уровне 2 — ваша эффективность упирается в него. Прогрессия уровней — командная задача.

### 4.2 Уровни 1-2: начальные

**Уровень 1: Tab Complete** — автодополнение строк, работа в рамках одного файла, минимальное взаимодействие.

**Уровень 2: Agent IDE** — чат с контекстом проекта, мультифайловые правки, plan mode. Потолок — контекст: модель либо не видит нужное, либо видит слишком много лишнего.

### 4.3 Уровень 3: Context Engineering

Мантра: «Каждый токен должен заслужить своё место в промпте.»

Включает: system prompts и rules-файлы, описания инструментов, управление историей диалога, выбор инструментов. Контекст по-прежнему критичен для маленьких моделей, токеноёмких инструментов и агентов с десятками инструментов.

### 4.4 Уровень 4: Compounding Engineering

Замкнутый цикл: спланировать -> делегировать -> оценить -> зафиксировать результат (обновить правила для будущих сессий).

Пример: агент добавил `lodash` — вы удаляете его и добавляете в CLAUDE.md запрет. Ошибка больше не повторится. Но инстинкт фиксировать всё подряд может навредить — слишком много инструкций равно ни одной.

### 4.5 Уровень 5: MCP и Skills

MCP и кастомные Skills дают агенту доступ к базе данных, API, CI, дизайн-системе, Playwright, Slack. Вместо того чтобы просто думать о кодовой базе, модель может **на ней действовать**.

Тенденция: CLI-инструменты вместо MCP для экономии токенов. Командный реестр Skills проходит тот же процесс, что и код: PR, ревью, версии.

> Уровни 3-5 — фундамент для всего последующего. Без них уровни 6-8 усиливают хаос.

### 4.6 Уровень 6: Harness Engineering

Переломный момент. Context engineering — курирование того, что модель видит. Harness engineering — построение всей среды и feedback loops для автономной работы.

**Feedback loops:** агент не просто пишет код — он видит результат (тесты, линтеры, DevTools) и итерирует.

**Backpressure:** системы типов, тесты, линтеры, pre-commit hooks — автоматические механизмы, позволяющие агентам обнаруживать и исправлять ошибки без человека.

**Два совета:**
1. Проектируйте с заделом на throughput, а не на совершенство — допускайте мелкие неблокирующие ошибки, делайте финальный quality pass
2. Ограничения лучше инструкций: «Работай, пока не пройдёшь тесты» эффективнее чеклиста из 20 пунктов

### 4.7 Уровень 7: Background Agents

Критический сдвиг: от «нескольких вкладок» к «работе, которая происходит без меня». Агент может генерировать план и исполнять его асинхронно.

**Ralph Loop** — автономный цикл агента, запускающий CLI повторно, пока все пункты PRD не завершены. Каждая итерация — свежий инстанс с чистым контекстом.

**Инструменты оркестрации:**
- **Dispatch** — skill для Claude Code, превращающий сессию в командный центр с изолированными воркерами
- **Inspect (Ramp)** — облачные песочницы для долгих автономных задач

**Разделение реализатора и ревьюера** — один инстанс модели не должен проверять свою работу.

**CI-интеграция:** docs-бот, security-ревьюер, dependency-бот.

### 4.8 Уровень 8: Автономные агентные команды

Агенты координируются друг с другом напрямую: забирают задачи, делятся находками, разрешают конфликты. Anthropic использовала 16 параллельных агентов для написания C-компилятора. Cursor запустил сотни конкурентных агентов.

Текущая рекомендация: вкладывайте энергию в уровень 7. Уровень 8 — авангард, а не мейнстрим.

### 4.9 Карта прогрессии

| Уровень | Название | Ключевая идея |
|---------|----------|---------------|
| 1 | Tab Complete | Автодополнение |
| 2 | Agent IDE | Чат с кодовой базой |
| 3 | Context Engineering | Каждый токен заслуживает место в промпте |
| 4 | Compounding Engineering | Каждая сессия улучшает следующую |
| 5 | MCP и Skills | Агент действует, а не только думает |
| 6 | Harness Engineering | Feedback loops для автономной итерации |
| 7 | Background Agents | Агенты работают без вас |
| 8 | Автономные команды | Агенты координируются друг с другом |

**Зависимости:** уровни 3-5 — фундамент. Без них 6-8 усиливают хаос вместо продуктивности.

### 4.10 Как определить свой уровень

| Если вы... | Уровень |
|------------|---------|
| Используете только tab complete | 1 |
| Активно работаете в чате IDE | 2 |
| Тщательно настраиваете CLAUDE.md | 3 |
| После ошибок обновляете правила | 4 |
| Используете MCP, пишете Skills | 5 |
| Настроили тесты/линтеры как feedback | 6 |
| Запускаете агентов в фоне | 7 |
| Оркестрируете команду агентов | 8 |

## 5. Сводка ключевых идей модуля

1. **SDD** превращает хаотичный вайб-кодинг в управляемый процесс через спецификации как артефакты намерения
2. **AIDD** организует LLM как команду ролей с quality-гейтами между этапами
3. **Триада** (человек + стратег + исполнитель) — проверенная модель распределения ролей для двух агентов
4. **DRY-flow** — итеративный workflow на базе Spec-Kit с constitution.md и промптингом через md-файлы
5. **6 практик** формируют повседневную дисциплину: контекст, планирование, тесты, верификация, git-гигиена
6. **8 уровней зрелости** — от tab complete до автономных команд; уровни 3-5 — фундамент
7. **Нейросеть умножает экспертизу, не заменяет её** — агент берёт рутину, человек занимается тем, где нужна голова
