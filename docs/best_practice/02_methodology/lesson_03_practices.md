# Практики и процессы разработки с AI

## Введение

В предыдущих уроках мы разобрали теоретическую основу: SDD как подход к спецификациям и AIDD как систему ролей. Теперь перейдём к практике: как конкретно организовать ежедневную работу с AI-агентами, какие паттерны работают, а какие приводят к потере времени.

В этом уроке — три проверенных подхода:

1. **Триада** — методология работы с двумя AI-агентами (стратег + исполнитель)
2. **DRY-flow** — итеративный workflow на основе Spec-Kit с constitution.md и промптингом через md-файлы
3. **6 практик AI-кодинга** — подготовка, делегирование, планирование, приёмка, верификация и git-гигиена

---

## 1. Методология триады

### 1.1. Происхождение и суть

Триада родилась из раздражения. Типичная ситуация: разработчик сидит с AI-агентом и понимает, что они оба занимаются одним и тем же. Человек думает над архитектурой — агент тоже думает. Человек начинает писать код — агент параллельно что-то генерирует. В итоге два полуфабриката вместо одного готового результата. Плюс бесконечные уточнения: «Какой шаблон использовать?», «Куда положить файл?». Половина времени уходит на объяснения вместо работы.

Решение — **разделить роли**. Получилась триада из трёх участников.

### 1.2. Три роли, три задачи

| Участник | Роль | Что делает | Где работает |
|----------|------|------------|--------------|
| **Человек** | Владелец продукта | Решает что делаем, принимает результат, проводит ревью, разрабатывает HLD-схемы | — |
| **Claude Opus** («шеф») | Стратег | Думает, планирует, пишет инструкции для исполнителя, проверяет результаты | Веб-интерфейс |
| **Claude Code** («коллега») | Исполнитель | Пишет код, запускает команды, фиксирует изменения | Редактор кода |

**Главный принцип:** Opus думает, Code делает, человек направляет и проверяет.

Если смешать роли, начинается хаос:
- Opus выдаёт код, который некуда деть
- Code уходит в рассуждения вместо работы
- Человек теряется в деталях вместо того, чтобы думать о продукте

**Хороший признак правильной настройки:** Code не тратит время на размышления, а сразу выдаёт результат. Если агент начинает философствовать — инструкции недостаточно чёткие.

### 1.3. Почему именно два агента, а не три или пять

Попытки добавить третьего специализированного агента (один для тестов, один для кода, один для документации) приводят к каше. Человек начинает терять контекст, пропускать ошибки, делать допущения вместо проверки.

В менеджменте есть понятие **нормы управляемости**: руководитель эффективно управляет 5-7 людьми. С агентами это число **меньше**, потому что они работают быстрее и генерируют больше результатов для проверки.

С двумя агентами — чёткие роли и понятные границы. Две параллельные нити, видно как они взаимодействуют. Ограничение на два агента — это не недостаток, а защита. Естественный ограничитель, который не даёт превратиться в бездумного диспетчера задач.

### 1.4. Что реально ускоряется

**Заметный эффект:**
- Настройка инфраструктуры
- Написание типовых автотестов
- Анализ и исправление стандартных ошибок
- Генерация кода по шаблонам
- Рефакторинг по известным подходам

**Что НЕ ускоряется:**
- Стратегическое планирование
- Определение приоритетов
- Проверка архитектурных решений
- Общение с командой и бизнесом
- Исследовательское тестирование
- Понимание предметной области

> **Нейросеть умножает экспертизу, не заменяет её.** Если вы не понимаете, как должен выглядеть хороший код, агент вам не поможет. Если понимаете — поможет писать его быстрее.

### 1.5. Настройка: три файла, которые меняют всё

#### Файл 1: CLAUDE.md

Главный контекст проекта. Агент читает его автоматически при старте. Кладётся в корень репозитория.

Содержание (200-400 строк):
- Описание проекта и архитектуры
- Стек технологий
- Структура директорий
- Ключевые команды (сборка, тесты, запуск)
- Правила (нарушать нельзя)
- Текущий статус (спринт, активные задачи)

**Пример:**

```markdown
# Warehouse Management System

## Проект
Система управления складом. Микросервисная архитектура,
12 сервисов, ~50k пользователей.

## Стек
- Backend: Java 17, Spring Boot 3.2, PostgreSQL 15
- Frontend: Vue 3, TypeScript, Pinia
- Тесты: JUnit 5, REST Assured, Selenide

## Правила (нарушать нельзя)
1. Не менять версии зависимостей без согласования
2. Не трогать файлы в /legacy до рефакторинга
3. Коммиты через feature-ветку → MR → проверка
4. Тесты обязательны для нового кода

## Текущий статус
Спринт 14: рефакторинг inventory-service
```

#### Файл 2: settings.json

Настройки агента в `.claude/settings.local.json`. Определяет что можно делать без спроса, что требует подтверждения, что запрещено:

```json
{
  "permissions": {
    "allow": [
      "Bash(mvn test *)",
      "Bash(cat *)",
      "Bash(ls *)",
      "Bash(git status)",
      "Bash(git diff *)",
      "Bash(git add *)",
      "Bash(git commit *)"
    ],
    "deny": [
      "Bash(git push *)",
      "Bash(rm -rf *)",
      "Bash(docker system prune *)",
      "Bash(mvn deploy *)",
      "Bash(kubectl delete *)"
    ]
  }
}
```

**Совет:** Начните с разрешений на чтение и тесты — это безопасно. Запретите всё, что связано с продакшеном. Добавляйте разрешения постепенно.

#### Файл 3: Инструкции к задачам

Папка `docs/instructions/` с детальными инструкциями для крупных задач. Каждая задача — отдельный файл.

**Шаблон хорошей инструкции:**

```markdown
# [Номер задачи]: [Название]

## Контекст
Одним абзацем: что не так сейчас и почему нужно исправить.

## Задачи

### Task 1: [Конкретное действие]
Файл: [точный путь]
По образцу: [путь к примеру]
Что сделать:
- [конкретный пункт 1]
- [конкретный пункт 2]
Тест: [название теста]
Коммит: [текст коммита]

### Task 2: [Следующее действие]
...

## Критерии готовности
- [ ] [Проверяемый критерий 1]
- [ ] [Проверяемый критерий 2]

## Чего НЕ делать
- Не трогать [файл/модуль]
- Не менять [что-то конкретное]
```

### 1.6. Рабочий цикл на практике

#### «Утро»: планирование с Opus

Открываете чат с Opus, загружаете контекст проекта:

**Вы:** Сегодня работаем над WH-234. Посмотри текущее состояние inventory-service, составь план рефакторинга на 4-5 задач.

**Opus:** Посмотрел структуру. InventoryService на 800 строк, делает слишком много. Предлагаю разбить так:
1. Выделить InventoryRepository (работа с БД)
2. Создать InventoryValidator (валидация)
3. Вынести InventoryMapper (преобразование DTO)
4. Оставить в сервисе только бизнес-логику
5. Добавить тесты

Opus создаёт файл `docs/instructions/WH-234.md` с детальным планом.

#### «День»: работа с Code

Открываете редактор с Claude Code:

**Вы:** Прочитай CLAUDE.md и docs/instructions/WH-234.md. Начинай с Task 1.

**Code:** Прочитал. Создаю InventoryRepository по образцу OrderRepository.
*[создаёт файл, показывает код]*
Готово. Запустить тесты?

**Вы:** Запускай.

**Code:** Тесты прошли. Коммит?

**Вы:** Коммить. Дальше Task 2.

Короткие команды работают лучше длинных, когда агент уже в контексте.

#### Полезные команды

```
# Начало работы
Прочитай CLAUDE.md и docs/instructions/[задача].md

# Переключение
Мы на WH-234, Task 3

# Быстрые команды в контексте
Дальше
Task 2, поехали
Коммить
Запусти тесты
Покажи diff

# Восстановление контекста после паузы
Прочитай CLAUDE.md, мы на WH-234, Task 3.
```

### 1.7. Типичные грабли

| Проблема | Плохо | Хорошо |
|----------|-------|--------|
| Агент застрял | «Нужно сделать репозиторий с правильной структурой» | «Сделай как в order-service/OrderRepository.java, только для inventory» |
| Результат неправильный | «Поправь вот тут и вот тут» | `git checkout -- .` Теперь давай заново с правильным ТЗ |
| Агент философствует | «Нужно улучшить производительность» | «Добавь кэширование в getInventory(). Используй @Cacheable как в ProductService» |
| Много переспросов | Продолжать объяснять | Улучшить CLAUDE.md — там не хватает контекста |

### 1.8. Метрики качества сессии

**Хорошая сессия:**
- Агент выполняет задачи без лишних вопросов
- Результат соответствует ожиданиям с первого раза
- Время уходит на проверку, а не на объяснения

**Плохая сессия:**
- Много переписки и переспросов
- Результаты приходится переделывать
- Ощущение, что быстрее было бы самому

Если сессии плохие — проблема обычно в подготовке. Улучшайте CLAUDE.md и инструкции.

## 2. DRY-flow: итеративный workflow со Spec-Kit

### 2.1. Суть подхода

DRY-flow — это гибридный подход к использованию Spec-Kit, который убирает лишние приседания из стандартного workflow и фокусируется на предсказуемости результата. Ключевые элементы:

- Итеративный `constitution.md`
- Промптинг фич с помощью md-файлов
- Git-ветки для контроля урона
- Авто-кодинг с код-ревью финального MR

### 2.2. Проблематика

Многие задачи можно делать на автопилоте с чистым LLM. Но для проекта со зрелыми процессами нужно, чтобы в разработке появлялись декомпозированные, расписанные требования, и не было пространства для фантазий: есть спецификация, кодстайл, покрытие, архитектура.

Цель DRY-flow: **сэмулировать проект со зрелыми процессами**, где агент работает строго по спецификациям — «агент попал на галеру».

### 2.3. Пошаговый workflow

#### Шаг 1: Конституция

Файл `constitution.md` — железобетонные правила проекта. Создаётся один раз, итеративно улучшается.

**Лайфхак:** За несколько циклов работы собирается полный файл, и команду `/speckit.constitution` можно больше не вызывать. Файл содержит ограничения, общие для всех проектов:

- Фреймворк и инструменты
- Не усложнять задачи
- Новые фичи — в отдельных ветках
- Асинхронный код
- Тесты на всё
- Линтеры после изменений
- Документация по-русски
- Именованные аргументы вместо позиционных
- Архитектура Controller → Service → CRUD
- Структура проекта

Конституция — это живой документ. После каждого код-ревью обновляйте его, чтобы агент в следующий раз работал точнее.

#### Шаг 2: User Stories в md-файлах

Вместо CLI-ввода создаём `specify_prompt.md` с отформатированным текстом:

```markdown
# 005-authenticate-user

## User Story 1: Аутентификация
**Как пользователь системы**, я хочу аутентифицироваться
с помощью email и пароля, чтобы пользоваться системой.

### Acceptance Criteria:
1. Запрос авторизует по совпадению email и password.
2. Ответ API выдаёт JWT-токен.
3. В случае ошибки — соответствующий статус-код.

### UI:
- Не делаем. Задача только для API.
```

**Правила декомпозиции:**
- Несколько мелких фич лучше, чем одна большая
- Порядок историй имеет значение — зависимости сверху вниз
- Составьте план внедрения заранее, не «по наитию»
- Связанные логические блоки подавайте вместе

**Лайфхак:** Для формулировки user stories используйте отдельное окно ChatGPT для предварительного обсуждения.

#### Шаг 3: Генерация бизнес-требований

Запуск: `/speckit.specify @specify_prompt.md`

Агент создаёт полноценную бизнес-фичу с corner-кейсами, способами самопроверки, негативными сценариями.

**Важно:** Обязательно вычитывайте результат. Всё, что агент написал, пойдёт в формирование задач. Вычистите лишнее, не дайте усложнить задачу.

**Совет по языку:** Агент всегда пишет по-английски. Отправляйте переписывать по-русски — чем проще читать требования, тем вдумчивее будете это делать.

#### Шаг 4: Git-ветка

Критически важный элемент. Ветка — изоляция проекта от вреда:

- Если агент уйдёт не туда — откат до коммита
- Удобно смотреть историю изменений
- MR для финального ревью
- Спеки остаются даже при полном откате

#### Шаг 5: План внедрения

Запуск: `/speckit.plan @constitution.md`

Генерирует технические спецификации. Самые ценные артефакты:

- **`data-model.md`** — модели БД, которые нельзя интерпретировать двояко
- **`openapi.yaml`** — спецификация API в формате OpenAPI
- **`research.md`** — выбранные технологии и отвергнутые альтернативы

> Интерпретировать модели БД и схему API двояко нельзя. Эти требования либо есть, либо нет. Именно эти документы делают имплементацию наиболее детерминированной.

#### Шаг 6: Нарезка задач

Запуск: `/speckit.tasks`

Создаёт нумерованные фазы и задачи (T001, T002...). Наглядный пошаговый алгоритм.

**Лайфхак:** Лишние задачи попросите агента выкинуть. Среди задач обязательно должны быть пункты про запуск тестов, расширение покрытия, проверку линтерами.

#### Шаг 7: Реализация по фазам

Запуск: `/speckit.implement Phase 1-2`

Лайфхаки:

1. **Чистить контекст перед запуском** — `/compact` или новая сессия
2. **Запускать фазами** — итерации с коммитами между ними
3. **Автопилот** — правок кода много, одобрять каждую не нужно (git-ветка страхует)
4. **Коммиты после фаз** — чекпоинты для отката
5. **Промежуточный контроль** — после нескольких фаз пробежаться по файлам и сопоставить с tasks.md

#### Шаг 8: Финальное код-ревью

1. Отсмотреть все диффы в IDE
2. Проверить архитектурные принципы
3. Проверить зависимости
4. Запустить тесты вручную
5. **Обновить constitution.md** — живой документ должен становиться точнее
6. Пуш ветки и мерж

### 2.4. Модификации Spec-Kit

Из практического опыта — что можно убрать из стандартного workflow:

| Что убрано | Почему |
|------------|--------|
| `checklists.md` | Нейросеть просто ставит галочки, жрёт токены без ценности |
| `quickstart.md` | Непонятно для кого нужен — у агента есть проект и constitution |
| `Parallel tasks [P]` | Параллельное выполнение ненадёжно, усложняет детерминированность |
| Команды самопроверки (`/speckit.analyze`, `/speckit.clarify`) | Бомбят вопросами за скоупом фичи, лучше вдумчиво вычитать на других этапах |

### 2.5. ChatOps: уведомления в Telegram

Чтобы не тратить время на простои агента в ожидании реакции, полезно настроить вебхуки:

- Когда агенту нужно разрешение на действие
- Когда агент выполнил задачу

Это позволяет запускать агента на автопилоте и заниматься другими делами, получая нотификации о важных моментах.

## 3. Шесть практик AI-кодинга

### 3.1. Практика 1: Ошибки агента — ваша ответственность

Первое и самое сложное — отпустить вожжи. Теперь не вы пишете код. Теперь вы организуете работу того, кто пишет код.

**Смена мышления:**

| Неправильно | Правильно |
|-------------|-----------|
| «Агент написал плохой код, AI плохой» | «Я дал плохие инструкции или кривой контекст» |

Современные фронтирные модели для распространённых стеков — это уровень уверенного мидла. Если мидл-разработчик не справился, проблема почти всегда в процессе:

- **Контекст:** не собран, противоречив или содержит мусор
- **Инструкции:** их мало, они абстрактные или перегружены деталями
- **Критерии:** нет автотестов, не объяснено, что такое «хороший результат»

**Для редких стеков:** придётся работать техничнее — с примерами, с образцами, с поиском в интернете для конкретных кейсов.

### 3.2. Практика 2: Контекст — это топливо

Агент не телепат. Он не знает ваш проект так, как вы.

**Принцип «Сверху-вниз»:**
Перед стартом любой задачи дайте агенту изучить проект. Пусть пройдёт от архитектуры к подсистемам. Не кидайте его сразу в бой с багом в 500-й строке непонятного файла.

**Прогрев:**
Идеально — Memory Bank (файлы с описанием архитектуры и правил проекта). Если нет — попросите агента исследовать проект сверху вниз: какие подсистемы существуют, как взаимодействуют, для чего предназначены.

**Узкий фокус:**
Когда общее понимание есть, задавайте вопрос по более узкой теме. «Подготовься к задачам по модулю аутентификации» — пусть агент подтянет в контекст только релевантные файлы и связи.

### 3.3. Практика 3: Планирование — обязательный этап

Никогда не начинайте реализацию без утверждённого плана.

**Автономность плана:**
Агент должен включить в план всё необходимое:
- Тонкие моменты
- Ссылки на стандарты и правила проекта
- Правильное взаимодействие подсистем
- Принятые в проекте зависимости
- Максимально простые решения без оверинжиниринга

**Закрытие пробелов:**
Сформировав план, попросите агента:
- Перепроверить его
- Дополнить забытое
- Прояснить все непонятные моменты
- Закрыть непроработанные gaps

Только когда у агента нет вопросов, план считается доработанным.

**Фиксация:**
Всегда сохраняйте план в .md файл. Даже если агент хорошо управляется с контекстом — это необходимый шаг.

**Реализация плана — самая простая вещь.** Просите агента сделать пункты плана и ведите его до полной реализации.

### 3.4. Практика 4: Приёмка через тесты, а не через Code Review

Вычитывать тысячи строк кода за AI утомительно и опасно — глаз «замыливается».

**Новый подход к приёмке:**
Больше думайте о критериях качества работы:
- Какие тесты сделать
- Какие приёмочные сценарии прогнать

**Ключевой принцип:** Обратная связь должна быть доступна агенту — критерий оформлен тестом, и агент может запускать его для самопроверки. А не просто «эстетичный экран корзины».

**Самый важный тест — e2e сценарий использования фичи.** Агент должен делать интеграционный e2e тест. Все остальные тесты хоть и полезны, но менее важны.

Прорабатывая фичу, сразу обсуждайте, как будете её тестировать.

### 3.5. Практика 5: Верификация — сверка с реальностью

Очень важный шаг в конце. Нужен даже для внимательных агентов.

> Творческие натуры, как Claude, без верификации могут недосчитаться реализации части пунктов.

**Процедура:**

1. Попросите агента взять оригинальный .md файл плана
2. Систематично по пунктам сверить реализацию с конкретными файлами кода, документации и тестов
3. Проверить каждую деталь
4. Дать отчёт: «Пункт 1 — сделано (файл А), Пункт 2 — сделано (тест Б)»
5. На базе отчёта составить план доработки

**Правило:** Когда агент говорит «Готово» — не верьте. Это спасает от 80% недоделок.

### 3.6. Практика 6: Git-гигиена

Git — единственный способ отменить «инициативу» агента, которая пошла не туда.

**Правило:**

| Событие | Действие |
|---------|----------|
| Сформировали план | Commit |
| Реализовали пункт плана | Commit |
| Прошли тесты | Commit |

Коммиты — это чекпоинты. Без них любая ошибка агента может унести часы работы.

## 4. Антипаттерны и как их избегать

### 4.1. «Сделай всё сразу»

**Симптом:** Один промпт на всю фичу. Агент генерирует 2000 строк кода за один раз.

**Почему плохо:** Невозможно ревьюить, невозможно откатить частично, ошибки накапливаются.

**Решение:** Декомпозиция на задачи по 50-200 строк с отдельными коммитами.

### 4.2. «Починить починку»

**Симптом:** Агент сделал ошибку. Просим «поправь вот тут». Агент правит, ломает другое. Просим ещё раз...

**Почему плохо:** Каждая итерация «починки» добавляет энтропию. После 3-4 итераций код превращается в спагетти.

**Решение:** `git checkout -- .` и начать задачу заново с более точной инструкцией. Переделать с нуля с правильным ТЗ дешевле, чем «чинить починку».

### 4.3. «Абстрактные инструкции»

**Симптом:** «Улучши производительность сервиса», «Сделай код более читаемым», «Добавь нормальную обработку ошибок».

**Почему плохо:** Модель интерпретирует абстрактные указания по-своему — результат непредсказуем.

**Решение:** Конкретные указания с примерами: «Добавь кэширование в метод getInventory(). Используй @Cacheable как в ProductService.getProduct()».

### 4.4. «Контекстная перегрузка»

**Симптом:** Огромный CLAUDE.md на 1000+ строк. Десятки правил, противоречащих друг другу.

**Почему плохо:** Слишком много инструкций — всё равно что ни одной. Модель теряется.

**Решение:** CLAUDE.md — 200-400 строк. Только самое важное. Детали — в отдельных файлах по запросу.

### 4.5. «Автопилот без страховки»

**Симптом:** Запуск агента на автопилоте в main-ветке без коммитов.

**Почему плохо:** Нет возможности откатить. Один неудачный run — и часы работы потеряны.

**Решение:** Всегда отдельная ветка. Коммиты после каждой фазы. MR для финального ревью.

## 5. Первый запуск на новом проекте

### 5.1. Пошаговый план

1. **Opus анализирует проект** — создаёт CLAUDE.md
2. **Выбираете простую первую задачу** — для калибровки
3. **Code выполняет** — вы смотрите на результат
4. **Корректируете инструкции** — по итогам

После 2-3 задач процесс налаживается и дальше идёт на автомате.

### 5.2. Чек-лист для старта

- [ ] Создать CLAUDE.md с описанием проекта, стека, структуры и правил
- [ ] Настроить settings.json с разрешениями и запретами
- [ ] Создать шаблон инструкции к задачам
- [ ] Выбрать простую первую задачу для калибровки
- [ ] Выполнить задачу в триаде: Opus планирует → Code выполняет → вы ревьюите
- [ ] По итогам — обновить CLAUDE.md

|--------|--------|----------|-----------|
| **Фокус** | Распределение ролей между агентами | Итеративный workflow со спецификациями | Повседневные привычки |

## 7. Итоги

### Ключевые тезисы

1. **Триада** (Человек + Opus + Code) — эффективная модель распределения ролей. Стратег думает, исполнитель делает, человек направляет.

2. **DRY-flow** превращает Spec-Kit в предсказуемый конвейер: constitution.md → user stories в md → бизнес-требования → план → задачи → реализация по фазам → код-ревью.

3. **Шесть практик** формируют повседневную дисциплину:
   - Ошибки агента — ваша ответственность
   - Контекст — это топливо
   - Планирование обязательно
   - Приёмка через тесты
   - Верификация по пунктам
   - Git-гигиена

4. **Антипаттерны** (делай всё сразу, починка починки, абстрактные инструкции, перегрузка контекста, автопилот без страховки) — главные враги продуктивности.

5. **Нейросеть умножает экспертизу, не заменяет её.** Агент берёт на себя рутину, человек занимается тем, где нужна голова.

### Что делать дальше

- Попробуйте триаду на одной задаче: Opus создаёт инструкцию, Code выполняет
- Настройте CLAUDE.md и settings.json для вашего проекта
- Примените 6 практик к следующей AI-сессии
- Переходите к уроку 4 для понимания уровней зрелости AI-разработчика
